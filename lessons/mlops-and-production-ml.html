Experiment tracking

Detailed explanation

Experiment tracking is the practice of recording the details of every ML training run — hyperparameters, metrics, code version, dataset version, and artifacts — so you can compare runs, reproduce results, and share findings with teammates. Without tracking, ML experiments are ephemeral: you try 20 model configurations and forget which achieved the best validation F1 three weeks later.

MLflow is the most popular open-source experiment tracking tool. You instrument your training code with mlflow.log_param() for hyperparameters, mlflow.log_metric() for metrics per step/epoch, mlflow.log_artifact() for model files and plots, and mlflow.sklearn.log_model() for the model itself. All runs are stored in a structured database and viewable in a web UI.

Weights & Biases (wandb) is the most popular cloud-based tracking tool. It auto-logs GPU metrics, system stats, and provides real-time dashboards, model comparison tables, and sweep (hyperparameter search) integration. import wandb; wandb.init(project='my-project'); wandb.log({'train_loss': loss}) is all you need to start. The free tier is sufficient for individual projects.

What to track in every experiment: (1) all hyperparameters, (2) train/val/test metrics per epoch, (3) model checkpoint of the best validation epoch, (4) training data version (git hash of data preprocessing script), (5) code version (git commit hash), (6) hardware (GPU type, batch size), (7) training duration. This information is the minimum needed to reproduce the experiment.

Model registry is the next step beyond experiment tracking. After identifying the best experiment, you register the model to a registry (MLflow Model Registry, HuggingFace Hub) with a version number and metadata. The registry provides lifecycle management: a model can be in 'staging' (being evaluated), 'production' (serving live traffic), or 'archived' (replaced by a newer version). This enables controlled, auditable model promotion.

Why this matters in AI
Without experiment tracking, ML development is chaos. You cannot reproduce your best result, cannot tell a teammate how you achieved it, and cannot safely improve upon it. Experiment tracking is the minimum viable infrastructure for any serious ML project.
Common mistakes
  • Not tracking dataset version alongside model version — if the dataset changes and you retrain, the model version should increment. Without dataset versioning, you cannot reproduce historical experiments.
  • Only logging final metrics and not per-epoch curves — per-step metrics reveal whether training converged, diverged, or overfitted. Final metrics alone hide the training dynamics.
  • Not logging random seeds — an unreproducible 'best result' is useless. Always log: random_seed, numpy_seed, torch_seed.

Model registry

Detailed explanation

A model registry is a centralized repository for managing ML model versions. It stores model artifacts (weights, code, configs), metadata (training metrics, hyperparameters, dataset versions), and lifecycle state (staging, production, archived). Think of it as Git for ML models — it tracks what was trained, when, on what data, with what performance, and which version is currently deployed.

MLflow Model Registry: the most widely used open-source model registry. Core concepts: registered model (a named model with multiple versions), model version (a specific trained artifact with metadata), stage (None → Staging → Production → Archived), tags (key-value metadata: dataset_version, training_date, owner). MLflow tracks experiments (training runs) and lets you promote the best run's model to the registry.

Benefits of a model registry: (1) reproducibility — know exactly which model version is in production and how to recreate it, (2) auditability — track who deployed what model and when (required for regulated industries like healthcare, finance), (3) rollback — if a new model degrades performance, instantly revert to the previous production version, (4) collaboration — data scientists share models; engineers deploy directly from the registry.

Model versioning strategy: version each significant change. Semantic versioning (major.minor.patch): major = breaking change in API or data format, minor = improvement in accuracy, patch = bug fix. Each version should include: model artifact, training code (git commit hash), dataset hash, evaluation metrics, inference requirements (Python version, dependencies). This makes every version independently reproducible.

Registry in production: cloud providers offer managed model registries: MLflow on Databricks, AWS SageMaker Model Registry, GCP Vertex AI Model Registry, Azure ML Model Registry. These integrate with CI/CD pipelines: a new model is automatically registered after successful training, validation gates ensure only models that meet quality thresholds are promoted to production, canary deployments route 5% of traffic to the new model before full rollout.

Why this matters in AI
A model registry is the foundation of production MLOps. Without it, deploying ML models becomes ad hoc and risky — you don't know which model is in production, can't roll back, and can't reproduce results. It is a non-negotiable component of any production ML system.
Common mistakes
  • Logging models without logging the training data version — if the model is retrained on different data, the registry entry must record which dataset version it was trained on. Without this, you can't reproduce the model.
  • Not setting up model validation gates — automatically promoting any new model to production is dangerous. Always gate on validation metrics: promote only if val_auc >= baseline_auc × 0.995.
  • Treating the model registry as a one-way pipeline — model versions should also record deployment metadata (when deployed, to which environment, performance after deployment). Closed-loop feedback enables model monitoring and improvement.

Model serving

Detailed explanation

Model serving is the infrastructure that makes a trained ML model accessible to users or other systems in production. At its core, serving means: accepting an input request (a sentence, an image, a structured record), running it through the model, and returning the prediction with acceptable latency and reliability. Everything else in serving is about doing this efficiently at scale.

REST APIs are the standard interface for model serving. A FastAPI or Flask server exposes an endpoint (e.g., POST /predict) that accepts JSON input, preprocesses it, runs the model, and returns JSON output. For synchronous requests (user waits for response), REST APIs with appropriate timeout handling work well. For asynchronous workloads (batch processing, background inference), queue-based architectures (Celery + Redis, SQS + Lambda) are more appropriate.

Serving infrastructure options: self-hosted (PyTorch serve, TorchServe, BentoML, Triton Inference Server) for full control, managed cloud (AWS SageMaker, Google Vertex AI, Azure ML endpoints) for ease of deployment, and specialized LLM serving (vLLM, TGI — Text Generation Inference) for transformers with optimizations like continuous batching and KV-cache management.

Latency optimization in serving: model quantization (reduce weight precision from float32 to int8 or int4 — 2–4× speedup at small accuracy cost), ONNX export (convert PyTorch model to optimized ONNX format for faster inference), TensorRT (NVIDIA's inference optimizer that fuses layers and uses GPU-specific optimizations), model distillation (train a smaller student model to mimic the teacher), and request batching (group multiple requests into one batch for GPU efficiency).

Production serving requires more than just a model endpoint: health checks (is the model server alive?), metrics collection (latency percentiles, throughput, error rates), graceful degradation (return cached response or fallback if model fails), autoscaling (scale up GPU instances when traffic spikes), model versioning (support A/B testing between model versions), and logging (record every prediction for monitoring and debugging).

Why this matters in AI
A trained model that is not deployed delivers zero value. Model serving is the bridge between ML research and real-world impact. Every ML engineer must know how to wrap a model in an API, deploy it, and ensure it remains reliable in production.
Common mistakes
  • Loading the model inside the request handler — each request would load the model from disk, taking 2–10 seconds. Load once at startup.
  • Not handling concurrent requests — by default, a single Python server is single-threaded. Use async/await, multiple workers, or a thread pool for concurrent prediction.
  • Not testing the serving endpoint under load — a model that works in development may crash under production traffic due to memory constraints or thread contention.

Batch inference

Detailed explanation

Batch inference processes a large dataset through a model offline, producing predictions for all items at once. Unlike online (real-time) inference where predictions are needed immediately, batch inference runs on a schedule (nightly, weekly) and precomputes predictions that are stored and served later. Examples: generating product recommendations for all users each night, scoring all loan applications submitted in the past 24 hours, generating personalized email content for a marketing campaign.

Batch inference architecture: (1) trigger (scheduled cron job, event-driven), (2) data fetch (read from data warehouse or feature store), (3) batch prediction (process data through model), (4) store results (write predictions to database), (5) serve (application reads precomputed predictions from database, returns in milliseconds). The latency at request time is the database read latency (< 5ms), not the model inference time.

Throughput optimization: batch inference should maximize throughput (items processed per second) rather than minimize per-item latency. Key techniques: batching (group items into batches of 32-256 for GPU utilization), vectorization (process entire batches as matrix operations, not one item at a time), parallel processing (multiple workers, each processing a chunk of the dataset), model optimization (TorchScript, ONNX, TensorRT for faster inference).

Pandas batch inference: for tabular models, iterate over DataFrames in chunks: df.iterrows() (slow, avoid), df.apply() (medium), vectorized prediction: model.predict(df[features]) on the full DataFrame or large chunks. For sklearn models, predict() accepts the full test matrix — process 1M rows at once if RAM allows, or in chunks of 100K.

Batch inference for LLMs: generating text for 10,000 items with an LLM is expensive. Optimize: use smaller, faster models when quality allows, cache common patterns (identical prompts get cached responses), batch prompts together (vLLM's continuous batching), run inference on GPU (100× faster than CPU for LLMs), and parallelize across multiple GPUs.

import joblib
from tqdm import tqdm
# Efficient batch inference for sklearn model
model = joblib.load('model.pkl')
def batch_predict(df: pd.DataFrame, batch_size: int = 100_000) -> pd.Series:
    predictions = []
    n_batches = (len(df) + batch_size - 1) // batch_size
    for i in tqdm(range(n_batches), desc='Batch inference'):
        batch = df.iloc[i*batch_size:(i+1)*batch_size]
        preds = model.predict_proba(batch[feature_cols])[:,1]
        predictions.extend(preds)
    return pd.Series(predictions, index=df.index)

from joblib import Parallel, delayed
def predict_chunk(chunk):
    model = joblib.load('model.pkl')  # load once per worker
    return model.predict_proba(chunk[features])[:,1]
n_cores = 8
chunks = [df.iloc[i::n_cores] for i in range(n_cores)]

Why this matters in AI
Batch inference is the correct architecture for any use case where predictions can be precomputed — recommendations, scoring, content generation for campaigns. It decouples prediction latency (which can be slow) from serving latency (which must be fast). Understanding batch inference architecture prevents the mistake of performing expensive model inference synchronously during a user request.
Common mistakes
  • Using df.iterrows() for batch inference — iterrows() processes one row at a time and is 100-1000× slower than vectorized predict(df). Always process the full batch or large chunks with vectorized operations.
  • Not caching predictions between batch runs — if 80% of items haven't changed since the last batch, recomputing predictions for all 100% wastes compute. Track which items changed and only recompute for those.
  • Not monitoring batch inference job health — a failed batch job means serving stale predictions from the previous run. Set up monitoring for: job completion time, number of items processed, prediction distribution vs previous run.

Online inference

Detailed explanation

Online inference (also called real-time inference) produces predictions in response to individual requests, typically requiring a response within 100-500ms for user-facing applications. Unlike batch inference (precomputed, stored), online inference requires the model to be loaded in memory, ready to process any input at any time. The architecture: user request → API endpoint → preprocessing → model inference → postprocessing → response.

Serving infrastructure: ML models are served via REST APIs (FastAPI, Flask) or gRPC servers (TensorFlow Serving, Triton Inference Server). The model is loaded into memory at server startup and kept resident for the lifetime of the server. Multiple server instances run behind a load balancer for scalability. Stateless design: each request is independent, servers can be scaled horizontally.

Latency components: total request latency = network round trip + preprocessing + model inference + postprocessing. For a typical tabular model: preprocessing (10ms) + model inference (1-5ms) + serialization (1ms) ≈ 15ms. For a BERT model: tokenization (5ms) + inference on 1 GPU (20-50ms) ≈ 30-60ms. For an LLM: time-to-first-token (200-1000ms) + streaming generation (50 tokens/second) ≈ 1-10 seconds.

GPU serving for deep learning: inference on GPU is 10-100× faster than CPU for neural networks. But GPU memory is limited (24GB for an RTX 4090) and GPUs are expensive. Optimization strategies: model quantization (4-bit/8-bit reduces memory 4-8×), ONNX Runtime (2-3× faster than PyTorch for inference), TensorRT (up to 10× faster than PyTorch, NVIDIA GPU only), vLLM (state-of-the-art LLM serving with continuous batching and PagedAttention).

Autoscaling: production serving systems must scale up when traffic spikes and scale down when traffic decreases. Container orchestration (Kubernetes, AWS ECS) monitors request queue depth and CPU/GPU utilization, adding more server instances when load is high. For bursty traffic patterns, prewarming (keeping some instances running in advance) prevents cold start latency.

Why this matters in AI
Online inference is the architecture for all user-facing ML applications: search ranking, product recommendations, content moderation, autocomplete, fraud detection. Understanding how to build, deploy, and scale online inference systems is the bridge between an ML model and a production AI product.
Common mistakes
  • Loading the model on every request — loading a 500MB model takes 2-5 seconds. Load the model once at server startup and keep it in memory. Never load inside the request handler.
  • Not measuring latency under load — a model that responds in 20ms for 1 request may respond in 500ms when 100 requests arrive simultaneously (if they all compete for the same GPU). Always load test before deploying.
  • Running inference synchronously in a web request handler — for long-running inference (> 200ms), use async workers: return a job ID immediately, run inference in a background worker, and let the client poll for the result. This prevents timeouts and enables graceful backpressure.

FastAPI

Detailed explanation

FastAPI is a modern Python web framework for building APIs quickly and efficiently. It is built on Starlette (ASGI framework) and Pydantic (data validation). FastAPI's key features: automatic API documentation (Swagger UI at /docs and ReDoc at /redoc, generated from type annotations), request validation (Pydantic models validate input types automatically), async support (built-in async/await for non-blocking I/O), and performance comparable to Node.js and Go (due to ASGI and Starlette).

Pydantic models are the core of FastAPI's input/output handling. Define a class inheriting from pydantic.BaseModel with typed fields. FastAPI automatically: validates incoming JSON against the model (returns HTTP 422 Unprocessable Entity for invalid input with a detailed error message), parses validated data into the model instance, and uses the model to generate the API schema. Output models ensure you never accidentally expose internal fields.

Dependency injection in FastAPI enables: shared database connections, authentication, configuration loading, and logging — all as reusable dependencies. def get_db(): db = SessionLocal(); try: yield db; finally: db.close(). Any endpoint that declares db: Session = Depends(get_db) automatically gets a fresh database session, properly closed after the request. This pattern is much cleaner than global connections.

Async endpoints: for I/O-bound operations (database queries, external API calls), use async def instead of def. async def predict(request: PredictRequest, db: Session = Depends(get_db)): result = await model.async_predict(request.text); return result. Async endpoints can serve thousands of concurrent requests on a single worker because awaiting I/O releases the thread to serve other requests. For CPU-bound operations (running ML inference), use a thread pool executor to avoid blocking the event loop.

Testing FastAPI with TestClient: from fastapi.testclient import TestClient; client = TestClient(app). client.post('/predict', json={'text': 'hello'}) — this runs the full request through FastAPI including validation. Use pytest fixtures to set up the client and mock dependencies. FastAPI's dependency override (app.dependency_overrides) replaces production dependencies with test mocks — no real database or model needed in unit tests.

Why this matters in AI
FastAPI is the standard for ML model serving in Python. It combines rapid development, automatic validation, async performance, and self-documenting APIs. Every ML engineer deploying models via REST API should know FastAPI.
Common mistakes

Docker

Detailed explanation

Docker is a containerization platform that packages an application and all its dependencies (Python version, libraries, CUDA version, config files) into a portable container image. The container runs identically on any machine that has Docker installed — your laptop, a teammate's machine, a cloud VM, or a Kubernetes cluster. Containers solve the 'it works on my machine' problem that plagues ML deployment.

Key concepts: a Docker image is a read-only blueprint (like a class in OOP) that specifies the filesystem, environment variables, and startup command. A container is a running instance of an image (like an object). Images are built from a Dockerfile — a text file with instructions: FROM (base image), RUN (execute commands to install software), COPY (copy files into the image), WORKDIR (set working directory), ENV (environment variables), EXPOSE (declare port), CMD (default startup command).

A well-structured ML model Dockerfile: start from a Python base image (python:3.11-slim or an NVIDIA CUDA image for GPU models), install system dependencies with apt-get, copy requirements.txt and run pip install, copy the model artifacts and application code, and specify the uvicorn/gunicorn command to start the serving API. Keep images small by using slim base images, combining RUN commands to reduce layers, and using .dockerignore to exclude large files.

Docker Compose orchestrates multi-container applications: your ML API container, a Redis cache container, a monitoring container (Prometheus + Grafana), and a database container can all be defined in a single docker-compose.yml file and started with one command: docker-compose up. This replicates a production-like environment locally for testing.

GPU support requires the NVIDIA Container Toolkit. After installing it, docker run --gpus all your_image makes all GPUs available to the container. For multi-GPU training, docker run --gpus all -e CUDA_VISIBLE_DEVICES=0,1 restricts to specific GPUs. In Kubernetes, GPU scheduling requires the NVIDIA device plugin.

Why this matters in AI
Docker is the standard packaging and deployment mechanism for ML models in industry. Without containerization, deploying a model requires manually replicating its environment on every target machine — an error-prone, time-consuming process. With Docker, deployment is reproducible and takes minutes.
Common mistakes
  • Not using a .dockerignore file — without it, large files (datasets, .git directory, __pycache__) are included in the build context, making builds slow.
  • Installing packages in separate RUN commands — each RUN command creates a new image layer. Combine all installs into one RUN command to keep the image smaller.
  • Not pinning package versions in requirements.txt — requirements.txt without versions (just 'torch') will install the latest version, making the image non-reproducible when rebuilt later.

CI/CD

Detailed explanation

CI/CD (Continuous Integration / Continuous Delivery) is the practice of automatically testing and deploying code changes. In traditional software, CI/CD runs tests whenever code is pushed and deploys to production when tests pass. For ML, CI/CD extends to include: model training, evaluation, performance validation, model registration, and deployment — the complete ML pipeline from code commit to serving.

Continuous Integration for ML: when a developer pushes code (model code, preprocessing code, training script), the CI pipeline automatically: (1) runs linting and unit tests, (2) runs integration tests on a small data sample, (3) trains the model on a fixed training set, (4) evaluates on a fixed test set, (5) compares metrics to a baseline threshold, (6) flags failures for human review. Popular CI tools: GitHub Actions, GitLab CI, Jenkins, CircleCI. Config is a YAML file in the repository.

Continuous Delivery for ML: after a model passes all CI checks, CD automatically promotes it: (1) register the model in the model registry (MLflow, Vertex AI Model Registry) with version and performance metrics, (2) deploy to a staging environment for integration testing, (3) run canary deployment in production (serve 5% of traffic with the new model, 95% with the old model), (4) monitor production metrics for 24–48 hours, (5) gradually increase the new model's traffic share if metrics hold.

Model validation gates are automated checks that block deployment if the model fails quality standards. Common gates: (1) minimum test set AUC (e.g., AUC > 0.85), (2) performance parity on protected groups (AUC difference < 0.03 across demographic groups), (3) inference latency (P99 < 100ms), (4) no regression from production model (new model must beat current prod model by > 0.01 AUC), (5) data distribution check (training data is not stale). Each gate failure blocks promotion to the next stage.

ML-specific CI/CD tools: MLflow for experiment tracking and model registry. DVC (Data Version Control) for versioning datasets and model files alongside code. Weights & Biases for experiment tracking with CI integration. Vertex AI Pipelines and SageMaker Pipelines for managed ML pipelines. Metaflow and Prefect for orchestrating Python-based ML workflows. CML (Continuous Machine Learning) from Iterative AI adds model metrics directly to pull request comments.

Monitoring

Detailed explanation

ML model monitoring tracks the performance, reliability, and behavior of deployed models in production. Unlike traditional software monitoring (is the server up? is response time < 100ms?), ML monitoring also tracks prediction quality metrics, data drift, and model drift — aspects unique to statistical learning systems that can degrade silently without any software errors.

Monitoring dimensions: operational metrics (latency, throughput, error rate, memory usage — standard infrastructure metrics), prediction metrics (confidence score distribution, prediction class distribution — should be stable if input distribution is stable), data quality metrics (null rates, feature value distributions against training baseline — PSI, KS-test), and ground truth metrics (actual performance vs predictions — requires delayed labels from production outcomes).

Latency monitoring with percentiles is more informative than average latency. P50 latency is the median (50% of requests are faster). P95 latency is the 95th percentile (only 5% of requests are slower). P99 latency captures tail latency — the worst 1% of users. Set alerts on P99 latency, not average — a fast average can hide pathologically slow tail requests that users experience as failures.

Feedback loops enable ground truth monitoring. In fraud detection, labeled fraud outcomes arrive days later when human reviewers confirm or deny model flags. Log every prediction with a unique request ID, then join with outcome labels when they arrive. Compute model accuracy, precision, and recall on this labeled production sample. When production metrics drop below a threshold, trigger retraining.

Alerting strategy: too few alerts = problems slip through unnoticed. Too many alerts = alert fatigue and real alerts get ignored. Start with: error rate > 1%, P99 latency > 5s, confidence score mean drops more than 2 standard deviations, any feature PSI > 0.2. Add or remove alert rules based on incident postmortems.

Data drift

Detailed explanation

Data drift (also called covariate shift) occurs when the statistical distribution of input features in production differs from the distribution seen during training. The model was trained on one distribution and now receives inputs from a different distribution — its predictions may be unreliable even if the model itself has not changed. Data drift is one of the most common reasons ML models degrade in production over time.

Types of drift: feature drift (input feature distributions change — e.g., customers now have higher average income than when the model was trained), label drift (the distribution of target values changes — fraud patterns evolve), and concept drift (the relationship between inputs and outputs changes — what used to predict churn no longer predicts churn because product features changed). Feature and label drift are measurable; concept drift requires monitoring prediction error on labeled data.

Detecting feature drift: compare production input distributions against training distributions using statistical tests. Population Stability Index (PSI) is widely used in credit risk: PSI < 0.1 = no significant shift, 0.1–0.2 = moderate shift, >0.2 = significant drift requiring model review. Kolmogorov-Smirnov test and chi-square test can also detect distributional shifts. Tools: Evidently AI, Alibi Detect, WhyLabs.

Monitoring drift in production: sample incoming requests (or all if volume allows) and compare feature statistics (mean, std, percentiles, value distributions for categoricals) against training set statistics. Set up alerts when PSI or KS-test p-values exceed thresholds. Log these metrics to a dashboard (Grafana, Datadog) so you can visualize drift trends over time.

Responding to drift: retrain the model on recent data. Implement a sliding window retraining schedule (retrain monthly on the last 6 months of data). Or use online learning — update the model continuously with new examples as they arrive. For severe concept drift, collect new labeled data and start a new development cycle.

Concept drift

Detailed explanation

Concept drift occurs when the statistical relationship between inputs (features) and outputs (labels) changes over time. A model trained on historical data becomes inaccurate because the world has changed. Examples: a fraud detection model trained in 2022 fails in 2024 because fraud patterns evolved, a recommendation model trained on pre-pandemic behavior fails in 2024 because user preferences changed, a demand forecasting model fails during economic shocks.

Types of drift: (1) data drift (feature drift, covariate shift) — the distribution of input features P(X) changes but the relationship P(Y|X) stays the same. (2) Concept drift (label drift) — P(Y|X) changes: the same features now map to different labels. (3) Label drift — P(Y) changes (class frequency shifts). (4) Upstream data drift — the data pipeline changes: a new feature engineering step, a schema change, or an upstream data source change that alters feature distributions.

Detecting drift: (1) statistical tests — compare feature distributions between training and inference: KL divergence, Population Stability Index (PSI), Kolmogorov-Smirnov (KS) test, chi-square test for categorical features. (2) Model performance monitoring — if ground truth labels are available (often delayed), track AUC, accuracy, or RMSE over time. A degrading performance trend signals drift. (3) Prediction drift — monitor the distribution of model outputs/confidence scores over time. Shifts in output distribution often precede measurable performance degradation.

Responding to drift: (1) continuous retraining — retrain periodically on recent data (rolling window: last 90 days), (2) triggered retraining — retrain when drift is detected above a threshold, (3) online learning — update the model incrementally with each new example (useful for fast-drifting systems like ads), (4) ensemble with time-aware weighting — weight recent models higher than old models.

Evidently AI and Arize: open-source tools for drift detection and monitoring. Evidently AI generates drift reports comparing training and production data distributions. Arize and WhyLabs are commercial platforms with real-time drift alerting. MLflow and W&B also have monitoring dashboards. In production: monitor feature distributions daily, alert on PSI > 0.2 (significant drift) or PSI > 0.1 (moderate drift), trigger retraining when drift is detected.

def compute_psi(expected, actual, bins=10):
    expected_perc = np.histogram(expected, bins=bins)[0] / len(expected)
    actual_perc = np.histogram(actual, bins=bins, range=(expected.min(), expected.max()))[0] / len(actual)
    psi = np.sum((actual_perc - expected_perc) * np.log((actual_perc + 1e-8) / (expected_perc + 1e-8)))
    return psi

A/B testing

Detailed explanation

A/B testing (also called randomized controlled experiment) is the method of comparing two versions of something (A = control, B = treatment) by randomly assigning users to one version and measuring outcomes. In ML, A/B tests compare two models, two recommendation algorithms, two UI designs, or two pricing strategies. Randomization ensures that differences in outcomes are caused by the version, not by confounding factors.

A correctly designed A/B test requires: (1) a clear metric (conversion rate, session duration, revenue per user), (2) a pre-specified significance level α (typically 0.05) and power (1-β, typically 0.8), (3) a power analysis to determine the required sample size before collecting data, (4) random assignment to groups, and (5) analysis after the pre-specified sample size is reached (not before). Stopping early or peeking at results leads to false positives.

Sample size calculation: the required number of samples depends on the baseline rate, the minimum detectable effect (MDE — the smallest improvement worth detecting), desired power, and significance level. For a conversion rate test (baseline=5%, MDE=1%, α=0.05, power=0.8), you need ~5000 samples per group. Underpowering leads to inconclusive results; overpowering wastes resources.

Pitfalls: novelty effect (new versions get higher engagement temporarily because users are curious, not because they are genuinely better — wait until behavior stabilizes), Simpsons paradox (the winning version in total may lose in every subgroup — check segmented results), and network effects (in social platforms, user A assigned to B may interact with user A2 assigned to A, contaminating the groups).

Multi-armed bandit (MAB) testing is an alternative to A/B testing that dynamically allocates more traffic to better-performing variants during the experiment, reducing regret (lost conversions from showing inferior variants). MAB is appropriate when the cost of showing the worse version is high (medical treatment, high-value recommendations). Pure A/B testing is appropriate when you need clean statistical inference.

Why this matters in AI

Common mistakes
  • Peeking at results before the predetermined sample size is reached and stopping early — this inflates false positive rates dramatically. The p-value is only valid at the predetermined sample size.
  • Not checking group balance — if randomization is imperfect (assignment bug), the groups may differ on baseline characteristics, biasing results. Always check that groups have similar baseline metrics.
  • Running too many simultaneous tests — testing 20 features simultaneously at α=0.05 produces ~1 false positive. Use family-wise error rate control or FDR correction.

Canary deployment

Detailed explanation

Canary deployment is a gradual rollout strategy where a new model version is deployed to a small fraction of traffic (the 'canary') while the majority still uses the old model. If the new model performs well on the canary traffic, the rollout percentage is gradually increased until the new model handles 100% of traffic. If the new model degrades performance, the canary is immediately rolled back — only a small fraction of users were affected.

Why canary deployment: ML models can look great in validation but fail on production traffic due to distribution shift, edge cases, or infrastructure issues. Testing on production traffic (with real users) is the most reliable way to validate a model. Canary deployment limits the blast radius of a bad model to the canary percentage — a 5% canary means at most 5% of users see degraded performance before rollback.

Implementation: a traffic routing layer (load balancer, feature flag service, or ML serving infrastructure) routes requests between models based on the configured split percentage. Tools: AWS SageMaker deployment with model weights, Kubernetes with Istio traffic splitting, Seldon Core, Argo Rollouts, or simple A/B testing infrastructure. Each request is labeled with which model version served it, enabling per-version metric tracking.

Monitoring during canary: define rollback triggers before deployment. Common triggers: new model latency p99 > 2× old model, new model AUC drops > 1% below old model (if labels available), new model error rate > 0.5%, prediction distribution shift (new model outputs very different probability distributions). Automated rollback triggers prevent overnight failures when no engineer is watching.

Shadow mode deployment: an even safer variant — the new model receives a copy of production traffic but its predictions are NOT used. The new model's predictions are logged and compared to the serving model's predictions (and eventually to ground truth labels). Shadow mode detects problems before any user is affected. The downside: you see no real-world impact, and some bugs only appear when predictions affect downstream systems.

Rollback

Detailed explanation

Rollback is the process of reverting a deployed model to a previous version when the new version underperforms or causes issues in production. Rollback capability is a safety net that enables confident deployment: if you can always roll back in 5 minutes, the cost of a bad deployment is low. Without rollback, each deployment is high-risk, and teams become overly cautious about deploying changes.

When to rollback: (1) model performance metrics drop significantly (AUC drops > 5%, conversion rate drops > 2%), (2) latency increases beyond SLA (p99 > 500ms), (3) error rate spikes (> 1%), (4) prediction distribution shift (model starts predicting extreme values), (5) business KPI degradation (upstream business metrics that the model influences). Set up automated alerts and rollback triggers for all of these.

Blue-green deployment: run two versions simultaneously (blue=current, green=new). Route 100% traffic to blue while green is being tested. Switch traffic to green when confident. Rollback = switch traffic back to blue. Achieves zero-downtime deployment and instant rollback. The tradeoff: running two versions simultaneously doubles inference infrastructure cost.

Canary deployment: route a small percentage of traffic (5-10%) to the new version, monitor, and gradually increase the percentage if metrics are good. Rollback = route 100% back to the old version. This is more resource-efficient than blue-green and reduces blast radius (only 5% of users see a bad model). Most production ML deployments use canary releases.

Shadow mode: run the new model in parallel with the production model. The new model's predictions are not served to users — they are just logged. This enables measuring the new model's predictions and metrics without any user impact, before making any traffic routing decisions. Ideal for validating a new model before any user exposure.

Cost monitoring

Detailed explanation

Cost monitoring for AI systems tracks and controls the financial costs of LLM API calls, compute resources, and cloud infrastructure. Without monitoring, AI applications can silently accumulate thousands of dollars in unexpected costs — a prompt that generates 10,000 output tokens per call at $0.015/1K tokens costs $150 per 1,000 calls.

LLM cost components: input tokens (cheaper — reading context), output tokens (more expensive — generating text), and number of requests. For Claude Opus: ~$15/1M input tokens, ~$75/1M output tokens (verify current pricing). A system that processes a 2,000-token document and generates a 500-token summary costs: 2000×$0.015/1000 + 500×$0.075/1000 = $0.03 + $0.0375 = ~$0.067 per call. At 100,000 calls/day: $6,700/day.

Cost reduction strategies: (1) Caching — cache LLM responses for repeated identical or similar requests. A cache hit costs nothing vs a cache miss costing $0.067. (2) Prompt compression — remove unnecessary tokens from the system prompt and context. Shorter prompts cost less. (3) Smaller models — route simple requests to Claude Haiku (5-10x cheaper than Opus) and complex requests to Opus. (4) Batch processing — Anthropic's message batches API reduces cost by 50% for non-real-time workloads.

Monitoring infrastructure: track per-request token usage from the API response object (response.usage.input_tokens, response.usage.output_tokens). Aggregate by: user, feature, time period. Set alerts: alert when daily cost > $X, alert when cost per request increases > 20% (indicates prompt bloat or unexpected token generation). Use Prometheus + Grafana or DataDog for visualization.

Cost attribution: attribute costs to the features/teams that generate them. Without attribution, you don't know which feature is responsible for 80% of your LLM costs. With attribution: team A's chatbot generates $500/day, team B's document summarizer generates $3,000/day. Now you know where to focus optimization efforts.

    cost_by_feature: Dict[str, float] = field(default_factory=dict)
    def track_call(self, response, model: str, feature: str = 'default'):
        input_tokens = response.usage.input_tokens
        output_tokens = response.usage.output_tokens
        prices = self.PRICE_PER_1M.get(model, {'input': 0, 'output': 0})
        cost = (input_tokens * prices['input'] + output_tokens * prices['output']) / 1_000_000
        self.total_cost += cost
        self.total_input_tokens += input_tokens
        self.total_output_tokens += output_tokens
        self.calls_by_feature[feature] = self.calls_by_feature.get(feature, 0) + 1
        self.cost_by_feature[feature] = self.cost_by_feature.get(feature, 0.0) + cost
        return cost
    def report(self):
        print(f'Total cost: ${self.total_cost:.4f}')
        print(f'Total tokens: {self.total_input_tokens:,} input, {self.total_output_tokens:,} output')
        for feature, cost in sorted(self.cost_by_feature.items(), key=lambda x: -x[1]):
            print(f'  {feature}: ${cost:.4f} ({self.calls_by_feature[feature]} calls)')
Why this matters in AI
Cost monitoring is what keeps AI products profitable. An LLM feature that's free to build can become a money sink at scale without cost controls. A product that is free to users but costs $10,000/day in LLM calls is not viable. Cost monitoring reveals which features are expensive (for optimization) and which are cheap (not worth optimizing). Without monitoring, you're flying blind on one of the largest variable costs in AI applications.
Common mistakes
  • Not tracking output tokens separately from input tokens — output tokens are typically 5x more expensive than input tokens. A model that generates 10,000 output tokens per call costs 10x more than one that generates 1,000 tokens. Monitoring only total tokens hides this difference.
  • Not setting cost alerts — without alerts, you discover unexpected costs in the monthly invoice. Set real-time alerts: 'alert if hourly LLM spend > $100' catches runaway costs before they become runaway bills.
  • Optimizing prompts without measuring the token reduction — 'I shortened the prompt so it should cost less.' Measure: count tokens before and after with tiktoken (OpenAI) or the tokenizer count_tokens endpoint. Confirm the reduction was significant.
