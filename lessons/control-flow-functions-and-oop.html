Conditions and loops

Detailed explanation

Conditional statements (if/elif/else) and loops (for/while) are the two fundamental control-flow structures in Python. Together they allow programs to make decisions based on data and repeat operations over collections. In ML, conditions appear constantly: checking if validation loss improved, if a threshold is exceeded, if a key exists in a dictionary. Loops drive training iterations, batch processing, dataset loading, and hyperparameter sweeps.

Python if-elif-else evaluates conditions top to bottom and executes the first branch that evaluates to True. Python's truthiness rules: 0, 0.0, empty string, empty list, empty dict, and None are all falsy. Everything else is truthy. This enables idiomatic checks like 'if results:' instead of 'if len(results) > 0:'.

The for loop iterates over any iterable — list, tuple, string, dict, set, range, generator, or custom object implementing __iter__. Python's enumerate(iterable) gives both index and value: for i, item in enumerate(items). zip(a, b) pairs corresponding elements from two iterables. These patterns are used constantly in ML: for epoch, (batch_x, batch_y) in enumerate(dataloader).

The while loop repeats while a condition is True. It is used for: training until convergence (while val_loss > tol), game playing loops in RL (while not done), and retry loops (while retries < max_retries). Be careful: a while loop without a guaranteed exit condition is an infinite loop. Always ensure the loop variable changes on each iteration.

Loop control: break exits the loop immediately (useful for early stopping). continue skips the rest of the current iteration (useful for skipping invalid samples). pass is a no-op placeholder (used in empty class bodies or as a stub). else on a for/while loop executes if the loop completed without hitting a break — unusual but useful for 'search and not found' patterns.

Why this matters in AI
Conditions and loops are the skeleton of every ML training script. Epoch loops, batch loops, early stopping logic, metric threshold checks, and data preprocessing pipelines are all built on these two structures. Mastering them is prerequisite to writing any training code.
Common mistakes
  • Infinite loops from forgetting to update the loop variable — always verify that a while loop has a path to termination.
  • Modifying a list while iterating over it — leads to skipped or repeated elements. Iterate over a copy: for item in list(my_list):.
  • Using == to compare floats — floating point imprecision makes 0.1 + 0.2 != 0.3. Use abs(a - b) < 1e-9 or math.isclose(a, b).

break and continue

Detailed explanation

break exits the enclosing loop immediately — execution continues with the first statement after the loop. continue skips the rest of the current iteration and moves to the next — execution jumps back to the loop condition. These two statements provide fine-grained control over loop execution.

break is used for early exit when a condition is met: early stopping in training (stop when validation loss stops improving), finding the first item matching a condition, exiting an infinite game loop when the game ends. continue is used to skip invalid or irrelevant items without nesting if-else blocks: skip corrupted training samples, skip already-processed records, skip None values.

In nested loops, break and continue only apply to the innermost enclosing loop. To break out of multiple nested loops, use a flag variable (found = True; if found: break), wrap in a function and use return, or (rarely) use Python's for-else: the else clause of a for loop executes only if the loop completed without hitting a break.

For loops with break and else: for item in items: if condition(item): result = item; break; else: result = None — the else block runs if and only if no break occurred. This is a clean 'search and not found' pattern. 'if result is None: print(Not found)' achieves the same thing but is less idiomatic.

Performance note: in inner loops over large sequences, an early break can dramatically improve performance. Searching for the first matching element in a list of 1 million items takes O(n) in the worst case. With a break as soon as the match is found, average case is O(n/2) — 2× faster for random data.

Why this matters in AI
break and continue are essential for writing efficient, readable loops. Without them, loops require deeply nested conditionals or sentinel values. Early stopping in ML training is a prototypical break use case — continuing to train when the model has converged wastes time and causes overfitting.
Common mistakes
  • Expecting break to exit multiple nested loops — break only exits the innermost loop. Use a flag variable or function return to exit multiple levels.
  • Using continue when break is meant — if you want to stop the loop, use break. continue just skips the current iteration and continues the loop.
  • Missing the for-else clause — Python's for-else is rarely known but useful for search patterns. Know it exists so you recognize it in others' code.

Functions

Detailed explanation

A function is a named, reusable block of code that takes inputs (parameters), performs a task, and optionally returns an output. Functions are the fundamental unit of code organization in Python. Every non-trivial program is built from functions. The rule of thumb: if you are writing the same logic twice, it belongs in a function; if a block of code grows beyond ~15 lines, it should be split into multiple functions.

Python functions are defined with the def keyword, followed by the function name, parentheses containing parameters, a colon, and an indented body. Parameters are variables that receive the values passed when the function is called. The return statement sends a value back to the caller. A function without an explicit return statement returns None.

Parameters can have default values, making them optional: def greet(name, greeting='Hello'): ... means you can call greet('Alice') or greet('Alice', 'Hi'). *args collects any number of positional arguments into a tuple. **kwargs collects any number of keyword arguments into a dictionary. These let you write flexible functions that accept variable numbers of arguments.

Functions should do one thing and do it well — this is the Single Responsibility Principle. A function called process_data that loads CSV, cleans nulls, engineers features, trains a model, and plots results is doing 6 things; split it into 6 functions. This makes each piece testable, debuggable, and reusable independently.

Docstrings document functions. Write a triple-quoted string immediately after the def line describing: what the function does, what each parameter is (type and meaning), and what it returns. Good docstrings are how you communicate with future-you and your teammates. In AI work, document the expected shape and dtype of tensor/array parameters explicitly.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

def compute_accuracy(y_true, y_pred):
    '''Compute classification accuracy.
    Args:
        y_true: list or array of true labels
        y_pred: list or array of predicted labels
    Returns:
        float: accuracy between 0.0 and 1.0
    '''
    return sum(t == p for t, p in zip(y_true, y_pred)) / len(y_true)
Call: acc = compute_accuracy([1,0,1,1], [1,0,0,1]) → 0.75
Default parameter: def train_model(X, y, lr=0.01, epochs=100): ...
Variable args: def log(*messages, level='INFO'): for msg in messages: print(f'[{level}] {msg}')
Return multiple values: def min_max(values): return min(values), max(values); lo, hi = min_max([3,1,4,1,5])
Why this matters in AI
Functions are how you structure any program larger than 20 lines. In AI work, every step of the pipeline — load data, preprocess, train, evaluate, deploy — should be its own function. Unfactored code is untestable and unmaintainable.
Common mistakes
  • Writing functions that do too many things — a function that loads data, trains a model, and plots results is three functions combined. Split it.
  • Not returning a value when the caller needs one — forgetting return means the function returns None silently, causing bugs where the caller gets None unexpectedly.
  • Mutable default arguments like def f(lst=[]): — the list is shared across all calls. Use def f(lst=None): if lst is None: lst = [] instead.

Arguments and return values

Detailed explanation

Python functions accept parameters (defined in the function signature) and receive arguments (actual values passed at call time). Positional arguments are passed in order. Keyword arguments are passed by name: preprocess(data, normalize=True) — normalize gets its value by name. Default values: def preprocess(data, normalize=True, max_len=512) — normalize and max_len have defaults and are optional.

*args packs extra positional arguments into a tuple. **kwargs packs extra keyword arguments into a dict. def log(*args, **kwargs): print(args, kwargs). This pattern is used in ML for flexible APIs: trainer = Trainer(*dataset_args, **config_dict) passes an arbitrary number of positional and keyword arguments.

Return values: a function can return any Python object (or None if no return statement). Return multiple values as a tuple: def split(X, y, ratio): return X[:n], X[n:], y[:n], y[n:]. The caller unpacks: X_train, X_test, y_train, y_test = split(X, y, 0.8). This is the most common Python multi-return pattern.

Variable scope: variables defined inside a function are local — they cannot be accessed outside. Variables defined outside a function are global. The global keyword allows a function to modify a global variable (avoid this — it creates hard-to-debug side effects). Use function parameters and return values instead of global state.

Type hints (Python 3.5+) annotate argument and return types: def train(X: np.ndarray, y: np.ndarray, lr: float = 0.01) -> dict. Type hints are not enforced at runtime but enable IDE autocompletion, static analysis (mypy), and serve as documentation. They are increasingly standard in professional Python code.

Why this matters in AI
Functions with clear arguments and return values are the building blocks of maintainable ML code. Poorly designed function signatures (using global variables, returning None when a value is expected) make code hard to test, debug, and reuse. Consistent function design is what separates professional ML code from notebook-only code.
Common mistakes
  • Using mutable default arguments — def fn(data=[]): data.append(1) shares the list across all calls. Use def fn(data=None): if data is None: data = [].
  • Not returning values from functions that compute results — a function that modifies a global variable instead of returning a result is hard to test and reuse.
  • Over-using **kwargs to avoid documenting arguments — **kwargs hides the function's interface. Prefer explicit named parameters with defaults for discoverability.

Lambda map filter reduce

Detailed explanation

Lambda functions are anonymous single-expression functions written inline. The syntax is: lambda arguments: expression. Lambda x: x**2 is a function that squares its input. They are useful for short functions passed as arguments — sorted(items, key=lambda x: x['score']) sorts a list of dicts by their 'score' key. Lambdas cannot contain statements (no if-elif-else blocks, no assignments), only expressions.

map(function, iterable) applies a function to every element and returns an iterator. result = list(map(lambda x: x**2, [1,2,3,4])) gives [1,4,9,16]. In ML, map is used to apply the same transformation to every element: token_ids = list(map(tokenizer.encode, texts)). List comprehensions are often more Pythonic: [x**2 for x in range(1,5)].

filter(function, iterable) returns an iterator of elements where the function returns True. long_texts = list(filter(lambda t: len(t) > 100, texts)) keeps only texts longer than 100 characters. In ML: valid_samples = list(filter(lambda s: s['label'] is not None, dataset)). Again, comprehensions often read better: [s for s in dataset if s['label'] is not None].

reduce(function, iterable) from functools applies a function cumulatively to combine all elements into a single result. reduce(lambda a, b: a*b, [1,2,3,4,5]) computes 1*2*3*4*5 = 120. In ML, reduce is used to combine metrics across batches: total_loss = reduce(lambda a, b: a+b, batch_losses). Python's built-in sum() and max() cover most reduce use cases without importing functools.

When to use lambda vs def: use lambda for simple one-liners passed to higher-order functions (sorted, map, filter, key= arguments). Use def for anything more complex — a named function is clearer, testable, and can contain multiple statements. Never use a lambda that spans multiple lines or contains complex logic. The Pythonic rule: if it needs a comment, give it a name.

Why this matters in AI
Lambda, map, filter, and reduce enable functional programming patterns — applying transformations to data without explicit loops. They appear in data preprocessing pipelines, metric computation, and dataset processing code. Understanding them makes you fluent in reading and writing idiomatic Python ML code.
Common mistakes
  • Using lambda for complex logic — lambda factorial = lambda n: 1 if n==0 else n * factorial(n-1) works but is unreadable. Write a named function.
  • Not converting map/filter results to list — map() and filter() return lazy iterators; accessing them twice gives empty results on the second access.
  • Using reduce for simple aggregations — use sum(), max(), min(), any(), all() instead of reduce(lambda a,b: a+b, ...) for clarity.

Comprehensions

Detailed explanation

List comprehensions are a concise, Pythonic way to create lists: [expression for item in iterable if condition]. squared_evens = [x**2 for x in range(20) if x % 2 == 0]. This is equivalent to a for loop with an append, but more concise and often faster (CPython optimizes comprehensions). Comprehensions are idiomatic Python — professional code uses them extensively.

Dict comprehensions create dictionaries: {key_expr: value_expr for item in iterable if condition}. label_to_idx = {label: idx for idx, label in enumerate(unique_labels)}. word_freq = {word: text.count(word) for word in set(text.split())}. Set comprehensions: {item.lower() for item in words}.

Generator expressions are lazy comprehensions — they compute values on demand without building a full list in memory. (x**2 for x in range(1_000_000)) creates a generator that yields values one at a time. This is critical when processing large datasets that don't fit in memory. sum(x**2 for x in data) uses a generator — no intermediate list is created.

Nested comprehensions handle 2D data: matrix = [[1,2,3],[4,5,6],[7,8,9]]; flat = [x for row in matrix for x in row] → [1,2,3,4,5,6,7,8,9]. For creating 2D lists: identity = [[1 if i==j else 0 for j in range(n)] for i in range(n)]. Nested comprehensions can become hard to read — if nested more than two levels deep, use explicit for loops.

When NOT to use comprehensions: (1) when the logic is complex (use a named function instead), (2) when you need to handle exceptions inside the comprehension (use a try/except in a helper function), (3) when the comprehension has side effects — comprehensions should produce a value, not cause side effects.

Why this matters in AI
Comprehensions are the Pythonic way to transform and filter data. They are faster than equivalent for loops (especially list comprehensions), more readable when simple, and enable writing data transformations in a functional style. Every Python ML engineer uses them constantly.
Common mistakes
  • Using a list comprehension when a generator would suffice — for sum(), any(), all(), and lazy operations, use a generator expression to avoid building a large intermediate list.
  • Nesting comprehensions more than 2 levels deep — beyond two levels, use explicit for loops. Readability matters more than conciseness.
  • Adding side effects inside comprehensions — [print(x) for x in items] works but is wrong. Use a for loop when side effects are intended.

Modules and packages

Detailed explanation

A Python module is a single .py file containing functions, classes, and variables. A package is a directory of modules with an __init__.py file. When you write 'import numpy as np', Python finds the numpy package (directory) in your Python path, loads its __init__.py, and makes it available as 'np'. Packages can be nested: torch.nn.functional is the 'functional' module inside the 'nn' sub-package of 'torch'.

Import styles: 'import module' — access with module.function(). 'from module import function' — access directly as function(). 'from module import *' — imports all public names into the current namespace (avoid this — pollutes the namespace and makes it unclear where names came from). 'import module as alias' — useful for long names (import numpy as np, import pandas as pd).

The PYTHONPATH and sys.path determine where Python looks for modules. Virtual environments (venv, conda) provide isolated Python installations with their own site-packages directory, preventing version conflicts between projects. Always use a virtual environment for each project: python -m venv venv; source venv/bin/activate.

Creating your own module: create utils.py with your helper functions. In main.py: from utils import preprocess, evaluate. For larger projects, organize into packages: src/data/loader.py, src/models/train.py, src/utils/metrics.py. Add an __init__.py to each directory. The __init__.py can be empty or can import key names for a cleaner public API.

Circular imports: module A imports from module B which imports from module A. This causes an ImportError. Solution: move shared code to a third module C, restructure dependencies, or use late imports (import inside functions). Circular imports are a design smell indicating modules are too tightly coupled.

Why this matters in AI
Modules and packages are how Python projects are organized. As ML projects grow beyond a single script, you need to split code into reusable components: data loading, model definition, training, evaluation. Understanding imports enables you to navigate large codebases and structure your own projects professionally.
Common mistakes
  • Naming your own file the same as a standard library module (e.g., random.py, os.py) — Python finds your file first, shadowing the standard library module and causing cryptic errors.
  • Using 'from module import *' — this makes the code hard to understand because the origin of names is unclear, and conflicts silently override names.
  • Not using virtual environments — global Python installations lead to version conflicts between projects. Always use venv or conda.

OOP

Detailed explanation

Object-Oriented Programming (OOP) organizes code around objects — entities that combine data (attributes) and behavior (methods) into a single unit. Python is a fully object-oriented language: everything, including numbers and functions, is an object. OOP helps you model real-world entities and build reusable, maintainable systems.

A class is a blueprint for creating objects. You define the blueprint with class ClassName: and create individual objects (instances) by calling the class like a function: obj = ClassName(). Each instance has its own copy of the attributes defined in __init__ (the constructor). Methods are functions defined inside the class that automatically receive the instance as their first argument (self).

Encapsulation means bundling data and the methods that operate on it together, and controlling access. In Python, a single underscore prefix (_attribute) signals 'internal, don't use outside the class' by convention; double underscore (__attribute) triggers name mangling to make it harder to access from outside. Encapsulation hides implementation details and exposes a clean interface.

Inheritance lets a subclass reuse and extend the behavior of a parent class. class Dog(Animal): means Dog inherits all attributes and methods from Animal, and can add its own or override existing ones. super().__init__() calls the parent class's constructor. This enables code reuse — you define common behavior once in the parent and specialize in children.

In AI engineering, OOP is pervasive: PyTorch models are nn.Module subclasses, scikit-learn estimators follow a fit/predict/transform interface, HuggingFace datasets use a Dataset class. Understanding OOP makes you able to read and extend these frameworks rather than treating them as black boxes.

Classes and objects

Detailed explanation

Object-oriented programming (OOP) in Python organizes code around objects — instances of classes that combine data (attributes) and behavior (methods). A class is a blueprint; an object is an instance. class Dog: defines a blueprint; fido = Dog() creates an instance. This pattern is used extensively in ML: sklearn estimators (RandomForestClassifier), PyTorch modules (nn.Linear), and dataset classes all follow the same OOP pattern.

The __init__ method is the constructor — called when an instance is created. self is the first parameter of every instance method, referring to the current instance. self.attribute stores instance data. Example: class Model: def __init__(self, lr=0.01, epochs=100): self.lr = lr; self.epochs = epochs. Methods: def train(self, X, y): ... — access instance attributes with self.lr.

Encapsulation bundles data and methods together. Private attributes (by convention: _name or __name) are intended for internal use only. Properties (@property decorator) allow attribute-like access to computed values or controlled setters. class Circle: def __init__(self, radius): self._radius = radius; @property def area(self): return 3.14 * self._radius ** 2.

PyTorch uses OOP extensively. Every neural network is a class inheriting from nn.Module: class MLP(nn.Module): def __init__(self): super().__init__(); self.fc1 = nn.Linear(784, 256); self.fc2 = nn.Linear(256, 10). def forward(self, x): return self.fc2(F.relu(self.fc1(x))). The forward() method defines the computation graph. model = MLP() creates an instance; model(x) calls forward(x).

Magic methods (dunder methods) customize class behavior. __str__ defines the string representation: print(model) calls __str__. __len__ defines len(obj). __getitem__ defines obj[i] (enables iteration and PyTorch DataSet indexing). class MyDataset: def __len__(self): return len(self.data); def __getitem__(self, idx): return self.data[idx], self.labels[idx]. DataLoader calls these to iterate over batches.

class Trainer:
    def __init__(self, model, lr=0.001, epochs=10):
        self.model = model; self.lr = lr; self.epochs = epochs
        self.history = {'train_loss': [], 'val_loss': []}
    def fit(self, X_train, y_train, X_val, y_val):
        for epoch in range(self.epochs):
            loss = self._train_epoch(X_train, y_train)
            val_loss = self._evaluate(X_val, y_val)
            self.history['train_loss'].append(loss)
            self.history['val_loss'].append(val_loss)
    def __repr__(self):
        return f'Trainer(model={type(self.model).__name__}, lr={self.lr}, epochs={self.epochs})'
PyTorch: class TwoLayerNet(nn.Module):
    def __init__(self, in_dim, hidden, out_dim):
        super().__init__(); self.net = nn.Sequential(nn.Linear(in_dim, hidden), nn.ReLU(), nn.Linear(hidden, out_dim))
    def forward(self, x): return self.net(x)
Why this matters in AI
OOP is how large ML systems are organized. Sklearn estimators, PyTorch modules, and Hugging Face models all follow the class interface. Understanding classes enables you to: extend existing models (inheritance), create custom datasets (DataSet subclasses), and write reusable training infrastructure (Trainer class).
Common mistakes
  • Forgetting super().__init__() in subclasses — PyTorch nn.Module requires calling super().__init__() to initialize the module's internals. Forgetting it causes cryptic errors.
  • Accessing instance attributes without self — inside a method, x = 5 creates a local variable; self.x = 5 sets an instance attribute. Forgetting self. means the attribute is not stored.
  • Not separating model logic from training logic — put model architecture in one class, training loop in another (or a function). This separation makes the code reusable and testable.

Inheritance polymorphism basics

Detailed explanation

Inheritance lets a class inherit attributes and methods from a parent class, eliminating code duplication and establishing an is-a relationship. class Dog(Animal): means Dog IS-A Animal and inherits all of Animal's methods. Polymorphism means different classes can be used interchangeably if they share the same interface — you can call model.fit(X, y) on any sklearn estimator because they all implement the same interface.

Inheritance syntax: class ChildClass(ParentClass): super().__init__() calls the parent's __init__ from inside the child's __init__. This initializes the parent's attributes before adding the child's own. You can override any parent method: defining the same method name in the child class replaces the parent's version. You can extend a parent method: call super().method_name() and add more behavior.

Method resolution order (MRO): Python uses C3 linearization to determine which method to call when there are multiple levels of inheritance. Child methods take precedence over parent methods. Python's built-in type() shows the class, and ClassName.__mro__ shows the method resolution order. Multiple inheritance (class C(A, B)) is supported but should be used carefully — mixins are the cleaner pattern.

Polymorphism in ML: sklearn's estimator API is the canonical example of polymorphism. LogisticRegression, RandomForestClassifier, SVC, and XGBClassifier all implement fit(X, y), predict(X), and predict_proba(X). Code that uses these methods works identically regardless of which classifier is used — you can swap classifiers without changing any other code. This is polymorphism enabling modular design.

Abstract base classes: the abc module allows you to define interfaces that subclasses MUST implement. @abstractmethod on a method in the parent class means any subclass that doesn't implement it will raise a TypeError when instantiated. This is how sklearn's BaseEstimator enforces that all estimators implement fit and predict.

Worked example

from abc import ABC, abstractmethod
class Model(ABC):
    @abstractmethod
    def fit(self, X, y): pass
    @abstractmethod
    def predict(self, X): pass
    def score(self, X, y):  # not abstract: has a default implementation
        return (self.predict(X) == y).mean()
class LinearModel(Model):
    def __init__(self, lr=0.01):
        super().__init__()
        self.lr = lr
        self.weights = None
    def fit(self, X, y):
        # Gradient descent for logistic regression
        self.weights = np.zeros(X.shape[1])
        for _ in range(1000):
            y_pred = 1/(1+np.exp(-X @ self.weights))
            self.weights -= self.lr * X.T @ (y_pred - y) / len(y)
        return self
    def predict(self, X):
        return (1/(1+np.exp(-X @ self.weights)) > 0.5).astype(int)
class TreeModel(Model):
    def __init__(self): super().__init__()
    def fit(self, X, y): from sklearn.tree import DecisionTreeClassifier; self._model = DecisionTreeClassifier().fit(X, y); return self
    def predict(self, X): return self._model.predict(X)
# Polymorphism: same code works with both models
for model in [LinearModel(lr=0.1), TreeModel()]:
    model.fit(X_train, y_train)
    print(f'{model.__class__.__name__}: accuracy = {model.score(X_val, y_val):.4f}')
Why this matters in AI
Inheritance and polymorphism are the OOP patterns that make sklearn's design so elegant: write code once that works with any estimator. In your own ML code, these patterns let you swap components (different models, different preprocessors) without rewriting the surrounding logic.
Common mistakes
  • Forgetting to call super().__init__() in the child's __init__ — if the parent class has an __init__ that sets up important attributes, forgetting super() means those attributes are never set, causing AttributeError later.
  • Overriding methods when you should be extending them — if you override a parent method completely, you lose the parent's behavior. If the parent's behavior should still run, call super().method_name() inside your override.
  • Using deep inheritance hierarchies — more than 2-3 levels of inheritance becomes confusing. Prefer composition ('has-a') over deep inheritance ('is-a'). An MLPipeline that has a FeatureExtractor and a Model is easier to understand than a hierarchy of 4 classes.
