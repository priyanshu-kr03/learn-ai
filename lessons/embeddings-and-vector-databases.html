Embeddings

Detailed explanation

An embedding is a dense, fixed-size vector of real numbers that represents the semantic meaning of a piece of text, an image, or other data in a continuous vector space. Items that are semantically similar should have embeddings that are close together (measured by cosine similarity or dot product), while dissimilar items should be far apart. Embeddings capture meaning in a form that mathematical operations can manipulate.

Text embeddings are produced by encoder models like BERT, sentence-transformers, or OpenAI's text-embedding-ada-002. These models encode the input text into a vector of typically 768 or 1536 dimensions. The embedding captures not just keywords but semantic concepts: 'king' and 'monarch' will be nearby even if they share no characters. 'Paris is to France as Berlin is to ___' can be answered by vector arithmetic: Berlin_embedding + (France_embedding - Paris_embedding) ≈ Germany_embedding.

Embeddings enable semantic search. Instead of keyword matching (which fails when different words express the same concept), you embed the query and all documents, then find documents whose embeddings are nearest to the query embedding. This is the core of RAG systems and semantic search engines. Vector databases (Pinecone, Weaviate, Chroma, FAISS) are optimized for storing millions of embeddings and performing approximate nearest-neighbor search in milliseconds.

Beyond text, embeddings exist for images (CLIP embeds images and text in the same space), audio (Whisper encoder produces audio embeddings), tabular data (entity embeddings for categorical features), and graphs (node2vec for graph nodes). The crucial property in multimodal systems: if an image and its caption have nearby embeddings, you can search images with text queries and vice versa.

Embedding fine-tuning (creating domain-specific embeddings) improves semantic search on specialized domains. General embeddings trained on web text may not capture domain-specific vocabulary well — in medical text, 'MI' means myocardial infarction, not Michigan. Fine-tuning embeddings on domain-specific text or contrastive pairs (this document is relevant to this query, that one is not) can dramatically improve retrieval precision.

Why this matters in AI
Embeddings are the mathematical foundation of modern semantic search, RAG systems, recommendation engines, and zero-shot classification. Every time you use semantic search or RAG, embeddings are doing the heavy lifting.
Common mistakes
  • Using keyword search when you need semantic search — keywords miss synonyms and paraphrases; embeddings capture semantic equivalence.
  • Embedding very long documents as a single vector — the embedding averages all content and the specific relevant passage is diluted. Chunk documents into 256–512 token pieces and embed each chunk separately.
  • Comparing embeddings from different models — embeddings from different models live in different vector spaces; they cannot be directly compared.

Sentence embeddings

Detailed explanation

Sentence embeddings (also called sentence vectors) map a full sentence or paragraph to a dense vector representation where semantically similar texts are close together in vector space. Unlike word embeddings (word2vec, GloVe) which embed individual words, sentence embeddings embed complete utterances — capturing sentence-level meaning including word order, negation, and context.

Sentence-BERT (SBERT): the foundational sentence embedding model, trained by fine-tuning BERT with siamese network training on NLI (Natural Language Inference) and STS (Semantic Textual Similarity) datasets. Given sentence pairs labeled as entailment/contradiction/neutral, SBERT learns embeddings where entailing sentences are close and contradicting sentences are far apart. SBERT embeddings dramatically outperform naive [CLS] pooling or average pooling of BERT token embeddings for similarity tasks.

Pooling strategies: how to go from token embeddings (one embedding per token) to a single sentence embedding. Mean pooling: average all token embeddings weighted by attention mask (ignore padding). [CLS] pooling: use only the [CLS] token's embedding (original BERT). Max pooling: take the maximum activation across all positions for each dimension. Mean pooling is most common and generally best for sentence similarity. WeightedMean pooling weights more recent layers higher.

Modern sentence embedding models: OpenAI text-embedding-3-small (1536 dimensions, strong performance), OpenAI text-embedding-3-large (3072 dimensions, state-of-the-art), all-MiniLM-L6-v2 (384 dimensions, fast, open-source), all-mpnet-base-v2 (768 dimensions, best quality open-source general-purpose), E5-large, BGE-large (Chinese + English), Cohere Embed v3. For production RAG systems: text-embedding-3-small is a great default (fast, cheap, high quality).

Dimensionality and normalization: sentence embeddings range from 384 to 3072 dimensions. For cosine similarity: normalize embeddings to unit length so that cosine similarity = dot product (faster). For storage: reduce to 256 or 512 dimensions with PCA or MRL (Matryoshka Representation Learning) for 4-8× storage reduction with minimal quality loss. OpenAI's text-embedding-3 models support dimensions parameter for built-in truncation.

Why this matters in AI
Sentence embeddings are the foundation of modern semantic search, RAG systems, recommendation engines, and document clustering. Any system that needs to find semantically similar content (not just keyword matches) relies on sentence embeddings. Understanding how they work enables building powerful, scalable semantic search at any scale.
Common mistakes
  • Using word embeddings (GloVe/word2vec) for sentence similarity by averaging word vectors — this loses word order and context. 'The dog bit the man' and 'The man bit the dog' get the same embedding. Use sentence-specific models.
  • Using [CLS] token from BERT without fine-tuning for similarity — vanilla BERT's [CLS] embedding is not trained for similarity (only for NSP). It performs poorly. Use SBERT or sentence-transformers.
  • Mixing embeddings from different models — all FAISS indices and embeddings must come from the same model with the same normalization. Adding OpenAI embeddings to an SBERT index gives nonsensical results.

Cosine similarity

Detailed explanation

Cosine similarity measures the cosine of the angle between two vectors: cos(θ) = (A·B)/(||A|| * ||B||). It ranges from -1 (opposite direction) to +1 (same direction). A cosine similarity of 1 means the vectors point in the same direction (regardless of magnitude). This makes it ideal for text and embedding similarity — two documents about 'machine learning' should be similar whether they are short or long.

Why cosine similarity is preferred for text: a long document about machine learning and a short document about machine learning have different TF-IDF vector magnitudes (the long one has larger values because terms appear more times). But their direction is similar. Cosine similarity normalizes by magnitude — it measures direction (topic) rather than magnitude (length). Euclidean distance would incorrectly find the long document dissimilar from the short one.

In embedding spaces (word2vec, BERT, sentence transformers), semantically similar concepts have high cosine similarity. king - man + woman ≈ queen: the difference vectors are parallel in the embedding space. Semantic search retrieves documents with high cosine similarity to the query embedding. This is how all modern embedding-based retrieval works.

Computing cosine similarity efficiently: for large sets of vectors, precompute normalized embeddings (divide each by its L2 norm), then use matrix multiplication to compute all pairwise dot products: similarities = X_norm @ X_norm.T. Since vectors are normalized, dot products equal cosine similarities. This vectorized approach is 100–1000× faster than computing similarities in a Python loop.

Cosine similarity vs dot product: normalized dot product = cosine similarity. For semantic search, normalizing embeddings before computing dot products enables using efficient MIPS (Maximum Inner Product Search) algorithms — the same algorithms used in FAISS and vector databases — for cosine similarity search.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

a = np.array([1.0, 0.0, 1.0])  # document about topic A
b = np.array([2.0, 0.0, 2.0])  # same topic, longer document
c = np.array([0.0, 1.0, 0.0])  # different topic
print(cosine_similarity(a, b))  # 1.0 — same direction
print(cosine_similarity(a, c))  # 0.0 — orthogonal
Batch similarities: X_norm = X / np.linalg.norm(X, axis=1, keepdims=True)
similarity_matrix = X_norm @ X_norm.T  # (n × n) all-pairs
sklearn: from sklearn.metrics.pairwise import cosine_similarity
sim = cosine_similarity(query_embeddings, doc_embeddings)  # (n_queries, n_docs)
Why this matters in AI
Cosine similarity is the universal metric for semantic similarity in NLP and embedding-based search. Understanding it enables you to build semantic search, find similar users/items in recommendation systems, and cluster documents by topic.
Common mistakes
  • Using Euclidean distance instead of cosine similarity for text embeddings — Euclidean distance penalizes different magnitudes (document lengths) rather than topic similarity.
  • Not normalizing before using dot products as similarities — raw dot products are scale-dependent. Normalize to unit vectors first.
  • Expecting cosine similarity to work for vectors with many near-zero dimensions — sparse vectors (TF-IDF) need dot product computation on only non-zero dimensions for efficiency. Use sparse matrix operations.

Semantic search

Detailed explanation

Semantic search retrieves documents based on meaning rather than exact keyword match. Traditional keyword search (BM25, Elasticsearch) finds documents containing the exact query words. Semantic search finds documents with similar meaning even if they use completely different words. Query: 'dog diseases' → semantic search also finds 'canine health conditions', 'veterinary illnesses' — keyword search would miss these.

Dense retrieval pipeline: (1) offline indexing: embed all documents with a sentence encoder, store embeddings in a vector index (FAISS, Pinecone, Weaviate, Chroma). (2) Online query: embed the query, search the vector index for the nearest neighbor embeddings, return the top-K most similar documents. The entire process takes milliseconds even for millions of documents using approximate nearest neighbor (ANN) search.

Hybrid search: combining dense (semantic) and sparse (keyword/BM25) retrieval. BM25 is strong for exact matches, acronyms, proper nouns, and out-of-vocabulary terms. Dense retrieval is strong for synonyms, paraphrases, and conceptual matches. Hybrid search uses both and combines scores (Reciprocal Rank Fusion, or convex combination). Most production search systems use hybrid search — it outperforms either alone.

Embedding-based semantic search vs cross-encoder reranking: semantic search with bi-encoders is for retrieval (candidate generation from a large corpus), cross-encoder reranking is for precision (rescore a small candidate set). The two-stage architecture — retrieve top-100 with bi-encoder, rerank with cross-encoder — combines speed and accuracy.

Evaluation metrics: NDCG@k (Normalized Discounted Cumulative Gain), MRR (Mean Reciprocal Rank), Recall@k (did the relevant document appear in top-k results?), MAP (Mean Average Precision). BEIR benchmark evaluates retrieval models across 18 diverse IR datasets. Run evaluation on your specific domain — general-purpose models may underperform domain-specific fine-tuned models on specialized corpora.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from sentence_transformers import SentenceTransformer
import faiss
import numpy as np
model = SentenceTransformer('all-MiniLM-L6-v2')
documents = ['BERT is a bidirectional transformer.', 'PyTorch is a deep learning framework.',
             'Dogs are loyal companions.', 'NLP processes natural language text.']
# Index documents
doc_embeddings = model.encode(documents, normalize_embeddings=True).astype('float32')
index = faiss.IndexFlatIP(doc_embeddings.shape[1])  # Inner product (= cosine for normalized)
index.add(doc_embeddings)
# Query
query = 'transformers for NLP'
q_emb = model.encode([query], normalize_embeddings=True).astype('float32')
scores, indices = index.search(q_emb, k=3)
for i, idx in enumerate(indices[0]):
    print(f'{i+1}. [{scores[0,i]:.3f}] {documents[idx]}')
# Hybrid search: BM25 + dense retrieval with RRF
from rank_bm25 import BM25Okapi
# BM25 retrieval
corpus = [doc.lower().split() for doc in documents]
bm25 = BM25Okapi(corpus)
bm25_scores = bm25.get_scores(query.lower().split())
bm25_rank = np.argsort(bm25_scores)[::-1]
dense_rank = indices[0].tolist()
# Reciprocal Rank Fusion
def rrf(ranks, k=60):
    scores = {}
    for rank_list in ranks:
        for position, doc_id in enumerate(rank_list):
            scores[doc_id] = scores.get(doc_id, 0) + 1/(k + position + 1)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)
hybrid = rrf([bm25_rank, dense_rank])
print('Hybrid results:', [(documents[i], round(s,4)) for i,s in hybrid[:3]])
Why this matters in AI
Semantic search is the foundation of modern RAG systems, enterprise search, and recommendation engines. It solves the vocabulary mismatch problem that has plagued information retrieval for decades. Understanding dense retrieval, vector indices, and hybrid search enables building search experiences that actually understand what users mean.
Common mistakes
  • Using cosine similarity on non-normalized embeddings with dot product — if embeddings are not normalized to unit length, dot product is not cosine similarity. Either normalize before storing or use faiss.IndexFlatL2 (Euclidean distance) and convert.
  • Indexing chunks without metadata — when retrieving a chunk, you need to know which document it came from, its position, and how to fetch surrounding context. Always store metadata alongside embeddings.
  • Not evaluating retrieval quality separately from generation quality — in a RAG system, a poor retriever will always produce poor answers even with a perfect LLM. Evaluate retrieval (Recall@k) independently from end-to-end answer quality.

Vector database

Detailed explanation

A vector database is a database system optimized for storing and searching high-dimensional vector embeddings. Traditional databases support exact lookups (find records where id=123) or text search (find records containing 'machine learning'). Vector databases support approximate nearest neighbor (ANN) search: find the K vectors most similar to a query vector by cosine or Euclidean distance. This enables semantic search over large collections of embeddings.

Why approximate nearest neighbor? Exact nearest neighbor search in d dimensions requires comparing the query against all n vectors — O(n*d) — which is feasible for n < 1M but too slow for large collections. ANN algorithms trade a small accuracy loss (not always finding the exact nearest neighbor) for dramatic speedups. HNSW (Hierarchical Navigable Small World) is the current state-of-the-art ANN algorithm: it builds a multi-layer graph where higher layers provide long-range connections and lower layers provide precise short-range connections. Query time is approximately O(log(n)).

Vector database options: FAISS (Facebook AI Similarity Search) is an open-source library (not a full database) — fast, supports GPU, excellent for research and prototyping. Chroma is an open-source database designed for LLM applications — simple API, metadata filtering, good for development. Pinecone is a managed cloud vector database — fully managed, scales to billions of vectors, expensive. Weaviate and Qdrant are open-source full-featured vector databases with REST APIs, metadata filtering, and hybrid search. pgvector is a PostgreSQL extension — add vector search to your existing PostgreSQL database with minimal new infrastructure.

Metadata filtering enables combining vector similarity with structured attribute constraints. A document database might store embeddings alongside metadata: {document_id, date, author, topic, department}. A query like 'find the 5 most similar documents about compliance written after 2023-01-01 by the legal team' requires both vector similarity and metadata filtering. All modern vector databases support this combination — it is essential for production RAG systems.

Choosing a vector database: for development/prototyping, use Chroma (easy setup, local persistence). For small production deployments (< 10M vectors), use pgvector (no new infrastructure if you already use PostgreSQL). For large production deployments (> 100M vectors) with SLA requirements, use Pinecone or Qdrant managed cloud. For privacy-sensitive use cases with full data control, self-hosted Qdrant or Weaviate.

Why this matters in AI
Vector databases are the infrastructure layer for all semantic search and RAG applications. Without one, embedding-based retrieval requires comparing the query to every stored embedding — O(n) per query. With an ANN index, the same query runs in milliseconds even across millions of embeddings.
Common mistakes
  • Not normalizing embeddings before using inner product similarity — for cosine similarity via inner product, embeddings must be L2-normalized. Un-normalized vectors give arbitrary results.

FAISS

Detailed explanation

FAISS (Facebook AI Similarity Search) is an open-source library for efficient similarity search over dense vectors. It solves the fundamental problem: given millions of embedding vectors, find the K most similar vectors to a query vector in milliseconds. Without FAISS, brute-force cosine similarity over 1 million 768-dimensional vectors takes ~1 second per query; FAISS ANN search takes < 1 millisecond.

FAISS index types: (1) IndexFlatL2 / IndexFlatIP — exact search. Computes exact Euclidean (L2) or inner product (IP = cosine for normalized) distance. Accurate but O(n) per query. For < 100K vectors, use this. (2) IndexIVFFlat — Inverted File Index. Clusters vectors into nlist buckets (using K-means), searches only nprobe buckets at query time. Fast (10-100× speedup) with slight accuracy loss. For 100K-10M vectors. (3) IndexHNSWFlat — Hierarchical Navigable Small World graph. Very fast ANN search. Better accuracy-speed trade-off than IVF. (4) IndexIVFPQ — IVF + Product Quantization. Compresses vectors to ~1/32 their size — fits 100M vectors in RAM. Some accuracy loss.

IndexIVF training: IVF indices require training a K-means clustering (quantizer) before adding vectors. faiss.index_factory creates composite indices. Rule of thumb: nlist = 4 × sqrt(n_vectors). nprobe (number of clusters searched at query time): higher nprobe = higher accuracy, lower speed. Tune nprobe for your accuracy-speed requirement.

GPU acceleration: FAISS has GPU implementations (faiss-gpu) that can index and search 100× faster than CPU. For large production systems: offline indexing on GPU, serve the index from CPU (GPU servers are expensive). Or use managed vector databases (Pinecone, Weaviate, Qdrant) which handle infrastructure.

FAISS vs vector databases: FAISS is a library (Python, C++) — you manage the file I/O, updates, and serving yourself. Vector databases (Pinecone, Weaviate, Chroma, Qdrant, pgvector) add: persistent storage, CRUD operations (add/update/delete), metadata filtering, HTTP API, authentication, and horizontal scaling. For production: use a vector database. For research or on-premises: use FAISS directly.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import faiss
import numpy as np
d = 384  # embedding dimension
n = 100000  # number of documents
np.random.seed(42)
xb = np.random.randn(n, d).astype('float32')  # document embeddings
faiss.normalize_L2(xb)  # normalize for cosine similarity
# Exact search (baseline)
index_flat = faiss.IndexFlatIP(d)
index_flat.add(xb)
# ANN search with HNSW
index_hnsw = faiss.IndexHNSWFlat(d, 32)  # 32 neighbors per node
index_hnsw.add(xb)
# Query
xq = np.random.randn(1, d).astype('float32')
faiss.normalize_L2(xq)
D_flat, I_flat = index_flat.search(xq, k=5)
D_hnsw, I_hnsw = index_hnsw.search(xq, k=5)
print('Exact:', I_flat[0], '\nHNSW:', I_hnsw[0])
print('HNSW recall:', len(set(I_flat[0]) & set(I_hnsw[0]))/5)
# IVF index for large scale
nlist = 100  # number of clusters
quantizer = faiss.IndexFlatIP(d)
index_ivf = faiss.IndexIVFFlat(quantizer, d, nlist)
index_ivf.train(xb)  # K-means clustering
index_ivf.add(xb)
index_ivf.nprobe = 10  # search 10 clusters at query time
D, I = index_ivf.search(xq, k=5)
# Save and load index
faiss.write_index(index_ivf, 'my_index.faiss')
index_loaded = faiss.read_index('my_index.faiss')
Why this matters in AI
FAISS is the building block behind every vector similarity search at scale. Whether you use LangChain, LlamaIndex, Weaviate, or Chroma — they all use FAISS or its algorithms under the hood. Understanding FAISS directly gives you full control over index configuration, accuracy-speed trade-offs, and memory usage.
Common mistakes
  • Not normalizing embeddings before adding to IndexFlatIP — IndexFlatIP computes inner product, which equals cosine similarity ONLY for unit-norm vectors. Call faiss.normalize_L2(xb) before adding to IP indices.
  • Using IndexFlatL2/IP for millions of vectors — exact flat indices are O(n) per query. For > 100K vectors, switch to HNSW or IVF for acceptable query latency.
  • Not training the IVF index before adding vectors — IndexIVFFlat.train(xb) must be called before add(). Skipping this raises an error or produces wrong results.

Chroma

Detailed explanation

ChromaDB is an open-source, lightweight vector database designed specifically for embedding-based retrieval in LLM applications. Unlike production vector databases (Pinecone, Weaviate, Qdrant), Chroma prioritizes ease of use and runs in-process (no separate server) or as a lightweight server. It's the default vector store for LangChain and LlamaIndex in development and small-scale deployment.

Core concepts: Collection — a named container for embeddings, similar to a table in a relational database. Each collection stores: embeddings (vectors), documents (the original text), metadata (key-value pairs for filtering), and IDs (unique identifiers). Documents, embeddings, and metadata are stored together so you can retrieve the original text alongside similarity scores.

Embedding functions: Chroma has a default embedding function (sentence-transformers/all-MiniLM-L6-v2) that is called automatically when you add documents. You can also provide precomputed embeddings. For production quality, override with a better embedding model: `embedding_function = embedding_functions.OpenAIEmbeddingFunction(api_key=..., model_name='text-embedding-3-large')`.

Persistence: by default, Chroma runs in-memory (no persistence across sessions). For persistence: `client = chromadb.PersistentClient(path='./chroma_db')` — stores the database on disk and can be reloaded in future sessions. For multi-process or multi-machine access: use `chromadb.HttpClient(host='localhost', port=8000)` with a running Chroma server.

When to use Chroma vs other options: Chroma is perfect for: local development, prototyping, small-scale applications (< 1M vectors), educational use. Switch to managed services (Pinecone, Weaviate Cloud) when: you need cloud hosting, very large collections (> 10M vectors), production reliability (uptime SLAs), or advanced features (hybrid search, multi-tenancy).

Why this matters in AI
ChromaDB is the fastest way to add vector search to an LLM application. Setting up Chroma takes 10 lines of code; setting up Pinecone or Weaviate takes 30+ lines and requires external accounts. For learning RAG, building prototypes, or running locally, Chroma is the right default. Knowing when to graduate to a managed vector database is the next skill.
Common mistakes
  • Using in-memory Chroma for anything that should persist — `chromadb.Client()` loses all data when the process ends. Use PersistentClient for any data you want to keep between sessions.
  • Not specifying an embedding function — if you don't provide an embedding function when creating a collection, Chroma uses its default. If you later query with a different embedding model, similarity scores will be meaningless (comparing apples to oranges in embedding space). Always explicitly specify and document which embedding model a collection uses.
  • Not handling IDs carefully — Chroma requires each document to have a unique ID. Adding a document with the same ID as an existing one will raise an error. Use deterministic IDs (hash of content) to enable idempotent upserts.

Pinecone Milvus pgvector

Detailed explanation

Pinecone, Milvus, and pgvector are production vector databases for storing and querying high-dimensional embeddings at scale. Unlike ChromaDB (developer tool), these are designed for production workloads: millions to billions of vectors, high query throughput, low latency, and enterprise features like access control and monitoring.

Pinecone: a fully managed, serverless vector database. Pros: zero infrastructure management, automatic scaling, REST API, handles billions of vectors. Cons: vendor lock-in, cost (serverless pricing by usage). Best for: teams without ML infrastructure expertise, startups that want to move fast. An index in Pinecone is a fully managed collection: you specify dimension, metric (cosine/euclidean/dotproduct), and serverless configuration.

Milvus: an open-source vector database (Apache 2.0 license). Can be self-hosted (Docker, Kubernetes) or used via Zilliz Cloud (managed service). Supports many index types (HNSW, IVF_FLAT, IVF_SQ8) for different performance-accuracy tradeoffs. Supports multiple vector fields per collection, hybrid search (scalar + vector filtering), and multi-tenancy. Best for: teams with infrastructure capability and cost sensitivity.

pgvector: a PostgreSQL extension that adds vector similarity search to Postgres. `CREATE EXTENSION vector`. Pros: no new infrastructure if you already use Postgres, SQL queries with vector similarity, ACID transactions, combine with relational data natively. Cons: performance degrades for very large vector collections (> 10M vectors) without careful tuning. Best for: existing Postgres users with moderate vector search needs (< 10M vectors, < 100 QPS).

Choosing a vector database: start with pgvector if you're already using Postgres and have < 5M vectors. Use Pinecone for fast prototyping with < 1M vectors on a managed service. Use Milvus when you have > 10M vectors, need maximum query performance (HNSW index), and have DevOps capability to manage the deployment.

# pgvector setup (SQL)

# 2. CREATE EXTENSION vector;  -- in Postgres
# 3. Create table with vector column
from pgvector.psycopg2 import register_vector
import psycopg2
import numpy as np
conn = psycopg2.connect(database='mydb', user='postgres', host='localhost')
register_vector(conn)
cur = conn.cursor()
cur.execute('CREATE TABLE IF NOT EXISTS docs (id serial PRIMARY KEY, content text, embedding vector(384))')
# Insert embeddings
embeddings = np.random.rand(100, 384).astype('float32')
for i in range(100):
    cur.execute('INSERT INTO docs (content, embedding) VALUES (%s, %s)', (f'doc_{i}', embeddings[i].tolist()))
conn.commit()
# Create HNSW index for fast search
cur.execute('CREATE INDEX ON docs USING hnsw (embedding vector_cosine_ops)')
# Query: find 5 most similar docs
query_vec = np.random.rand(384).tolist()
cur.execute('SELECT id, content, embedding <=> %s AS distance FROM docs ORDER BY embedding <=> %s LIMIT 5', (query_vec, query_vec))
results = cur.fetchall()
for id, content, dist in results: print(f'{content}: distance={dist:.4f}')
# Pinecone setup
from pinecone import Pinecone, ServerlessSpec
pc = Pinecone(api_key='your-api-key')
# Create index
if 'my-index' not in pc.list_indexes().names():
    pc.create_index(
        name='my-index', dimension=384, metric='cosine',
        spec=ServerlessSpec(cloud='aws', region='us-east-1')
    )
index = pc.Index('my-index')
# Upsert vectors
vectors = [(f'id_{i}', np.random.rand(384).tolist(), {'category': 'product'}) for i in range(100)]
index.upsert(vectors=vectors)
# Query
results = index.query(vector=np.random.rand(384).tolist(), top_k=5, filter={'category': 'product'})
for match in results.matches: print(f'id={match.id}, score={match.score:.4f}')
Why this matters in AI
Production vector databases are the infrastructure layer that makes RAG applications scalable. ChromaDB running locally on a single machine can handle thousands of vectors; Pinecone can handle billions. Understanding the tradeoffs between managed services (Pinecone), open-source self-hosted (Milvus), and database extensions (pgvector) lets you choose the right infrastructure for each project's scale, budget, and operational constraints.
Common mistakes
  • Choosing Pinecone for a personal project with a small budget — Pinecone's serverless pricing is fine for prototyping, but costs increase significantly with traffic. For a personal project with low traffic, pgvector or ChromaDB is free.
  • Not creating a vector index in pgvector — without `CREATE INDEX USING hnsw`, similarity queries in pgvector do full table scans (O(n)). With an HNSW index, queries are O(log n). The HNSW index must be explicitly created.
  • Not normalizing vectors for cosine similarity — cosine similarity measures the angle between vectors. If your embedding model doesn't output unit-normalized vectors, you must normalize before inserting: `embedding / np.linalg.norm(embedding)`. Without normalization, cosine similarity scores are inaccurate.

Metadata filtering

Detailed explanation

Metadata filtering combines vector similarity search with structured attribute filtering — retrieve documents that are both semantically relevant AND match specific attribute conditions. 'Find product documentation for iPhone 15 models' requires: semantic match ('iPhone 15') AND attribute filter (product_line='iPhone', version='15').

Why metadata filtering matters: without filtering, vector search returns the most similar documents globally. In a knowledge base with documents for multiple products, a query about Product A may retrieve documents from Products B and C if they're topically similar. Metadata filtering restricts the search to the relevant subset, improving precision without sacrificing recall.

Pre-filtering vs post-filtering: pre-filtering — restrict the search space to documents matching the metadata condition, then find the most similar within that subset. Faster for large collections when the filter is selective. Post-filtering — retrieve top-k by similarity, then filter by metadata. Works when the filter is not very selective (returns many documents). Most vector databases use pre-filtering for indexed metadata fields.

Metadata design: which fields to include as metadata depends on your access patterns. For customer support: product_name, product_version, category, language, date_created. For news: source, date, topic, sentiment. For code: language, repository, file_type, last_modified. Index fields you'll frequently filter on; store infrequently-used fields as additional context.

Combining filters: AND, OR, NOT conditions. 'category=electronics AND price<500 AND in_stock=true.' Vector databases implement this as structured query syntax — Pinecone uses MongoDB-style operators: {"$and": [{"category": "electronics"}, {"price": {"$lt": 500}}]}. pgvector uses SQL WHERE clause natively.

Why this matters in AI
Metadata filtering is the difference between a demo RAG system and a production RAG system. A customer support RAG without metadata filtering retrieves documentation for all products when a user asks about one specific product — generating answers that mix information from different products. Metadata filtering restricts retrieval to contextually appropriate documents, dramatically improving answer quality.
Common mistakes
  • Over-filtering to the point of retrieving nothing — too strict a filter (exact version match, exact category) may return zero relevant documents if the metadata is slightly inconsistent. Implement a fallback: if filtered results < 2, retry with a relaxed filter.
  • Not indexing frequently-filtered metadata fields — vector databases store metadata, but not all fields are indexed for fast filtering. Check your database's documentation — in Pinecone, all metadata fields are filterable; in Milvus, you must create a scalar index on fields used for filtering.
  • Storing too much in metadata — metadata should contain structured, filterable attributes. Long text content belongs in the document field, not in metadata. Storing a full document as metadata defeats the purpose of the document field and makes metadata filtering slower.

Hybrid search

Detailed explanation

Hybrid search combines dense vector search (semantic similarity via embeddings) with sparse lexical search (keyword matching via BM25 or TF-IDF) to retrieve documents that are either semantically related OR contain exact keywords. Neither method alone is optimal: dense search excels at semantic similarity but misses exact keyword matches; sparse search excels at exact terms but misses paraphrases.

Why the combination matters: query 'PyTorch gradient checkpointing' — dense search might return documents about 'memory optimization in deep learning' (semantically related but doesn't mention the exact technique). BM25 returns documents containing all three exact terms. The hybrid retrieves both: semantic matches for context, exact matches for technical precision. This is especially important for technical documentation, code search, and medical information.

Reciprocal Rank Fusion (RRF): the simplest way to combine results from multiple retrieval systems. For each retrieved document, assign a score: RRF(d) = Σ 1/(k + rank_in_retrieval_system_i) where k=60 is a constant. Documents appearing in both systems get double contribution. RRF doesn't require calibrated scores across systems — it only uses rank position.

Alpha-weighted fusion: alternatively, compute a weighted sum of dense and sparse scores: final_score = α × dense_score + (1-α) × sparse_score. Requires normalizing scores to [0,1] first. α is a hyperparameter that balances semantic vs keyword matching. Optimize α on a validation set — typical values are α=0.5 to 0.7 (slightly favoring dense).

Production hybrid search: Weaviate, Elasticsearch, and OpenSearch support hybrid search natively. Qdrant and Pinecone provide sparse+dense vector storage. For LangChain: `EnsembleRetriever` combines a BM25Retriever (sparse) and a VectorStoreRetriever (dense) with configurable weights.

from langchain.retrievers import BM25Retriever, EnsembleRetriever
from langchain.vectorstores import Chroma
from langchain.embeddings import HuggingFaceEmbeddings
# Setup
docs = [...]  # list of Document objects
# Dense retriever (semantic)
embeddings = HuggingFaceEmbeddings(model_name='all-MiniLM-L6-v2')
vectorstore = Chroma.from_documents(docs, embeddings)
dense_retriever = vectorstore.as_retriever(search_kwargs={'k': 5})
# Sparse retriever (keyword)

bm25_retriever.k = 5
# Ensemble with RRF
ensemble_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, dense_retriever],
    weights=[0.4, 0.6]  # 40% BM25, 60% dense
)
results = ensemble_retriever.invoke('PyTorch gradient checkpointing memory optimization')
for r in results: print(r.page_content[:100])
# RRF implementation from scratch
def reciprocal_rank_fusion(results_list, k=60):
    scores = {}
    for results in results_list:
        for rank, doc_id in enumerate(results):
            scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank + 1)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)
# Usage
dense_results = ['doc_3', 'doc_7', 'doc_1', 'doc_5', 'doc_2']  # by dense score
sparse_results = ['doc_1', 'doc_3', 'doc_8', 'doc_4', 'doc_6']  # by BM25 score
fused = reciprocal_rank_fusion([dense_results, sparse_results])
print(fused[:5])  # doc_3 and doc_1 appear in both → higher RRF score
Why this matters in AI
Hybrid search consistently outperforms either pure dense or pure sparse retrieval across most RAG benchmarks by 5-15% on recall@10. The improvement is highest for queries with specific technical terms or proper nouns that are sparsely represented in the embedding space. In production RAG systems, hybrid search is the recommended default retrieval strategy.
Common mistakes
  • Not normalizing scores before alpha-weighted combination — dense scores (cosine similarity: 0-1) and sparse scores (BM25: 0-50) have very different ranges. Without normalization, one dominates. Always normalize to [0,1] before weighted fusion.
  • Using BM25 alone for short queries — BM25 gives very noisy results for single-word queries ('fraud', 'churn') where most documents contain that word. Dense search performs better for short or vague queries. Hybrid helps most for medium-length technical queries (4-15 words).
  • Not evaluating hybrid vs dense-only on your specific dataset — hybrid generally wins, but on datasets with very unique terminology or code, BM25 may dominate. Always measure: recall@10 for dense-only, BM25-only, and hybrid on your validation set.

Chunking

Detailed explanation

Chunking is the process of splitting long documents into smaller segments (chunks) before embedding and indexing them in a vector store. This is necessary because: (1) embedding models have context window limits (typically 512 tokens), (2) smaller chunks provide more precise retrieval (a 100-word chunk about 'attention mechanism' will rank higher for an attention query than a 5000-word chapter that mentions attention once), (3) the LLM context window limits how much retrieved text can be included in the prompt.

Fixed-size chunking: split every n tokens (or characters) with an optional overlap. Pros: simple, predictable chunk sizes. Cons: may split in the middle of a sentence, paragraph, or concept. overlap=50 tokens means consecutive chunks share 50 tokens — this ensures continuity at chunk boundaries and prevents important context from being split across chunks. Example: chunk_size=256 tokens, overlap=32 tokens.

Semantic chunking: split at natural semantic boundaries — sentence boundaries, paragraph breaks, section headings. spaCy's sentencizer splits at sentence boundaries. Markdown splitter splits at ## headings. HTML splitter splits at tags. These produce variable-length chunks that preserve semantic units. The trade-off: some chunks may be too long for the embedding model or too short to be meaningful.

Recursive character text splitter (LangChain default): tries to split at the most natural boundary available, in order of preference: \n\n (paragraph), \n (newline), '. ' (sentence), ' ' (word), '' (character). This produces chunks that respect document structure while enforcing a maximum size.

Optimal chunk size: depends on the use case. For dense factual documents (legal, medical): 128-256 tokens (precise retrieval of specific facts). For narrative documents (books, articles): 512-1024 tokens (preserves context). For code: split by function/class (variable size). Always evaluate retrieval quality at multiple chunk sizes — there is no single optimal size. Rule of thumb: match chunk size to the expected length of the relevant passage for your queries.

Why this matters in AI
Chunking strategy directly affects retrieval quality in RAG systems. Poor chunking (too large: imprecise retrieval; too small: loses context; splitting mid-sentence: incoherent chunks) is one of the most common causes of poor RAG performance. Thoughtful chunking is as important as embedding model choice.
Common mistakes
  • Using fixed character-based chunking without overlap — chunk at exactly 1000 characters means the last sentence of one chunk and first sentence of the next chunk are separated. A query spanning the boundary won't find either chunk. Use overlap of 10-20% of chunk size.
  • Using the same chunk size for all document types — code should be chunked by function, markdown by section heading, legal documents by clause. Use document-type-aware chunking.
  • Not storing metadata with chunks — when a chunk is retrieved, you need to know: which document it came from, which page, what section. This is needed for citations, context expansion, and debugging. Always preserve and store metadata.

Embedding evaluation

Detailed explanation

Embedding evaluation measures whether an embedding model captures the semantic properties relevant to your use case. Generic embedding benchmarks (MTEB — Massive Text Embedding Benchmark) measure across many tasks, but the best embedding model for generic tasks may not be best for your specific domain (legal contracts, medical notes, code).

Intrinsic evaluation: measures properties of the embedding space directly without a downstream task. Semantic similarity: compute embeddings for pairs of sentences with human-judged similarity scores (STS-Benchmark, STS-B). Measure Spearman correlation between cosine similarity and human scores. A high Spearman correlation means the model's similarity judgments match human intuition.

Extrinsic evaluation (most important for production): evaluate embeddings on your actual downstream task. For RAG: given a query and a corpus, retrieve top-k documents — what fraction contain the answer? (Recall@k). For semantic search: given queries with labeled relevant documents, compute NDCG@10 (Normalized Discounted Cumulative Gain). For clustering: cluster embeddings and compare to known labels with ARI.

Domain adaptation matters: an embedding model trained on Wikipedia performs well on general text. For legal contracts, medical notes, or scientific papers, a domain-specific model (Legal-BERT, BioBERT, SciBERT) or a general model fine-tuned on your domain often outperforms off-the-shelf general models by 10-20% on intrinsic metrics and up to 30% on retrieval metrics.

MTEB leaderboard: the HuggingFace MTEB leaderboard ranks embedding models across 56 datasets and 7 task types. For choosing an off-the-shelf model: sort by your task type (retrieval, clustering, classification). Consider: model size (tradeoff between quality and inference speed), dimension (higher dimension generally = better quality but higher storage/compute), license (commercial use?). Current top general models: bge-large-en-v1.5, e5-mistral-7b, text-embedding-3-large.

# Extrinsic evaluation: retrieval recall@k
def evaluate_retrieval(model_name, queries, corpus, relevant_doc_ids, k=10):
    from sentence_transformers import SentenceTransformer
    import numpy as np
    from sklearn.metrics.pairwise import cosine_similarity
    model = SentenceTransformer(model_name)
    corpus_embeddings = model.encode(corpus, batch_size=64)
    query_embeddings = model.encode(queries, batch_size=64)
    scores = cosine_similarity(query_embeddings, corpus_embeddings)
    recalls = []
    for i, rel_ids in enumerate(relevant_doc_ids):
        top_k = np.argsort(scores[i])[::-1][:k]
        recall = len(set(top_k) & set(rel_ids)) / len(rel_ids)
        recalls.append(recall)
    return np.mean(recalls)
print(evaluate_retrieval('all-MiniLM-L6-v2', queries, corpus, relevant_ids, k=10))

Why this matters in AI
Choosing the wrong embedding model can silently degrade RAG performance by 20-40%. A RAG system that retrieves 60% of relevant documents (recall@5 = 0.60) will generate answers with missing context. Without evaluation, you don't know whether switching from all-MiniLM-L6-v2 to bge-large-en-v1.5 would improve recall to 0.75. Evaluation gives you the evidence to make these decisions.
Common mistakes
  • Choosing an embedding model based on MTEB overall rank without checking task-specific performance — the top-ranked model on MTEB overall may not be the best for retrieval. Sort MTEB by 'Retrieval' task type when building a RAG system.
  • Not evaluating on domain-specific data — a model that ranks highly on MTEB general benchmarks may rank poorly on your specific domain (medical, legal, code). Build a domain-specific evaluation set (100 query-document pairs) and measure retrieval recall before choosing a model.
  • Ignoring inference speed — the largest and best embedding models (7B parameter models like e5-mistral) are 10-50x slower than small models (all-MiniLM-L6-v2). For a production search system with low latency requirements, a slightly worse but much faster model may be the right choice.
