Function calling

Detailed explanation

Function calling (also called tool use) allows LLMs to call external functions when generating a response. Instead of generating text, the model outputs a structured call with a function name and arguments: {'name': 'search_web', 'arguments': {'query': 'current weather in Paris'}}. The application executes the function and returns the result to the model, which then incorporates it into the final response. This turns LLMs from knowledge-retrieval systems into action-taking agents.

How function calling works in OpenAI/Claude API: (1) define tool schemas (function name, description, parameters with JSON Schema), (2) send the schema along with the user message, (3) model returns either a normal response OR a tool_use block with function name and arguments, (4) application executes the function, (5) send the result back to the model as a tool result, (6) model generates the final response using the tool result. This round-trip can repeat multiple times (multi-step tool use).

Tool schema design: the quality of the function description critically affects reliability. The description must explain WHEN to use the function (not just what it does). Parameter descriptions must be precise. Example: a poorly described 'search' function vs 'search_product_catalog(query: str) — use to find products matching the user's criteria. Returns a list of up to 10 matching products with prices and availability.' The second description guides the model to use the tool appropriately.

Parallel tool calls: modern APIs support parallel function calls — the model can call multiple tools simultaneously. Example: a user asks 'What is the weather in Paris and the current AAPL stock price?' The model calls weather_tool and stock_tool in parallel, waits for both results, then combines them in the response. This significantly speeds up multi-tool agents.

Tool use in Claude API (Anthropic): define tools as a list of dicts with 'name', 'description', 'input_schema'. Claude returns a response with stop_reason='tool_use' and a content block with type='tool_use', name, and input. The application runs the tool and sends back a message with type='tool_result'. Claude continues from this context to produce the final response.

from openai import OpenAI

    }, 'required': ['city']}

    tools=tools, tool_choice='auto')

    args = json.loads(response.choices[0].message.tool_calls[0].function.arguments)
    result = get_weather(args['city'])  # execute the actual function

Why this matters in AI
Function calling is what transforms LLMs from static knowledge sources into agents that can take actions in the world — search the web, execute code, query databases, call APIs, send emails. Understanding function calling is essential for building AI agents, chatbots with live data, and any LLM-powered application that needs access to external systems.
Common mistakes
  • Writing vague tool descriptions — the model decides WHEN to call a tool based on the description. 'Searches something' is not actionable. 'search_web(query) — use when you need current information not in your training data, such as today's news, current prices, or live events' gives the model a clear trigger.
  • Not handling the tool_use stop reason — if the model returns stop_reason='tool_use', the application must execute the tool and send back the result. Ignoring this leaves the model hanging with an incomplete response.
  • Returning unformatted tool results — the tool result should be a clean, structured string or JSON. Returning raw API responses (thousands of tokens of noise) wastes context and confuses the model.

Tool schema

Detailed explanation

A tool schema is the formal description of a function that an AI agent can call. It tells the LLM what the tool does, what parameters it takes, which parameters are required vs optional, and what data types are expected. Without a clear schema, the LLM cannot correctly format its tool calls — it may pass wrong types, miss required parameters, or call the wrong tool.

OpenAI function calling format (also used by Anthropic): a JSON Schema object with 'name', 'description', and 'parameters'. The parameters field uses JSON Schema to describe the input: type, properties, required, enum, description for each field. Example: a web_search tool has parameters: {'query': {'type': 'string', 'description': 'The search query'}, 'max_results': {'type': 'integer', 'default': 5}}. The LLM uses this schema to generate a valid JSON call.

Anthropic tool use: tools are passed in the API request as a list. The model responds with a tool_use content block containing: 'id' (unique call ID), 'name' (tool name), 'input' (dict of arguments). You execute the function and return the result in a 'tool_result' content block with the matching 'tool_use_id'. Multiple tools can be called in parallel — the model may return multiple tool_use blocks in one response.

Description quality matters: the tool description is the model's only guide for deciding WHEN to use the tool. Vague description: 'fetches data' — when would you use this? Clear description: 'Retrieves current weather data for a specific city. Use when the user asks about current weather, temperature, or forecast for any location.' The model uses the description to reason about whether this tool is appropriate for the current query.

Parameter constraints with JSON Schema: use 'enum' for categorical parameters: 'unit': {'type': 'string', 'enum': ['celsius', 'fahrenheit']}. Use 'minimum'/'maximum' for numeric ranges. Use 'pattern' for string format (e.g., 'pattern': '^\d{4}-\d{2}-\d{2}$' for ISO date). These constraints cause the model to generate valid inputs without needing additional validation.

# Full agentic loop: model calls tool, you return result, model generates final answer
def agent_loop(user_message):
    messages = [{'role': 'user', 'content': user_message}]
    while True:
        resp = client.messages.create(model='claude-opus-4-8', max_tokens=2048, tools=tools, messages=messages)
        if resp.stop_reason == 'end_turn': return resp.content[-1].text
        if resp.stop_reason == 'tool_use':
            messages.append({'role': 'assistant', 'content': resp.content})
            tool_results = []
            for block in resp.content:
                if block.type == 'tool_use':
                    result = execute_tool(block.name, block.input)
                    tool_results.append({'type': 'tool_result', 'tool_use_id': block.id, 'content': str(result)})
            messages.append({'role': 'user', 'content': tool_results})
Why this matters in AI
Tool schemas are the API contract between your application and the AI model. A well-designed tool schema makes the agent reliable: it calls the right tools with valid parameters. A poorly designed schema causes the agent to misuse tools, hallucinate parameters, or fail to call tools when needed. Mastering tool schema design is essential for building robust AI agents.
Common mistakes
  • Writing vague tool descriptions — 'a useful tool for getting information' does not tell the model when to use it. Write actionable descriptions: 'Use this to retrieve the current stock price for a ticker symbol when the user asks about stock prices or company valuations.'
  • Not using JSON Schema constraints — without 'enum' and type constraints, the model may pass 'Celsius' instead of 'celsius' (case mismatch), or pass a string where an integer is expected. Use schema constraints to enforce valid inputs.
  • Not handling tool errors — if the tool call fails (network error, invalid input), return the error in the tool_result content. The model will then retry with corrected parameters or explain the error to the user.

Tool validation

Detailed explanation

Tool validation in LLM agent systems means checking that the model's tool call inputs are valid before executing the tool. When an LLM generates a JSON function call, it may include: wrong types (string instead of int), missing required fields, out-of-range values, or completely invalid structure. Executing a tool with invalid inputs causes cryptic errors or silent failures.

JSON schema validation: tool definitions use JSON Schema to specify expected parameters. Before executing a tool call, validate the model's output against the schema using `jsonschema.validate()`. This catches type mismatches, missing required fields, and format violations before they reach your tool code.

Pydantic for runtime validation: Pydantic's data models are the standard way to validate tool call arguments in Python. Define a Pydantic model matching your tool's expected arguments, parse the model's output using it, and Pydantic will raise ValidationError for any type mismatch or missing field with a clear error message.

Retry on validation failure: when a tool call fails validation, send the error message back to the model and ask it to fix the input. This retry loop typically resolves 80-90% of tool call failures on the first retry. Include the validation error in the context: 'Your previous tool call failed with: field X is required. Please provide a valid value for X.'

Graceful degradation: not all tool call failures should crash the agent. Categorize failures: (1) validation failures (wrong format) — retry with error feedback. (2) execution failures (API error, timeout) — retry with backoff. (3) logical failures (tool returned empty results) — fall back to a different tool or acknowledge the limitation to the user.

Why this matters in AI
Tool validation is the difference between a brittle demo agent and a production agent. In a demo, you always test with well-formed inputs and the tool always succeeds. In production, users ask questions in unexpected ways, the model generates slightly malformed tool calls, and external APIs return errors. Tool validation with retry turns these failures into recoverable events instead of agent crashes.
Common mistakes
  • Executing tool calls without validation — trusting the LLM to always generate valid JSON with correct types is too optimistic. Models make mistakes. Always validate before executing, especially for tools that have side effects (send email, execute code, modify database).
  • Not including the error in the retry message — sending back 'tool call failed, please retry' is unhelpful. Include the specific validation error: 'field max_results must be between 1 and 20, you provided 100.' The model needs the specific error to generate a correct retry.
  • Infinite retry loops — without a maximum retry count, a tool call that always fails (because of a misunderstood tool schema) will loop forever. Cap retries at 3 and fail gracefully after that.

Agents

Detailed explanation

An AI agent is a system where an LLM controls a loop: it observes the current state, decides what action to take (which tool to call, what query to issue, what to write), takes the action, observes the result, and iterates until the task is complete. Unlike a single LLM call, an agent can take multiple steps, use multiple tools, and adapt its strategy based on intermediate results. Agents enable open-ended problem solving.

The ReAct (Reasoning and Acting) framework is the most common agent pattern. At each step, the LLM outputs: Thought (I need to find the current population of France), Action (search Wikipedia for 'France population'), Observation (Wikipedia says 67 million as of 2024), Thought (now I have the information I need), Final Answer (France has approximately 67 million people). The alternating Thought-Action-Observation loop enables the agent to reason about multi-step problems.

Tool calling is the mechanism that gives agents capabilities beyond text generation. A tool is a function with a name, description, and parameter schema. The LLM sees the tool descriptions and selects which tool to call and with what parameters. Common tools: web search, code interpreter (Python REPL), calculator, calendar, email, database query, API calls. Tool validation ensures the model's requested parameter values are type-checked and safe before execution.

Agent reliability challenges: hallucinating tool parameters (the LLM calls a tool with plausible but wrong arguments), infinite loops (the agent keeps taking actions without making progress), context exhaustion (long multi-step tasks exhaust the context window), and error propagation (one wrong tool call produces wrong information that propagates through subsequent reasoning). Mitigations: output parsing with retry, step limits, structured intermediate state, and human-in-the-loop checkpoints.

Multi-agent systems use multiple specialized agents collaborating: a Planner agent decomposes a complex task into subtasks, Executor agents handle each subtask (one for web search, one for code writing, one for data analysis), and a Critic agent reviews and validates the combined output. LangGraph, CrewAI, and AutoGen are frameworks for building multi-agent systems. This architecture enables parallelism and specialization for complex workflows.

Planner executor

Detailed explanation

The planner-executor pattern separates an agent's reasoning (what to do) from its execution (how to do it). A planner LLM generates a structured plan (a list of steps with tool calls), and an executor LLM or deterministic function executes each step in order. This separation improves reliability: the planner can reason globally about the task; the executor focuses on one step at a time.

When to use planner-executor: tasks that require multi-step reasoning with dependencies between steps. 'Research this company, analyze their financials, write a report' requires: (1) search for basic info, (2) find financial data, (3) synthesize a report using both. A single-pass agent would try to do all three simultaneously and often miss dependencies. A planner generates the explicit three-step plan first.

Plan structure: a plan is typically a list of subtasks, each with: step number, description, required tools, dependencies (which prior steps must complete first), and expected output. In LangGraph, each step in the plan becomes a node in the graph, with edges representing dependencies.

Plan validation before execution: before starting execution, the planner's plan should be validated. Check: are all referenced tools available? Are dependency relationships acyclic (no circular dependencies)? Are step descriptions complete enough for the executor? Invalid plans are caught before any expensive tool calls are made.

Dynamic re-planning: a static plan assumes the world is predictable. When a step fails or returns unexpected results, the agent should re-plan. Dynamic re-planning: after each step, re-evaluate whether the remaining plan is still valid given the new information. If a tool returned empty results ('no financial data found'), revise the plan to search for alternative data sources before proceeding.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import anthropic
import json
client = anthropic.Anthropic()
# Planner: generate a structured plan
def generate_plan(task: str) -> list:
    response = client.messages.create(
        model='claude-opus-4-8',
        max_tokens=1024,
        system='You are a planning agent. Given a task, output a JSON list of steps. Each step has: {step: int, description: str, tool: str, depends_on: list[int]}. Be specific and concrete.',
        messages=[{'role': 'user', 'content': f'Create a detailed plan to: {task}'}]
    )
    text = response.content[0].text
    # Parse JSON from response
    import re
    json_match = re.search(r'\[.*?\]', text, re.DOTALL)
    if json_match: return json.loads(json_match.group())
    return []
plan = generate_plan('Research Python web frameworks and write a comparison report')
for step in plan: print(f'Step {step["step"]}: {step["description"]} (tool: {step.get("tool", "none")})')
Why this matters in AI
The planner-executor pattern is how production AI agents handle complex multi-step tasks reliably. Without explicit planning, an LLM agent tries to reason about all steps simultaneously, loses track of state, and produces inconsistent outputs. By separating 'what to do' (planner) from 'how to do it' (executor), you get more reliable execution, better error handling, and clearer auditability of what the agent did.
Common mistakes
  • Over-specifying the plan — a plan with 15 micro-steps for a simple task is harder to execute correctly than a plan with 4 high-level steps. Plan at the right granularity: each step should be independently executable and verifiable.
  • Not handling plan step failures — the executor must decide: if step 3 fails, skip step 4 (which depends on step 3), report the failure, and try the alternative path if one exists. Without failure handling, a failed step silently propagates incorrect state.
  • Using the planner model for both planning and execution — planning requires global reasoning (understand the full task); execution requires precise action (correct tool call format). Using a less capable model for execution and a more capable model for planning often gives better results at lower cost.

ReAct concept

Detailed explanation

ReAct (Reasoning + Acting) is an agent framework where the LLM alternates between Reasoning (thinking about what to do next) and Acting (calling a tool or taking an action). The model produces explicit intermediate reasoning steps (Thought: ...) before each action (Action: tool_name, args), and observes the result (Observation: tool output) before deciding on the next step. This explicit reasoning trace makes the agent's decisions interpretable and correctable.

ReAct loop: Thought 1 → Action 1 → Observation 1 → Thought 2 → Action 2 → Observation 2 → ... → Final Answer. The model reasons about what it knows, what it needs, which tool to use, and why — before calling the tool. After observing the result, it reasons again about whether it has enough information or needs more. This reasoning-before-acting pattern dramatically reduces tool misuse compared to direct function calling without explicit reasoning.

ReAct vs chain-of-thought vs tool use: chain-of-thought (CoT) enables complex reasoning without tool use — the model thinks step by step but cannot look up external information. Tool use without reasoning produces direct API calls but no interpretable rationale. ReAct combines both: reasoning traces + external tool calls. The key insight: external tools ground reasoning in facts (reduce hallucination), and explicit reasoning makes tool selection smarter.

Implementation: in early ReAct implementations, the model was prompted to format its output as 'Thought: ... Action: [tool name] Input: [json args]'. The framework parsed the Action/Input and executed the tool. Modern LLM APIs (Anthropic, OpenAI) have native tool use with structured JSON calls — they implement ReAct-style reasoning internally without requiring the text-format prompt template.

ReAct in LangChain and LangGraph: LangChain's AgentExecutor implements the ReAct loop. LangGraph provides a more flexible graph-based agentic framework where nodes are processing steps and edges are transitions. LangGraph enables branching (different paths based on tool output), loops (retry until success), and parallel tool execution — going beyond the simple sequential ReAct loop.

# Minimal ReAct from scratch with Anthropic tool use
def react_agent(question, tools, max_steps=10):
    messages = [{'role': 'user', 'content': question}]
    for step in range(max_steps):
        resp = client.messages.create(model='claude-opus-4-8', max_tokens=2048, tools=tools, messages=messages)
        messages.append({'role': 'assistant', 'content': resp.content})
        if resp.stop_reason == 'end_turn':
            return next(b.text for b in resp.content if hasattr(b, 'text'))
        tool_results = []
        for block in resp.content:
            if block.type == 'tool_use':
                result = execute_tool(block.name, block.input)
                tool_results.append({'type': 'tool_result', 'tool_use_id': block.id, 'content': str(result)})
        messages.append({'role': 'user', 'content': tool_results})
    return 'Max steps reached'
Why this matters in AI
ReAct is the foundational paradigm for building agents that can reason about and use tools reliably. The explicit reasoning trace before each action makes agents debuggable — you can see WHY the agent chose to call a specific tool, and identify where the reasoning went wrong when things fail.
Common mistakes
  • Not setting max_iterations — a ReAct agent can get stuck in a reasoning loop, calling the same tool repeatedly. Always set a maximum number of steps (10-20) and return a clear error message when exceeded.
  • Using ReAct for simple single-step tasks — for a question that requires exactly one tool call, the overhead of the reasoning-acting loop adds latency without benefit. Use direct tool use for predictable, structured tasks.
  • Not handling tool errors in the ReAct loop — when a tool fails, the Observation should contain the error message so the model can reason about the failure and try a different approach.

Memory

Detailed explanation

Memory in AI agents refers to the ability to retain and recall information across turns in a conversation (short-term memory) and across separate sessions (long-term memory). Without memory, every conversation starts from scratch — the agent has no context from past interactions. Memory enables personalization, continuity, and the ability to build on prior work.

Short-term memory (within-conversation): the conversation history stored in the messages list. Every new message is appended, and the full history is passed to the LLM on each turn. Limitation: context window limit — at some point, the history must be summarized or truncated. Strategies: summarize old messages (replace old messages with a summary), sliding window (keep only the last N messages), or message compression (compress verbose messages).

Long-term memory (cross-session): information stored persistently in a database and retrieved when relevant. Types: episodic memory (specific past events — 'user mentioned they are vegetarian last Tuesday'), semantic memory (general facts about the user — 'user prefers Python'), procedural memory (how to perform tasks — 'user's workflow for deploying to production'). Implemented with a vector store — retrieve relevant memories based on semantic similarity to the current query.

Memory in LangChain: ConversationBufferMemory (stores full history), ConversationSummaryMemory (summarizes old history with LLM), ConversationSummaryBufferMemory (keeps recent messages verbatim + summarizes older ones), ConversationEntityMemory (extracts entities from conversation: names, preferences, facts), VectorStoreRetrieverMemory (stores messages in vector store, retrieves relevant ones).

Memory update vs retrieval: memory must be explicitly written (save important facts as the conversation progresses) and explicitly read (retrieve relevant facts given the current context). An agent that only retrieves memory but never updates it will never learn. Implement a memory manager that: at conversation end, extracts key facts and stores them, at conversation start, retrieves relevant memories based on the user's opening query.

LangChain

Detailed explanation

LangChain is a framework for building LLM-powered applications. It provides: abstractions for LLM calls (works with OpenAI, Anthropic, local models), memory (store conversation history), chains (compose multiple LLM calls), agents (LLMs that use tools to complete tasks), and document loaders/text splitters/vector stores for RAG. LangChain reduces boilerplate when building complex LLM applications.

Core LangChain concepts: LLMs and Chat Models are the wrappers around model providers. Prompts are templates with input variables (PromptTemplate). Chains are sequences of operations: LLMChain = prompt → LLM → output_parser. Sequential chains combine multiple LLMChains where output of one becomes input of the next. LCEL (LangChain Expression Language) uses pipe syntax: chain = prompt | llm | output_parser — cleaner than explicit chain classes.

Memory in LangChain stores conversation history for multi-turn interactions. ConversationBufferMemory keeps all messages (for short conversations). ConversationSummaryMemory uses an LLM to summarize old messages (for long conversations — stays within context window). ConversationKGMemory extracts knowledge graph entities from conversation history. Memory is attached to a ConversationChain and automatically included in the prompt.

LangChain agents use LLMs as reasoning engines that decide which tools to call and in what order to answer a query. The LLM is given a set of tool descriptions and a task; it iteratively: reasons about what to do next (Thought), calls a tool (Action), observes the result (Observation), and repeats until it has a final answer. Agents enable: web search, calculator, code execution, database queries, and custom tools — combining them dynamically based on the task.

LangGraph (part of the LangChain ecosystem) is a more powerful framework for building stateful multi-agent systems. It represents the agent's workflow as a directed graph of nodes (functions) and edges (control flow). This enables: conditional branching (if the retriever finds nothing, fall back to web search), loops (retry up to 3 times), parallel node execution, and human-in-the-loop interruption points. LangGraph is more explicit and controllable than LangChain agents.

LangGraph

Detailed explanation

LangGraph is a framework for building stateful, multi-step LLM applications as directed graphs where nodes are processing functions (agent steps, tool calls, conditionals) and edges are transitions between steps. Unlike LangChain's linear chains, LangGraph supports cycles (loops for iterative refinement), branching (conditional paths based on output), parallel execution (multiple tools at once), and human-in-the-loop (pause and wait for human input).

Core concepts: State — a typed dict (TypedDict) that flows through the graph, accumulating information at each step. Nodes — Python functions that take State and return updated State. Edges — static (A always goes to B) or conditional (go to B or C based on state content). StateGraph — the graph object where you add_node() and add_edge(). Compiled graph — calling graph.compile() produces a runnable that can be invoked, streamed, or run asynchronously.

The supervisor pattern: a meta-agent (supervisor) decides which specialized sub-agent to call next based on the current state. The supervisor routes to: 'research_agent', 'calculation_agent', or 'writer_agent' depending on what the task requires. The supervisor sees the accumulated state (what has been done) and decides the next step. This enables complex multi-agent orchestration with explicit routing logic.

Human-in-the-loop: LangGraph supports interrupt_before and interrupt_after for human review. compile(checkpointer=MemorySaver(), interrupt_before=['sensitive_action_node']) pauses the graph before executing that node. The human reviews the pending action (e.g., 'agent is about to send an email'), approves or modifies, and resumes. Essential for autonomous agents with real-world consequences.

Streaming: LangGraph's compiled graph supports async streaming — you can stream intermediate tokens and state updates to the user in real-time. graph.astream_events() yields events as they happen: model tokens, tool call starts, tool results, state updates. This enables responsive UX where users see the agent's reasoning and tool results as they occur.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator
class AgentState(TypedDict):
    messages: Annotated[list, operator.add]  # append-only message history
    next: str  # next node to route to
# Node functions
def call_model(state):
    response = llm.invoke(state['messages'])
    return {'messages': [response]}
def call_tool(state):
    last_message = state['messages'][-1]
    tool_result = execute_tool(last_message.tool_calls[0])
    return {'messages': [ToolMessage(content=str(tool_result), tool_call_id=last_message.tool_calls[0]['id'])]}
def should_continue(state):
    last = state['messages'][-1]
    if last.tool_calls: return 'action'
    return END
# Build graph
graph = StateGraph(AgentState)
graph.add_node('agent', call_model)
graph.add_node('action', call_tool)
graph.set_entry_point('agent')
graph.add_conditional_edges('agent', should_continue, {'action': 'action', END: END})
graph.add_edge('action', 'agent')  # after tool, go back to agent
app = graph.compile()
# Invoke with streaming
import asyncio
async def stream_agent(question):
    async for event in app.astream_events({'messages': [HumanMessage(content=question)]}, version='v1'):
        if event['event'] == 'on_chat_model_stream':
            print(event['data']['chunk'].content, end='', flush=True)
asyncio.run(stream_agent('Research the latest AI developments in 2024'))
Why this matters in AI
LangGraph addresses the limitations of linear agent frameworks: real-world agent tasks require loops (retry until correct), branching (different tools for different situations), and human oversight (approval before consequential actions). LangGraph's graph-based approach makes complex agent behavior explicit, debuggable, and reliable.
Common mistakes
  • Not defining State with proper annotations — use Annotated[list, operator.add] for message lists to enable append-only semantics. Without annotations, each node's output overwrites the state rather than merging.
  • Creating cycles without a termination condition — add a counter to the state and check MAX_ITERATIONS to prevent infinite loops. LangGraph will recurse indefinitely if there is no path to END.
  • Not using checkpointers for long-running agents — without a checkpointer (MemorySaver or SqliteSaver), the state is lost if the process crashes mid-execution. Add a checkpointer for any agent that runs more than a few seconds.

Fallbacks

Detailed explanation

Fallbacks are alternative paths taken when the primary approach fails — a model call times out, returns an error, produces an invalid response, or the confidence is too low. Robust production systems always have fallbacks; brittle systems fail completely when any component fails.

Model fallback chain: if the primary model (e.g., Claude Opus) is unavailable or times out, fall back to a faster/cheaper model (Claude Haiku). If that also fails, fall back to a cached response, a rules-based response, or a graceful 'I cannot answer this right now' message. Each level of fallback is less capable but more reliable.

Tool fallback: if the primary search tool returns empty results, try an alternative search strategy (broader query, different data source). If OCR fails to parse a document, try a different OCR engine. If a translation API fails, try a different translation service or skip translation. The key: always have at least one backup path that cannot fail (even if the backup is 'return None and log the failure').

Confidence-based fallback: use model confidence (prediction probability) to route requests. High confidence (>0.9): auto-approve the response. Medium confidence (0.5-0.9): add a disclaimer. Low confidence (<0.5): fallback to a human reviewer or a more capable model. This approach maximizes automation for confident cases while ensuring quality for uncertain cases.

LangChain fallback chains: LangChain provides `.with_fallbacks([fallback_chain])` syntax for automatic fallback. If the primary chain raises an exception, the fallback chain is tried. You can chain multiple fallbacks: primary_model → backup_model → rules_based_responder → 'service unavailable' message.

import anthropic
import time
# Model fallback chain
client = anthropic.Anthropic()
def call_with_fallback(messages, primary='claude-opus-4-8', fallback='claude-haiku-4-5-20251001', timeout=10):
    for model in [primary, fallback]:
        try:
            response = client.messages.create(
                model=model, max_tokens=1024, messages=messages
            )
            return response.content[0].text
        except anthropic.APITimeoutError:
            print(f'Timeout on {model}, trying fallback...')
        except anthropic.APIStatusError as e:
            if e.status_code == 529:  # API overloaded
                print(f'{model} overloaded, trying fallback...')
                time.sleep(1)
            else:
                raise
    return 'I am temporarily unable to process your request. Please try again shortly.'
# LangChain fallback pattern
from langchain_anthropic import ChatAnthropic
from langchain_core.output_parsers import StrOutputParser
primary_llm = ChatAnthropic(model='claude-opus-4-8')
fallback_llm = ChatAnthropic(model='claude-haiku-4-5-20251001')

Why this matters in AI
Production AI systems fail without fallbacks. LLM API outages happen. Search APIs go down. OCR fails on corrupted images. A system without fallbacks shows error pages to users when any component fails. A system with fallbacks degrades gracefully: primary model → backup model → rules-based → 'try again later'. Fallbacks are the implementation of reliability at the application layer.
Common mistakes
  • Only having one fallback level — if both the primary and the first fallback fail (concurrent API outage), the system crashes. Always have at least 3 levels: primary, backup, and a hardcoded default response.
  • Not logging fallback activations — fallbacks that activate silently hide production failures. When a fallback is used, log: why (timeout/error/confidence), which level was used, and the request that triggered it. Use this log to identify reliability issues in the primary system.
  • Using the same provider for primary and fallback — if your primary and fallback both use OpenAI, an OpenAI outage takes both down simultaneously. Diversify across providers: Anthropic primary, OpenAI fallback (or vice versa).

Tracing and debugging

Detailed explanation

Tracing in LLM applications means recording every step in a chain or agent run: the input prompt, the model called, the tokens used, the response, tool calls made, and the final output. Without tracing, debugging an agent that calls 5 tools across 10 LLM calls is nearly impossible — you see only the final (wrong) output but not which step failed.

LangSmith for tracing: LangSmith (by LangChain) is the most widely used tracing tool for LLM applications. Set environment variables: `LANGCHAIN_TRACING_V2=true`, `LANGCHAIN_API_KEY=...`. All LangChain calls are automatically traced. The LangSmith UI shows: full prompt → model response timeline, token counts, latency per step, child spans for tool calls.

What to trace: each LLM call (input tokens, output tokens, model name, latency), each tool call (tool name, input, output, success/failure), each retrieval operation (query, retrieved documents, scores), and the final output. For multi-step chains: capture the full trace tree so you can see exactly where the chain went wrong.

Debugging failures: common LLM application failures and how to trace them: (1) model hallucination — check if the retrieved context actually contained the answer (retrieval failure) or if the model ignored the context (generation failure). (2) tool call failure — check the exact tool input the model generated (often a JSON format issue). (3) infinite loops — check the agent's decision trace: it kept calling the same tool because a previous tool returned an error it didn't handle.

Production monitoring: tracing should continue in production, not just during development. Log all traces, monitor: latency percentiles (p50, p90, p99), error rates, token cost per request, number of LLM calls per user request. Alert on: p99 latency > threshold, error rate > 1%, cost anomalies.

import time
from dataclasses import dataclass, field
from typing import List, Any
@dataclass
class Span:
    name: str; start: float = field(default_factory=time.time)
    end: float = None; inputs: dict = field(default_factory=dict)
    outputs: dict = field(default_factory=dict); children: List = field(default_factory=list)
class Tracer:
    def __init__(self): self.spans = []
    def start_span(self, name, inputs=None):
        span = Span(name=name, inputs=inputs or {}); self.spans.append(span); return span
    def end_span(self, span, outputs=None):
        span.end = time.time(); span.outputs = outputs or {}
    def print_trace(self):
        for span in self.spans:
            duration = (span.end or time.time()) - span.start
            print(f'{span.name}: {duration*1000:.0f}ms | inputs: {span.inputs} | outputs: {span.outputs}')
tracer = Tracer()
Why this matters in AI
Debugging an LLM application without tracing is like debugging a distributed system without logs. The chain might call 5 tools, make 8 LLM requests, and produce a wrong answer — without tracing, you have no idea which step was responsible. Tracing reduces debugging time from hours to minutes by showing exactly what happened at each step.
Common mistakes
  • Only tracing in development — production failures often occur on edge-case inputs that never appeared in testing. Keeping tracing on in production (with appropriate data retention policies) means you can always debug any production issue.
  • Not tracing tool inputs and outputs — the most common agent failures are incorrect tool inputs (model generates malformed JSON) or unexpected tool outputs (tool returns an error message the model doesn't handle). Always log both the raw tool input and the raw tool output.
  • Not setting a project name in LangSmith — without a project name, all traces go to a default project and become impossible to organize as your application grows. Use meaningful project names like 'rag-customer-support-v2'.

Human-in-the-loop

Detailed explanation

Human-in-the-loop (HITL) is a system design pattern where a human reviewer is embedded in the automated workflow for high-stakes or uncertain decisions. Instead of full automation, the system runs autonomously for routine cases and escalates to a human for cases that are uncertain, novel, or high-risk. This combines the speed of automation with the judgment of human expertise.

When HITL is necessary: medical diagnosis (AI suggests, doctor confirms), legal document review (AI flags issues, lawyer reviews), financial fraud decisions (AI scores risk, analyst confirms before blocking), content moderation (AI classifier, human reviewer for borderline cases), AI agent actions that are irreversible (send an email, execute a trade, delete data).

HITL in LLM agents: use LangGraph's interrupt mechanism to pause agent execution before critical actions. The agent plans an action (e.g., 'send email to customer'), raises an interrupt, a human reviews the planned action, approves or modifies it, and the agent continues. This prevents autonomous agents from taking irreversible wrong actions.

Active learning: HITL is not just for deployment — it's also for training data creation. Active learning uses model uncertainty to identify which examples would be most valuable for a human to label. Instead of labeling randomly, a human labels the examples the model is most uncertain about — dramatically reducing the number of labels needed to achieve target performance.

Confidence-based routing: a practical HITL implementation — route requests to human review when model confidence falls below a threshold. For a classification model: if max(softmax_output) < 0.7, escalate to human. Tune the threshold to balance automation rate (cost) vs accuracy (quality). Track: automation rate, human review queue depth, and error rate on auto-approved vs human-reviewed cases.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from langgraph.graph import StateGraph, END
from langgraph.checkpoint.memory import MemorySaver
# LangGraph with human-in-the-loop interrupt
def should_interrupt(state):
    'Return True if action requires human approval'
    action = state.get('planned_action', {})
    return action.get('type') in ['send_email', 'delete_record', 'execute_payment']
def plan_action(state): return {'planned_action': {'type': 'send_email', 'to': 'user@example.com', 'body': 'Your order shipped.'}}
def execute_action(state): return {'result': f'Executed: {state["planned_action"]}'}
# Build graph with interrupt
builder = StateGraph(dict)
builder.add_node('plan', plan_action)
builder.add_node('execute', execute_action)
builder.add_edge('plan', 'execute')
graph = builder.compile(checkpointer=MemorySaver(), interrupt_before=['execute'])
# Run: will pause before execute
config = {'configurable': {'thread_id': '1'}}
result = graph.invoke({'query': 'Notify customer'}, config)
print('Paused. Planned action:', result['planned_action'])
# Human reviews and approves by resuming
result = graph.invoke(None, config)  # resume from checkpoint
print('Result:', result['result'])
Why this matters in AI
HITL is the responsible deployment pattern for AI systems in high-stakes domains. A fully automated loan approval system that is 95% accurate still makes mistakes on 5% of cases — with HITL, those uncertain cases get human review, the 95% routine cases are automated, and the system is both faster (for routine cases) and safer (for edge cases) than either pure automation or pure human review.
Common mistakes
  • Setting confidence thresholds without measuring impact — 'escalate if confidence < 0.7' is arbitrary. Measure: at your chosen threshold, what fraction of errors does human review catch? What fraction of cases are escalated? Tune the threshold to hit your target quality and automation rate.
  • Not logging which cases were escalated and the human decision — HITL generates valuable data. The cases that required human review are the hardest cases for the model. Using human decisions on these cases for retraining continuously improves the automation rate over time.
  • Creating brittle HITL queues — if humans can't keep up with the volume of escalated cases (queue grows faster than it's drained), the system degrades. Design HITL workflows with queue monitoring, SLA tracking, and overflow handling.
