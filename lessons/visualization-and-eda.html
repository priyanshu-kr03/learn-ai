EDA

Detailed explanation

Exploratory Data Analysis (EDA) is the process of investigating a dataset before modeling — understanding its structure, distributions, missing values, outliers, and relationships between variables. EDA prevents costly mistakes: building a model on corrupted data, making wrong assumptions about distributions, missing obvious feature engineering opportunities, or misunderstanding the target variable.

The EDA checklist: (1) shape and dtypes — how many rows, columns, what type is each feature? (2) missing values — what fraction of each column is null? what pattern do missings follow? (3) target distribution — is it balanced? skewed? (4) numerical feature distributions — histograms, boxplots to spot outliers and skewness (5) categorical feature distributions — bar charts, value counts (6) correlations — heatmap to find strong feature-feature and feature-target relationships (7) outlier analysis — identify extreme values that may be data errors.

Pandas profiling tools automate EDA. ydata-profiling (formerly pandas-profiling) generates an HTML report with all EDA outputs in one command: from ydata_profiling import ProfileReport; ProfileReport(df).to_file('eda.html'). This report includes: alerts for high cardinality, high missing rate, high correlation pairs, and potential outliers — a comprehensive first look in seconds.

Bivariate analysis reveals relationships between pairs of variables. Scatter plots show the relationship between two continuous variables. Box plots compare the distribution of a continuous variable across categories (e.g., income by education level). Pivot tables compute aggregates (mean, count) of one variable split by another. Correlation heatmaps show linear relationships between all continuous variables simultaneously.

EDA insights drive feature engineering. A highly skewed numerical feature (e.g., income, price) performs better after log transformation in linear models. A categorical feature with 200 unique values needs target encoding or embedding rather than one-hot encoding. Two features with correlation > 0.95 are nearly redundant — consider dropping one. An obvious linear relationship between a feature and target suggests it will be highly predictive with minimal engineering.

Why this matters in AI
EDA is the most important step in any ML project. Models trained without EDA often fail because of unnoticed data problems: wrong data types, severe class imbalance, data leakage features, target definition errors. An hour of EDA prevents days of debugging.
Common mistakes
  • Skipping EDA and jumping straight to modeling — this leads to models that memorize data artifacts rather than real patterns.
  • EDA on the test set — only perform EDA on training data. Looking at test distributions influences feature engineering decisions, leaking information.
  • Interpreting correlation as causation — a high correlation between two features means they move together, not that one causes the other.

Histogram

Detailed explanation

A histogram visualizes the distribution of a numerical variable by dividing the range of values into bins and counting how many values fall in each bin. The x-axis shows the value range; the y-axis shows frequency (count) or density (probability per unit width). Histograms answer: what is the shape of this distribution? Is it normal, skewed, bimodal, or uniform?

Distribution shapes reveal important properties. A right-skewed distribution (long tail to the right) is common for income, price, and count data — log transformation often normalizes it. A left-skewed distribution (long tail to the left) is less common. A bimodal distribution (two peaks) suggests two different subpopulations are mixed in the data — perhaps different user segments or product categories. A uniform distribution (flat) may indicate a numerical ID column that should be categorical.

Bin width choice dramatically affects the interpretation. Too few bins (e.g., 5) hides the distribution shape. Too many bins (e.g., 1000) creates a noisy, hard-to-read histogram. The Freedman-Diaconis rule: bin width = 2*IQR/n^(1/3) is a principled default. Matplotlib's plt.hist(data, bins='fd') applies it automatically.

Comparing distributions: overlay histograms with alpha < 1 to compare two groups. Or use kernel density estimation (KDE) plots — a smooth approximation of the histogram — for cleaner visual comparison. sns.histplot(df, x='age', hue='target', kde=True) shows the age distribution for each target class with a smoothed density curve.

For large datasets, the exact bin counts matter less than the overall shape. For small datasets (n < 100), histograms can be misleading — a few samples in a bin make it look important. Use KDE instead, which smooths over the sparse data.

Why this matters in AI
The histogram is the most fundamental visualization in data analysis. Before choosing a preprocessing method, you must see the distribution. Before assuming normality, you must check the histogram. It takes 30 seconds to make and prevents hours of misguided modeling.
Common mistakes
  • Using the default number of bins (10) without checking if it is appropriate — always try several bin counts.
  • Comparing raw count histograms for groups of different sizes — use density=True (normalize to probability) for fair comparison.
  • Confusing histogram with bar chart — histograms are for continuous data (bins are ranges); bar charts are for categorical data (bars are categories).

Bar chart

Detailed explanation

A bar chart displays the frequency, count, or aggregate value of a categorical variable. The x-axis shows categories; the y-axis shows the value. Bar charts answer: how many samples belong to each class? which product category has the highest revenue? what is the distribution of educational levels in the dataset? They are the go-to visualization for categorical data.

Horizontal bar charts (barh) work better when category names are long — they prevent label overlap. Sort bars by value for easier reading: df['category'].value_counts().sort_values().plot(kind='barh'). Stacked bar charts show composition within groups: plot multiple series with stacked=True to show how each category's total breaks down into sub-categories.

For comparing two groups across categories (e.g., survival rate by class for male vs female), use a grouped bar chart with hue: sns.barplot(data=df, x='pclass', y='survived', hue='sex'). This shows two bars per category, one for each group. The error bars show confidence intervals by default in seaborn — remove them with ci=None for cleaner plots.

Choose between bar chart and pie chart: bar charts are almost always better than pie charts. Humans compare heights much more accurately than angles. Pie charts become unreadable with more than 5–6 categories. The only case for pie charts: showing one or two proportions that sum to 100% for a general audience (e.g., 'our market share is 23%').

In ML reporting, bar charts are used to: visualize class distribution in training data (balanced vs imbalanced), compare model performance across different algorithms (bar chart of F1 scores), show feature importance (horizontal bar chart sorted by importance), and display confusion matrix class-wise metrics.

Why this matters in AI
Bar charts are the most versatile and universally understood chart type for categorical data. In ML, they are used at every stage: EDA (data distribution), model comparison (benchmark results), and explainability (feature importance).
Common mistakes
  • Using 3D bar charts — the added dimension adds visual complexity without information. Use flat 2D bars.
  • Not sorting bars — unsorted bars make comparison difficult. Sort by value for immediate visual ranking.
  • Using pie charts instead of bar charts — angle perception is less accurate than length perception. Prefer bar charts for any comparison task.

Line chart

Detailed explanation

A line chart shows the change of a variable over time or a continuous ordered dimension. In ML, line charts are used for: training/validation loss curves (does the model converge? is it overfitting?), performance metrics over epochs (accuracy, AUC), hyperparameter sensitivity (performance vs learning rate), and time series data visualization.

Creating line charts with matplotlib: plt.plot(x, y, marker='o', linewidth=2, label='Training loss'). Multiple lines on the same plot: call plt.plot() multiple times before plt.show(). plt.legend() shows the labels. Essential elements: plt.xlabel('Epoch'), plt.ylabel('Loss'), plt.title('Training curve'), plt.grid(True, alpha=0.3) for readability. Tight layout: plt.tight_layout() prevents label clipping.

Seaborn line charts: sns.lineplot(data=df, x='epoch', y='loss', hue='split', style='split') creates a line chart from a DataFrame with confidence intervals automatically computed and shaded. hue='split' colors by 'train' vs 'val'. errorbar='sd' or errorbar=('ci', 95) shows standard deviation or confidence interval bands.

Training curve best practices: plot both training and validation metrics on the same chart. Use log scale for loss: plt.yscale('log'). Mark the best validation epoch with a vertical line: plt.axvline(x=best_epoch, color='r', linestyle='--', label='Best model'). Smooth noisy curves with a moving average: pd.Series(losses).rolling(10).mean().

Interactive line charts with Plotly: px.line(df, x='epoch', y=['train_loss', 'val_loss'], title='Training curves') creates an interactive chart. Users can hover to see exact values, zoom in/out, toggle series. Use for exploratory analysis and sharing results. For publication: matplotlib/seaborn (static PDF/PNG). For web dashboards: Plotly/Bokeh (interactive HTML).

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import matplotlib.pyplot as plt
import numpy as np
epochs = range(1, 101)
train_loss = np.exp(-np.array(epochs) / 20) + np.random.randn(100) * 0.02
val_loss = np.exp(-np.array(epochs) / 15) + np.random.randn(100) * 0.03 + 0.1
plt.figure(figsize=(10, 5))
plt.plot(epochs, train_loss, 'b-', label='Training loss', linewidth=2)
plt.plot(epochs, val_loss, 'r-', label='Validation loss', linewidth=2)
plt.axvline(x=np.argmin(val_loss)+1, color='g', linestyle='--', label='Best epoch')
plt.xlabel('Epoch'); plt.ylabel('Loss'); plt.title('Training Curves')
plt.legend(); plt.grid(True, alpha=0.3); plt.tight_layout()
plt.savefig('training_curves.png', dpi=150)
# Smooth noisy training curves
smoothed = pd.Series(train_loss).rolling(window=10, min_periods=1).mean()
plt.plot(epochs, train_loss, alpha=0.3, color='blue')
plt.plot(epochs, smoothed, color='blue', linewidth=2, label='Smoothed train loss')
Why this matters in AI
Training loss curves are the primary diagnostic tool during model training. They reveal: convergence (is loss decreasing?), overfitting (validation loss increasing while training loss decreasing), learning rate issues (loss oscillates = too high, barely decreases = too low), and learning rate scheduling effects. Every training run should be accompanied by loss curves.
Common mistakes
  • Not using plt.tight_layout() — labels often overlap or are clipped. Always call plt.tight_layout() before plt.savefig() or plt.show().
  • Using the same color for multiple lines without labels — different lines must be distinguishable by color AND/OR style (dashed, dotted) for accessibility (colorblind users).
  • Not labeling axes — a chart with unlabeled axes is meaningless. Always add xlabel, ylabel, and a descriptive title. Include units in axis labels: 'Time (seconds)', 'Loss (cross-entropy)'.

Scatter plot

Detailed explanation

A scatter plot displays the relationship between two continuous variables. Each data point is plotted at its (x, y) coordinates. Scatter plots reveal: linear relationships (points form a diagonal line), non-linear relationships (points form a curve), clusters (groups of dense points), outliers (isolated points far from the cloud), and no relationship (random cloud of points).

Adding color (hue) and size encodes additional dimensions. Coloring by the target variable reveals whether two features are jointly predictive of the target. plt.scatter(X[:,0], X[:,1], c=y, cmap='coolwarm') shows which regions of feature space correspond to each class. For continuous targets, a color gradient shows how the target varies across feature space.

Pair plot (seaborn's pairplot) creates a grid of scatter plots for every pair of features simultaneously. sns.pairplot(df, hue='target', vars=['age','income','score']) shows a 3×3 grid. The diagonal shows each variable's histogram (within-group). This is the most comprehensive EDA visualization for continuous features — it reveals correlations, distributions, and class separability in one plot.

Regression line overlay: sns.regplot(x='x', y='y', data=df) overlays a fitted regression line with confidence interval on the scatter plot. This visualizes the linear relationship and its uncertainty. For non-linear relationships, use sns.regplot(lowess=True) for a locally-weighted smoother.

Handling overlapping points (overplotting): when many points overlap, the scatter plot shows density of the cloud incorrectly. Solutions: alpha parameter for transparency (plt.scatter(x, y, alpha=0.3)), jitter (add small random noise to discrete variables: sns.stripplot), hexbin plot (2D histogram as hexagons, plt.hexbin(x, y, gridsize=30, cmap='Blues')), and kernel density estimation (sns.kdeplot with fill=True).

plt.colorbar(label='Count')
Why this matters in AI
Scatter plots are the primary tool for understanding relationships between continuous variables. Before building any regression or correlation-based model, visualize the data — a scatter plot may reveal that the relationship is non-linear, or that there are two clusters that need different models.
Common mistakes
  • Not using alpha for dense scatter plots — overplotting makes the chart meaningless. Always use alpha < 1 for datasets with more than a few hundred points.
  • Using scatter plots for categorical vs continuous — use a strip plot, violin plot, or box plot instead.
  • Not plotting both feature space and prediction space — in classification, plot the decision boundary overlaid on the scatter plot to understand where the model makes mistakes.

Box plot

Detailed explanation

A box plot (box-and-whisker plot) summarizes the distribution of a continuous variable using five statistics: minimum whisker, Q1 (25th percentile), median (Q2, 50th percentile), Q3 (75th percentile), and maximum whisker. The box shows the interquartile range (IQR = Q3-Q1). Whiskers typically extend to Q1 - 1.5*IQR and Q3 + 1.5*IQR. Points beyond the whiskers are plotted individually as outliers.

Box plots are excellent for comparing distributions across groups. sns.boxplot(data=df, x='category', y='revenue') shows the revenue distribution for each category side-by-side. You can immediately see: median level, spread (IQR), outlier presence, and skewness (is the median closer to Q1 or Q3?) for each group. This comparison is impossible to make from bar charts of means alone.

Violin plots (sns.violinplot) combine a box plot with a kernel density estimate — you see both the summary statistics and the full distribution shape. This is more informative than a box plot when distributions are multimodal (two peaks). However, violin plots require more data to be interpretable — use them for n > 100 per group.

Categorical + continuous analysis with box plots: 'how does age distribute differently for churned vs non-churned customers?' sns.boxplot(x='churned', y='age', data=df). If the two boxes are clearly separated (medians far apart, boxes barely overlapping), age is a potentially useful feature. If the boxes completely overlap, age may not discriminate between classes.

When to use box plot vs histogram: for comparing multiple groups, box plots are more compact — you can show 10 groups in one chart. For one group, histograms show more distributional detail. For small samples (n < 20), scatter/strip plots show individual points better than box plots (which can be misleading with few data points).

Why this matters in AI
Box plots reveal outliers, spread, and skewness at a glance — information that bar charts (means only) or single statistics hide. In ML, comparing feature distributions across classes with box plots is a fast way to identify discriminative features and detect data quality issues.
Common mistakes
  • Using box plots with very few data points — a box plot from 5 data points is meaningless. Use strip plots or just show the raw points.
  • Not showing the raw data alongside box plots for small-to-medium datasets — overlay a jitter/strip plot: sns.stripplot(x='class', y='age', data=df, alpha=0.3, jitter=True, ax=ax).
  • Ignoring the whisker definition — the whiskers are NOT the min/max by default; they extend to Q1-1.5*IQR and Q3+1.5*IQR. Points beyond are outliers.

Correlation heatmap

Detailed explanation

A correlation heatmap visualizes the pairwise Pearson correlation coefficients between all numerical features simultaneously. Pearson correlation measures linear relationships: +1 is perfect positive linear correlation, -1 is perfect negative linear correlation, 0 is no linear relationship. The heatmap uses color intensity to encode the correlation value — red for positive, blue for negative, white for near-zero.

Interpreting the heatmap: features with high absolute correlation with the target (the 'target' row) are potentially strong predictors. Features with high correlation with each other are redundant — keeping both adds noise and causes multicollinearity problems in linear models. A correlation > 0.9 between two features is a strong signal to drop one or combine them.

Limitations: Pearson correlation only captures linear relationships. A U-shaped (quadratic) relationship between two variables would show correlation ≈ 0, even though there is a strong relationship. Always supplement correlation heatmaps with scatter plots to visualize the actual relationship. For non-linear relationships, Spearman rank correlation (non-parametric) or mutual information are better metrics.

Point-biserial correlation measures the linear relationship between a binary categorical variable and a continuous variable. Cramér's V measures the association between two categorical variables. For a complete correlation analysis including categoricals, compute Pearson for continuous-continuous pairs, point-biserial for continuous-binary pairs, and Cramér's V for categorical-categorical pairs.

Handling multicollinearity: in linear regression, two highly correlated features create unstable coefficient estimates (large variance). Options: (1) drop one feature from each highly correlated pair, (2) apply PCA to reduce correlated features to orthogonal components, (3) use regularization (Ridge or Lasso) which handles multicollinearity naturally. Tree-based models (Random Forest, XGBoost) are naturally robust to correlated features.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

corr = df.select_dtypes('number').corr()
fig, ax = plt.subplots(figsize=(12,10))
sns.heatmap(corr, annot=True, fmt='.2f', cmap='RdYlBu_r', center=0, vmin=-1, vmax=1, ax=ax)
ax.set_title('Feature Correlation Matrix')
Highlight high correlations: high_corr = corr.where(np.triu(np.ones(corr.shape), k=1).astype(bool))
high_corr_pairs = high_corr.stack().sort_values(ascending=False)
print('Most correlated pairs:', high_corr_pairs[high_corr_pairs.abs() > 0.7])
Spearman (non-parametric): corr_spearman = df.corr(method='spearman')
Correlation with target: target_corr = df.corr()['target'].drop('target').sort_values(key=abs, ascending=False)
print('Top predictors by correlation:', target_corr.head(10))
Why this matters in AI
Correlation heatmaps are the fastest way to identify multicollinearity, redundant features, and strong predictors. Removing highly correlated features simplifies the model, reduces training time, and prevents coefficient instability in linear models.
Common mistakes
  • Dropping features with correlation 0 with the target — correlation only measures linear relationships. A feature with correlation 0 may still have a strong non-linear relationship.
  • Not masking the lower triangle — the correlation matrix is symmetric. Show only the upper triangle to avoid visual clutter.
  • Confusing correlation strength with statistical significance — for small n, even a correlation of 0.5 may not be statistically significant. Compute p-values alongside correlations.

Outlier analysis

Detailed explanation

Outliers are data points that differ significantly from the rest of the data. They can be: real (an exceptionally rich customer, a genuine disease event), data entry errors (age=999, salary=-1000), or measurement errors (a sensor reading of 10,000°C). Handling outliers incorrectly — either ignoring them when they are errors or removing them when they are real — degrades model quality.

Detecting univariate outliers: IQR method — compute Q1 (25th percentile) and Q3 (75th percentile); outliers are values below Q1 - 1.5*IQR or above Q3 + 1.5*IQR. This is robust because IQR is not influenced by the outliers themselves. Z-score method: values more than 3 standard deviations from the mean are outliers. Z-scores are not robust because both mean and std are influenced by outliers — prefer IQR for skewed data.

Multivariate outlier detection identifies points that are unusual in their combination of features (e.g., an employee who is 25 years old with 30 years of experience). Mahalanobis distance accounts for correlations between features and identifies multivariate outliers. Isolation Forest is an ensemble method that isolates outliers by building random trees — outliers are isolated with fewer splits. DBSCAN labels low-density points as outliers (class -1).

How to handle outliers: (1) investigate — is this an error or a genuine value? Contact the data source. (2) Remove if data error — drop the row. (3) Cap/winsorize if extreme but real — replace values above the 99th percentile with the 99th percentile value: df['feature'] = df['feature'].clip(upper=df['feature'].quantile(0.99)). (4) Transform — apply log transform to compress the scale of right-skewed features. (5) Use robust models — tree-based models (Random Forest, XGBoost) are naturally robust to outliers because they split on thresholds, not values.

In production, outlier handling at inference time is critical. If your model was trained on capped features, incoming predictions must apply the same caps. Inputs far outside the training range should trigger a warning or flag (out-of-distribution detection). An unexpected surge in outlier counts in incoming data is often an early signal of data drift.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

IQR outlier detection: Q1 = df['feature'].quantile(0.25)
Q3 = df['feature'].quantile(0.75)
IQR = Q3 - Q1
outliers = df[(df['feature'] < Q1 - 1.5*IQR) | (df['feature'] > Q3 + 1.5*IQR)]
Box plot: sns.boxplot(x=df['salary']); plt.title('Salary distribution with outliers')
Winsorize: lower, upper = df['salary'].quantile([0.01, 0.99])
df['salary_capped'] = df['salary'].clip(lower, upper)
Isolation Forest: from sklearn.ensemble import IsolationForest
iso = IsolationForest(contamination=0.05, random_state=42)
preds = iso.fit_predict(df_numeric)  # -1 = outlier, 1 = inlier
Why this matters in AI
Outlier handling is one of the most impactful preprocessing steps. Linear regression, PCA, KNN, and SVM are all highly sensitive to outliers. A single data entry error (age=999) in a linear regression dataset can severely distort the fitted coefficients.
Common mistakes
  • Removing all outliers without investigation — genuine extreme values (fraud transactions, disease events) may be exactly what the model needs to learn to detect.
  • Using z-score outlier detection on non-normal data — z-scores assume normality. For skewed distributions, use IQR or percentile-based thresholds.
  • Applying the same outlier treatment to training and test data separately — compute thresholds from training data only, then apply to both train and test.

Class imbalance visualization

Detailed explanation

Visualizing class imbalance is the critical first step before choosing any ML technique for imbalanced data. A bar chart of class counts immediately reveals the degree of imbalance: 9,000 legitimate vs 100 fraud transactions is a 90:1 ratio. This visualization drives all subsequent modeling decisions.

Bar chart of class distribution: `df['target'].value_counts().plot(bar)` shows absolute counts. `df['target'].value_counts(normalize=True).plot(bar)` shows proportions. For binary classification: annotate each bar with the count and percentage. For multiclass: sort by count (most to least frequent) to identify the long tail.

Confusion matrix visualization: after training, the confusion matrix shows where the model's errors concentrate. For 90:1 imbalanced data, a naive model that predicts majority class always achieves 98.9% accuracy but has 0% recall on the minority class. The confusion matrix heatmap (seaborn.heatmap) reveals this: it shows the actual count correctly predicted vs missed.

Precision-Recall curve: the standard ROC curve is misleading for extreme imbalance because the false positive rate uses the majority class as denominator. The PR curve uses precision (TP/(TP+FP)) and recall (TP/(TP+FN)) — both computed only on positive class. For imbalanced data, the PR curve is more informative: a model with AUC-PR=0.8 on a 1:100 imbalanced dataset is genuinely good.

Stratified sampling visualization: when you apply SMOTE, undersample, or use class weights, visualize the resulting class distribution to confirm the imbalance was addressed. Also visualize the feature distributions per class — do the two classes have overlapping features (hard problem) or distinct features (easy problem)?

Why this matters in AI
Class imbalance is the leading cause of silent ML failures. A model with 99% accuracy sounds excellent until you realize the dataset is 99% negative class — the model learned to predict 'no' for everything. Visualizing the class distribution before modeling forces you to confront this issue upfront and choose appropriate metrics (precision-recall, not accuracy) and techniques (SMOTE, class weights).
Common mistakes
  • Reporting accuracy on imbalanced data — accuracy is meaningless for imbalanced datasets. Always report: precision, recall, F1-score, and AUC-PR. `sklearn.metrics.classification_report` gives all three per class.
  • Not checking per-class recall — a model can achieve good macro-F1 by having excellent precision on the majority class and poor recall on the minority class. Always report per-class metrics, not just weighted averages.
  • Applying SMOTE before train/test split — SMOTE creates synthetic minority class samples. If applied before splitting, synthetic samples from the test set appear in training — data leakage. Always split first, then apply SMOTE only on the training set.

Data storytelling

Detailed explanation

Data storytelling is the practice of communicating insights from data analysis in a way that is compelling, clear, and actionable for a non-technical audience. Raw charts and numbers are not enough — they need narrative context, clear visual hierarchy, and actionable takeaways. The best data story answers: so what? Why does this matter? What should we do?

The structure of a data story: (1) Context — what question are we answering and why does it matter? (2) Conflict — what's the problem or surprising finding? (3) Resolution — what does the data tell us and what action should be taken? This three-act structure (borrowed from screenwriting) works for data presentations, reports, and dashboards.

Visualization choices: each chart type communicates something specific. Use line charts for trends over time. Bar charts for comparisons between categories. Scatter plots for relationships between two continuous variables. Heatmaps for two-dimensional distributions. Choose the chart that best communicates the insight, not the most impressive-looking one. Avoid pie charts (hard to compare similar-sized slices).

Annotation and titles: every visualization should have: a descriptive title that states the finding ('Revenue grew 40% after feature launch' not 'Revenue over time'), axis labels with units, and annotations highlighting the key insight. The viewer should understand the point of the chart without reading surrounding text.

Reducing cognitive load: effective visualizations minimize the mental effort required to extract the insight. Remove chart junk (gridlines, unnecessary borders, 3D effects). Use color intentionally (one color for the highlighted element, gray for context). Order categories meaningfully (sorted by value, not alphabetically). Keep text minimal — every word must earn its place.

Worked example

import matplotlib.pyplot as plt
import matplotlib.patches as mpatches
import pandas as pd
import numpy as np
# Before: default chart
df = pd.DataFrame({'month': range(1, 13), 'revenue': [120, 115, 125, 130, 128, 145, 160, 155, 170, 185, 195, 220]})
feature_launch_month = 6
# After: story-driven chart
fig, ax = plt.subplots(figsize=(12, 6))
# Gray context, colored highlight
before = df[df['month'] <= feature_launch_month]
after = df[df['month'] > feature_launch_month]
ax.plot(before['month'], before['revenue'], color='#888', linewidth=2.5)
ax.plot(after['month'], after['revenue'], color='#2563eb', linewidth=2.5)
# Annotate the key moment
ax.axvline(x=feature_launch_month, color='#16a34a', linestyle='--', alpha=0.7)
ax.text(feature_launch_month + 0.2, 145, 'Feature\nlaunch', color='#16a34a', fontsize=11, fontweight='bold')
# Annotate the growth
ax.annotate('+52% revenue\nsince launch', xy=(12, 220), fontsize=12, fontweight='bold', color='#2563eb')
# Title that states the finding
ax.set_title('New feature drove 52% revenue growth in second half', fontsize=15, fontweight='bold', pad=15)
ax.set_xlabel('Month'); ax.set_ylabel('Revenue ($K)')
ax.spines['top'].set_visible(False); ax.spines['right'].set_visible(False)
Why this matters in AI
Data storytelling is what separates ML practitioners who drive business impact from those who produce analysis that no one acts on. A technically perfect model evaluation that is presented as a table of numbers will not influence decisions. The same evaluation framed as 'our model will prevent 500 fraudulent transactions per week, saving $2.3M annually' gets immediate stakeholder buy-in.
Common mistakes
  • Starting with the data instead of the question — don't show all your analysis and hope the audience finds the insight. Start with the question and conclusion, then use the data to support it.
  • Using too many charts — a 30-slide deck with a different chart on each slide is overwhelming. The best presentations have 3-5 key charts, each making one point clearly. More is not more.
  • Not removing default chart styling — matplotlib defaults produce professional-looking charts. Remove the top and right spines, use a neutral gray for context data and a bold color for the key data, and increase font sizes for readability at presentation scale.
