Recommendation systems

Detailed explanation

Recommendation systems predict what items (products, movies, songs, articles) a specific user is likely to engage with, based on the user's history and the behavior of similar users. They are among the highest-value ML applications: Netflix's recommendation system is estimated to save $1B/year in prevented churn; Amazon attributes 35% of revenue to recommendations.

Two fundamental approaches: Collaborative filtering — 'users who liked what you liked also liked X'. Analyzes the user-item interaction matrix (ratings, clicks, purchases) to find similar users or items. Content-based filtering — 'because you liked this, you'll like other things similar to it'. Analyzes item features (movie genre, director, cast) to find similar items.

User-based vs item-based collaborative filtering: user-based: find users most similar to the target user (cosine similarity of rating vectors), recommend items those similar users rated highly. Item-based: find items most similar to items the user has rated (cosine similarity of item rating vectors), recommend highly similar items. Item-based scales better (item similarity is stable; user similarity changes as users rate more items).

Matrix factorization: the core technique in modern collaborative filtering. Factorize the user-item matrix R (n_users × n_items) into U (n_users × k) × V^T (k × n_items). k is the latent factor dimension (typically 20-200). U encodes each user's 'taste profile' in k dimensions; V encodes each item's 'topic profile' in k dimensions. Predicted rating: R_hat[u,i] = U[u,:] · V[i,:]. Learn U and V by minimizing reconstruction error on observed ratings (SGD or ALS).

Evaluation metrics: offline metrics: RMSE on held-out ratings (regression), Precision@K, Recall@K, NDCG@K (ranking quality). Online metrics: click-through rate, add-to-cart rate, purchase rate, time spent, return visits. Offline metrics don't always predict online performance — a serendipitous recommendation (slightly unexpected but delightful) may score lower on RMSE but outperform online.

# Train-test split and SVD matrix factorization
trainset, testset = train_test_split(data, test_size=0.2, random_state=42)
svd = SVD(n_factors=100, lr_all=0.005, reg_all=0.02, n_epochs=20)
svd.fit(trainset)

unrated = [i for i in all_item_ids if i not in rated_items]
preds = [(i, svd.predict(user_id, i).est) for i in unrated]
top_10 = sorted(preds, key=lambda x: x[1], reverse=True)[:10]
print('Top recommendations:', top_10)

Why this matters in AI
Recommendation systems are one of the most economically valuable ML applications. Every major consumer platform (Amazon, Netflix, Spotify, YouTube, TikTok) uses sophisticated recommendation systems. Understanding matrix factorization, collaborative filtering, and evaluation metrics is essential for anyone working in consumer ML.
Common mistakes
  • Using RMSE as the only evaluation metric — RMSE measures rating prediction accuracy but not recommendation list quality. A system that predicts 4.5 instead of 5 but recommends the right items is better than one with lower RMSE that recommends obvious or irrelevant items. Use Precision@K and NDCG@K alongside RMSE.
  • Not handling the cold start problem — new users and new items have no interaction history. Fallback strategies: for new users, recommend popular items or ask for initial preferences. For new items, use content-based filtering (item features) until enough interaction data accumulates.
  • Evaluating on seen items only — if you test the recommender by asking it to predict ratings on items the user has already interacted with, you're testing memorization, not generalization. Use a temporal holdout: train on interactions before date T, test on interactions after date T.

Content-based filtering

Detailed explanation

Content-based filtering recommends items similar to what a user has previously liked, based on item features rather than the behavior of other users. 'You watched Inception (sci-fi, Christopher Nolan) — you might like Interstellar (sci-fi, Christopher Nolan).' Unlike collaborative filtering, content-based filtering does not require user interaction data from other users, making it ideal for cold-start scenarios.

Item representation: items are represented as feature vectors based on their attributes. For movies: genre (one-hot encoded), director, cast, plot synopsis (TF-IDF or embedding), rating (normalized). For articles: TF-IDF of the text, publication date, author, topic tags. For products: category, price, brand, product description embedding.

User profile: build a user profile by aggregating the features of items the user has positively interacted with. For movies: the user's profile = average of the feature vectors of all movies they rated highly. Similarity = cosine similarity between the user profile vector and candidate item vectors. Recommend items with the highest similarity to the user profile.

Pros and cons: pros — solves cold start for new items (only need item features, not interaction data), explainable ('recommended because you watched sci-fi movies'), no need for other users' data (privacy-preserving). Cons — filter bubble (only recommends similar items, no serendipity), requires rich item metadata, cannot learn from implicit collective behavior (what other users with similar taste prefer).

Hybrid approach: combine content-based and collaborative filtering for the best of both. For new users (cold start): use content-based filtering on their stated preferences or browsing history. For existing users with rich interaction history: switch to collaborative filtering which leverages collective behavior. This hybrid avoids both cold start problems (new user, new item) and filter bubbles.

import pandas as pd
import numpy as np
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.metrics.pairwise import cosine_similarity
# Item feature matrix
movies = pd.DataFrame({'title': ['Inception', 'Interstellar', 'The Matrix', 'Avengers'],
    'genres': ['Sci-Fi Thriller', 'Sci-Fi Drama', 'Sci-Fi Action', 'Action Superhero'],
    'description': ['dream heist multi-level', 'space time wormhole love', 'simulation red blue pill', 'superheroes infinity stones']})
# TF-IDF on combined features
movies['features'] = movies['genres'] + ' ' + movies['description']
tfidf = TfidfVectorizer(stop_words='english')
feature_matrix = tfidf.fit_transform(movies['features'])
# Cosine similarity between items
item_similarity = cosine_similarity(feature_matrix, feature_matrix)
pd.DataFrame(item_similarity, index=movies['title'], columns=movies['title']).round(3)
# User profile + recommendation
def get_content_recommendations(user_liked_titles, movies, item_similarity, n=5):
    liked_indices = [movies[movies['title'] == t].index[0] for t in user_liked_titles if t in movies['title'].values]
    if not liked_indices: return movies.head(n)['title'].tolist()
    # User profile = mean similarity to liked items
    user_profile = item_similarity[liked_indices].mean(axis=0)
    # Don't recommend already-liked items
    user_profile[liked_indices] = -1
    top_indices = np.argsort(user_profile)[::-1][:n]
    return movies.iloc[top_indices]['title'].tolist()
print(get_content_recommendations(['Inception', 'The Matrix'], movies, item_similarity))
Why this matters in AI
Content-based filtering solves the cold-start problem that collaborative filtering cannot: for a new item with no interaction history, you can still recommend it to users who like similar items based on features alone. Understanding the full recommendation system toolkit — content-based, collaborative, hybrid — enables designing the right system for each business context.
Common mistakes
  • Creating item feature vectors without domain-appropriate features — for movies, genre + director + cast are more informative than raw plot text. Choose features that humans would use to judge similarity.
  • Not normalizing item feature vectors — before computing cosine similarity, normalize feature vectors to unit length. Without normalization, items with more text (longer descriptions) have larger magnitude vectors and appear more similar to everything.
  • Building a pure content-based system for a mature product — once you have thousands of users with interaction history, collaborative filtering typically outperforms content-based. Use content-based as the cold-start fallback, switch to collaborative/hybrid once interaction data is available.

Collaborative filtering

Detailed explanation

Collaborative filtering (CF) is the most common approach to recommendation systems. It predicts a user's preferences by finding patterns across many users. The core idea: users who have agreed on many items in the past will likely agree on future items. If Alice and Bob both loved movies X, Y, and Z, and Bob also loved movie W, Alice will probably love W too — even though Alice has never seen W.

User-based collaborative filtering: find the K users most similar to the target user (by cosine similarity or Pearson correlation of their item ratings). Predict the target user's rating for an unseen item as the weighted average of similar users' ratings. Limitation: as the number of users grows to millions, computing pairwise similarity becomes expensive O(n²). Also, users rarely rate many items (sparse matrices) so similarity estimates are noisy.

Item-based collaborative filtering: compute item-item similarity (items rated similarly by the same users are similar). To predict a user's rating for item I, find the K most similar items that the user has rated, and compute a weighted average. Item-item similarity is more stable than user-user similarity (items' characteristics change less than user preferences) and enables precomputing similarities offline.

Matrix Factorization (MF) is the dominant CF approach. Represent users as vectors U ∈ R^(n_users × K) and items as vectors V ∈ R^(n_items × K). The predicted rating: R̂_{u,i} = U_u · V_i (dot product). Train U and V by minimizing the reconstruction error on observed ratings: Σ (R_{u,i} - U_u · V_i)² + λ(||U||² + ||V||²). The K-dimensional vectors encode latent factors (for movies: genre preference, preferred era, action vs drama). SVD and ALS (Alternating Least Squares) are classic MF algorithms; NCF (Neural Collaborative Filtering) uses neural networks for non-linear MF.

The cold-start problem: new users (no rating history) and new items (no ratings) cannot be recommended via CF. Solutions: content-based filtering for new items (use item features — genre, cast, description), demographic-based recommendations for new users, and hybrid systems that combine CF with content-based. Collecting initial ratings (onboarding survey, implicit feedback from first interactions) reduces cold start.

    def __init__(self, n_users, n_items, K=32):
        super().__init__()
        self.user_emb = nn.Embedding(n_users, K)
        self.item_emb = nn.Embedding(n_items, K)

Why this matters in AI
Collaborative filtering powers the recommendation systems at Netflix, Spotify, Amazon, and YouTube — the primary drivers of engagement and revenue for major platforms. Understanding CF, matrix factorization, and cold-start is essential for anyone building recommendation features.
Common mistakes
  • Evaluating recommendations with RMSE alone — RMSE measures rating prediction accuracy, not the quality of the top-K list that users actually see. Use ranking metrics: NDCG@K, Recall@K, Precision@K.
  • Not handling the cold-start problem — a pure CF model returns no recommendations for new users or items. Always have a fallback strategy.
  • Ignoring implicit feedback — most platforms cannot rely on explicit ratings (1–5 stars). Use implicit signals: clicks, views, purchases, time spent. Treat positives as (user, item) pairs and use negative sampling.

Matrix factorization intuition

Detailed explanation

Matrix factorization decomposes a large, sparse matrix (like a user-item ratings matrix) into two smaller dense matrices whose product approximates the original. The intuition: users can be described by a few latent factors (preferences for action, drama, comedy), and items by the same latent factors. A user's predicted rating for an item is the dot product of their latent factor vectors.

SVD and collaborative filtering: Singular Value Decomposition (SVD) is the mathematically pure form of matrix factorization. For ML: approximate SVD where only k factors are used (truncated SVD). The Netflix Prize famously showed that matrix factorization outperformed k-nearest-neighbor collaborative filtering — the winning solution was based on SVD++.

Latent factor model: User u rates item i. We want to learn: user embedding u ∈ R^k (k=10-100 dimensions) and item embedding v ∈ R^k. Predicted rating: r̂(u,i) = u^T × v. Learn u and v by minimizing squared error on observed ratings: min ||R - UV^T||² + λ||U||² + λ||V||² (L2 regularization prevents overfitting). This is equivalent to regularized SVD.

Implicit feedback: most interaction data is implicit (clicks, views, purchases) rather than explicit (star ratings). Implicit feedback matrix factorization: treat non-zero interactions as positive signals (weight=1) and unobserved interactions as uncertain negatives (weight=0). The Alternating Least Squares (ALS) algorithm efficiently handles large implicit feedback matrices by treating user and item embeddings as separate optimization problems solved alternately.

Modern extensions: Neural Collaborative Filtering (NCF) replaces the dot product with a neural network — learns more complex user-item interactions. BERT4Rec treats recommendation as a masked item prediction problem (like BERT's masked language modeling). These extensions improve ranking but at higher computational cost.

from sklearn.decomposition import TruncatedSVD
import numpy as np
# Simple matrix factorization with sklearn
# R = user-item matrix (rows=users, columns=items, values=ratings)
# sparse: most ratings are unknown (NaN or 0)
R = np.array([[4, 5, 0, 0, 1],
               [0, 3, 4, 0, 0],
               [1, 0, 5, 4, 0],
               [0, 0, 0, 5, 3]])
R_filled = np.where(R == 0, R.mean(), R)  # fill with mean for SVD
svd = TruncatedSVD(n_components=2, random_state=42)
U = svd.fit_transform(R_filled)    # user embeddings [4 users, 2 factors]
VT = svd.components_               # item embeddings [2 factors, 5 items]
R_pred = U @ VT                    # reconstructed matrix
print('User embeddings (2D):'); print(U.round(2))
print('\nOriginal vs Predicted for first user:')
print(f'Original: {R[0]}'); print(f'Predicted: {R_pred[0].round(2)}')
Why this matters in AI
Matrix factorization is the foundation of all collaborative filtering recommendation systems. Understanding the latent factor model — that users and items can be represented in a shared embedding space, and similarity in that space predicts preference — is the intuition behind Spotify's recommendations, Netflix's suggestions, and Amazon's 'customers also bought.' This same idea of learning embeddings extends to NLP (Word2Vec, BERT) and graph networks.
Common mistakes
  • Conflating the rating matrix with the latent factor matrix — the rating matrix R is input data; U and V are learned representations. The predicted rating R̂ = UV^T approximates R, but R̂ includes predictions for unobserved (missing) ratings — that's the point.
  • Not handling cold start — matrix factorization requires at least one observed interaction per user/item to learn embeddings. New users with zero history cannot be factorized. Handle with content-based features for cold-start users, then switch to collaborative filtering once interactions accumulate.
  • Treating implicit feedback as explicit ratings — a user viewing a product is not the same as giving it 5 stars. Implicit feedback models should distinguish 'saw but didn't click' (weak negative) from 'clicked and purchased' (strong positive). Weight interactions by confidence rather than treating all as positive.

Ranking metrics

Detailed explanation

Ranking metrics evaluate how well a system orders a list of items — specifically, how well it places relevant items near the top. Search engines, recommendation systems, and information retrieval systems are evaluated with ranking metrics, not classification accuracy, because the position of relevant items in the list matters as much as whether they're included.

Precision@k: of the top-k items returned, what fraction are relevant? P@5 = (# relevant in top 5) / 5. Simple but doesn't penalize for relevant items placed low. A system that returns all relevant items in position 1000 has P@1000 = 1 but is useless.

Mean Reciprocal Rank (MRR): for each query, compute the reciprocal rank of the FIRST relevant result: 1/rank. Average across queries. MRR=1 means the first result is always relevant. MRR=0.5 means the first relevant result is at rank 2 on average. Used when only one relevant item exists per query (e.g., factual QA: which document contains the answer?).

NDCG (Normalized Discounted Cumulative Gain): the most comprehensive ranking metric. Takes into account: whether relevant items are retrieved, the position of relevant items (earlier = better), and graded relevance (not all relevant items are equally relevant). DCG@k = Σ (2^rel_i - 1) / log2(i+1) for i=1..k. NDCG = DCG / IDCG (ideal DCG = DCG of perfect ranking). NDCG=1 is perfect; NDCG=0 is worst.

Recall@k: of all relevant items in the corpus, what fraction are in the top-k? Recall@k is the standard metric for RAG retrieval evaluation — 'of all the documents that could answer the query, how many did we retrieve in the top k?'. Recall@5 measures: was the relevant document retrieved at all? Precision@5 measures: if I return 5 documents, how many are relevant?

import numpy as np
# Ranking metric implementations
def precision_at_k(relevant, retrieved_k):
    return len(set(relevant) & set(retrieved_k)) / len(retrieved_k)
def recall_at_k(relevant, retrieved_k):
    return len(set(relevant) & set(retrieved_k)) / len(relevant)
def reciprocal_rank(relevant, retrieved):
    for rank, item in enumerate(retrieved, start=1):
        if item in relevant: return 1.0 / rank
    return 0.0

Why this matters in AI
Ranking metrics capture what matters for retrieval systems: not just whether a relevant item is retrieved, but whether it appears near the top. A recommendation system with Recall@10=0.9 (retrieves 90% of relevant items in top 10) but NDCG@5=0.4 (relevant items are scattered throughout position 3-10 rather than at positions 1-2) provides a poor user experience because users rarely look beyond the top 3 results.
Common mistakes
  • Using accuracy instead of ranking metrics for recommendation/retrieval — accuracy measures classification; for ranking, you need to know WHERE in the ranked list relevant items appear. Use precision@k, recall@k, MRR, or NDCG.
  • Not choosing k appropriately — evaluate at the k that matches your system's UX. For a search engine showing 10 results: evaluate P@10 and NDCG@10. For a chatbot showing the top RAG context document: evaluate Recall@1 and MRR.
  • Conflating MRR and MAP — MRR uses only the FIRST relevant result. MAP (Mean Average Precision) averages precision at each position where a relevant result appears — better for queries with multiple relevant documents. Use MRR for single-answer queries; MAP or NDCG for multiple-answer queries.

Anomaly detection

Detailed explanation

Anomaly detection identifies rare data points that differ significantly from normal observations. Unlike classification (labeled positives and negatives), anomaly detection is usually semi-supervised or unsupervised: you have abundant normal examples but few or no labeled anomalies. Applications: fraud detection (rare fraudulent transactions among millions of legitimate ones), system health monitoring (rare server failures among normal operation), manufacturing defect detection, and cybersecurity intrusion detection.

Statistical methods: Z-score (flag points more than 3 standard deviations from the mean — works for univariate Gaussian data), IQR fences (points below Q1-3*IQR or above Q3+3*IQR — robust to outliers), and Mahalanobis distance (multivariate Z-score accounting for feature correlations). These work well for low-dimensional, approximately Gaussian data but fail on high-dimensional and non-Gaussian distributions.

Isolation Forest: an ensemble method that isolates anomalies by randomly selecting a feature and a random split value, recursively until each sample is isolated in its own leaf. Anomalies (rare, unusual points) are isolated with fewer splits (shorter path length) than normal points. The anomaly score is the inverse of average path length across all trees. Isolation Forest is fast, scalable, and works well in practice — it is the go-to starting point for anomaly detection.

Autoencoders for anomaly detection: train an autoencoder (encoder compresses the input to a low-dimensional latent representation; decoder reconstructs the original input) exclusively on normal data. The autoencoder learns to reconstruct normal patterns well. At inference, anomalies have high reconstruction error because the model has not learned to reconstruct unusual patterns. The reconstruction error is the anomaly score. Variational Autoencoders (VAEs) provide a probabilistic alternative.

Evaluation challenge: anomaly detection is hard to evaluate because labels are often unavailable. If labeled anomaly data exists, use F1 score, precision at K, and AUC-PR (precision-recall AUC, more informative than ROC-AUC for rare anomalies). Without labels, use the contamination parameter (estimated fraction of anomalies) to tune the threshold. Qualitative review by domain experts is often the most reliable evaluation.

Why this matters in AI
Anomaly detection is critical for any system that must be reliable and secure. Fraud, intrusion, equipment failure — these rare events cause enormous damage when missed. Traditional classification fails because labeled anomaly data is rare. Unsupervised anomaly detection catches what cannot be labeled in advance.
Common mistakes
  • Setting contamination too high — if your actual anomaly rate is 0.1% but you set contamination=0.1, you flag 10% of data as anomalies with many false positives. Estimate the contamination carefully.
  • Training the autoencoder on data that includes anomalies — the autoencoder must train only on normal data, or it learns to reconstruct anomalies and loses discriminative power.
  • Ignoring class imbalance when evaluating — a model that flags everything as normal has 99.9% accuracy on 0.1% anomaly data. Always report precision, recall, and F1 on the anomaly class.

Isolation Forest

Detailed explanation

Isolation Forest is an anomaly detection algorithm based on the key insight: anomalies are rare and different — they are more easily isolated (separated from the rest) than normal points. It builds random decision trees where at each node, a feature and a split value are chosen randomly. A point is anomalous if it can be isolated in very few splits (short path length in the tree). Normal points require many splits to isolate because they are surrounded by other normal points.

Algorithm: (1) draw a subsample of n_samples (typically 256) from the dataset. (2) Build a random isolation tree: recursively select a random feature and a random split value until each point is isolated or a maximum depth is reached. (3) Repeat B times (n_estimators=100) to build a forest. (4) For each data point, compute its average path length across all trees. Short average path = anomaly. Normalize to an anomaly score between 0 (normal) and 1 (anomaly). Score > 0.6 is typically considered anomalous.

Why it works: imagine a 2D dataset with 1000 normal points clustered together and 10 outliers far from the cluster. To isolate a normal point (inside the cluster), many splits are needed — you need to split the cluster many times to separate one point from its neighbors. To isolate an outlier (far from everything else), 1-2 splits suffice — it's already separated. This path length difference is the anomaly signal.

Hyperparameters: n_estimators (number of trees, default=100 — more is better but diminishing returns), max_samples (subsample size, default='auto' = min(256, n_samples) — larger subsamples give more accurate estimates but slower training), contamination (expected fraction of anomalies in the data, used to set the threshold — default='auto' uses -0.5 as the threshold).

Isolation Forest vs One-Class SVM vs Autoencoder: Isolation Forest is computationally efficient (linear time and space), works well in high dimensions, and requires no training of a model in the traditional sense. One-Class SVM works well for low-dimensional data with a clear normal boundary but scales poorly. Autoencoder-based anomaly detection (high reconstruction error = anomaly) is more powerful for structured data (images, sequences) but requires more setup and hyperparameter tuning.

One-class SVM

Detailed explanation

One-Class SVM is an anomaly detection algorithm that learns the boundary of the normal class without requiring examples of anomalies. It finds a hypersphere (or hyperplane with RBF kernel) that tightly encloses the normal training data. At inference, points outside this boundary are classified as anomalies. The nu hyperparameter controls the fraction of training points allowed outside the boundary (upper bound on outliers + lower bound on support vectors).

Algorithm intuition: a standard SVM finds the maximum-margin boundary between two classes. One-Class SVM finds the tightest possible boundary around a single class — the training data. It maps data to a high-dimensional feature space (using RBF kernel) and finds the minimum-radius hypersphere enclosing most of the data. Points outside are anomalies.

Hyperparameters: nu — fraction of training points treated as outliers during training (0.01 to 0.1 typically). Lower nu = tighter boundary = more false positives. Higher nu = looser boundary = more false negatives. kernel — 'rbf' (default, handles non-linear boundaries), 'linear', 'polynomial'. gamma — RBF kernel bandwidth. High gamma = narrow kernel = tighter boundary. Use gamma='scale' or 'auto' to start.

When to use One-Class SVM vs Isolation Forest: One-Class SVM works well for low-dimensional data (< 20 features) with a clear geometric boundary for the normal class. It is particularly effective for novelty detection (training data is all normal, need to detect if new data is different). Isolation Forest is better for high-dimensional tabular data, is faster, and scales to millions of samples. One-Class SVM does not scale well beyond ~10K samples.

Practical limitations: One-Class SVM is slow for large datasets (quadratic in n_samples). sklearn's SGDOneClassSVM (online learning version) scales to larger datasets. One-Class SVM assumes the normal class is a connected region in feature space — if normal data has multiple distinct modes, it struggles. Mixture models or Isolation Forest handle multi-modal normal data better.

Time series forecasting

Detailed explanation

Time series forecasting predicts future values based on historical temporal data. Unlike regular tabular ML, time series data has a temporal ordering constraint: the past is known, the future is unknown, and future data must never be used to predict the past (no data leakage). Common applications: demand forecasting (predict product sales for inventory planning), financial forecasting (stock prices, economic indicators), capacity planning (predict server load), and anomaly detection (identify unusual patterns).

Classical time series methods: ARIMA (AutoRegressive Integrated Moving Average) models the series as a linear combination of past values and past errors. SARIMA extends ARIMA with seasonal components. Exponential smoothing (ETS) assigns exponentially decreasing weights to older observations. These classical methods are interpretable, fast, and effective for stationary or simply-seasonal series, but struggle with multiple seasonality, complex patterns, and incorporating external regressors.

ML-based forecasting: convert the time series problem into a supervised learning problem by creating lag features (the value at time t-1, t-2, ..., t-k become features for predicting t). Rolling statistics (rolling mean, rolling std, rolling max over windows of 7, 30, 90 days) capture trend and seasonality as features. This framing lets any ML model (XGBoost, LightGBM, LSTMs) be used for forecasting.

Walk-forward validation (also called time series cross-validation) is mandatory for honest model evaluation. Split the series chronologically: train on weeks 1–52, validate on weeks 53–54. Then train on weeks 1–54, validate on weeks 55–56. Continue rolling forward. Never use random cross-validation on time series — it allows future information to leak into training, dramatically inflating performance estimates.

Modern forecasting: Prophet (by Facebook/Meta) handles holidays, missing data, and multiple seasonality automatically with minimal tuning. Neural Forecast (Amazon Chronos, N-BEATS, TFT) uses deep learning to learn complex patterns. For production forecasting systems, often the best approach is an ensemble: classical model + ML model + neural model, weighted by validation performance on recent data.

Why this matters in AI
Time series forecasting is one of the most commercially valuable ML applications — every company that holds inventory, plans capacity, or manages resources needs forecasts. Understanding lag features, walk-forward validation, and seasonality handling is essential for time series ML.
Common mistakes
  • Using random cross-validation for time series — this allows future information to leak into training. Always use walk-forward/time series cross-validation.
  • Forgetting to lag features relative to the forecast horizon — if you are forecasting 7 days ahead, you cannot use values from days t-1 through t-6 (they would not be available at prediction time). Use lag_7, lag_8, etc.
  • Not checking for stationarity — ARIMA requires the series to be stationary (constant mean and variance over time). Apply ADF (Augmented Dickey-Fuller) test and difference the series if needed.

Lag features

Detailed explanation

Lag features (also called temporal features) use the value of a variable at a previous time step as a feature for predicting the current time step. If yesterday's sales were high, today's sales are likely also high — the lag-1 feature captures this autocorrelation. Lag features are fundamental to time series modeling and tabular ML on sequential data.

Creating lag features: df['sales_lag1'] = df['sales'].shift(1) creates a lag-1 feature (yesterday's sales). df['sales_lag7'] = df['sales'].shift(7) creates a lag-7 feature (same day last week). df['sales_lag30'] = df['sales'].shift(30) creates a lag-30 feature (last month). The .shift() method shifts values down by n rows, introducing NaN in the first n rows (which must be handled or dropped).

Lead and lag: shift(1) is a lag (past values). shift(-1) is a lead (future values — future data leakage!). Only use lags (positive shift) as features, never leads. Using shift(-1) as a feature means 'what is tomorrow's value' — this is direct data leakage and gives perfect prediction but is useless in production.

Group-level lags: for data with multiple entities (user_id, store_id), lags must be computed per entity to avoid lagging across entities. df.groupby('user_id')['event'].shift(1) computes the lag-1 feature for each user independently. Without groupby, the lag-1 for user B's first record would be user A's last record — meaningless cross-entity contamination.

Difference features: df['sales_diff1'] = df['sales'] - df['sales'].shift(1) measures the change from the previous period. Percentage change: df['sales_pct_change'] = df['sales'].pct_change(). Difference features capture trend and acceleration (how fast is the value changing?). They are often more informative than raw lag features for non-stationary time series.

A random forest cannot naturally learn from sequential data, but if you provide yesterday's value as a feature, it can learn autocorrelation patterns. Lag features are essential for demand forecasting, anomaly detection in time series, and any sequential prediction problem.
Common mistakes
  • Not sorting data before creating lags — if the data is not sorted by time, shift(1) gives the wrong value. Always sort by (entity_id, timestamp) before creating lag features.
  • Not using groupby for entity-level lags — without groupby, lag features 'bleed' across entities. User A's last record becomes the lag-1 for User B's first record — nonsensical.
  • Using future data (negative shift) — shift(-1) is a lead: 'what is tomorrow's value'. This is the most severe form of data leakage in time series. Only use positive shifts (past values) as features.

Rolling features

Detailed explanation

Rolling features compute aggregate statistics over a sliding time window: the mean, sum, max, min, or std of the past W periods. A 7-day rolling mean smooths out day-to-day noise and captures the recent trend. A 30-day rolling std captures recent volatility. Rolling features are the most informative features for time series ML alongside lag features.

Creating rolling features: df['sales_rolling7_mean'] = df.groupby('user_id')['sales'].rolling(7).mean().reset_index(level=0, drop=True). The window parameter specifies how many past periods to include. min_periods parameter: rolling(7, min_periods=1) computes the rolling mean even for the first 6 rows (using whatever data is available). Without min_periods, the first 6 rows get NaN.

Rolling window types: fixed window (rolling(7) — exactly the last 7 rows), expanding window (expanding() — all past rows), exponentially weighted (ewm(span=7) — exponentially decaying weights, more recent values have higher weight). The EWM is often more predictive than a simple rolling mean because it prioritizes recent data without fully ignoring older data.

Rolling aggregations: .rolling(W).mean() — average over W periods. .rolling(W).std() — standard deviation (captures volatility). .rolling(W).max() — maximum. .rolling(W).min() — minimum. .rolling(W).sum() — total. .rolling(W).median() — less sensitive to outliers. For time-indexed DataFrames: rolling('7D') uses 7 calendar days, not 7 rows — handles irregular time intervals correctly.

Careful with rolling and groupby: for multi-entity data, always apply rolling within each entity's group. Without groupby: rolling(7) at the boundary between entity A and entity B computes the average of entity B's first 3 records and entity A's last 4 records — meaningless. Use: df.groupby('entity_id')['value'].rolling(7).mean().

df['sales_roll30_std'] = df.groupby('user_id')['sales'].transform(lambda x: x.rolling(30, min_periods=1).std())

df['sales_ewm7'] = df.groupby('user_id')['sales'].transform(lambda x: x.ewm(span=7, adjust=False).mean())

In Kaggle time series competitions, rolling features are consistently among the top 3 most important features.

  • Not using .transform() for rolling in a groupby — df.groupby('id')['value'].rolling(7).mean() returns a multi-index Series. Use .transform() to get a flat Series aligned with the original DataFrame index.
  • Computing rolling features across entity boundaries — without groupby, rolling windows span entity boundaries, mixing data from different users/products. Always use groupby for multi-entity data.

Walk-forward validation

Detailed explanation

Walk-forward validation (also called time series cross-validation or rolling origin validation) is the correct cross-validation procedure for time series data. Standard k-fold cross-validation randomly assigns samples to folds — this means future data can be in the training fold and past data in the test fold, violating temporal causality. Walk-forward validation ensures the test set is always AFTER the training set.

Fixed-origin walk-forward: train on all data from time 0 to t, test on t to t+h, then expand the training window to t+h and test on t+h to t+2h. Continue until the end of the data. The training window grows with each fold. This uses all available historical data for the final model but gives less data in early folds.

Rolling-window walk-forward: instead of expanding the training window, maintain a fixed-size window that slides forward. Train on data from t-W to t, test on t to t+h. Then train on t+1-W to t+1, test on t+1 to t+1+h. This simulates a production system that retrains on the past W periods (e.g., last 1 year). Useful when old data may be less relevant than recent data (concept drift).

Horizon-aware evaluation: the test set size (forecast horizon h) should match the intended deployment scenario. If you deploy a model that forecasts 7 days ahead, test on 7-day horizons. If you deploy monthly, test on 30-day horizons. Mismatching the evaluation horizon and the deployment horizon gives misleading results.

sklearn's TimeSeriesSplit: implements expanding-window walk-forward. n_splits=5 creates 5 folds, each larger than the previous. gap parameter introduces a gap between train and test (to simulate real-world delay in label availability). Use with cross_val_score() like any other CV strategy but with time-ordered data.
