Missing values

Detailed explanation

Missing values (NaN — Not a Number, or None) appear in almost every real-world dataset. They arise from: data collection failures (sensor outage, form field skipped), data merging (joining tables where some rows have no match), data truncation (values beyond a range set to missing), or truly unknown information (a field that did not apply to that record). Ignoring missing values causes most ML libraries to crash or silently produce wrong results.

Before handling missing values, understand WHY they are missing. Missing Completely At Random (MCAR): missingness has no relationship to any variable — randomly corrupted sensor readings. Missing At Random (MAR): missingness depends on observed variables — richer customers skip income question. Missing Not At Random (MNAR): missingness depends on the value itself — people with very high income are more likely to hide income. MCAR and MAR can be handled with imputation; MNAR is harder and may require modeling the missingness as a feature.

Imputation strategies: for numeric columns, impute with median (robust to outliers) or mean (sensitive to outliers). For categorical columns, impute with mode or a new 'Unknown' category. For time series, forward-fill (propagate last valid value) or interpolation. KNN imputation uses the values of the k most similar rows (measured by non-missing features) to impute missing values — more accurate but slower.

Adding a missingness indicator column alongside imputation is often valuable: df['age_was_missing'] = df['age'].isnull().astype(int); df['age'] = df['age'].fillna(df['age'].median()). This lets the model learn different patterns for rows where age was originally missing vs. rows with observed ages. The missingness itself can be predictive of the target.

Never impute before splitting train and test — fit the imputer on training data only, then transform both. If you fit on all data, you leak information from the test set into the imputer's learned statistics (mean, median, k-nearest neighbors). Always use sklearn's Pipeline or fit_transform on train followed by transform on test.

Why this matters in AI
Missing value handling is one of the most impactful preprocessing decisions. Wrong handling (dropping rows, leaking test statistics) causes data loss and evaluation inflation. Correct handling requires understanding why values are missing and using the pipeline pattern to prevent leakage.
Common mistakes
  • Dropping all rows with any missing value — if 40% of rows have at least one missing value, this destroys your dataset. Impute strategically.
  • Fitting the imputer on all data before train/test split — leaks test set statistics into training. Always fit only on training data.
  • Using mean imputation for skewed distributions — an outlier-heavy income column's mean is pulled by outliers. Use median for skewed numeric features.

Outliers

Detailed explanation

Outliers are data points that are far from the main distribution of a variable. They can arise from measurement errors, data entry mistakes, rare but legitimate events, or systemic bugs in data collection. Outlier handling is a critical preprocessing step because outliers disproportionately influence certain ML algorithms (linear regression, KNN, neural networks) while being robust to others (tree-based models).

Detecting outliers in univariate data: IQR method — Q1 - 1.5×IQR to Q3 + 1.5×IQR defines the 'normal' range (the standard Tukey fence). Points outside are outliers. Z-score method — values with |z-score| > 3 are outliers (more than 3 standard deviations from the mean). Modified z-score (MAD) — more robust version using median instead of mean: |0.6745 × (x - median) / MAD| > 3.5.

Detecting outliers in multivariate data: Mahalanobis distance — measures how many standard deviations a point is from the centroid in all dimensions simultaneously, accounting for correlations. Isolation Forest — an anomaly detection algorithm that isolates outliers by randomly partitioning data. Outliers require fewer splits to isolate (they're rare and different). Local Outlier Factor (LOF) — measures whether a point is less dense than its neighbors.

Handling outliers — options and tradeoffs: (1) Remove — only if the outlier is clearly a data error (salary = -50,000). (2) Cap/winsorize — replace values above the 99th percentile with the 99th percentile value. Preserves the sample size while reducing influence. (3) Transform — log transform compresses the tail of skewed distributions. (4) Robust model — use tree-based models (random forest, gradient boosting) that are not sensitive to outliers. (5) Investigate — ask WHY a point is an outlier. A customer with $1M purchase is an outlier but also a high-value customer — removing them would be a serious mistake.

Domain-dependent treatment: outlier handling is not mechanical — it requires domain knowledge. In fraud detection, outliers ARE the signal (unusual transactions are fraud). In financial data, extreme values (stock crashes, earnings surprises) are legitimate and important. In sensor data, a temperature reading of 9999°C is clearly a sensor error. Always understand what your data represents before deciding whether an outlier is noise or signal.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import pandas as pd
import numpy as np
from sklearn.ensemble import IsolationForest
# IQR-based outlier detection
def iqr_outliers(series):
    Q1, Q3 = series.quantile(0.25), series.quantile(0.75)
    IQR = Q3 - Q1
    lower, upper = Q1 - 1.5 * IQR, Q3 + 1.5 * IQR
    return (series < lower) | (series > upper)
# Z-score based detection
def zscore_outliers(series, threshold=3):
    z = (series - series.mean()) / series.std()
    return z.abs() > threshold
# Compare detections
for col in df.select_dtypes(include='number').columns:
    iqr_mask = iqr_outliers(df[col])
    z_mask = zscore_outliers(df[col])
    print(f'{col}: IQR={iqr_mask.sum()}, Z-score={z_mask.sum()}')
# Winsorize at 1%-99%
def winsorize(df, cols, lower_pct=0.01, upper_pct=0.99):
    df = df.copy()
    for col in cols:
        lower, upper = df[col].quantile(lower_pct), df[col].quantile(upper_pct)
        df[col] = df[col].clip(lower=lower, upper=upper)
    return df
Why this matters in AI
Unhandled outliers are one of the most common causes of poor linear model performance. A salary column with values {$30K, $45K, $60K, $65K, $2M} will have mean ≈ $440K — completely unrepresentative of the typical employee. A linear regression trained on this data will predict $440K for everyone. Detecting and handling outliers makes your preprocessing robust and your model performance predictable.
Common mistakes
  • Removing outliers blindly without investigation — always ask: is this a data error or a real but extreme value? Removing a $1M customer order because it's an outlier destroys a high-value business insight. Investigate before removing.
  • Using z-score for non-normally distributed data — z-score assumes normality. For highly skewed data (income, transaction amounts), use the IQR method or log-transform first, then compute z-scores.
  • Not considering outliers in categorical columns — numerical outliers are visible in distributions, but categorical outliers (typos: 'Unitde States', 'Germay') are invisible without value_counts(). Always check categorical columns for rare or misspelled values.

Scaling

Detailed explanation

Feature scaling transforms numerical features to a consistent range or distribution so that distance-based and gradient-based algorithms are not dominated by large-scale features. Without scaling, a feature with range 0–10,000 (e.g., income in dollars) dominates a feature with range 0–1 (e.g., age in normalized years) in KNN distance computation and gradient descent, causing the optimizer to behave poorly.

Min-Max Scaling (Normalization) maps values to [0, 1]: x_scaled = (x - x_min) / (x_max - x_min). This preserves the exact shape of the distribution. Use when: the algorithm requires bounded inputs (neural networks with sigmoid output, images normalized to [0,1]), the distribution is not Gaussian, and outliers are not a concern. Limitation: one extreme outlier can compress the rest of the data into a narrow range.

Standardization (Z-score normalization) maps to mean=0, std=1: x_scaled = (x - μ) / σ. This is the most commonly used scaling method. Use for: linear models (logistic regression, SVM, linear regression), neural networks (stable gradients), PCA (requires zero mean for correct variance decomposition). Limitation: the output has no fixed range — extreme outliers will still have large values.

Robust Scaling uses the median and IQR: x_scaled = (x - median) / IQR. Since median and IQR are not influenced by outliers, Robust Scaling is insensitive to extreme values. Use when the data has outliers that should not be removed but should not dominate scaling. sklearn's RobustScaler computes this.

Critical rule: fit the scaler on training data only, then transform both train and test. Never fit on the full dataset — this leaks test set statistics into training. In sklearn Pipelines, StandardScaler().fit_transform(X_train) then StandardScaler.transform(X_test) is handled automatically and correctly.

Why this matters in AI
Scaling is mandatory for: logistic regression, SVM, KNN, neural networks, and PCA. Failing to scale causes gradient descent to converge slowly or not at all, distance metrics to give wrong results, and regularization to be applied unevenly across features. Tree-based models (RandomForest, XGBoost) do not require scaling.
Common mistakes
  • Fitting the scaler on the test set or full dataset — always fit on training data only. fit_transform on test data is one of the most common ML bugs.
  • Applying MinMax scaling when the test data has values outside the training range — the scaled value will be outside [0,1], violating the assumption. Use StandardScaler for more robust behavior.
  • Scaling the target variable (y) — for regression, you may scale y; but for classification, never scale the class labels.

Standardization

Detailed explanation

Standardization (Z-score normalization) transforms a feature so it has mean = 0 and standard deviation = 1. Formula: z = (x - μ) / σ where μ is the feature mean and σ is the standard deviation computed from the training set. Result: values typically fall in [-3, 3]. Standardization does NOT bound the range — outliers remain outliers, just scaled differently.

Why standardization is necessary: many ML algorithms use distance metrics (K-Means, KNN, SVM) or gradient descent (logistic regression, neural networks). Distance-based algorithms are dominated by features with large scales. Example: age (18–90) and income (20,000–500,000) — income dominates distance computations. Gradient descent converges faster when features are on similar scales.

StandardScaler in sklearn: fit on training set only (scaler.fit(X_train)), then transform both train and test. Never fit on the test set — this leaks test statistics into preprocessing. Always use inside a Pipeline to guarantee proper train-only fitting.

When to use standardization vs normalization: use StandardScaler when the data is roughly Gaussian, or you use algorithms that assume zero-mean features (PCA, linear regression with ridge). Use MinMaxScaler when you need values in [0, 1] (neural networks with sigmoid/tanh). RobustScaler (uses median/IQR) is better when outliers are present.

RobustScaler: z = (x - median) / IQR where IQR = Q3 - Q1. Less sensitive to outliers than StandardScaler. Use for income data, housing prices, or any feature with extreme outliers.

Why this matters in AI
Standardization is required for most ML algorithms except tree-based models (decision trees, random forest, XGBoost use threshold-based splits, not distances). Without standardization, KNN, SVM, logistic regression, PCA, and neural networks all perform significantly worse.
Common mistakes
  • Fitting the scaler on the full dataset (train + test) — this leaks test statistics. Always fit only on training data.
  • Using StandardScaler on features with many outliers — use RobustScaler instead.
  • Forgetting to save the fitted scaler for inference — incoming data must be scaled with the same mean/std used during training. Save: joblib.dump(scaler, 'scaler.pkl').

Normalization

Detailed explanation

Normalization (min-max scaling) transforms features to a fixed range, typically [0, 1]. Formula: x_scaled = (x - x_min) / (x_max - x_min). The minimum value maps to 0, the maximum to 1, all others are proportionally distributed. Unlike standardization, normalization preserves the original distribution shape.

When normalization is preferred: (1) neural networks with sigmoid or tanh activations that saturate outside [0,1] and [-1,1] — normalized inputs prevent saturation at initialization, (2) image preprocessing: pixel values (0–255) normalized to [0,1] by dividing by 255, (3) when the algorithm requires a bounded input range, (4) when you need range [-1, 1]: MinMaxScaler(feature_range=(-1, 1)).

Sensitivity to outliers: normalization is very sensitive to outliers. An outlier at x_max = 1,000,000 compresses all other values near 0. Example: incomes [30000, 50000, 80000, 500000] → [0.0, 0.042, 0.105, 1.0]. Normal incomes cluster near 0, losing discrimination. Solution: clip outliers before normalizing, or use standardization.

Per-sample vs per-feature normalization: MinMaxScaler normalizes per feature (across all samples). sklearn's Normalizer normalizes per sample (each sample gets unit L2 norm — sum of squares = 1). Per-sample normalization is used for text features (TF-IDF vectors) and computing cosine similarity.

Training vs test consistency: min and max must come from the training set. Test values outside the training range will be scaled outside [0,1] — this is expected and correct. Do NOT recompute min/max on test data. Use MinMaxScaler().fit(X_train).transform(X_test).

Why this matters in AI
Normalization is the standard preprocessing step for neural networks and image models. Understanding when to use normalization vs standardization leads to more reliable models.
Common mistakes
  • Using MinMaxScaler when data has outliers — outliers dominate the range. Clip outliers first: X = np.clip(X, X.quantile(0.01), X.quantile(0.99)).
  • Applying MinMaxScaler to constant features (zero variance) — division by zero produces NaN. Drop constant features first.
  • Confusing feature normalization (MinMaxScaler) with sample normalization (Normalizer) — completely different operations.

Encoding

Detailed explanation

Encoding converts categorical features (strings like 'red', 'blue', 'green') into numerical representations that ML models can process. No sklearn or PyTorch model accepts raw strings — all inputs must be numbers. The choice of encoding affects model performance, especially for tree-based models (which can handle ordinal encoding naturally) vs linear models (which require more careful encoding to prevent spurious linear relationships).

One-hot encoding (OHE): creates a binary column for each unique category. 'color' with values [red, blue, green] → [is_red, is_blue, is_green] with exactly one 1 per row. Pros: no artificial ordering, works with linear models. Cons: sparse high-dimensional matrix for high-cardinality features (1000 cities → 1000 columns). Use drop_first=True or drop='first' to avoid the dummy variable trap (perfect multicollinearity in linear regression).

Label encoding: assigns integers 0, 1, 2, ... to categories in alphabetical order (or custom order). LabelEncoder in sklearn. Pros: compact (1 column). Cons: implies an order that doesn't exist — a model might learn 'green > blue > red' from the 2, 1, 0 encoding when no such order exists. Use ONLY for tree-based models (they don't care about the numerical value, only the threshold) or when there is a genuine ordinal relationship.

Target encoding (mean encoding): replaces each category with the mean target value for that category. Pros: compact, captures predictive information, handles high-cardinality. Cons: severe data leakage if computed on training data including the sample itself — must use cross-validation-based target encoding. sklearn's TargetEncoder (sklearn >= 1.3) and category_encoders library handle this correctly.

Frequency encoding: replaces each category with its frequency in the training set. Captures popularity information — a city that appears 10,000 times in the data is different from one that appears 5 times. Less likely to overfit than target encoding. Binary/hashing encoding: encodes k categories into log2(k) binary columns — more compact than OHE. Useful for very high cardinality (10K+ unique values).

Why this matters in AI
Choosing the right encoding strategy is one of the most impactful feature engineering decisions. Using label encoding for nominal features in linear models can silently harm performance. Target encoding for high-cardinality features with proper cross-validation can provide large AUC improvements over OHE.
Common mistakes
  • Using label encoding for nominal categories in linear models — linear regression and logistic regression assume numerical ordering. '3' (green) is not greater than '1' (blue) for a color feature.
  • One-hot encoding high-cardinality features — 'country' with 200 unique values → 200 binary columns that are mostly 0. Use target encoding or frequency encoding instead.
  • Target encoding without cross-validation — computing mean target for each city using the same training set (including the sample being encoded) leaks the target. Use sklearn's TargetEncoder or category_encoders.TargetEncoder with cv=5.

One-hot encoding

Detailed explanation

One-hot encoding converts a categorical feature with K unique values into K binary columns, one per category. For a 'color' feature with values ['red', 'green', 'blue'], one-hot encoding creates three columns: is_red, is_green, is_blue — each row has exactly one 1 and two 0s. This representation allows linear models to use categorical features without imposing any ordinal relationship (red > green > blue would be wrong).

The dummy variable trap: with K categories, you only need K-1 binary columns — the Kth is always determined by the others (if not red and not green, then blue). Including all K creates perfect multicollinearity, which causes issues for linear models. sklearn's OneHotEncoder with drop='first' or drop='if_binary' avoids this. For tree-based models, multicollinearity does not matter — use drop=None.

High cardinality categories (city names with 10,000 unique values, user IDs) create a huge number of columns with one-hot encoding. A 10,000-category feature becomes 10,000 columns — making the model huge and sparse. Alternatives: (1) target encoding — replace each category with the mean target value for that category, (2) frequency encoding — replace with category frequency, (3) embedding layers (for neural networks) — learn a dense vector representation, (4) hashing — hash to a fixed-size vector (feature hashing).

handle_unknown='ignore' is a critical parameter. At inference, the model may encounter a category unseen during training. Without this parameter, sklearn's OneHotEncoder throws a ValueError. With it, the unknown category is encoded as all zeros — the model has seen no training examples for this category, so predicting from prior knowledge is reasonable.

Ordinal encoding assigns integers 1, 2, 3, ... to categories with a natural order (size: small=1, medium=2, large=3; education: high_school=1, bachelor=2, master=3, phd=4). This preserves the ordinal relationship. If there is no natural order, ordinal encoding imposes an arbitrary one — use one-hot encoding instead. sklearn's OrdinalEncoder supports specifying the category order explicitly.

Common mistakes

  • Fitting the encoder on the test set — always fit_transform on training data, transform on test data. This ensures test categories are handled correctly.
  • Not using handle_unknown='ignore' in production — new category values in production inputs will crash the encoder without this parameter.

Ordinal encoding

Detailed explanation

Ordinal encoding assigns integers 0, 1, 2, ... to categories with a meaningful natural order. The order is specified explicitly to preserve the ordinal relationship. Examples: education level (primary=0, secondary=1, bachelor=2, master=3, phd=4), satisfaction rating (very_unsatisfied=0, unsatisfied=1, neutral=2, satisfied=3, very_satisfied=4), t-shirt size (XS=0, S=1, M=2, L=3, XL=4).

The key requirement: there must be a meaningful, domain-justified order. 'Small < Medium < Large' is meaningful because it corresponds to actual physical size. 'Cat < Dog < Fish' (for animal type) is NOT meaningful — assigning numbers to unordered categories creates a false relationship (dog - cat ≠ fish - dog). Use one-hot encoding for unordered categories.

sklearn's OrdinalEncoder: specify the order explicitly with categories=[['XS','S','M','L','XL']]. Without explicit ordering, OrdinalEncoder assigns integers alphabetically — often wrong. Always specify the category order.

Ordinal encoding works well with: tree-based models (they use thresholds, so the order matters but specific numerical values don't), gradient boosting, random forests. For linear models, ordinal encoding assumes equal spacing between levels (master is exactly as far from bachelor as bachelor is from secondary). If this assumption is wrong, consider target encoding or treating as nominal.

Target encoding (also called mean encoding) replaces each category with the mean target value for that category. It creates a continuous feature that captures the relationship between the category and the target. Risk: it leaks label information and must be computed on training data only. Use cross-validation-based target encoding (sklearn's TargetEncoder or category_encoders) to prevent leakage.

Why this matters in AI
Ordinal encoding preserves meaningful relationships between ordered categories. One-hot encoding ignores the ordering — a model using one-hot cannot infer that XL is larger than L without seeing many examples. Ordinal encoding gives tree-based models the ability to learn threshold-based rules like 'if size >= L (3): ...'
Common mistakes
  • Using ordinal encoding for nominal categories — assigning 1 to cat, 2 to dog, 3 to fish implies cat < dog < fish, which is nonsensical. This can mislead linear models.
  • Using the default alphabetical ordering — OrdinalEncoder without explicit categories assigns integers alphabetically. 'XL' comes after 'S' alphabetically but before 'S' in size order.
  • Not specifying handle_unknown for test data — if a new category appears at inference, OrdinalEncoder raises an error. Set handle_unknown='use_encoded_value', unknown_value=-1.

Target encoding caution

Detailed explanation

Target encoding replaces each category value with the mean of the target variable for that category. 'city=Paris' becomes the mean fraud rate for Paris transactions. This captures the predictive relationship directly and handles high-cardinality categoricals (10,000 zip codes) where one-hot encoding would be infeasible. The caution: target encoding causes severe data leakage if done incorrectly.

The leakage problem: if you compute the mean target for 'Paris' using ALL examples (including the test set), then use that mean as a feature, you've implicitly told the model the target values. The feature mean_fraud_rate_Paris=0.12 was computed knowing which Paris transactions were fraud — including the test examples. This inflates training performance but produces a model that doesn't generalize.

Correct implementation: compute target means ONLY from the training set, then apply to validation and test sets. Within the training set, use leave-one-out (LOO) encoding or k-fold target encoding: compute the mean for each example using all OTHER examples in the training set (never including the example itself). sklearn's TargetEncoder (added in v1.3) implements this correctly with cross-validation.

Smoothing: categories with few examples produce noisy means. If only 3 transactions are from 'TinyCity' and all 3 happen to be fraud, the mean=1.0 will wildly overestimate the fraud rate. Smoothing blends the category mean with the global mean: smooth_mean = (n_cat × cat_mean + m × global_mean) / (n_cat + m) where m is the smoothing factor (typically 10-50). Larger m means more shrinkage toward the global mean for small categories.

Alternatives: when to avoid target encoding: if you have < 50 examples per category, target encoding estimates will be noisy even with smoothing — use ordinal encoding or frequency encoding instead. If interpretability is required, leave-one-out encoding is more transparent than cross-validation encoding.

# Manual leave-one-out encoding (for understanding)
def loo_target_encode(X_train, y_train, col, smoothing=10):
    df = X_train.copy(); df['__target__'] = y_train.values
    global_mean = y_train.mean()
    result = np.zeros(len(df))
    for i in range(len(df)):
        cat = df[col].iloc[i]
        mask = (df[col] == cat)
        mask.iloc[i] = False  # leave this example out
        n_cat = mask.sum(); cat_mean = df[mask]['__target__'].mean() if n_cat > 0 else global_mean
        # smooth toward global mean
        result[i] = (n_cat * cat_mean + smoothing * global_mean) / (n_cat + smoothing)
    return result
Why this matters in AI
Target encoding leakage is one of the most common silent bugs in ML. The model appears to have excellent performance in cross-validation, but performs poorly in production. The bug: target encoding was applied before the train/validation split, so validation performance was inflated by leaked target information. Understanding and preventing this is critical for building reliable ML pipelines.
Common mistakes
  • Fitting TargetEncoder on the full dataset before cross-validation — any sklearn transform must be inside a Pipeline so it's refit on each fold's training data. `cross_val_score(Pipeline([('enc', TargetEncoder()), ('model', ...)]), X, y)` is correct. `enc.fit_transform(X); cross_val_score(model, X_enc, y)` leaks.
  • Not using smoothing for rare categories — without smoothing, a city with 1 transaction that happens to be fraud gets city_fraud_rate=1.0. This single data point will make the model extremely suspicious of all transactions from that city.
  • Target encoding an ordinal variable — if the categorical variable has a natural ordering (low, medium, high), use ordinal encoding instead of target encoding. Target encoding destroys the ordering information.

Imbalanced data

Detailed explanation

Imbalanced classification is when one class is much rarer than the other. Examples: fraud detection (0.1% of transactions are fraud), medical diagnosis (1% of patients have the disease), defect detection (0.5% of manufactured parts are defective). A naive model that always predicts the majority class achieves 99.9% accuracy on fraud detection — but catches 0% of fraud. Accuracy is a meaningless metric for imbalanced problems.

Evaluation metrics for imbalanced problems: use Precision, Recall, F1 score, or the ROC-AUC and PR-AUC curves. PR-AUC (Area Under Precision-Recall Curve) is more informative than ROC-AUC for severely imbalanced datasets. ROC-AUC can be misleadingly high even for a model that catches very few positives. Always check the confusion matrix and compute precision and recall separately.

Handling imbalance: oversampling minority class (replicate minority examples — SMOTE creates synthetic minority examples by interpolating between existing ones), undersampling majority class (randomly remove majority examples — loses information but reduces training time), class weighting (train with class_weight='balanced' or explicit weights — penalizes mistakes on minority class more, effectively oversampling in loss space). Of these, class weighting is the safest starting point.

SMOTE (Synthetic Minority Oversampling Technique) creates new minority examples by: selecting a minority example, finding its k nearest minority neighbors, and creating a new example at a random point along the line segment between the original and a random neighbor. This creates synthetic but realistic-looking examples. from imblearn.over_sampling import SMOTE; X_resampled, y_resampled = SMOTE().fit_resample(X_train, y_train). Warning: apply SMOTE only to the training set, never to validation or test.

Threshold tuning: most classifiers output a probability score. The default threshold of 0.5 is not optimal for imbalanced problems. Plot the precision-recall curve and choose the threshold that maximizes F1 (or another task-relevant metric). For fraud detection, you might choose a lower threshold (0.2) to increase recall (catch more fraud) at the cost of lower precision (more false alarms for human review).

Why this matters in AI
Imbalanced data is the rule, not the exception, in high-value applications (fraud, medical diagnosis, churn). A model that ignores imbalance catches 0% of the rare events that matter. Every ML practitioner must know how to handle imbalanced data correctly.
Common mistakes
  • Using accuracy as the metric for imbalanced data — 99% accuracy for a 1% positive rate means the model predicts all negatives. Report F1, AUC, or PR-AUC instead.
  • Applying SMOTE to the full dataset before splitting — this causes data leakage. Split first, then apply SMOTE only to the training set.
  • Not tuning the decision threshold — most classifiers default to 0.5, which is rarely optimal for imbalanced problems. Always tune the threshold based on business requirements.

Class weights

Detailed explanation

Class weights adjust the importance of each class during training to compensate for class imbalance. Instead of each training sample contributing equally to the loss, samples from the minority class contribute more (get higher weight). This makes the model more sensitive to minority class errors, effectively teaching it to pay attention to rare events.

class_weight='balanced' in sklearn: automatically computes weights inversely proportional to class frequency. weight_class_k = n_samples / (n_classes × n_samples_in_class_k). For 95% negative, 5% positive: weight_positive = 1/(2 × 0.05) = 10, weight_negative = 1/(2 × 0.95) ≈ 0.53. So each positive sample has ~19× the weight of each negative sample. Available in: LogisticRegression, DecisionTreeClassifier, RandomForestClassifier, SGDClassifier.

scale_pos_weight in XGBoost: the ratio of negative to positive samples. scale_pos_weight = sum(negative_samples) / sum(positive_samples). For 95/5 split: scale_pos_weight = 0.95/0.05 = 19. This tells XGBoost to upweight positive samples by 19×. Similar to sklearn's class_weight but as a single ratio (for binary classification only).

Class weights vs resampling: class weights achieve a similar effect to SMOTE (oversampling) but without creating new synthetic samples. Class weights are computationally cheaper (no new samples to store or process) and work for all algorithms. SMOTE may help when the minority class is in a low-density region that the model needs more examples to learn. In practice, class_weight='balanced' is the first thing to try for imbalanced classification.

Manual class weights: sometimes the balanced class weights are not optimal. If false negatives are 5× more costly than false positives, set class_weight={0: 1, 1: 5}. This allows domain knowledge about the costs of different types of errors to be incorporated directly into the training objective.

Why this matters in AI
Class weights are the simplest and most effective first-line treatment for class imbalance. Adding class_weight='balanced' is a one-line change that often significantly improves minority class recall without any other modifications. It is the first imbalance technique to try before SMOTE, undersampling, or ensembles.
Common mistakes
  • Not using class_weight='balanced' as the baseline — always compare with and without class weights before trying more complex techniques (SMOTE, undersampling). The improvement from balanced weights alone often makes other techniques unnecessary.
  • Using class_weight='balanced' without re-evaluating the threshold — balanced class weights shift the decision boundary. The optimal threshold for the balanced model may be different from 0.5. Use precision_recall_curve to re-evaluate.
  • Using scale_pos_weight > 100 in XGBoost — very high scale_pos_weight makes the model extremely sensitive to false negatives, potentially resulting in almost all samples predicted as positive. Start with the natural ratio and tune from there.

SMOTE concept

Detailed explanation

SMOTE (Synthetic Minority Oversampling Technique) generates synthetic training examples for the minority class by interpolating between existing minority class samples. Unlike random oversampling (which just duplicates existing samples), SMOTE creates NEW synthetic samples that are novel combinations of real minority examples. This provides the model more diverse examples of minority class behavior.

SMOTE algorithm: (1) for each minority class sample x_i, find its k nearest minority-class neighbors (default k=5). (2) Randomly select one neighbor x_j. (3) Generate a synthetic sample: x_synthetic = x_i + λ × (x_j - x_i), where λ ~ Uniform(0, 1). The synthetic sample is on the line segment between x_i and x_j — it is a plausible interpolation of two real minority examples. Repeat until the desired oversampling ratio is achieved.

SMOTE variants: SMOTE can create samples in unsafe regions (between minority and majority samples). Variants: Borderline-SMOTE — only oversample minority samples near the decision boundary (borderline samples), ADASYN (Adaptive Synthetic Sampling) — generates more samples in harder-to-learn regions (where density of minority is lower), SVMSMOTE — uses SVM boundary to focus oversampling on borderline samples. These variants improve performance on specific types of imbalance.

When SMOTE helps and when it doesn't: SMOTE helps when the minority class forms coherent, dense clusters — interpolation between real samples creates meaningful new samples. SMOTE hurts when the minority class is scattered in isolated points — interpolation between far-apart minority samples creates unrealistic synthetic samples. Always evaluate SMOTE against class_weight='balanced' to see if it provides additional benefit.

SMOTE with pipelines: CRITICAL — SMOTE must only be applied to training data. If applied before the train/test split, synthetic samples based on test data points leak information. Use imbalanced-learn's Pipeline (not sklearn's) which correctly applies SMOTE only during fit, not during transform/predict.

Preprocessing pipelines

Detailed explanation

A preprocessing pipeline encapsulates all data transformations in a single object that can be fitted once (on training data) and applied consistently to new data. sklearn's Pipeline chains transformers (preprocessors) and an estimator (model) into one object: Pipeline([('scaler', StandardScaler()), ('model', LogisticRegression())]). When you call pipe.fit(X_train, y_train), the scaler is fit on X_train, X_train is transformed, and the model is trained on the transformed data. When you call pipe.predict(X_test), the scaler transforms X_test using the training data statistics, then the model predicts.

ColumnTransformer: handles different preprocessing for different feature types. ColumnTransformer([('numeric', StandardScaler(), numeric_cols), ('categorical', OneHotEncoder(), categorical_cols)]) applies StandardScaler to numeric columns and OneHotEncoder to categorical columns simultaneously. The outputs are concatenated into a single feature matrix. This is the backbone of a production-ready preprocessing pipeline.

Pipeline prevents leakage: the most important benefit. Without a pipeline, fit_transform(X_all) before train_test_split leaks test statistics. With a pipeline, scaler.fit(X_train) is called only on training data — the test data is transformed using training statistics. Pipelines make this correct behavior automatic and non-negotiable.

Pipeline with cross-validation: when using cross_val_score(pipe, X, y, cv=5), each fold correctly: fits the preprocessor on the 4 training folds, transforms the validation fold using those statistics, trains the model, and evaluates on the validation fold. Without a pipeline in cross-validation, you'd need to manually redo fitting/transforming for each fold — error-prone and tedious.

Feature union: FeatureUnion combines multiple transformers in parallel and concatenates their outputs. Useful for combining different feature extraction strategies: FeatureUnion([('tfidf', TfidfVectorizer()), ('sentiment', SentimentExtractor())]) applies both extractors and concatenates the results. FunctionTransformer wraps arbitrary functions as sklearn transformers for inclusion in pipelines.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

    ('cat', categorical_transformer, cat_cols)
])

    ('preprocessor', preprocessor),
    ('model', LogisticRegression(class_weight='balanced', max_iter=1000))
])
full_pipe.fit(X_train, y_train)
print(f'Val AUC: {roc_auc_score(y_val, full_pipe.predict_proba(X_val)[:,1]):.4f}')
# Save the entire pipeline

# Load at inference time

Why this matters in AI
Preprocessing pipelines are the correct, reproducible way to handle feature preprocessing in production. They prevent data leakage, ensure consistent preprocessing between training and inference, and make the entire transformation-prediction workflow inspectable, versionable, and deployable as a single artifact.
Common mistakes
  • Fitting the scaler on all data before the pipeline — pipe = Pipeline([('scaler', StandardScaler()), ('model', LR())]); scaler.fit(X_all); X_scaled = scaler.transform(X_all); pipe.fit(X_train_scaled) — this defeats the purpose of the pipeline.
  • Not saving the fitted pipeline — at inference, you need the same scaler parameters (mean/std) used during training. If you retrain the scaler on production data, it will transform data differently from how the model was trained.
  • Using sklearn's Pipeline with imbalanced-learn's resamplers — sklearn's Pipeline does not call resamplers. Use imblearn.pipeline.Pipeline instead, which supports fit_resample for samplers (SMOTE, RandomUnderSampler).
