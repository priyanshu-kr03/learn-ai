RAG

Detailed explanation

Retrieval-Augmented Generation (RAG) is a technique that combines information retrieval with language model generation. Instead of relying solely on the LLM's parametric knowledge (what it memorized during training), RAG retrieves relevant documents from an external knowledge base at query time and provides them as context in the prompt. The LLM then generates a response grounded in those retrieved documents.

RAG solves two fundamental LLM limitations: knowledge cutoff (the LLM cannot know about events after its training data cutoff) and hallucination (the LLM makes up facts it doesn't know). By providing accurate, up-to-date documents in the prompt, RAG grounds the LLM's response in real information. The LLM's role shifts from knowledge source to reasoning engine — it synthesizes the retrieved documents into a coherent answer.

The RAG pipeline has three stages: indexing (split documents into chunks, embed each chunk, store embeddings in a vector database), retrieval (embed the user query, find top-k most similar chunks from the vector database using approximate nearest-neighbor search), and generation (concatenate retrieved chunks with the query in the LLM prompt, generate a grounded response).

Advanced RAG techniques improve on naive RAG. Hybrid search combines dense vector search with sparse BM25 keyword search — capturing both semantic similarity and keyword relevance. Re-ranking applies a cross-encoder model to re-rank the top-k retrieved chunks by relevance to the query. Query expansion generates multiple query reformulations and retrieves for each, improving recall. Contextual compression extracts only the relevant sentences from each retrieved chunk rather than including the full chunk.

RAG evaluation uses RAGAS (Retrieval Augmented Generation Assessment): faithfulness (does the answer contain only information from retrieved docs?), answer relevance (does the answer actually address the question?), context precision (are the retrieved chunks relevant to the query?), and context recall (do the retrieved chunks contain the information needed to answer?). These four metrics give a comprehensive picture of RAG quality.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

Index: for chunk in split_documents(docs): embedding = embed_model.encode(chunk); vector_store.add(embedding, chunk).
Retrieve: query_emb = embed_model.encode(user_query); top_chunks = vector_store.search(query_emb, k=5).

Call LLM: response = llm.complete(prompt). Return response.
Why this matters in AI
RAG is the most impactful technique for building reliable, grounded LLM applications on private or specialized knowledge. Every enterprise LLM application — internal knowledge bases, document Q&A, code search — uses RAG or a variant.
Common mistakes
  • Not chunking documents — embedding entire documents averages content across the whole document, making retrieval imprecise. Chunk into 256–512 token pieces with 50-token overlap.
  • Retrieving too many chunks — if you retrieve 20 chunks, the prompt may overflow the context window and the model loses precision. Retrieve 3–5 highly relevant chunks.
  • Not evaluating retrieval quality separately from generation quality — if retrieval is bad, generation cannot compensate. Evaluate both independently.

Create 20 test questions with ground truth answers. Measure: retrieval recall (does the correct chunk appear in top-5?), faithfulness (does the generated answer contain only information from retrieved docs?), and answer accuracy.

Document loading

Detailed explanation

Document loading is the first step of a RAG pipeline: reading documents from various sources and formats into a consistent text representation. Different document types require different loaders: PDFs (text extraction, handling multi-column layouts, extracting tables), Word documents (.docx), web pages (HTML parsing), CSV/Excel (tabular data), code files (preserve indentation and structure), and unstructured text files.

LangChain document loaders: PyPDFLoader (text extraction from PDFs), WebBaseLoader (web scraping), UnstructuredFileLoader (handles many formats), DirectoryLoader (load all files from a directory), GitHubLoader (load code from repositories), S3FileLoader (load from AWS S3). Each loader returns Document objects with page_content (extracted text) and metadata (source, page number, last modified date).

PDF challenges: text-based PDFs (created by word processors) extract text reliably. Scanned PDFs (images of text) require OCR. PDFs with complex layouts (multi-column, tables, footnotes, headers/footers) require specialized extraction. pdfplumber handles tables well. pymupdf (fitz) is fast for text extraction. unstructured.io provides the most comprehensive extraction with layout understanding.

Chunking after loading: after loading, documents are typically long (a 100-page PDF = 50,000 words). These must be split into smaller chunks for embedding and retrieval. The key challenge: preserve semantic context at chunk boundaries. Use RecursiveCharacterTextSplitter with sentence-aware splitting, and always carry 50-100 token overlap between consecutive chunks to handle boundary cases.

Metadata preservation: document metadata (source file, page number, section heading, author, date) should be preserved through loading and chunking. When a retriever returns a chunk, you need to know exactly which document, page, and section it came from — for citation generation, quality auditing, and debugging.

Why this matters in AI
Document loading is where most RAG failures originate. Poor extraction (garbled text from OCR, missing tables, lost structure) means the subsequent retrieval and generation steps are working with incomplete or corrupted information. Robust document loading is the foundation of reliable RAG.
Common mistakes
  • Using PyPDFLoader for scanned PDFs — PyPDF extracts text from text-based PDFs but produces empty strings for scanned PDFs (which are images). Use unstructured.io or pytesseract for OCR-based extraction.
  • Not preserving metadata during chunking — when splitting a 100-page PDF into 500 chunks, each chunk must know which page it came from. Pass documents (with metadata) to text_splitter.split_documents(), not raw strings to split_text().
  • Loading the entire document as one chunk — passing a 50,000-word document to the embedding model (which has a 512-token limit) results in truncation. The embedding represents only the first 512 tokens, missing all subsequent content. Always split before embedding.

Chunking

Detailed explanation

Chunking is the process of splitting long documents into smaller segments (chunks) before embedding and indexing them in a vector store. This is necessary because: (1) embedding models have context window limits (typically 512 tokens), (2) smaller chunks provide more precise retrieval (a 100-word chunk about 'attention mechanism' will rank higher for an attention query than a 5000-word chapter that mentions attention once), (3) the LLM context window limits how much retrieved text can be included in the prompt.

Fixed-size chunking: split every n tokens (or characters) with an optional overlap. Pros: simple, predictable chunk sizes. Cons: may split in the middle of a sentence, paragraph, or concept. overlap=50 tokens means consecutive chunks share 50 tokens — this ensures continuity at chunk boundaries and prevents important context from being split across chunks. Example: chunk_size=256 tokens, overlap=32 tokens.

Semantic chunking: split at natural semantic boundaries — sentence boundaries, paragraph breaks, section headings. spaCy's sentencizer splits at sentence boundaries. Markdown splitter splits at ## headings. HTML splitter splits at tags. These produce variable-length chunks that preserve semantic units. The trade-off: some chunks may be too long for the embedding model or too short to be meaningful.

Recursive character text splitter (LangChain default): tries to split at the most natural boundary available, in order of preference: \n\n (paragraph), \n (newline), '. ' (sentence), ' ' (word), '' (character). This produces chunks that respect document structure while enforcing a maximum size.

Optimal chunk size: depends on the use case. For dense factual documents (legal, medical): 128-256 tokens (precise retrieval of specific facts). For narrative documents (books, articles): 512-1024 tokens (preserves context). For code: split by function/class (variable size). Always evaluate retrieval quality at multiple chunk sizes — there is no single optimal size. Rule of thumb: match chunk size to the expected length of the relevant passage for your queries.

from langchain.text_splitter import RecursiveCharacterTextSplitter
# Recursive splitter — preserves semantic structure
splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,       # max characters per chunk
    chunk_overlap=64,     # overlap for continuity at boundaries
    length_function=len,  # or use tiktoken for token counting
    separators=['\n\n', '\n', '. ', ' ', '']
)
document = open('my_document.txt').read()
chunks = splitter.split_text(document)
print(f'{len(chunks)} chunks, avg length: {sum(len(c) for c in chunks)/len(chunks):.0f} chars')
# Token-based splitting with tiktoken (for OpenAI embedding limits)
import tiktoken
from langchain.text_splitter import RecursiveCharacterTextSplitter
enc = tiktoken.get_encoding('cl100k_base')
def token_count(text): return len(enc.encode(text))
splitter = RecursiveCharacterTextSplitter(
    chunk_size=256, chunk_overlap=32, length_function=token_count,
)
chunks = splitter.split_documents(documents)  # preserves metadata
# Each chunk: {page_content: '...', metadata: {source: 'file.pdf', page: 5}}
Why this matters in AI
Chunking strategy directly affects retrieval quality in RAG systems. Poor chunking (too large: imprecise retrieval; too small: loses context; splitting mid-sentence: incoherent chunks) is one of the most common causes of poor RAG performance. Thoughtful chunking is as important as embedding model choice.
Common mistakes
  • Using fixed character-based chunking without overlap — chunk at exactly 1000 characters means the last sentence of one chunk and first sentence of the next chunk are separated. A query spanning the boundary won't find either chunk. Use overlap of 10-20% of chunk size.
  • Using the same chunk size for all document types — code should be chunked by function, markdown by section heading, legal documents by clause. Use document-type-aware chunking.
  • Not storing metadata with chunks — when a chunk is retrieved, you need to know: which document it came from, which page, what section. This is needed for citations, context expansion, and debugging. Always preserve and store metadata.

Indexing

Detailed explanation

Indexing in the context of RAG systems is the process of computing embeddings for all document chunks and storing them in a vector database for fast similarity search. It is the offline step that prepares the knowledge base for online retrieval. Indexing is typically done once (or periodically when documents are updated), while retrieval happens at query time.

Indexing pipeline: (1) load documents from sources, (2) split into chunks (chunk_size=512 tokens, overlap=64 tokens), (3) compute embeddings for each chunk using a sentence encoder (all-MiniLM-L6-v2, text-embedding-3-small), (4) store (embedding vector, chunk text, metadata) in a vector database (Chroma, FAISS, Pinecone, Weaviate). Total cost: embedding computation (API cost or GPU time) + storage cost.

Incremental indexing: for large, frequently updated document collections, re-indexing everything from scratch is expensive. Implement incremental indexing: hash each document's content, compare with the stored hash, only re-embed documents that have changed. This reduces indexing cost by 90%+ for collections where most documents are stable.

Index management: vector databases must be maintained. Operations: add (new documents), update (re-embed modified documents), delete (remove outdated documents). Most vector databases support these operations. Implement a document registry: store document_id, content_hash, last_indexed_at. Compare with current document state before indexing.

Indexing efficiency: embedding 100,000 chunks with a local model on GPU can be parallelized with batching. sentence-transformers encode() accepts a list of texts and computes embeddings in batches (batch_size=64 by default). For 100K chunks: batch encoding takes ~5 minutes on GPU vs ~30 minutes single-threaded. For production scale (millions of chunks): use distributed embedding pipelines (Apache Beam, Ray) or commercial APIs with high throughput.

Worked example

from langchain_community.vectorstores import Chroma
from langchain_community.embeddings import HuggingFaceEmbeddings
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.document_loaders import DirectoryLoader, PyPDFLoader
# Full indexing pipeline
loader = DirectoryLoader('./docs', glob='**/*.pdf', loader_cls=PyPDFLoader)
documents = loader.load()
splitter = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=64)
chunks = splitter.split_documents(documents)
print(f'Indexing {len(chunks)} chunks...')
embeddings = HuggingFaceEmbeddings(model_name='all-MiniLM-L6-v2', encode_kwargs={'batch_size': 64})
vectorstore = Chroma.from_documents(
    documents=chunks, embedding=embeddings,
    persist_directory='./chroma_db',
    collection_name='my_documents'
)
vectorstore.persist()
print(f'Indexed {vectorstore._collection.count()} chunks')
# Incremental indexing
import hashlib, json
def get_doc_hash(doc_path):
    return hashlib.sha256(open(doc_path, 'rb').read()).hexdigest()
registry_path = 'index_registry.json'
try:
    registry = json.load(open(registry_path))
except FileNotFoundError:
    registry = {}
def needs_reindexing(doc_path):
    current_hash = get_doc_hash(doc_path)
    return registry.get(doc_path) != current_hash
# Only re-index changed files
docs_to_index = [f for f in all_doc_paths if needs_reindexing(f)]
print(f'{len(docs_to_index)} docs need re-indexing')
Why this matters in AI
Indexing quality determines RAG retrieval quality. A well-indexed knowledge base with appropriate chunking and embeddings provides relevant context for every query. A poorly indexed knowledge base (wrong chunking, low-quality embeddings, outdated documents) causes the retriever to return irrelevant or stale content, no matter how good the LLM.
Common mistakes
  • Re-indexing everything on every document update — for a 10K document collection, full re-indexing takes 30+ minutes and costs significant API money. Implement content hashing and incremental indexing.
  • Not testing retrieval quality after indexing — after building the index, run 20 test queries and manually check the top-5 retrieved chunks. Are they relevant? If not, the chunking or embedding strategy needs adjustment.
  • Mixing documents from different languages in one index without language filtering — embedding models optimized for English produce poor embeddings for non-English text. Either use a multilingual model or create separate indices per language.

Retrieval

Detailed explanation

Retrieval is the process of finding relevant context from a knowledge base given a query, as the 'R' in RAG (Retrieval Augmented Generation). The retrieved context is provided to the LLM alongside the user's question so the LLM can answer with up-to-date, specific, and grounded information rather than relying solely on its training data.

Dense retrieval (vector search): embed the query and find the most similar document chunks by cosine similarity of embeddings. Captures semantic similarity — 'what is the capital of France?' retrieves 'Paris is the capital of France.' even if the exact words 'capital of France' don't appear. Powered by bi-encoder models + FAISS/vector database.

Sparse retrieval (BM25/TF-IDF): classical term-frequency based retrieval. BM25 scores documents by the frequency of query terms, with saturation (diminishing returns for term repetition) and length normalization. Strong for: exact term matches, rare proper nouns, technical terminology, code identifiers. Returns zero for documents with no query term overlap.

Hybrid retrieval: combine dense and sparse scores using Reciprocal Rank Fusion (RRF) or a learned linear combination. Most production RAG systems use hybrid retrieval to get both semantic understanding (dense) and exact match (sparse). OpenSearch, Weaviate, Chroma, and LangChain all support hybrid retrieval.

Retrieval quality metrics: Context Recall (what fraction of the ground truth facts can be found in the retrieved context?), Context Precision (what fraction of the retrieved context is relevant to the question?). These are measured by the RAGAS framework. High context recall means the retriever is finding all relevant information. High context precision means it is not flooding the LLM with irrelevant context.

Worked example

from langchain_community.vectorstores import Chroma
from langchain_community.embeddings import HuggingFaceEmbeddings
from langchain.text_splitter import RecursiveCharacterTextSplitter
embeddings = HuggingFaceEmbeddings(model_name='all-MiniLM-L6-v2')
splitter = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=64)
docs = splitter.split_documents(loaded_docs)  # from document loaders
vectorstore = Chroma.from_documents(docs, embeddings, persist_directory='./chroma_db')
# Retrieve top-5 most relevant chunks
retriever = vectorstore.as_retriever(search_type='similarity', search_kwargs={'k': 5})
results = retriever.invoke('What is the attention mechanism?')
for doc in results:
    print(doc.page_content[:200], '\n---')
# MMR retrieval: diverse results (Maximum Marginal Relevance)
retriever_mmr = vectorstore.as_retriever(
    search_type='mmr', search_kwargs={'k': 5, 'fetch_k': 20, 'lambda_mult': 0.5}
)
# lambda_mult: 0=max diversity, 1=max relevance
results_mmr = retriever_mmr.invoke('What is the attention mechanism?')
# MMR returns 5 diverse results from the 20 most similar
Why this matters in AI
Retrieval quality is the most critical factor in RAG system performance. A perfect LLM cannot compensate for poor retrieval — if the relevant information is not in the retrieved context, the LLM will either hallucinate or say 'I don't know'. Improving retrieval (better embeddings, chunking, hybrid search) typically provides more improvement than upgrading the LLM.
Common mistakes
  • Retrieving too few chunks — with k=3, the retriever may miss relevant context. Start with k=5-10 and measure context recall. More chunks increase context recall at the cost of more noise in the prompt.
  • Not using MMR for diverse queries — with standard similarity retrieval, the top-5 results may be near-duplicate chunks from the same section. MMR ensures diversity by penalizing redundant results.
  • Not evaluating retrieval separately from generation — always measure context recall@k before optimizing generation. A retrieval recall of 60% creates a hard ceiling on answer accuracy that no amount of prompt engineering can overcome.

Query rewriting

Detailed explanation

Query rewriting transforms the user's original query into one or more better queries for retrieval. Users often ask questions in a way that is ambiguous ('how does it work?'), uses pronouns ('tell me more about it'), lacks context (follow-up questions in a conversation), or is too long/complex. Query rewriting improves retrieval precision by generating queries that better match the language and style of the indexed documents.

HyDE (Hypothetical Document Embeddings): instead of embedding the query directly, use an LLM to generate a hypothetical document that would answer the query ('What would a document that answers this question look like?'), then embed that hypothetical document. The idea: the hypothetical document's embedding is closer to relevant real documents than the sparse query's embedding. For technical queries with sparse vocabulary, HyDE often significantly improves retrieval.

Multi-query retrieval: use an LLM to generate 3-5 alternative phrasings of the query, retrieve results for each, and merge the result sets. Example: query='how does attention work?' → LLM generates: 'attention mechanism in transformers', 'self-attention explanation', 'query key value computation'. Retrieve for each and union the results. This increases recall at the cost of more retrieval calls.

Step-back prompting: for complex multi-hop questions, generate a more general 'step-back' question first. 'What were the side effects of Drug X in the 2020 trial?' → step-back: 'What are the trial results for Drug X?'. Retrieve for the general question first, then the specific. This helps when documents don't directly match the specific query.

Contextual compression: instead of just retrieving full chunks, have an LLM extract only the relevant portion of each retrieved chunk. A 500-token chunk retrieved for 'what is the learning rate?' may be mostly irrelevant — the LLM extracts only the 2 sentences about learning rate. This reduces prompt noise and improves answer accuracy.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from langchain.retrievers import MultiQueryRetriever
from langchain_openai import ChatOpenAI
# Multi-query retriever
llm = ChatOpenAI(model='gpt-4o-mini', temperature=0)
multi_retriever = MultiQueryRetriever.from_llm(
    retriever=vectorstore.as_retriever(search_kwargs={'k': 5}),
    llm=llm
)
# Generates 3 alternative queries, retrieves for each, deduplicates
results = multi_retriever.invoke('What is the attention mechanism?')
print(f'{len(results)} unique documents retrieved')
# HyDE: generate a hypothetical document, embed it
from langchain.prompts import PromptTemplate
hyde_prompt = PromptTemplate.from_template(
    'Write a technical paragraph that would answer this question: {question}'
)
hyde_chain = hyde_prompt | ChatOpenAI(model='gpt-4o-mini') 
hypothetical_doc = hyde_chain.invoke({'question': 'How does multi-head attention work?'})
hyde_embedding = embeddings.embed_query(hypothetical_doc.content)
# Search with hypothetical doc embedding instead of query embedding
results = vectorstore.similarity_search_by_vector(hyde_embedding, k=5)
Why this matters in AI
Query rewriting is one of the highest-ROI improvements for RAG systems. Users rarely phrase queries in a way that exactly matches indexed document language. Multi-query retrieval alone can improve context recall by 10-20% with minimal additional cost. HyDE is particularly effective for technical queries where the question uses different vocabulary than the answer.
Common mistakes
  • Generating too many alternative queries — 3-5 is optimal. 10+ queries means more LLM calls (latency + cost) with diminishing recall improvement as queries become redundant.
  • Not deduplicating multi-query results — when merging results from 5 queries, the same chunk may appear multiple times. Deduplicate by document ID or content hash before passing to the LLM.
  • Applying HyDE to simple, direct queries — HyDE adds an extra LLM call (generate hypothetical doc). For simple queries like 'what is an API key?', direct embedding works just as well. Use HyDE for complex, technical, or terminology-heavy queries.

Reranking

Detailed explanation

Reranking is the second stage of a two-stage retrieval pipeline: after the bi-encoder retrieves a candidate set of 50-100 documents quickly, a cross-encoder reranker rescores each document against the query with high accuracy. Reranking is critical because bi-encoder retrieval misses fine-grained relevance — it can retrieve semantically similar but not specifically relevant documents. The cross-encoder sees both query and document simultaneously, enabling much more precise relevance scoring.

Cross-encoder reranker: takes (query, document) pair and outputs a relevance score. Models like ms-marco-MiniLM-L-6-v2 are trained on the MS MARCO passage ranking dataset with 500K (query, relevant passage, non-relevant passage) triples. At inference: for each of the k retrieved candidates, run one forward pass and get a relevance score. Sort by score, take the top-n. Cost: k cross-encoder forward passes per query.

Cohere Rerank API: a commercial API endpoint for reranking. Faster to integrate than running a local cross-encoder. Input: query + list of documents. Output: relevance scores. Cohere Rerank 3 is state-of-the-art. Also available: Jina Reranker, VoyageAI Reranker, mixedbread.ai rerankers.

When reranking helps most: (1) when bi-encoder retrieval brings in many topically similar but not specifically relevant chunks, (2) for multi-hop questions where the relevant chunk discusses related concepts but uses different vocabulary, (3) for domain-specific retrieval where a general bi-encoder does not distinguish fine-grained domain relevance. When reranking adds cost without much benefit: simple factual queries where bi-encoder precision is already high.

Reranking in a full RAG pipeline: retrieve top-50 with bi-encoder (fast), rerank to top-5 with cross-encoder (accurate), use top-5 as context for LLM generation. The 50→5 funnel reduces prompt noise while the cross-encoder ensures the 5 best documents are selected. This architecture gives near-optimal retrieval accuracy with manageable latency (~100ms for cross-encoder over 50 docs).

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from sentence_transformers import CrossEncoder
reranker = CrossEncoder('cross-encoder/ms-marco-MiniLM-L-6-v2')
# After bi-encoder retrieval: top-50 candidates
candidate_docs = vectorstore.similarity_search(query, k=50)
# Score each with cross-encoder
pairs = [(query, doc.page_content) for doc in candidate_docs]
scores = reranker.predict(pairs)
# Sort by score, take top-5
import numpy as np
top_5_idx = np.argsort(scores)[::-1][:5]
reranked_docs = [candidate_docs[i] for i in top_5_idx]
print(f'Top-5 after reranking:')
for i, doc in enumerate(reranked_docs):
    print(f'{i+1}. [{scores[top_5_idx[i]]:.3f}] {doc.page_content[:100]}')
# Cohere Rerank (API)
import cohere
co = cohere.Client('your-api-key')
results = co.rerank(
    model='rerank-english-v3.0',
    query=query,
    documents=[doc.page_content for doc in candidate_docs],
    top_n=5
)
reranked_docs = [candidate_docs[r.index] for r in results.results]
Why this matters in AI
Reranking provides the most cost-effective accuracy improvement for RAG systems. A cross-encoder can distinguish between 'BERT uses attention' (relevant to a question about attention) and 'BERT's training used attention to mask tokens' (less directly relevant) — the bi-encoder would score both similarly. Adding reranking typically improves end-to-end answer accuracy by 5-15%.
Common mistakes
  • Retrieving too few candidates for the reranker — reranking only improves ranking within the candidate set. If the relevant document is not in the top-50 retrieved by the bi-encoder, the reranker cannot help. Always retrieve 5-10× more candidates than you need after reranking.
  • Using a general reranker for a specialized domain — a reranker trained on general web search (MS MARCO) may not correctly assess relevance for medical, legal, or code retrieval. Fine-tune a reranker on domain-specific data if available.
  • Not measuring reranker latency — cross-encoder inference is O(k) per query. For k=50 and a 6-layer cross-encoder, this adds ~50-100ms latency. Measure and budget for this in your system's SLA.

Context construction

Detailed explanation

Context construction is the process of assembling the retrieved chunks into the context window provided to the LLM for generation. This is not just concatenation — how you format, order, and select chunks significantly affects the quality of LLM responses. The goal: provide all necessary information in a format the LLM can effectively use, without exceeding the context window limit.

Context window budget: modern LLMs have large context windows (GPT-4: 128K, Claude 3: 200K) but using more context costs more tokens (money + latency). For every query, decide: how many chunks (typically 3-10), how to format them (with headers? document titles? page numbers?), and whether to compress them (extract only relevant sentences).

Lost in the middle effect: LLMs pay more attention to information at the beginning and end of the context window than information in the middle. If you have 10 retrieved chunks, put the most relevant chunk first and second-most-relevant last, with less relevant chunks in the middle. Alternatively, use contextual compression to trim less relevant chunks before inclusion.

Context formatting: include metadata with each chunk — document title, page number, section heading. Example: '---\nSource: Annual Report 2023, Section 4: Revenue\nPage: 47\n---\n[chunk text]'. This helps the LLM attribute information to sources and enables citation generation.

Context deduplication: when using multi-query retrieval, the same chunk may appear multiple times in the retrieved set. Deduplicate by content hash or source+position before constructing the context. Including the same chunk twice wastes context budget and can confuse the LLM.

def build_context(retrieved_docs, max_tokens=2000):
    """Assemble retrieved docs into a formatted context string."""
    context_parts = []
    token_count = 0
    for i, doc in enumerate(retrieved_docs):
        header = f'---\nSource {i+1}: {doc.metadata.get("source", "unknown")}\nPage: {doc.metadata.get("page", "N/A")}\n---'
        text = doc.page_content
        part_tokens = len(text.split()) * 1.3  # rough token estimate
        if token_count + part_tokens > max_tokens:
            break
        context_parts.append(f'{header}\n{text}')
        token_count += part_tokens
    return '\n\n'.join(context_parts)

prompt = ChatPromptTemplate.from_template(
    'Answer the question based only on the following context.\n\n'
    'Context:\n{context}\n\n'
    'Question: {question}\n\n'
    'Answer: If the context does not contain the answer, say "I cannot find this information in the provided context."'
)
chain = ({'context': retriever | format_docs, 'question': RunnablePassthrough()} | prompt | llm | StrOutputParser())

Why this matters in AI
Context construction is where retrieval meets generation. Even with perfect retrieval, a poorly constructed prompt (wrong ordering due to lost-in-the-middle, missing metadata, redundant chunks, truncated chunks) can degrade answer quality. Thoughtful context construction extracts maximum value from retrieved information.
Common mistakes
  • Truncating chunks at a hard token limit mid-sentence — the last retrieved chunk may be cut off in the middle of a sentence, making it incoherent. Truncate at sentence boundaries or use a soft limit (include the chunk if adding it doesn't exceed 110% of the limit).
  • Not including source metadata — the LLM cannot provide citations if it doesn't know which document each chunk came from. Always include source, page, and section metadata.
  • Including too many redundant chunks — 10 chunks from the same section all saying slightly different things about the same topic overwhelm the LLM with redundant context. Use MMR retrieval or cross-encoder reranking to select diverse, relevant chunks.

Grounded generation

Detailed explanation

Grounded generation is the process of generating LLM responses that are strictly based on the provided context, with citations to the source material. The goal is to prevent hallucination: every factual claim in the response should be traceable to a specific retrieved chunk. Grounded generation is the 'G' part of RAG — not just Retrieval Augmented Generation, but strictly grounded generation.

Prompt engineering for grounding: the system prompt must explicitly instruct the LLM to: (1) only use information from the provided context, (2) say 'I cannot find this information in the context' if the answer is not present, (3) cite specific sources for each claim. Without these instructions, LLMs will supplement context information with training knowledge, producing accurate-sounding but potentially hallucinated responses.

Citation generation: ask the LLM to output not just the answer but also the specific chunk number (Source 1, Source 3) that supports each statement. Post-process the response to hyperlink citations to the original document. This makes responses verifiable — users can check each claim against the source.

Faithfulness evaluation: RAGAS's Faithfulness metric measures whether every claim in the generated response can be traced to the retrieved context. LLM-as-a-judge approach: ask an LLM to decompose the answer into individual claims, then verify each claim against the context. A faithfulness score of 0.9 means 90% of claims in the response are supported by the context.

Anti-hallucination techniques: (1) lower temperature (temperature=0 for factual queries), (2) instruct to say 'I don't know' rather than guess, (3) use structured output with required 'sources' field, (4) implement post-generation verification: automatically check each claim in the response against the retrieved chunks, flag unsupported claims.

Why this matters in AI
Grounded generation is what makes RAG trustworthy in high-stakes applications: customer support, legal research, medical information, financial analysis. Without strict grounding, an RAG system might retrieve the correct document but then generate a response that mixes retrieved facts with hallucinated details. Citation support makes every claim verifiable.
Common mistakes
  • Not instructing the LLM to cite sources — without explicit citation instructions, LLMs blend context and training knowledge seamlessly, making it impossible to distinguish grounded from hallucinated statements.
  • Using high temperature for factual queries — temperature > 0.3 increases creative generation but also increases hallucination. For factual RAG queries, use temperature=0.
  • Not distinguishing 'I don't know' from a wrong answer — an LLM that says 'I cannot find this in the context' is more trustworthy than one that confidently hallucinates. Reward the model for appropriate uncertainty.

Citations

Detailed explanation

Citations in RAG systems let users verify that each claim in the generated response is grounded in a specific retrieved document. Citations transform RAG from a 'trust me' system into a verifiable one: instead of 'the contract renewal fee is $500' you get 'the contract renewal fee is $500 [Source: MSA_2024.pdf, Section 3.2, Page 7]'. This is essential for regulated industries and for building user trust.

Citation generation strategies: (1) source attribution in prompt — include source labels in the context and instruct the model to cite them: 'Use [Source 1], [Source 2] in your response to cite specific claims'. (2) Post-hoc citation matching — generate the response, then match each claim to the most similar retrieved chunk using semantic search. (3) Structured output — require the model to return JSON with claims and their source_ids separately, then render with citations.

Source labeling: format each retrieved chunk with a clear source label: '--- [Source 3: Annual Report 2023, Section Revenue, Page 47] ---\n[chunk text]'. The model can then reference [Source 3] in its response. Include enough metadata for the user to find the original document: file name, page, section heading.

Citation rendering: map citation references ([Source 3]) to clickable links or expandable quotes in the UI. When a user clicks [Source 3], show the relevant chunk highlighted in the original document. This makes the system fully auditable.

Citation accuracy evaluation: automatic evaluation of citation accuracy — does each citation actually support the claim? — requires an LLM to judge. For each (claim, cited chunk) pair, ask a judge model: 'Does this document excerpt support this claim?' Return a faithfulness score. Low scores mean the model is citing irrelevant chunks or generating claims not in any cited source.

import re
def parse_citations(response_text, retrieved_docs):
    citations = re.findall(r'\[Source (\d+)\]', response_text)
    cited_sources = []
    for c in set(citations):
        idx = int(c) - 1
        if 0 <= idx < len(retrieved_docs):
            cited_sources.append({'citation_num': c, 'source': retrieved_docs[idx].metadata.get('source'), 'page': retrieved_docs[idx].metadata.get('page'), 'text': retrieved_docs[idx].page_content[:200]})
    return cited_sources
Why this matters in AI
Citations are the accountability layer of RAG systems. In high-stakes domains — legal, medical, financial — every claim must be verifiable. Citations also help users build trust incrementally: seeing that a claim is supported by a specific page in a specific document is far more convincing than a confident-sounding response with no backing.
Common mistakes
  • Generating citations without verifying they support the claim — the model may cite [Source 3] for a claim that appears in a completely different chunk. Always build an automated citation accuracy checker.
  • Using citation numbers that don't match the context — if the context says '[Source 1]' but the model generates '[Document 3]', citations don't match. Standardize the citation format in both the context and the instruction.
  • Not including enough metadata in the source label — '[Source 3]' is not useful if the user doesn't know what document it is. Include file name, page number, section title.

RAG evaluation

Detailed explanation

RAG evaluation requires measuring both the retrieval component and the generation component independently, as failures can occur at either stage. The key metrics: Context Recall, Context Precision (retrieval quality), Faithfulness, Answer Relevancy, Answer Correctness (generation quality). The RAGAS framework provides automated LLM-based evaluation for all of these.

Context Recall: what fraction of the ground truth answer's information can be found in the retrieved context? Context Recall = 1.0 means all the information needed to answer is in the retrieved chunks. Context Recall < 0.8 indicates the retriever is missing important information — improving chunking, embedding, or retrieval strategy will help.

Context Precision: what fraction of the retrieved chunks are actually relevant to the question? Context Precision = 1.0 means every retrieved chunk is relevant. Context Precision < 0.7 means the retriever is bringing in too much noise — the LLM must sift through irrelevant context, increasing the chance of mistakes.

Faithfulness: what fraction of the generated answer's claims are supported by the retrieved context? Faithfulness < 1.0 indicates hallucination — the model is adding information not present in the context. For high-stakes applications, target Faithfulness > 0.95.

Answer Correctness: how similar is the generated answer to the ground truth answer? Computed as a combination of semantic similarity and factual overlap. This is the end-to-end quality metric that combines retrieval and generation quality. Build a test set of 50-100 (question, ground_truth_answer) pairs for reliable evaluation.

RAG failure modes

Detailed explanation

RAG systems fail in predictable, diagnosable ways. Understanding the failure modes helps you build more robust systems and debug effectively when RAG output quality is poor. The major failure modes map to different stages of the pipeline: document ingestion, retrieval, and generation.

Failure mode 1 — Retrieval failure (low context recall): the correct information is in the knowledge base but the retriever fails to retrieve it. Causes: chunking strategy splits the relevant information across chunk boundaries, embedding model produces poor representations for domain-specific terminology, query vocabulary doesn't match document vocabulary. Fixes: better chunking (overlap, semantic splitting), domain-specific embedding models, query rewriting (multi-query, HyDE).

Failure mode 2 — Context contamination (low context precision): the retriever returns irrelevant chunks alongside relevant ones, confusing the LLM. Causes: dense retrieval returns topically related but not specifically relevant chunks, embedding model conflates similar topics. Fixes: reranking (cross-encoder), metadata filtering (filter by date, document type), hybrid search (BM25 + dense for exact match precision).

Failure mode 3 — Hallucination despite context: the LLM generates claims not supported by the retrieved context. Causes: insufficient grounding instructions in the system prompt, LLM defaulting to training knowledge when context is unclear, context window too long (lost in the middle effect). Fixes: stronger grounding instructions ('ONLY use the provided context'), shorter context (reranking to top-3), faithfulness evaluation with automated checking.

Failure mode 4 — Stale knowledge: retrieved documents are outdated, and the system returns outdated information confidently. Fix: include document date in metadata, filter by recency, display document date alongside citations. Failure mode 5 — Context window overflow: the retrieved context + prompt exceeds the LLM's context limit, causing truncation. Fix: contextual compression (extract only relevant sentences from each chunk), aggressive reranking to top-5.

# Diagnostic framework: test each failure mode
def diagnose_rag_failure(question, retrieved_docs, generated_answer, ground_truth):
    diagnostics = {}
    # Test 1: Is the answer in the retrieved context?
    context_text = ' '.join(d.page_content for d in retrieved_docs)
    answer_in_context = any(phrase in context_text for phrase in ground_truth.split()[:5])
    diagnostics['retrieval_failure'] = not answer_in_context
    # Test 2: Is the answer faithful to context?
    if generated_answer and context_text:
        # Use LLM to check faithfulness
        faithfulness_check = check_faithfulness_with_llm(generated_answer, context_text)
        diagnostics['hallucination'] = faithfulness_check < 0.8
    # Test 3: Did truncation occur?
    total_context_length = sum(len(d.page_content) for d in retrieved_docs)
    diagnostics['context_overflow'] = total_context_length > 100000
    return diagnostics
Why this matters in AI
RAG systems are complex pipelines where failures can originate at any stage. Without a systematic understanding of failure modes, debugging is guesswork. Knowing the 5 major failure modes turns debugging into a structured diagnostic process: measure context recall (failure mode 1?), context precision (failure mode 2?), faithfulness (failure mode 3?).
Common mistakes
  • Blaming the LLM for failures that are actually retrieval failures — if the retrieved context doesn't contain the answer, the LLM cannot give a correct answer even with a perfect prompt. Always diagnose retrieval quality before changing the generation prompt.
  • Not building an automated failure detection system — manually checking every response for failures doesn't scale. Implement automated scoring: RAGAS for all 5 metrics, running daily against a test set of known Q&A pairs.
  • Treating all failures equally — a hallucination failure is more serious than a retrieval failure: the system is confidently wrong. Prioritize reducing hallucination (failure mode 3) before optimizing retrieval recall (failure mode 1).
