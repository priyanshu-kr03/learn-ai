PyTorch tensors

Detailed explanation

A tensor is the fundamental data structure in PyTorch. It is a multi-dimensional array of numbers, similar to a NumPy ndarray, but with two superpowers: it can live on a GPU and it can track gradient computations automatically. Everything in PyTorch — inputs, weights, biases, activations, losses — is a tensor.

Tensors are described by three properties: dtype (the data type of elements, like float32, int64, bool), shape (the size in each dimension, like [32, 3, 224, 224] for a batch of 32 RGB images of 224×224 pixels), and device (cpu or cuda:0). A mismatch in any of these is one of the most common sources of PyTorch errors.

You create tensors with torch.tensor() from Python lists, torch.zeros(), torch.ones(), torch.randn() for random values, or torch.from_numpy() to wrap a NumPy array. Operations on tensors are identical to NumPy: you can add, multiply, reshape, slice, concatenate, and more. Most NumPy operations have a direct PyTorch equivalent.

The key difference from NumPy is requires_grad. When you set requires_grad=True on a tensor, PyTorch records every operation applied to it in a computation graph. When you later call .backward() on a scalar output, PyTorch walks backward through this graph and computes gradients for every tensor that has requires_grad=True. Model parameters always have requires_grad=True; input data usually does not.

Tensor shape management is the single biggest source of bugs in PyTorch. A 2D tensor of shape [32, 128] and a 1D tensor of shape [128] behave differently in matrix multiplication. Use .shape to check, .view() or .reshape() to change shape, .unsqueeze(dim) to add a dimension, and .squeeze(dim) to remove a size-1 dimension. Print shapes at every step when debugging.

Broadcasting in PyTorch works like NumPy: when two tensors have compatible shapes, operations are applied element-wise with automatic expansion along size-1 dimensions. [32, 1] + [32, 128] broadcasts the first tensor to [32, 128] before adding. This is very useful but can produce silent wrong results if shapes are unexpectedly compatible.

Why this matters in AI
Everything in deep learning is a tensor. Wrong shapes, wrong dtypes, and wrong devices cause cryptic errors. Fluency with tensors is the first skill that separates a working PyTorch implementation from one that breaks mysteriously.
Common mistakes
  • Confusing .view() and .reshape() — .view() requires contiguous memory; .reshape() works regardless. When in doubt, use .reshape().
  • Forgetting .detach() when converting a gradient-tracked tensor to NumPy — PyTorch raises an error if you try to convert a requires_grad tensor directly.
  • Creating tensors from Python lists of mixed types — PyTorch infers dtype from the list; mixed int/float lists default to float64, not float32, which may mismatch model weights.

Autograd

Detailed explanation

Autograd is PyTorch's automatic differentiation engine. When you perform operations on tensors that have requires_grad=True, PyTorch silently builds a directed acyclic computation graph recording every operation. When you call .backward() on a scalar (usually the loss), PyTorch traverses this graph in reverse, applying the chain rule at each node, and deposits the computed gradients into the .grad attribute of each leaf tensor (the original parameters).

A leaf tensor is one that was not produced by any operation — it is a tensor you created directly, like a model weight. Intermediate tensors (activations, losses computed during forward pass) are non-leaf. Only leaf tensors accumulate gradients in .grad. This is important: printing .grad on an intermediate tensor returns None unless you call .retain_grad() on it first.

The computation graph is rebuilt fresh every time you do a forward pass. This is called dynamic computation graphs (define-by-run). Every time your training loop runs forward, a new graph is constructed. This means your model can include Python if-statements, for-loops, and dynamic shapes — the graph follows whatever the Python code actually executes. TensorFlow 1.x required you to define the static graph before running; PyTorch's dynamic approach is much more flexible for research.

Gradient accumulation is a feature, not a bug — gradients add up in .grad across multiple backward() calls. This is intentional when you want to simulate larger batches than fit in memory (accumulate gradients over N batches, then step). But if you forget to zero gradients between normal training steps, gradients from past batches corrupt current updates. Always call optimizer.zero_grad() (or zero out manually) before each backward() call.

torch.no_grad() is a context manager that disables gradient tracking inside its block. Use it for validation, inference, and any computation where you do not need gradients. It saves memory (no graph is built) and speeds up computation. model.eval() alone does not disable autograd — you need both model.eval() and torch.no_grad() during inference.

Gradients of non-scalar tensors: calling .backward() on a non-scalar tensor requires passing a gradient tensor of the same shape as the upstream gradient. When your loss is always a scalar (which it should be after calling .mean() or .sum()), this is handled automatically. Never call loss.backward() on a non-reduced loss tensor without thinking carefully about what gradient you are passing.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

x = torch.tensor(3.0, requires_grad=True). y = x**2 + 2*x + 1. Graph built: y depends on x via squaring and linear ops.
y.backward(). Gradient: dy/dx = 2x + 2 = 2*3 + 2 = 8. Check: x.grad → tensor(8.0).
Accumulation: y.backward() again without zeroing → x.grad → tensor(16.0) (doubled!).
Zero grad: x.grad.zero_() then y = x**2 + 2*x + 1 → y.backward() → x.grad = tensor(8.0) again.
No grad inference: with torch.no_grad(): pred = model(x_test). No graph built, fast, no memory overhead.
Why this matters in AI
Autograd is the engine that makes all of deep learning work automatically. Understanding it prevents the most common training bug (gradient accumulation) and lets you write custom differentiable operations.
Common mistakes
  • Forgetting optimizer.zero_grad() before backward() — the most common PyTorch training bug, causing gradients to double or triple on each step.
  • Printing intermediate tensor values inside a training loop using Python float() or .item() — these detach from the graph, which is fine for logging but confusing if you think you are computing gradients through them.
  • Using .numpy() on a requires_grad tensor — raises RuntimeError. Must call .detach().numpy() or .detach().cpu().numpy() for GPU tensors.

nn.Module

Detailed explanation

nn.Module is the base class for all neural networks in PyTorch. When you create a model, you subclass nn.Module, define your layers in __init__, and define the forward pass in the forward() method. PyTorch automatically handles parameter registration, device movement, training/eval mode, and serialization for any class that subclasses nn.Module.

In __init__, every layer assigned as an attribute is automatically registered as a submodule with its parameters tracked. This is why you must use nn.Linear(64, 32) rather than a plain weight tensor — nn.Linear knows its parameters are trainable and registers them. If you store a plain list of layers, use nn.ModuleList to ensure they are properly registered.

The forward() method defines the computation. You call it implicitly when you do model(input) — Python's __call__ mechanism runs forward() but also triggers registered hooks. You should never call model.forward(input) directly; always use model(input). This distinction matters because hooks (useful for inspecting gradients and activations) are only triggered via __call__.

model.parameters() returns an iterator over all learnable parameters in the model. This is what you pass to the optimizer: optimizer = Adam(model.parameters(), lr=3e-4). model.named_parameters() gives (name, tensor) pairs so you can inspect or selectively freeze specific layers. model.state_dict() returns a dictionary of all parameters and buffers — the complete serializable state of the model.

nn.Sequential is a convenience wrapper for linear stacks of layers where the output of one feeds directly into the next. Instead of writing forward manually, you can define model = nn.Sequential(nn.Linear(10, 64), nn.ReLU(), nn.Linear(64, 1), nn.Sigmoid()). This is clean for simple architectures but cannot express branching, skip connections, or multi-input/output models.

Buffers are tensors registered in a module that are not parameters (not trained), but should be part of state_dict and move with the model when .to(device) is called. BatchNorm's running_mean and running_var are buffers. Register them with self.register_buffer('name', tensor).

    def __init__(self):

Why this matters in AI
nn.Module is the contract every PyTorch model must follow. Understanding it is essential for reading other people's model code, debugging shape errors, and writing models that work correctly with checkpointing, distributed training, and quantization.
Common mistakes
  • Storing layers in a plain Python list instead of nn.ModuleList — parameters in a plain list are not registered and will not appear in model.parameters(), so the optimizer will not train them.
  • Calling model.forward(x) instead of model(x) — skips hooks and may produce wrong behavior with some modules.
  • Forgetting super().__init__() in __init__ — the module's internal state is not initialized, causing cryptic attribute errors.

Dataset and DataLoader

Detailed explanation

PyTorch's Dataset and DataLoader abstractions separate data storage from data delivery. A Dataset knows how to retrieve one example by index. A DataLoader wraps a Dataset and handles batching, shuffling, parallel loading, and delivering mini-batches to the training loop. Keeping these concerns separate makes your code reusable and testable.

To create a custom Dataset, subclass torch.utils.data.Dataset and implement two methods: __len__ (returns the total number of examples) and __getitem__ (takes an integer index and returns one example, usually a tuple of (input, label)). Any transformation, augmentation, or preprocessing should happen inside __getitem__ — not before dataset creation — so it applies lazily and efficiently.

DataLoader wraps your Dataset with parameters: batch_size controls how many examples per batch, shuffle=True randomizes order each epoch (use True for training, False for validation), num_workers controls how many CPU processes load data in parallel (0 means single process, 4–8 is typical for GPU training), and pin_memory=True speeds up CPU-to-GPU data transfer.

The standard built-in datasets (MNIST, CIFAR-10, ImageNet) live in torchvision.datasets. You specify transforms to apply: torchvision.transforms.Compose([transforms.ToTensor(), transforms.Normalize(mean, std)]). The ToTensor() transform converts PIL images to [0,1] float tensors. Normalize() standardizes to zero mean and unit variance per channel.

When data does not fit in memory, __getitem__ should load from disk on demand. When data fits in memory, load everything once in __init__ and store in an array, then just index in __getitem__. The difference in training speed can be 10× or more depending on disk speed vs memory access.

Handling variable-length sequences (text, audio) requires a custom collate_fn passed to DataLoader. The default collate_fn stacks tensors along a new batch dimension, which requires all examples to have the same shape. A custom collate_fn can pad sequences to the longest in the batch using torch.nn.utils.rnn.pad_sequence.

class TabularDataset(Dataset):
    def __init__(self, X, y): self.X=torch.tensor(X, dtype=torch.float32); self.y=torch.tensor(y, dtype=torch.float32)
    def __len__(self): return len(self.y)
    def __getitem__(self, i): return self.X[i], self.y[i]
train_ds = TabularDataset(X_train, y_train)
train_loader = DataLoader(train_ds, batch_size=32, shuffle=True, num_workers=4)

Test __len__: len(train_loader) gives number of batches = ceil(len(train_ds)/batch_size).
Inspect one batch: X_b, y_b = next(iter(train_loader)); print(X_b.shape, y_b.shape).
Why this matters in AI
Properly structured data loading prevents memory errors, enables parallel loading, and keeps training fast. Bottlenecked data loading (slow __getitem__) is a common reason GPU utilization stays low.
Common mistakes
  • Doing heavy preprocessing inside __getitem__ every epoch — if the same transform is deterministic, precompute it once in __init__ to avoid repeating expensive work.
  • Setting num_workers > 0 on Windows without the if __name__ == '__main__' guard — causes multiprocessing errors on Windows.
  • Using shuffle=True for a time-series dataset — shuffling breaks temporal order; use a custom sampler or manual split instead.

Training loop

Detailed explanation

The training loop is the core of any deep learning project. It repeatedly feeds batches of training data through the model, computes a loss, propagates gradients backward, and updates the model weights. A correct training loop has exactly six steps that must happen in the right order every iteration: zero gradients, forward pass, compute loss, backward pass, clip gradients (optional), optimizer step.

Zero gradients first (optimizer.zero_grad()) because PyTorch accumulates gradients — calling backward() twice without zeroing doubles the gradient. Next, forward pass: predictions = model(X_batch). Then loss = loss_fn(predictions, y_batch). Then loss.backward() computes all gradients via autograd. Optionally clip gradients if training RNNs or transformers (torch.nn.utils.clip_grad_norm_). Finally optimizer.step() updates all parameters.

The training loop runs inside a nested structure: outer loop over epochs, inner loop over batches from the DataLoader. One epoch means one complete pass through all training data. Multiple epochs allow the model to refine its parameters repeatedly. Tracking training loss per batch and validation loss per epoch gives you the learning curve that shows whether the model is learning, overfitting, or diverging.

Logging is important. Record loss and metric (accuracy, F1, etc.) every N batches to stdout or a tool like TensorBoard/Weights & Biases. Silent training loops are hard to debug — you cannot tell if something is wrong until training finishes. Use tqdm for a progress bar that shows loss in real time.

The training loop must call model.train() before training begins and model.eval() before validation. This toggles dropout and BatchNorm behavior. A common bug is running validation with model still in training mode — dropout makes validation stochastic and BatchNorm uses batch statistics, both of which inflate validation loss.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

model.train()  # enable dropout, BN train mode
for X_b, y_b in train_loader:
    optimizer.zero_grad()      # 1. zero grads
    pred = model(X_b)          # 2. forward
    loss = loss_fn(pred, y_b)  # 3. loss
    loss.backward()            # 4. backward
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)  # 5. clip (optional)
    optimizer.step()           # 6. update
    running_loss += loss.item()
After each epoch: avg_loss = running_loss / len(train_loader). Log it.
Validate: model.eval(); with torch.no_grad(): val_loss = ... Save best model if val_loss improves.
Why this matters in AI
A correctly written training loop is the difference between a model that learns and one that silently produces garbage. Every step has a reason; skipping or reordering any step produces wrong results.
Common mistakes
  • Calling optimizer.step() before loss.backward() — weights update using zero gradients from the previous batch.
  • Forgetting optimizer.zero_grad() — gradients from previous batches add up and training diverges after a few steps.
  • Logging loss.item() outside the training loop after the tensor has been freed — cache the float value with loss.item() inside the loop.

Validation loop

Detailed explanation

The validation loop evaluates model performance on data the model did not train on. It runs after each epoch (or after every N training steps) and produces validation loss and metrics. These numbers tell you whether the model is generalizing or memorizing the training data.

The validation loop differs from training in three ways: model.eval() is called to disable dropout and switch BatchNorm to use running statistics, torch.no_grad() wraps the loop to disable gradient computation (saving memory and time), and optimizer.step() is never called — we only observe, not update.

Validation metrics tell you when to stop training (early stopping), when to save a model checkpoint (save the best validation loss/accuracy so far), and how to compare different hyperparameter configurations. If training loss keeps decreasing but validation loss starts increasing, the model is overfitting — stop training or add regularization.

The validation set must remain truly held out. Never use validation performance to change the model weights or architecture in ways that would not generalize — that is validation set overfitting. Reserve a final test set that you only evaluate once after all model selection is done.

Computing metrics at validation time: accuracy is straightforward — count correct predictions / total. For imbalanced classes, compute precision, recall, and F1 per class. For regression, compute MAE, RMSE, and R². Always evaluate on the full validation set, not just one batch, because batch-level metrics are noisy.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

model.eval()  # disable dropout and BN batch stats
val_loss = 0; correct = 0; total = 0
with torch.no_grad():
    for X_b, y_b in val_loader:
        pred = model(X_b)
        val_loss += loss_fn(pred, y_b).item()
        correct += (pred.argmax(1) == y_b).sum().item()
        total += len(y_b)
avg_val_loss = val_loss / len(val_loader)
val_acc = correct / total
print(f'Val Loss: {avg_val_loss:.4f}, Val Acc: {val_acc:.4f}')
Early stopping: if val_loss did not improve for 5 consecutive epochs, stop training.
Why this matters in AI
Training loss alone is not a reliable signal. A model that memorizes training data has near-zero training loss and poor generalization. The validation loop is your window into true model quality.
Common mistakes
  • Forgetting model.eval() before validation — dropout makes predictions stochastic, giving artificially bad validation metrics.
  • Computing validation loss on only one batch instead of the full validation set — one batch can be unrepresentative.
  • Using validation performance to directly select data cleaning or feature engineering — this is a subtle form of leakage because those decisions are now tuned to the validation set.

Saving loading model

Detailed explanation

Saving a PyTorch model means persisting its learned parameters so you can resume training later or deploy the model for inference. The standard approach is to save and load the model's state_dict — the dictionary mapping layer names to their parameter tensors. This is preferred over saving the entire model object because it is more portable and does not depend on the current class definition being in the same file.

torch.save(model.state_dict(), 'model.pt') saves the state dict to disk. torch.load('model.pt') loads it back as a Python dictionary. model.load_state_dict(state_dict) applies the loaded weights to a freshly created model instance. The model architecture must match exactly — the same layer names and shapes — otherwise load_state_dict raises an error.

During training, save a checkpoint whenever validation loss improves. A checkpoint typically includes: the model state_dict, the optimizer state_dict (so you can resume training with the correct optimizer momentum and adaptive rates), the current epoch number, and the best validation metric seen so far. This lets you resume training exactly where you stopped, even after a crash.

For inference deployment, you often want to save a minimal file. If you have a fixed architecture defined in code, saving only state_dict is enough. torch.jit.script() converts the model to TorchScript — a serializable, optimizable format that does not require the Python class definition. This is important when deploying to environments without Python or to C++ inference servers.

map_location is important when loading a model trained on GPU to a CPU machine (or vice versa). state_dict = torch.load('model.pt', map_location='cpu') prevents the CUDA tensors from being loaded on a machine without GPU. Without this, torch.load on a CPU machine with a GPU-trained model raises a CUDA error.

Why this matters in AI
Saving checkpoints is insurance against hardware failure, runtime crashes, and cloud preemptions. Without checkpoints, a 12-hour training run that crashes at hour 11 is lost entirely.
Common mistakes
  • Saving the entire model object (torch.save(model, ...)) instead of state_dict — if you rename or move the class file, loading fails with a ModuleNotFoundError.
  • Forgetting to save the optimizer state — when resuming, the optimizer starts from scratch with no momentum history, which degrades the first few resumed epochs.
  • Not using map_location when loading — code crashes on CPU machines trying to load GPU tensors.

GPU usage

Detailed explanation

GPUs accelerate deep learning because neural network training is dominated by matrix multiplications on large tensors. A modern GPU has thousands of small cores that execute these multiplications in parallel, whereas a CPU has a handful of powerful cores optimized for sequential code. Training a ResNet50 on ImageNet takes ~days on CPU and ~hours on a modern GPU — often 50–100× speedup.

To use GPU in PyTorch, you move both the model and the data to the GPU device. device = torch.device('cuda' if torch.cuda.is_available() else 'cpu'). model = model.to(device). In the training loop: X_batch, y_batch = X_batch.to(device), y_batch.to(device). If model is on GPU but data is on CPU (or vice versa), PyTorch raises a device mismatch error.

The most common GPU bottleneck is data loading, not computation. If your GPU utilization (check with nvidia-smi) is less than 80% during training, the GPU is waiting for data. Fix this by increasing DataLoader's num_workers, using pin_memory=True to speed up CPU-GPU transfer, and caching preprocessed data to avoid repeated disk reads.

GPU memory is limited. A GPU with 16GB VRAM sounds large, but model parameters, optimizer states (Adam stores 2 extra tensors per parameter), activations, and gradients all compete for this memory. OOM (Out of Memory) errors are common. Solutions: reduce batch size, use gradient checkpointing (trade compute for memory by recomputing activations during backward pass), or use mixed precision (float16 for activations, float32 for critical operations).

Multiple GPUs: torch.nn.DataParallel wraps a model and splits each batch across multiple GPUs. torch.nn.parallel.DistributedDataParallel (DDP) is faster and more scalable — it uses separate processes per GPU and is the standard for multi-GPU training in production. For most learners, a single GPU is sufficient through early projects.

GPU training is 50–100× faster for real models. Without it, training a transformer or a ResNet on a real dataset is impractical. Understanding device management prevents the most common GPU-related errors.
Common mistakes

  • Logging tensors directly (writer.add_scalar('loss', loss, step)) while loss is on GPU — some loggers need CPU tensors; use loss.item() or loss.cpu() first.

Mixed precision concept

Detailed explanation

Mixed precision training uses 16-bit floating point (float16 or bfloat16) for most operations while keeping float32 for numerically sensitive ones like gradient accumulation and optimizer updates. This halves memory usage (float16 takes 2 bytes vs float32's 4 bytes), doubles memory throughput, and on modern GPUs with Tensor Cores (NVIDIA Ampere, Hopper) achieves 2–4× speedup in matrix multiplications.

The key challenge with float16 is numerical range. float32 can represent values from ~10^{-38} to ~10^{38}. float16 can only represent ~10^{-8} to ~65504. Gradients during backpropagation can be very small — smaller than float16 can represent — and would be stored as 0 (underflow). Loss scaling solves this: multiply the loss by a large constant before backward pass, so gradients are in float16 range, then divide by the same constant before the optimizer step.

PyTorch's torch.cuda.amp (Automatic Mixed Precision) handles all of this automatically with two components: autocast (context manager that automatically casts operations to float16 where safe) and GradScaler (manages loss scaling dynamically, increasing scale when there are no NaN/Inf gradients and decreasing when there are). You add about 5 lines to an existing training loop.

bfloat16 (brain floating point) is an alternative 16-bit format with the same exponent range as float32 but lower precision. It avoids underflow without loss scaling, making it simpler to use. It is the default half-precision format on Google's TPUs and is supported on NVIDIA Ampere+ GPUs. Most modern LLM training uses bfloat16.

Mixed precision does not always improve accuracy — and rarely hurts it significantly when implemented correctly. For tasks requiring extreme numerical precision (some scientific computing, certain loss functions), staying in float32 is safer. For standard vision and NLP tasks, mixed precision is a free speedup.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from torch.cuda.amp import autocast, GradScaler; scaler = GradScaler()
Training loop with AMP:
  optimizer.zero_grad()
  with autocast():
    pred = model(X_b)
    loss = loss_fn(pred, y_b)  # computed in float16
  scaler.scale(loss).backward()  # scales loss, backprops in float16
  scaler.step(optimizer)          # unscales grads, steps in float32
  scaler.update()                 # adjusts scale factor
Memory check: with float32, batch_size=64 may OOM. With AMP, same GPU can handle batch_size=128.
Speed check: time one epoch with and without AMP. On Tensor Core GPUs, expect 1.5–3× speedup.
Why this matters in AI
Mixed precision is a nearly free speedup for any GPU training project. On modern hardware it is expected practice, not an optimization — it is the default in most training frameworks.
Common mistakes
  • Using AMP on CPU — AMP only accelerates on GPU with Tensor Core support. On CPU, float16 is slower than float32.
  • Forgetting scaler.update() — the dynamic scale factor is never adjusted and may stay at a value that causes consistent NaN gradients.
  • Using float16 for the optimizer state (Adam's m and v terms) — this causes inaccurate parameter updates. Always keep optimizer states in float32.

Checkpoints

Detailed explanation

A checkpoint is a snapshot of a model's complete training state at a specific point in time, saved to disk so training can be resumed or the model can be deployed. A complete checkpoint contains more than just the model weights — it contains everything needed to resume training identically: model state_dict, optimizer state_dict, scheduler state_dict (if using LR scheduling), current epoch, current best metric, and any random state (for reproducibility).

Checkpointing strategy matters. Saving after every epoch is safe but wastes disk space. Save only when validation loss improves (best model checkpoint) and also save a 'latest' checkpoint every epoch for crash recovery. The best model checkpoint is what you deploy; the latest checkpoint is for resuming after a preemption.

In cloud training (AWS spot instances, Google Preemptible VMs, Colab), your compute can be terminated at any moment. Checkpoints every epoch guarantee you lose at most one epoch of training. Without checkpoints, a 48-hour training run that gets preempted at hour 47 requires starting over.

For very large models (billions of parameters), saving the full state_dict takes significant time and disk space. Techniques: save only changed layers (useful in fine-tuning where only a few layers change), use safetensors format (safer and faster than pickle-based .pt files), or save every N epochs rather than every epoch.

Checkpoint compatibility: if you rename a layer or add a new layer to your model architecture, loading the old checkpoint will fail because the keys do not match. Plan architecture changes carefully, or use strict=False in load_state_dict() (which loads only matching keys and ignores mismatches — useful for partial loading).

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

best_val_loss = float('inf'); best_epoch = 0
After each epoch:
  if val_loss < best_val_loss:
    best_val_loss = val_loss
    torch.save({'epoch': epoch, 'model': model.state_dict(), 'optimizer': opt.state_dict(), 'scheduler': sched.state_dict(), 'val_loss': best_val_loss}, 'best.pt')
  torch.save({'epoch': epoch, 'model': model.state_dict(), 'optimizer': opt.state_dict()}, 'latest.pt')
Resume: ckpt = torch.load('latest.pt'); model.load_state_dict(ckpt['model']); opt.load_state_dict(ckpt['optimizer']); start_epoch = ckpt['epoch'] + 1
Deploy: load 'best.pt', model.eval(), wrap in torch.no_grad() for inference.
Why this matters in AI
Without checkpointing, every crash, preemption, or experiment reset throws away hours or days of compute. Checkpointing is the professional practice that separates research code from production training pipelines.
Common mistakes
  • Saving only model weights and not optimizer state — resumed training uses stale/zero momentum, slowing convergence for the first few epochs.
  • Saving checkpoints inside the DataLoader loop every batch — disk I/O every batch is much slower than training itself, and checkpoint files may be partially written during crashes.
  • Not validating that the loaded checkpoint resumes correctly — run 2 epochs, save checkpoint, load it fresh, run 2 more epochs and confirm loss is continuous.
