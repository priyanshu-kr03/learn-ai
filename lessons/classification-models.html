Logistic regression

Detailed explanation

Logistic regression is a classification algorithm (despite its name containing 'regression') that models the probability that an input belongs to class 1. It passes a linear combination of features through the sigmoid function σ(z) = 1/(1 + e^{-z}) to produce a probability between 0 and 1. The decision boundary is p ≥ 0.5, which corresponds to z = 0 — a linear boundary in feature space.

The model: p = σ(w·x + b) where w is the weight vector and b is the bias. Training minimizes the binary cross-entropy loss: L = -[y log(p) + (1-y) log(1-p)]. This is a convex optimization problem — there is one global minimum, guaranteed by the convexity of the loss. Gradient descent (or second-order methods like L-BFGS) finds it reliably.

Regularization is essential. Without it, logistic regression can overfit — especially when features are high-dimensional or correlated. L2 regularization (Ridge) adds λΣw_i^2 to the loss, shrinking all weights toward zero. L1 regularization (Lasso) adds λΣ|w_i|, driving some weights to exactly zero (automatic feature selection). sklearn's LogisticRegression uses L2 by default; parameter C is the inverse of regularization strength (larger C = less regularization).

Logistic regression is interpretable: the weight w_i for feature x_i represents how much a one-unit increase in x_i changes the log-odds of class 1. exp(w_i) is the odds ratio — if exp(w_i) = 2, the odds of class 1 double for each unit increase in x_i. This interpretability makes logistic regression a first choice in high-stakes domains (medicine, credit scoring) where decisions must be explainable.

Logistic regression assumes a linear decision boundary. It cannot model non-linear relationships unless you manually engineer polynomial or interaction features. For non-linear classification tasks (XOR, circles within circles), use non-linear classifiers (SVM with kernel, random forest, neural network) or add engineered features to enable logistic regression to model the non-linearity.

Predict class: y_pred = model.predict(X_te_s)  # threshold at 0.5

Tune C with cross-validation: GridSearchCV(LogisticRegression(), {'C': [0.001, 0.01, 0.1, 1, 10, 100]}, cv=5, scoring='roc_auc')

model = LogisticRegression(max_iter=1000)
model.fit(X_train, y_train)
Why this matters in AI
Logistic regression is the essential baseline classifier and one of the most used models in production (especially in regulated industries). If it achieves acceptable accuracy, it should be preferred over complex models due to interpretability, fast inference, and well-understood failure modes.
Common mistakes
  • Not scaling features before logistic regression — gradient descent converges much slower when features are on different scales. Always standardize.
  • Using max_iter=100 (sklearn default) for large datasets or complex feature spaces — the solver may not converge. Increase to max_iter=1000 or more.
  • Interpreting coefficients directly as feature importance without controlling for scale — a coefficient of 0.5 for income in dollars is not comparable to 0.5 for age in years. Standardize first, then compare coefficients.

KNN

Detailed explanation

K-Nearest Neighbors (KNN) is the simplest possible ML algorithm. To classify a new point, find the K training examples closest to it (by some distance metric, usually Euclidean distance), and predict the majority class among those K neighbors. No training step is needed — the training data IS the model. This makes KNN a lazy learner: it does all computation at prediction time.

Distance metrics determine which points are 'nearest'. Euclidean distance sqrt(Σ(xi-yi)²) is most common for numerical features. Manhattan distance Σ|xi-yi| is more robust to outliers. For high-dimensional data, cosine similarity (1 - dot_product/(|a||b|)) is better than Euclidean because Euclidean distance becomes meaningless in high dimensions (curse of dimensionality). For categorical features, Hamming distance counts the number of positions where the values differ.

Choosing K: small K (K=1) overfits — the decision boundary is jagged and follows individual points. Large K underfits — predictions converge to the global majority class regardless of the query point. The optimal K is found via cross-validation. As a rule: K is usually odd (to avoid ties in binary classification) and K = sqrt(n_training) is a common heuristic starting point.

Prediction time complexity is O(n * d) per query: for each query, compute distances to all n training points, each of dimensionality d. This is slow for large n. KD-trees and Ball trees reduce the average case complexity to O(d * log(n)) by spatial partitioning. sklearn's KNeighborsClassifier automatically uses Ball trees for small d and brute force for large d (because KD-tree becomes slow in high dimensions).

KNN for regression: instead of majority vote, predict the mean of the K nearest neighbors' target values. KNN regression can capture any functional form as long as the neighborhood is representative. It is non-parametric — no assumptions about the functional form of the relationship between features and target. This is both a strength (flexible) and weakness (slow, requires all training data at prediction time).

Why this matters in AI
KNN is the conceptual baseline for all distance-based ML methods: embedding similarity search, recommendation via user-item similarity, anomaly detection via distance to nearest normal point, and semantic search with vector databases. Understanding KNN builds intuition for why feature scaling and distance metrics matter.
Common mistakes
  • Not normalizing features before KNN — a feature with range 0–1000 dominates a feature with range 0–1. StandardScaler is mandatory before KNN.
  • Using KNN on high-dimensional data without dimensionality reduction — in 100+ dimensions, all points are approximately equidistant (curse of dimensionality), making nearest neighbors meaningless.
  • Using KNN in production for large datasets — prediction is O(n) per query with brute force. With 1M training examples, each prediction requires 1M distance computations.

Naive Bayes

Detailed explanation

Naive Bayes is a probabilistic classifier based on Bayes theorem with the 'naive' assumption that features are conditionally independent given the class. Despite this unrealistic assumption (features are almost never truly independent), Naive Bayes works surprisingly well for text classification, spam detection, and document categorization. It is fast, scalable, and requires very little data to estimate parameters.

For spam detection: given an email with words [money, free, win, congratulations], Naive Bayes computes P(spam | words) ∝ P(spam) * P(money|spam) * P(free|spam) * P(win|spam) * P(congratulations|spam). Each factor is estimated from the training data: P(money|spam) = count(emails labeled spam containing 'money') / count(spam emails). The class with the highest posterior probability wins.

Types of Naive Bayes: Multinomial NB (MultinomialNB) — for word counts in text; features are counts (TF-IDF or bag-of-words counts). Best for text classification. Gaussian NB (GaussianNB) — for continuous features; assumes features follow a Gaussian distribution within each class. Good for continuous tabular data. Bernoulli NB (BernoulliNB) — for binary features; each feature is present or absent. Good for binary text features.

Laplace (additive) smoothing prevents zero probabilities for words not seen during training. If a test email contains a word never seen in the training spam emails, P(word|spam) = 0, making the entire product 0 regardless of other words. With Laplace smoothing (alpha=1): P(word|spam) = (count + alpha) / (total_words_in_spam + alpha * vocab_size). Setting alpha=1 is the standard; higher alpha provides more smoothing.

When to use Naive Bayes: (1) text classification with large vocabulary, (2) as a fast, strong baseline for any classification task, (3) when training data is scarce (few examples per class — Naive Bayes can work with as few as 20 examples per class), (4) when you need probability outputs (Naive Bayes is well-calibrated for text), (5) as an online learner for streaming data (parameters can be updated incrementally).

Why this matters in AI
Naive Bayes is the first classifier to try for text classification. It trains in milliseconds, works well with small data, and produces interpretable probability outputs. Understanding it teaches Bayes theorem, independence assumptions, and probability estimation — foundational ML concepts.
Common mistakes
  • Using MultinomialNB with negative features (e.g., TF-IDF differences) — MultinomialNB requires non-negative features. Use GaussianNB or BernoulliNB for features that can be negative.
  • Not applying Laplace smoothing (alpha=1) — without smoothing, words unseen in one class produce zero probability, making the classifier ignore all other words. Always use alpha > 0.
  • Expecting probability estimates to be calibrated for non-text data — GaussianNB probabilities on tabular data are often poorly calibrated (overconfident). Use CalibratedClassifierCV for calibrated probabilities.

SVM

Detailed explanation

Support Vector Machine (SVM) finds the maximum-margin decision boundary between classes. The margin is the distance between the decision boundary and the nearest training examples from each class (the support vectors). Maximizing the margin is a geometrically appealing objective — it creates the most robust separator: a small perturbation of the data is least likely to cross the boundary.

Hard-margin SVM assumes the data is linearly separable and finds the exact maximum margin boundary. Soft-margin SVM (C-SVM) allows misclassifications with a penalty. The C parameter controls the tradeoff: large C = small margin, few training errors (overfit); small C = large margin, more training errors (underfit). The optimal C is found via grid search with cross-validation.

The kernel trick is what makes SVM powerful. Many classification problems are not linearly separable in the original feature space but become linearly separable in a higher-dimensional space. The kernel function K(xi, xj) computes the dot product in the transformed space without explicitly computing the transformation. Common kernels: Linear K(x,z) = x·z (fast for text classification with TF-IDF), RBF/Gaussian K(x,z) = exp(-γ||x-z||²) (most common for non-linear problems), Polynomial K(x,z) = (x·z + c)^d.

SVMs were the dominant ML algorithm from ~1995–2010, before deep learning. They are still excellent for: (1) small to medium datasets (n < 100K) where training time is acceptable, (2) high-dimensional sparse data (text classification with linear SVM is fast and effective), (3) problems with clear margins and few support vectors. SVMs are less suited to: very large datasets (training is O(n²) to O(n³)), high-noise data, and problems requiring probability outputs (SVM does not naturally produce probabilities; Platt scaling is needed).

SVMs for regression (SVR) find a regression function within a tube of width ε around the true values. Points within the tube are not penalized; only points outside the ε-tube contribute to the loss. This makes SVR robust to outliers (insensitive loss) and often produces sparse solutions (only the support vectors determine the model).

Why this matters in AI
SVM is a mathematically elegant and powerful classifier. Understanding the margin, kernel trick, and C parameter builds intuition applicable to other models. Linear SVM is still one of the best baselines for text classification. Knowing SVMs helps you understand why deep learning kernels and attention mechanisms work.
Common mistakes
  • Not scaling features before SVM — RBF kernel uses Euclidean distance, making SVM extremely sensitive to feature scale. Always StandardScaler first.
  • Using SVC for large datasets — SVC training scales as O(n²) to O(n³). For n > 50K, use LinearSVC (scales as O(n)) or SGDClassifier with hinge loss.
  • Not searching over gamma — gamma controls the RBF kernel width. If gamma is too large, each support vector only influences its immediate neighborhood (overfit). If too small, the model is too smooth (underfit). Always grid search gamma with C.

Decision tree

Detailed explanation

A decision tree is a classification or regression model that makes predictions by learning a series of if-then-else rules from data. The tree splits the input space into regions, and predicts the majority class (classification) or mean value (regression) for the training examples in each region. Decision trees are highly interpretable — you can visualize and explain every prediction as a path through the tree.

The tree is built recursively by finding the best split at each node. 'Best' is defined by an impurity measure: Gini impurity (default in sklearn) measures how often a randomly chosen element would be misclassified if labeled randomly according to the class distribution in the node. Information Gain (Entropy-based) measures the reduction in entropy after the split. Both produce similar trees in practice.

Tree depth is the primary hyperparameter. An unconstrained tree grows until every leaf contains exactly one training example — achieving 100% training accuracy and massive overfitting. Limiting max_depth (to 3–10 for most tasks), min_samples_leaf (minimum examples per leaf), or min_samples_split (minimum examples to split a node) prevents overfitting. These are regularization hyperparameters.

Advantages: decision trees handle both numeric and categorical features without preprocessing, handle missing values naturally in some implementations, are robust to outliers (a split at x=1000 still works whether the outlier is 1000 or 1,000,000), and are trivially interpretable. Disadvantages: they are unstable (small data changes produce very different trees), have high variance, and tend to overfit even with regularization.

Decision trees are the building block of ensemble methods. Random forests and gradient boosting both train many trees and aggregate their predictions, correcting the variance problem of individual trees while retaining their interpretability at the individual split level.

Why this matters in AI
Decision trees are the starting point for ensemble methods (the most powerful tabular ML algorithms) and the first truly interpretable model to try. Understanding how trees split data and how depth controls overfitting is foundational for understanding random forests and gradient boosting.
Common mistakes
  • Not limiting max_depth — an unconstrained tree perfectly fits training data and fails on test data. Always set max_depth or min_samples_leaf.
  • Using decision trees for linear relationships — a tree needs many splits to approximate a diagonal boundary; logistic regression models it in one line. Use the right tool.
  • Trusting a visualized tree as an explanation without checking test performance — a beautiful tree that overfits is worse than a simple logistic regression.

Confusion matrix

Detailed explanation

A confusion matrix is a 2D table showing model performance for each combination of actual class and predicted class. For binary classification, it has 4 cells: True Positives (TP — predicted positive, actually positive), False Positives (FP — predicted positive, actually negative), True Negatives (TN — predicted negative, actually negative), False Negatives (FN — predicted negative, actually positive). Every other classification metric is computed from these 4 numbers.

Reading a confusion matrix: the rows represent actual classes, the columns represent predicted classes (or vice versa depending on the library). A perfect classifier has all predictions on the diagonal (TP and TN cells) and zeros off-diagonal (FP and FN cells). The off-diagonal cells are where the model is wrong — inspecting them tells you what the model confuses: does it confuse cats with dogs, or ships with planes?

For multi-class classification, the confusion matrix is N×N where N is the number of classes. Diagonal cells are correct predictions; off-diagonal cells show which classes are confused with which. Class imbalance is visible: if 90% of examples are class 0, the matrix shows 90% of predictions landing in the class 0 row regardless of model quality.

Visual inspection of the confusion matrix is more informative than accuracy alone. A confusion matrix for a medical diagnostic model might show: TP=90, TN=850, FP=50, FN=10. Accuracy = 940/1000 = 94%. But the model misses 10 cases that actually have the disease (FN=10) — this could be life-threatening. The confusion matrix makes this visible; accuracy hides it.

Normalization: row-normalized confusion matrix shows recall per class (fraction of actual class k examples predicted correctly). Column-normalized shows precision per class. Always normalize when classes are imbalanced — raw counts in large classes dominate the visualization.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from sklearn.metrics import confusion_matrix, ConfusionMatrixDisplay
y_pred = model.predict(X_test)
cm = confusion_matrix(y_test, y_pred)
disp = ConfusionMatrixDisplay(confusion_matrix=cm, display_labels=['Negative','Positive'])
disp.plot(cmap='Blues')
Manual extraction: TP=cm[1,1]; FP=cm[0,1]; TN=cm[0,0]; FN=cm[1,0]
Precision=TP/(TP+FP); Recall=TP/(TP+FN); F1=2*Precision*Recall/(Precision+Recall)
Normalized: cm_norm = confusion_matrix(y_test, y_pred, normalize='true')  # row-normalized (recall per class)
Why this matters in AI
Accuracy alone is not enough. A spam filter that labels everything as 'not spam' achieves 95% accuracy if 95% of emails are not spam — but catches zero spam. The confusion matrix exposes exactly what the model gets right and wrong for each class.
Common mistakes
  • Using accuracy as the sole metric for imbalanced datasets — in a dataset with 99% negatives, a model that always predicts negative has 99% accuracy but 0% recall. The confusion matrix immediately reveals this.
  • Not checking which type of error is more costly — FN and FP have different costs in medicine (FN = missed diagnosis), fraud detection (FP = false alarm), and content moderation. The confusion matrix guides threshold adjustment.
  • Confusing row-normalized and column-normalized — row-normalized gives recall; column-normalized gives precision. Know which you need before interpreting.

Accuracy

Detailed explanation

Accuracy is the fraction of predictions that are correct: accuracy = (TP + TN) / (TP + TN + FP + FN). It is the most intuitive classification metric — easy to compute and explain. However, accuracy is misleading for imbalanced datasets: a model that predicts 'no fraud' for every transaction achieves 99.5% accuracy on a dataset with 0.5% fraud rate, but is completely useless as a fraud detector.

When accuracy is appropriate: balanced classes (roughly equal class distribution), equal cost of false positives and false negatives. For MNIST digit classification (10 balanced classes), accuracy is a fine metric — each digit appears roughly equally and misclassifying any digit is equally bad. For medical diagnosis (rare disease detection), fraud detection, or any imbalanced binary classification, accuracy is almost always misleading.

Accuracy vs precision and recall: accuracy answers 'what fraction of all predictions are correct?' Precision answers 'when you predict positive, how often are you right?' Recall answers 'of all actual positives, how many did you catch?' For fraud detection: recall is critical (catch as many fraudulent transactions as possible), precision is also important (don't flag too many legitimate transactions). F1 score = harmonic mean of precision and recall.

Top-k accuracy: for multi-class problems with many classes (1000+ for ImageNet), top-1 accuracy (exact match) is supplemented with top-5 accuracy (correct class is among the 5 highest predicted probabilities). Top-5 accuracy is much higher than top-1 and gives a better sense of whether the model is 'close' in difficult cases.

Accuracy in deep learning training: training accuracy vs validation accuracy reveals overfitting. If training accuracy reaches 99% but validation accuracy is 80%, the model has memorized training examples rather than learning generalizable patterns. Always monitor both training and validation accuracy during training.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from sklearn.metrics import accuracy_score, classification_report, confusion_matrix
import numpy as np
# Standard accuracy
accuracy = accuracy_score(y_true, y_pred)
print(f'Accuracy: {accuracy:.4f}')
# Classification report: precision, recall, F1 per class
print(classification_report(y_true, y_pred, target_names=class_names))
# Confusion matrix
cm = confusion_matrix(y_true, y_pred)
print('Confusion matrix:\n', cm)
# Imbalanced example: why accuracy is misleading
y_true_imbalanced = np.array([0]*990 + [1]*10)  # 1% positive
y_pred_all_negative = np.zeros(1000)
print(f'Always-negative accuracy: {accuracy_score(y_true_imbalanced, y_pred_all_negative):.4f}')  # 0.99!
print(f'Always-negative recall: {recall_score(y_true_imbalanced, y_pred_all_negative):.4f}')  # 0.0!
# Top-k accuracy for multi-class
from sklearn.metrics import top_k_accuracy_score
y_prob = model.predict_proba(X_test)  # (n_samples, n_classes)
top1 = top_k_accuracy_score(y_true, y_prob, k=1)
top5 = top_k_accuracy_score(y_true, y_prob, k=5)
print(f'Top-1 accuracy: {top1:.4f}, Top-5 accuracy: {top5:.4f}')
Why this matters in AI
Understanding the limitations of accuracy is fundamental to not being misled by your model's performance. Choosing the wrong metric is one of the most common mistakes in applied ML — a model optimized for accuracy on an imbalanced problem may score 99% while completely failing at its actual purpose.
Common mistakes
  • Using accuracy as the primary metric for imbalanced classification — on a dataset with 1% positives, a model that predicts all negatives scores 99% accuracy. Use AUC-ROC, AUC-PR, F1 score, or recall at a fixed precision for imbalanced problems.
  • Reporting accuracy without the confusion matrix — accuracy hides the breakdown by class. A model with 90% accuracy but 0% recall for the minority class is worse than a model with 87% accuracy but 60% recall. Always show the confusion matrix alongside accuracy.
  • Not specifying the threshold — accuracy depends on the classification threshold (default 0.5). Different thresholds give different accuracies. Report accuracy at multiple thresholds or use threshold-independent metrics (AUC-ROC).

Precision

Detailed explanation

Precision is the fraction of positive predictions that are actually positive: Precision = TP / (TP + FP). Out of all examples the model predicted as positive, what fraction were really positive? High precision means: when the model says 'positive', it is usually right. Low precision means: the model generates many false alarms.

Precision is the right metric to optimize when false positives are costly. Spam filtering: a spam filter with low precision classifies legitimate emails as spam, causing users to miss important messages — a severe failure. Medical screening: if a positive test triggers expensive, painful follow-up procedures, false positives impose unnecessary cost and risk on healthy patients. Content moderation: false positive content removal censors legitimate speech.

The precision-recall tradeoff: increasing the decision threshold makes the classifier more selective — it only predicts positive when very confident, reducing FP (higher precision) but also catching fewer true positives (lower recall). Decreasing the threshold predicts positive more freely — more TP (higher recall) but also more FP (lower precision). The F1 score balances the two.

Macro vs micro precision: macro-precision averages precision across all classes equally (good for balanced classes). Micro-precision computes precision globally across all predictions (dominated by the majority class). Weighted-precision averages per-class precision weighted by class support (good for imbalanced data). sklearn's classification_report shows all three.

Precision without context is meaningless. A model that predicts positive for only one example, and gets it right, has precision=1.0 — but has only made one prediction. Always pair precision with recall (and ideally F1 or the precision-recall curve) to understand the full picture.

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

Why this matters in AI
Precision is essential when false positives have consequences. Every medical test, fraud flag, security alert, and content moderation decision involves a precision tradeoff. Knowing when to prioritize precision vs recall is a fundamental ML engineering judgment call.

  • Using macro precision on highly imbalanced data — macro precision treats all classes equally; if 98% of data is class 0, class 0 precision dominates the relevant metric. Use weighted precision.

Recall

Detailed explanation

Recall (also called sensitivity or true positive rate) is the fraction of actual positives that the model correctly identifies: Recall = TP / (TP + FN). Out of all examples that are actually positive, what fraction did the model find? High recall means: the model catches most of the true positives. Low recall means: the model misses many positive cases.

Recall is the right metric to optimize when false negatives are costly. Cancer screening: missing a cancer case (FN) means a patient goes untreated until the disease progresses. Fraud detection: missing a fraudulent transaction (FN) means direct financial loss. Failure mode detection in manufacturing: missing a defective part (FN) means a faulty product reaches customers. In all these cases, false alarms (FP) are acceptable; misses (FN) are not.

The recall-precision tradeoff mirrors the precision tradeoff in reverse. To increase recall, lower the decision threshold — the model predicts positive more freely, catching more true positives (higher recall) but also more false positives (lower precision). This is the core tradeoff in nearly every classification deployment decision.

Recall at K is used in information retrieval: out of the top-K items returned by a search or recommendation system, what fraction of all relevant items were included? For a document retrieval system that must find all relevant legal documents, recall@1000 (did the system retrieve all relevant documents in its top-1000 results?) is more important than precision@10.

Multi-class recall: per-class recall is the diagonal element of the row-normalized confusion matrix. If class 'cat' has recall=0.6, it means 40% of actual cats were misclassified as something else. High per-class recall variation reveals which classes the model struggles with most.

  • Not checking per-class recall for multi-class problems — overall recall can look good while a minority class has recall=0.0.

F1 score

Detailed explanation

The F1 score is the harmonic mean of precision and recall: F1 = 2 * (Precision * Recall) / (Precision + Recall). The harmonic mean punishes extreme values — if either precision or recall is very low, F1 is pulled toward that low value. This makes F1 a balanced metric that requires both precision and recall to be reasonably high, unlike arithmetic mean which would allow one to compensate for the other.

Why harmonic mean rather than arithmetic mean? If precision=1.0 and recall=0.0 (the model makes one correct prediction and misses everything else), the arithmetic mean is 0.5 (looks decent) but F1 = 0.0 (correctly shows terrible performance). The harmonic mean is more sensitive to the minimum of the two values.

F-beta score generalizes F1: F_β = (1+β²) * (Precision * Recall) / (β² * Precision + Recall). When β=1, it is F1 (equal weight to precision and recall). When β=2 (F2), recall is weighted twice as heavily — use for medical screening where recall matters more. When β=0.5 (F0.5), precision is weighted twice as heavily — use for spam detection where false alarms matter more.

Macro-F1 averages F1 across all classes equally — each class contributes equally regardless of how many examples it has. This is appropriate for multi-class problems where you care equally about performance on all classes. Micro-F1 computes F1 globally from aggregated TP, FP, FN — this is equivalent to accuracy for multi-class problems. Weighted-F1 weights by class support — appropriate for imbalanced datasets where you care more about performance on larger classes.

F1 is less useful than AUC-ROC when you want threshold-independent evaluation, or when the class imbalance is extreme. For very rare events (fraud, disease), F1 at the default threshold can look terrible because the model is calibrated for low-precision high-recall operation. Always report both F1 and AUC-ROC for imbalanced binary classification.

Why this matters in AI

Most ML competitions and production systems report F1 as the primary metric for classification tasks with imbalanced classes.
Common mistakes
  • Using F1 when classes are balanced and the cost of FP and FN are equal — in that case, accuracy is simpler and equally informative.
  • Reporting macro-F1 for a heavily imbalanced dataset where the rare class is the most important — a high macro-F1 can hide very low precision on the rare class if the common class pulls up the average.
  • Optimizing F1 via threshold=0.5 rather than finding the optimal threshold — the F1-maximizing threshold is often not 0.5 for imbalanced problems. Search over thresholds.

ROC-AUC

Detailed explanation

The ROC curve (Receiver Operating Characteristic) plots True Positive Rate (recall) on the y-axis against False Positive Rate (FPR = FP/(FP+TN)) on the x-axis as the decision threshold varies from 1 down to 0. At threshold=1, the model predicts positive for nothing (TPR=0, FPR=0). At threshold=0, it predicts positive for everything (TPR=1, FPR=1). The curve traces the tradeoff between sensitivity and specificity across all thresholds.

AUC (Area Under the Curve) measures the total area under the ROC curve. AUC=0.5 is a random classifier (diagonal line). AUC=1.0 is a perfect classifier. AUC=0.8 means the model correctly ranks 80% of positive-negative pairs — if you pick a random positive and a random negative from the dataset, there is an 80% chance the model assigns a higher score to the positive. AUC is threshold-independent and therefore a better summary of discriminative ability than F1 at a fixed threshold.

AUC is especially valuable for imbalanced binary classification because it is insensitive to class imbalance — it measures ranking ability, not calibration. A model with AUC=0.95 discriminates well even if its probability outputs are not well-calibrated (it may give all examples very high scores, but still ranks positives higher than negatives). Complement AUC with a calibration plot for probability-dependent applications.

The PR-AUC (Area Under the Precision-Recall curve) is an alternative to ROC-AUC for highly imbalanced datasets. ROC-AUC can be misleadingly high when negatives vastly outnumber positives because FPR has the large TN pool in its denominator. PR-AUC does not suffer from this — it only considers the positive class and penalizes false positives relative to true positives. For fraud detection (0.1% positive rate), PR-AUC is more informative.

Partial AUC (pAUC) measures the area under only part of the ROC curve — typically the part where FPR < 0.1. This is useful when false positives must be kept very low (security, medical screening): you care about the model's performance only in the low-FPR operating region, not across all thresholds.

Why this matters in AI
AUC-ROC is the standard evaluation metric for binary classification in most production systems. It measures how well the model can rank positives above negatives regardless of threshold, making it useful for comparing models, tuning, and reporting to stakeholders.
Common mistakes
  • Confusing AUC-ROC with PR-AUC — for highly imbalanced datasets (fraud, rare disease), ROC-AUC can be 0.99 while PR-AUC is 0.3. Always report both for imbalanced classification.
  • Using AUC to select the threshold — AUC summarizes all thresholds; you still need to choose a specific threshold for deployment based on business cost requirements.
  • Computing AUC from predicted classes rather than probabilities — AUC requires continuous scores (probabilities), not binary 0/1 predictions. Use predict_proba, not predict.

Threshold tuning

Detailed explanation

In binary classification, the model outputs a probability score (e.g., 0.73 for fraud). The classification threshold converts this score to a hard prediction: if score >= threshold → predict positive, else predict negative. The default threshold of 0.5 means: if the model is more than 50% confident it's positive, predict positive. This default is rarely optimal — changing the threshold trades off precision vs recall.

Precision-recall tradeoff: lowering the threshold (e.g., from 0.5 to 0.2) flags more samples as positive — this catches more true positives (higher recall) but also flags more false positives (lower precision). Raising the threshold (e.g., to 0.8) only flags very confident positives — fewer false alarms (higher precision) but many missed true positives (lower recall). The precision-recall curve shows all tradeoffs at once.

Choosing the threshold by business objective: the optimal threshold depends on the cost of false positives vs false negatives. For cancer screening (false negative = missed cancer = patient dies), prioritize high recall (low threshold). For a spam filter (false positive = blocked legitimate email = user loses important message), prioritize high precision (high threshold). The F-beta score weights recall β times more than precision — use β=2 for recall-priority, β=0.5 for precision-priority.

ROC curve: plots True Positive Rate (TPR = recall) vs False Positive Rate (FPR) at all thresholds. AUC-ROC measures the area under this curve — a classifier with AUC=1 perfectly separates classes at all thresholds, AUC=0.5 is random. Each point on the ROC curve corresponds to one threshold value. Choose the threshold at the 'knee' of the ROC curve (maximizes TPR while minimizing FPR) or use the Youden index: J = TPR - FPR, maximize J.

Calibration and threshold: threshold tuning only makes sense if the model is well-calibrated — probability outputs reflect true frequencies. A model that outputs 0.9 for 50% of actual positives is poorly calibrated. Use CalibratedClassifierCV to calibrate probabilities. After calibration, threshold tuning based on business costs is more reliable.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from sklearn.metrics import precision_recall_curve, roc_curve
from sklearn.metrics import f1_score
# Precision-Recall curve
prec, rec, thresh = precision_recall_curve(y_val, y_prob)
f1s = 2 * prec * rec / (prec + rec + 1e-8)
best_thresh = thresh[np.argmax(f1s)]
print(f'Best threshold (F1): {best_thresh:.3f}')
# For β=2 (recall-priority): Fβ score
beta = 2
fb = (1+beta**2) * prec * rec / (beta**2 * prec + rec + 1e-8)
best_thresh_fb = thresh[np.argmax(fb)]
print(f'Best threshold (F{beta}): {best_thresh_fb:.3f}')
# Apply chosen threshold
final_preds = (y_prob >= best_thresh).astype(int)
from sklearn.metrics import classification_report
print(classification_report(y_val, final_preds))
Why this matters in AI
Default 0.5 threshold is almost always wrong in production. Fraud detection, medical diagnosis, and content moderation all require threshold tuning based on business costs. Understanding how to systematically choose thresholds using precision-recall curves is a critical applied ML skill.
Common mistakes
  • Tuning the threshold on the test set — this turns the test set into a validation set. Tune threshold on the validation set, evaluate on the test set.
  • Not considering calibration — probability outputs must be calibrated for threshold tuning to work correctly. An uncalibrated model might output 0.9 for a sample that is only 50% likely to be positive.
  • Setting threshold to maximize F1 in all cases — F1 treats precision and recall equally. Choose the metric that matches the business objective: F2 for recall-priority, precision@recall=0.90 for detection systems.

Multiclass classification

Detailed explanation

Multiclass classification predicts one of K > 2 classes. Examples: digit recognition (0–9: K=10), image classification (1000 ImageNet classes: K=1000), language identification (K=100+ languages), document topic classification (K=20 topics). Most ML algorithms natively support multiclass; some (SVM, logistic regression) require strategies to extend to K classes.

Softmax output layer: for neural networks, multiclass classification uses a softmax output layer of K neurons. Softmax: P(class=k | x) = exp(zk) / Σ exp(zj). Each neuron outputs a probability; all K probabilities sum to 1. Training uses cross-entropy loss: L = -Σ yₖ log(P(class=k)) where y is a one-hot vector. PyTorch's CrossEntropyLoss combines log-softmax and NLL loss for numerical stability.

One-vs-rest (OvR): train K binary classifiers, each predicts 'is this class k or not?' At prediction time, choose the class with the highest score. For K=10 (digits), this trains 10 binary classifiers. LogisticRegression(multi_class='ovr') and SVC(decision_function_shape='ovr') use this approach. OvR is fast and simple but each classifier sees highly imbalanced data (1 class vs 9 others).

Softmax regression (multinomial logistic regression): a single model with K output neurons and softmax normalization. LogisticRegression(multi_class='multinomial', solver='lbfgs') trains this jointly. More powerful than OvR because all classes compete simultaneously — the probabilities are calibrated relative to each other.

Evaluation metrics for multiclass: accuracy (correct/total), per-class precision/recall/F1 (classification_report shows each class), macro-average (average of per-class metrics equally), weighted average (average weighted by class support), confusion matrix (K×K matrix showing which classes are confused with each other). For imbalanced multiclass, use macro-averaged F1.

Why this matters in AI
Multiclass classification is the dominant paradigm in computer vision (image classification), NLP (sentiment classification, topic classification, NER), and many tabular tasks (product category prediction). Understanding how to extend binary classifiers, evaluate with per-class metrics, and interpret confusion matrices is essential for production ML.

Imbalanced classification

Detailed explanation

Imbalanced classification occurs when classes are not equally represented. Examples: fraud detection (0.1% fraud, 99.9% legitimate), medical diagnosis (rare disease: 1% positive), churn prediction (5–15% churn). A classifier that always predicts the majority class achieves high accuracy (99.9%) but is useless — it never detects fraud. Standard algorithms trained on imbalanced data tend to ignore the minority class.

Resampling strategies: (1) oversampling — duplicate minority class samples or generate synthetic ones (SMOTE), (2) undersampling — remove majority class samples (random undersampling, Tomek links), (3) combined — both oversample minority and undersample majority. SMOTE (Synthetic Minority Oversampling Technique) generates synthetic minority samples by interpolating between existing minority samples. Use imbalanced-learn library.

Algorithmic approaches: (1) class_weight='balanced' — automatically adjusts sample weights inversely proportional to class frequency. Trains on the original imbalanced data but penalizes majority class misclassification less. Available in most sklearn classifiers, (2) scale_pos_weight in XGBoost — ratio of negative to positive samples, (3) ensemble methods (BalancedBaggingClassifier, EasyEnsemble) — each bag has a balanced sample.

Threshold tuning: the default 0.5 threshold is rarely optimal for imbalanced data. For 99.9% negative class, a model outputs low probabilities for positives. Lowering the threshold to 0.2 or 0.1 dramatically increases recall (catches more true positives) at the cost of more false positives. Use precision_recall_curve to find the threshold that achieves the desired recall.

Evaluation: NEVER use accuracy for imbalanced classification. Use: AUC-PR (area under precision-recall curve — directly measures performance on the minority class), AUC-ROC, F1-score, F-beta (β > 1 weights recall more than precision — for fraud detection, missing fraud is more costly than a false alarm), Matthews Correlation Coefficient (MCC — a balanced measure even for imbalanced data).

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from imblearn.over_sampling import SMOTE

    ('smote', SMOTE(random_state=42, k_neighbors=5)),
    ('model', LogisticRegression(class_weight='balanced', max_iter=1000))
])
pipe.fit(X_train, y_train)
y_prob = pipe.predict_proba(X_val)[:, 1]
from sklearn.metrics import average_precision_score
print(f'Val AUC-PR: {average_precision_score(y_val, y_prob):.4f}')
# Threshold tuning for desired recall

prec, rec, thresh = precision_recall_curve(y_val, y_prob)
target_recall = 0.90
idx = np.argmax(rec >= target_recall)
print(f'Threshold {thresh[idx]:.3f}: recall={rec[idx]:.3f}, precision={prec[idx]:.3f}')
Why this matters in AI
Imbalanced classification is the norm, not the exception, in fraud detection, medical diagnosis, fault detection, and many other high-value ML applications. Knowing how to handle imbalance — choosing the right evaluation metric, applying SMOTE, and tuning thresholds — separates practical ML engineers from those who just run models on benchmark datasets.
Common mistakes
  • Applying SMOTE before the train/test split — SMOTE generates synthetic samples that are interpolations of real training samples. If done before splitting, synthetic samples leak information into the test set. Always apply SMOTE inside a pipeline or after splitting.
  • Using accuracy to evaluate imbalanced models — a model that always predicts majority class has 99.9% accuracy but 0 recall on the minority class. Use AUC-PR or F1.
  • Not tuning the decision threshold — the default 0.5 threshold is almost always wrong for imbalanced problems. Always plot the precision-recall curve and choose a threshold that meets business requirements.
