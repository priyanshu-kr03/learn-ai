Q-learning

Detailed explanation

Q-learning is a model-free RL algorithm that learns the Q-function Q(s,a) — the expected cumulative reward of taking action a in state s and then acting optimally afterward. Once the Q-function is learned, the optimal policy is simply: always take the action with the highest Q-value: π*(s) = argmax_a Q(s,a). Q-learning is off-policy (can learn from experience generated by any policy) and model-free (doesn't need to know the environment's transition probabilities).

Q-learning update rule: Q(s,a) ← Q(s,a) + α × [r + γ × max_a' Q(s',a') - Q(s,a)]. The bracketed term is the TD error (Temporal Difference error): the difference between the current Q-estimate and the better estimate (r + γ × max Q(s',a')). α is the learning rate. With enough experience, tabular Q-learning converges to the optimal Q-function for finite state-action spaces.

Tabular Q-learning: for small, discrete state and action spaces, represent Q as a table (array) with rows=states and columns=actions. Initialize Q to 0. For each experience (s,a,r,s'), update Q[s][a] using the update rule. Requires visiting all state-action pairs many times — infeasible for large or continuous spaces.

Epsilon-greedy exploration: Q-learning requires exploring the environment to discover that some actions lead to high reward. Purely greedy policy (always take argmax Q) gets stuck if the initial Q-values are uninformative. Epsilon-greedy: with probability ε, take a random action (explore); with probability 1-ε, take the greedy action (exploit). Anneal ε from 1.0 (fully random) to 0.01 (mostly greedy) over training.

Deep Q-Network (DQN): Q-learning with a neural network as the Q-function approximator. Instead of a table Q[s][a], use Q_θ(s, a) = neural_network(s)[a]. Two critical innovations: experience replay (store transitions in a buffer, sample random minibatches for training — breaks correlations between consecutive transitions) and target network (keep a slowly-updated copy of the Q-network as the target — stabilizes training).

Worked example

import numpy as np
import gymnasium as gym
env = gym.make('FrozenLake-v1', is_slippery=False)
n_states, n_actions = env.observation_space.n, env.action_space.n
Q = np.zeros((n_states, n_actions))
alpha = 0.1; gamma = 0.99; epsilon = 1.0
for episode in range(10000):
    s, _ = env.reset()
    done = False
    while not done:
        # Epsilon-greedy action selection
        if np.random.rand() < epsilon:
            a = env.action_space.sample()  # explore
        else:
            a = Q[s].argmax()              # exploit
        s_, r, done, truncated, _ = env.step(a)
        # Q-learning update
        Q[s, a] += alpha * (r + gamma * Q[s_].max() * (not done) - Q[s, a])
        s = s_
    epsilon = max(0.01, epsilon * 0.999)  # anneal epsilon
# Evaluate
wins = sum(env.reset()[0] == 0 or True for _ in range(100))
print(f'Learned Q-table:\n{Q.reshape(4,4,4)[:,:,0].round(3)}')
# DQN with stable-baselines3
from stable_baselines3 import DQN
env = gym.make('CartPole-v1')
model = DQN('MlpPolicy', env, learning_rate=1e-3, buffer_size=10000, batch_size=64,
            exploration_fraction=0.1, exploration_final_eps=0.05, verbose=1)
model.learn(total_timesteps=100000)
from stable_baselines3.common.evaluation import evaluate_policy
mean_r, _ = evaluate_policy(model, env, n_eval_episodes=100)
print(f'DQN mean reward: {mean_r:.1f}')
Why this matters in AI
Q-learning is the foundational value-based RL algorithm. Understanding Q-learning from scratch gives intuition for all value-based methods (DQN, Rainbow, C51). The transition from tabular Q-learning to DQN illustrates the central challenge of deep RL: stabilizing gradient-based optimization of value functions.
Common mistakes
  • Not annealing epsilon — if epsilon stays at 1.0, the agent always acts randomly and never exploits its learned Q-values. If epsilon drops to 0 too fast, the agent stops exploring before finding the optimal policy. Anneal slowly: over 50-80% of total training steps.
  • Using the same network for both Q-estimate and Q-target in DQN — updating the Q-network while using it as the target creates a 'moving target' problem (the target changes with every update). Use a separate target network and update it slowly (every 1000 steps or with soft update τ=0.005).
  • Not using experience replay — sequential RL experience is highly correlated (s_t and s_{t+1} are almost identical). Training a neural network on correlated data causes the network to overfit to recent experience. Experience replay breaks this correlation by training on random historical samples.

Deep Q-Network

Detailed explanation

Deep Q-Network (DQN) is the algorithm that demonstrated deep RL works at scale — it learned to play 49 Atari games from raw pixels, in many cases reaching human-level performance. DQN extends Q-learning by using a neural network to approximate the Q-function (value of taking action a in state s) instead of a lookup table.

The Q-value and Bellman equation: Q(s,a) = expected total discounted reward when taking action a in state s, then following the optimal policy. The Bellman equation gives the recursive update: Q(s,a) = r + γ × max_a' Q(s', a'). In Q-learning, this update is applied iteratively; in DQN, it becomes the training target for the neural network.

Two stabilizing innovations: (1) Experience replay — store transitions (s, a, r, s', done) in a circular buffer. Sample random minibatches for training. This breaks temporal correlation between consecutive transitions (which would cause the neural network to catastrophically forget earlier states) and allows each transition to be used multiple times. (2) Target network — a copy of the Q-network with frozen weights, used to compute training targets. Without a target network, the target moves every gradient step, creating a moving target problem that destabilizes training. The target network is updated every C steps.

DQN architecture for Atari: input = 4 stacked grayscale frames (84×84×4 to capture motion). Three convolutional layers (32/64/64 filters) → Flatten → Two fully connected layers → Output: one Q-value per action. For simple environments (CartPole), a shallow MLP (2-3 layers) suffices. For pixel observations, convolutional layers are essential.

Modern DQN improvements: Double DQN — separates action selection (use online network) from value estimation (use target network) to reduce overestimation bias. Dueling DQN — decomposes Q(s,a) = V(s) + A(s,a) (state value + advantage of each action) — faster learning in states where actions don't matter much. Prioritized Experience Replay — sample transitions with higher TD-error more frequently — focuses learning on surprising transitions.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import torch
import torch.nn as nn
import numpy as np
from collections import deque
import random
# DQN components
class QNetwork(nn.Module):
    def __init__(self, state_dim, n_actions):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(state_dim, 128), nn.ReLU(),
            nn.Linear(128, 128), nn.ReLU(),
            nn.Linear(128, n_actions)
        )
    def forward(self, x): return self.net(x)
class ReplayBuffer:
    def __init__(self, capacity=100_000): self.buf = deque(maxlen=capacity)
    def push(self, *transition): self.buf.append(transition)
    def sample(self, batch_size): return zip(*random.sample(self.buf, batch_size))
    def __len__(self): return len(self.buf)
# Training step
def train_step(online_net, target_net, optimizer, replay_buf, gamma=0.99, batch_size=64):
    if len(replay_buf) < batch_size: return
    states, actions, rewards, next_states, dones = replay_buf.sample(batch_size)
    states = torch.FloatTensor(np.array(states))
    actions = torch.LongTensor(actions)
    rewards = torch.FloatTensor(rewards)
    next_states = torch.FloatTensor(np.array(next_states))
    dones = torch.FloatTensor(dones)
    # Current Q values
    current_q = online_net(states).gather(1, actions.unsqueeze(1)).squeeze(1)
    # Target Q values (use target network)
    with torch.no_grad():
        next_q = target_net(next_states).max(1)[0]
        target_q = rewards + gamma * next_q * (1 - dones)
    loss = nn.MSELoss()(current_q, target_q)
    optimizer.zero_grad(); loss.backward(); optimizer.step()
    return loss.item()
Why this matters in AI
DQN is the breakthrough that started modern deep RL. Understanding its two core innovations (experience replay and target networks) explains why naive Q-learning with neural networks fails, and why these specific fixes work. These concepts reappear in all modern RL algorithms — SAC, TD3, and Rainbow all use experience replay and target networks.
Common mistakes
  • Not using a target network — without a frozen target network, the Q-value targets change every gradient step. The network chases a moving target, causing oscillation or divergence. Always freeze the target network for C steps between updates.
  • Not normalizing/preprocessing states — if state features have very different scales (position: 0-5, velocity: -5 to 5, angle in radians: -0.4 to 0.4), the neural network struggles to learn. Normalize states to [-1, 1] or [0, 1] or standardize with mean/std computed from experience.
  • Forgetting to set torch.no_grad() when computing target values — without no_grad(), the target network's computation is included in the gradient graph. The loss would then include gradients through the target network weights, which should remain frozen during the update step.

Policy gradient

Detailed explanation

Policy gradient methods directly optimize the policy π(a|s) — the mapping from states to action probabilities — using gradient ascent on the expected return. Unlike Q-learning (which learns value functions and derives a policy implicitly), policy gradient methods parameterize the policy directly (usually as a neural network) and update the parameters to maximize expected reward.

REINFORCE algorithm (basic policy gradient): run an episode with the current policy, collect (state, action, reward) tuples. For each timestep t, compute the return G_t = sum of future discounted rewards from t onward. Update policy parameters: θ ← θ + α × G_t × ∇_θ log π_θ(a_t|s_t). Intuition: increase the probability of actions that led to high returns, decrease the probability of actions that led to low returns. The gradient log π_θ(a|s) is the 'direction' to change θ to increase action a's probability.

Actor-Critic: REINFORCE has high variance (returns depend on the full trajectory). Actor-Critic reduces variance by using a critic (value function V(s)) to compute the advantage A(s,a) = Q(s,a) - V(s). Instead of scaling the gradient by G_t (noisy total return), scale by A(s,a) (how much better was this action than average). The actor updates the policy; the critic updates the value function. A2C (Advantage Actor-Critic) is the synchronous version; A3C is asynchronous (parallel workers).

PPO (Proximal Policy Optimization): the most widely used policy gradient algorithm. Key insight: in standard policy gradient, a large gradient step may move the policy too far, causing catastrophic performance degradation. PPO clips the policy update: clip(π_new/π_old, 1-ε, 1+ε) — limits how much the policy can change in one step. This provides stable, monotonic improvement. PPO is used to train: game-playing agents, robotics control, and the RLHF step in ChatGPT/Claude training.

Policy gradient vs Q-learning: policy gradient works directly with continuous action spaces (robot joint angles, continuous force) — you just output a continuous action distribution (Gaussian) and take its gradient. Q-learning requires a discrete action space for the argmax. Policy gradient enables end-to-end differentiable learning of stochastic policies. Q-learning is more sample efficient for discrete action spaces.

import torch
import torch.nn as nn
import gymnasium as gym
import numpy as np
# REINFORCE policy network
class PolicyNet(nn.Module):
    def __init__(self, state_dim, action_dim):
        super().__init__()
        self.net = nn.Sequential(nn.Linear(state_dim, 128), nn.Tanh(), nn.Linear(128, action_dim))
    def forward(self, x): return torch.softmax(self.net(x), dim=-1)
env = gym.make('CartPole-v1')
policy = PolicyNet(4, 2)
optimizer = torch.optim.Adam(policy.parameters(), lr=1e-3)
# Training loop
for ep in range(1000):
    states, actions, rewards = [], [], []
    s, _ = env.reset()
    while True:
        s_t = torch.FloatTensor(s)
        probs = policy(s_t)
        a = torch.multinomial(probs, 1).item()  # sample action
        s, r, done, _, _ = env.step(a)
        states.append(s_t); actions.append(a); rewards.append(r)
        if done: break
    # Compute returns
    G = 0; returns = []
    for r in reversed(rewards):
        G = r + 0.99 * G; returns.insert(0, G)
    returns = torch.tensor(returns); returns = (returns - returns.mean()) / (returns.std() + 1e-8)
    # Policy gradient update
    optimizer.zero_grad()
    loss = -sum(torch.log(policy(s)[a]) * G for s,a,G in zip(states, actions, returns))
    loss.backward(); optimizer.step()
    if ep % 100 == 0: print(f'Ep {ep}: mean reward = {sum(rewards):.1f}')
# PPO with stable-baselines3
from stable_baselines3 import PPO
env = gym.make('CartPole-v1')
model = PPO('MlpPolicy', env, verbose=1, learning_rate=3e-4, n_steps=2048, batch_size=64)
model.learn(total_timesteps=200000)
mean_reward, _ = evaluate_policy(model, env, n_eval_episodes=100)
print(f'Mean reward: {mean_reward:.1f}')
Why this matters in AI
Policy gradient methods are fundamental to the most important AI applications today — RLHF (training ChatGPT to be helpful via human feedback), robotics control, game playing. Understanding policy gradients explains HOW models are aligned with human preferences and how robots learn complex motor skills from reward signals.
Common mistakes
  • Not normalizing returns in REINFORCE — without baseline subtraction or return normalization, gradients are dominated by the scale of rewards rather than relative performance. Normalize returns (subtract mean, divide by std) before computing the policy gradient.
  • Using policy gradient without a baseline for high-variance problems — REINFORCE's high variance makes it impractical for complex environments. Use Actor-Critic or PPO which incorporate a value function baseline.
  • Setting the learning rate too high in PPO — PPO is more stable than vanilla policy gradient but still sensitive to learning rate. Use learning rate scheduling (decay from 3e-4 to 0) and the default PPO hyperparameters from stable-baselines3 as a starting point.

REINFORCE algorithm

Detailed explanation

REINFORCE (Williams, 1992) is the foundational policy gradient algorithm in reinforcement learning. Unlike Q-learning which learns a value function and derives a policy, REINFORCE directly optimizes the policy parameters θ to maximize expected return. The policy is a neural network π(a|s; θ) that outputs action probabilities. The gradient of the expected return with respect to θ is computed and used to update the policy via gradient ascent.

Policy gradient theorem: the gradient of expected return J(θ) = E_π[Σ γᵗ rₜ] with respect to θ is: ∇J(θ) = E_π[∇ log π(a|s; θ) × G], where G is the return (cumulative future reward) from that state. Intuitively: if an action leads to high return G, increase its probability (positive gradient). If G is low or negative, decrease the action's probability. The log-probability trick converts the difficult expected-value gradient into a tractable expectation.

REINFORCE algorithm: (1) run the policy for one full episode, collecting (s_t, a_t, r_t) triples. (2) Compute the return G_t = Σ_{k≥t} γ^{k-t} r_k for each timestep. (3) For each timestep, compute the loss: L_t = -log π(a_t | s_t; θ) × G_t. (4) Backpropagate and update θ with gradient descent on the total loss Σ L_t. The negative sign converts maximization to minimization (standard gradient descent minimizes loss).

Baseline subtraction: the high variance of REINFORCE is its main problem — returns fluctuate a lot between episodes. Subtracting a baseline b from G_t: L_t = -log π(a_t | s_t; θ) × (G_t - b) reduces variance without introducing bias. The most common baseline is V(s_t) — the estimated value of the current state. G_t - V(s_t) is the advantage function A_t — how much better this action was vs the average action. REINFORCE with baseline is the precursor to actor-critic methods.

REINFORCE vs Q-learning: REINFORCE directly optimizes the policy (on-policy, policy-based). Q-learning learns a value function and derives a policy (off-policy, value-based). Policy gradient methods handle continuous action spaces naturally (output action distribution parameters like mean/std of a Gaussian), while Q-learning requires discrete actions. Modern RL algorithms (PPO, SAC, TD3) are policy gradient methods with various variance-reduction techniques.

Worked example

import torch
import torch.nn as nn
import gym

class PolicyNet(nn.Module):
    def __init__(self, n_states, n_actions):
        super().__init__()
        self.net = nn.Sequential(nn.Linear(n_states, 64), nn.ReLU(), nn.Linear(64, n_actions), nn.Softmax(dim=-1))
    def forward(self, x): return self.net(x)

env = gym.make('CartPole-v1')
policy = PolicyNet(env.observation_space.shape[0], env.action_space.n)
optimizer = torch.optim.Adam(policy.parameters(), lr=1e-3)
def train_episode():
    states, actions, rewards = [], [], []
    s, _ = env.reset()
    done = False
    while not done:
        probs = policy(torch.FloatTensor(s))
        a = torch.distributions.Categorical(probs).sample()
        s, r, done, _, _ = env.step(a.item())
        states.append(s); actions.append(a); rewards.append(r)
    # Compute returns
    G, returns = 0, []
    for r in reversed(rewards):
        G = r + 0.99 * G; returns.insert(0, G)
    returns = torch.FloatTensor(returns)
    returns = (returns - returns.mean()) / (returns.std() + 1e-8)  # normalize
    # Policy gradient update
    log_probs = torch.stack([torch.log(policy(torch.FloatTensor(s))[a]) for s, a in zip(states, actions)])
    loss = -(log_probs * returns).sum()
    optimizer.zero_grad(); loss.backward(); optimizer.step()
    return sum(rewards)
Why this matters in AI
REINFORCE is the conceptual foundation of all policy gradient methods. Understanding it makes PPO, A3C, TRPO, and SAC comprehensible — they all extend REINFORCE with variance reduction (baselines, advantage estimation) and stability improvements (clipping, trust region). Policy gradients are used in RLHF (Reinforcement Learning from Human Feedback) — the training technique behind ChatGPT and Claude.
Common mistakes
  • Not normalizing returns within an episode — raw returns have high variance (episode returns can range from 10 to 1000). Normalizing returns to zero mean, unit variance within an episode stabilizes training.
  • Using REINFORCE for continuous action spaces without Gaussian policy — for continuous actions, the policy should output the mean and std of a Gaussian: action = Normal(mean, std).sample(). Discrete softmax output only works for discrete actions.
  • Setting learning rate too high — policy gradient methods are sensitive to the learning rate. A too-large update can destroy the policy catastrophically. Start with lr=1e-3 and decrease if training is unstable.

Actor-critic

Detailed explanation

Actor-Critic combines two neural networks: the actor (the policy π(a|s) that decides which action to take) and the critic (the value function V(s) that estimates how good each state is). The key insight: instead of computing the raw return G_t (high variance in REINFORCE), use the advantage A(s,a) = Q(s,a) - V(s) — how much better is this action than the average action in this state. The critic provides A, which the actor uses to update the policy.

Why Actor-Critic reduces variance: in REINFORCE, the policy gradient is: ∇J ∝ G_t × ∇ log π(a_t|s_t). G_t is the total future return from step t, which is very noisy (depends on all future actions). Using the advantage A(s_t,a_t) = r_t + γV(s_{t+1}) - V(s_t) focuses on 'how much better was this specific action than expected' — much lower variance. The critic V(s) serves as a baseline that subtracts out the state-level noise.

A2C (Advantage Actor-Critic): the synchronous, on-policy Actor-Critic algorithm. Uses N parallel environments to collect experience simultaneously, computes advantages with GAE, and updates both actor and critic networks together. Actor loss: -A × log π(a|s) (maximize expected advantage). Critic loss: MSE between V(s) and bootstrapped target r + γV(s'). Entropy bonus: -β × H(π) to encourage exploration.

Training stability: Actor-Critic training is more stable than REINFORCE (advantage vs raw return) but less stable than PPO (no policy clipping). Common instabilities: actor diverges if the critic has not converged yet (the advantage estimates are wrong), critic diverges if the actor changes too fast (the target V(s') is stale). Use gradient clipping and small learning rates.

A3C (Asynchronous Advantage Actor-Critic): multiple asynchronous workers each run their own environment and update the global actor-critic asynchronously. The asynchronous updates introduce noise that acts as regularization. A3C was the first deep RL algorithm to solve Atari games at human level without a GPU, using only asynchronous CPU workers.

Worked example

import torch
import torch.nn as nn
import torch.nn.functional as F
class ActorCritic(nn.Module):
    def __init__(self, state_dim, action_dim):
        super().__init__()
        self.shared = nn.Sequential(nn.Linear(state_dim, 128), nn.Tanh())
        self.actor = nn.Linear(128, action_dim)   # output: action logits
        self.critic = nn.Linear(128, 1)           # output: state value V(s)
    def forward(self, state):
        h = self.shared(state)
        return self.actor(h), self.critic(h)
model = ActorCritic(4, 2)  # CartPole: 4 state dims, 2 actions
optimizer = torch.optim.Adam(model.parameters(), lr=3e-4)
# One update step
state_t = torch.FloatTensor(state)
logits, value = model(state_t)
action = torch.distributions.Categorical(logits=logits).sample()
next_value = model(torch.FloatTensor(next_state))[1].item()
advantage = reward + 0.99 * next_value * (not done) - value.item()
actor_loss = -torch.log_softmax(logits, dim=-1)[action] * advantage
critic_loss = F.mse_loss(value, torch.tensor([[reward + 0.99 * next_value * (not done)]]))
loss = actor_loss + 0.5 * critic_loss
optimizer.zero_grad(); loss.backward(); optimizer.step()

from stable_baselines3 import A2C
from stable_baselines3.common.env_util import make_vec_env
env = make_vec_env('CartPole-v1', n_envs=4)  # 4 parallel environments

model.learn(total_timesteps=200_000)
from stable_baselines3.common.evaluation import evaluate_policy
mean_r, _ = evaluate_policy(model, env, n_eval_episodes=100)
print(f'A2C mean reward: {mean_r:.1f}')
Why this matters in AI
Actor-Critic is the basis for all modern policy gradient algorithms (A3C, PPO, SAC). Understanding the actor-critic structure — and why the advantage function reduces variance compared to raw returns — is essential for understanding why PPO works and for debugging policy gradient training instability.
Common mistakes
  • Using the same loss weight for actor and critic — typical weights: actor_loss × 1.0, critic_loss × 0.5, entropy_loss × 0.01. The critic should be trained harder (0.5 coefficient on a typically smaller-magnitude loss) because accurate value estimates are critical for stable actor updates.
  • Not using gradient clipping — policy gradient losses can have large gradients that destabilize training. Use torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=0.5) before optimizer.step().
  • Training the actor before the critic has converged — early in training, the critic's value estimates are random. Using these random advantages to update the actor adds noise rather than signal. Optionally pre-train the critic for a few steps before training the actor.

Proximal policy optimization

Detailed explanation

Proximal Policy Optimization (PPO) is the most widely used deep RL algorithm in practice — used for training ChatGPT (via RLHF), robot locomotion, game playing, and resource management. PPO improves on the basic policy gradient by solving its key instability problem: too-large policy updates that destabilize training.

The problem with basic policy gradient: REINFORCE updates the policy using: θ ← θ + α × ∇log π_θ(a|s) × G. If the gradient step is too large, the new policy can be very different from the collection policy — but the advantage estimates were computed under the old policy. Using old estimates for a very different policy produces incorrect updates that can catastrophically degrade performance.

PPO's solution — clipped objective: PPO limits how much the policy can change per update. Define the probability ratio r_t(θ) = π_θ(a_t|s_t) / π_θ_old(a_t|s_t). PPO's clipped objective: L^CLIP = E[min(r_t × A_t, clip(r_t, 1-ε, 1+ε) × A_t)] where ε=0.2. The clip prevents r_t from exceeding 1.2 or going below 0.8 — the policy can change, but not too much per update.

PPO components: Actor-Critic architecture — actor (policy network) and critic (value network, estimates V(s) to compute advantages). Generalized Advantage Estimation (GAE) — smoothly interpolates between 1-step TD error and full Monte Carlo return for advantage estimation. Entropy bonus — adds H(π) to the objective to maintain exploration. Multiple epochs — unlike REINFORCE (one update per batch), PPO applies multiple gradient updates on each batch of transitions.

PPO hyperparameters: clip_range=0.2, n_epochs=10 (gradient updates per batch), batch_size=2048, learning_rate=3e-4, gamma=0.99, gae_lambda=0.95 (for GAE), entropy_coef=0.01. These are the defaults in Stable Baselines3 and work well for many environments. The most important: n_epochs (higher = more sample efficiency, lower = more stable).

Why this matters in AI
PPO is the foundation of modern RLHF — it's how ChatGPT, Claude, and other RLHF-trained models are fine-tuned with human feedback. Common mistakes
  • Using too many or too few n_epochs — n_epochs=1 is like REINFORCE (wasteful of data). n_epochs=50 reuses data so many times that the old policy and new policy diverge significantly — the clip_range can't fully prevent this. n_epochs=10 with clip_range=0.2 is the stable default.
  • Not normalizing advantages — advantages with high variance cause large gradient updates even with clipping. Always normalize advantages: (A - mean(A)) / (std(A) + 1e-8) within each batch. Stable Baselines3 does this automatically.
  • Using too few parallel environments — PPO's sample efficiency comes partly from diverse experience. Training with n_envs=1 means all 512 steps come from one trajectory (high temporal correlation). Using n_envs=8 or n_envs=16 provides more diverse transitions and faster learning.

Model-based vs model-free RL

Detailed explanation

Model-free RL learns a policy or value function directly from experience without building an explicit model of the environment. Model-based RL explicitly learns a model of the environment's dynamics (given state s and action a, predict the next state s' and reward r), then uses this model for planning.

Model-free pros and cons: pros — simpler, works for any environment, no model-bias. Cons — sample inefficient (requires millions of environment interactions for complex tasks), cannot plan ahead (each decision is reactive). Examples: Q-learning, DQN, PPO, SAC. Most widely used RL algorithms are model-free because they work without assumptions about environment structure.

Model-based pros and cons: pros — highly sample efficient (can simulate many trajectories using the learned model, reducing real-world interactions), can plan ahead over multiple timesteps. Cons — model errors compound over long horizons (a slightly wrong model produces wildly wrong rollouts after 100 steps), harder to implement correctly. Examples: AlphaGo (MCTS + learned model), Dreamer, World Models.

Dyna-Q: a hybrid approach — learn a model-free Q-function but use the learned environment model to generate simulated experience. Real transitions update the model; the model then generates K synthetic transitions per real transition to augment the experience buffer. This can improve sample efficiency 5-10x in environments where the model is accurate.

When to use model-based: simulation is available (robotics with a physics simulator, games, industrial control), real-world interactions are expensive or slow (physical robots, medical trials), the environment has known structure that can be encoded. When to use model-free: fast simulation environment, environment dynamics are complex or discontinuous, black-box environment with no access to internal state.

import gymnasium as gym
import numpy as np
# Model-based: learn transition dynamics
class LearnedModel:
    def __init__(self, state_dim, action_dim):
        import torch; import torch.nn as nn
        self.net = nn.Sequential(
            nn.Linear(state_dim + action_dim, 256), nn.ReLU(),
            nn.Linear(256, 256), nn.ReLU(),
            nn.Linear(256, state_dim + 1)  # predict next_state + reward
        )
        self.optimizer = torch.optim.Adam(self.net.parameters(), lr=1e-3)
    def train_step(self, states, actions, next_states, rewards):
        import torch
        inputs = torch.cat([states, actions], dim=-1)
        predictions = self.net(inputs)
        pred_states = predictions[:, :-1]
        pred_rewards = predictions[:, -1]
        loss = torch.nn.MSELoss()(pred_states, next_states) + torch.nn.MSELoss()(pred_rewards, rewards)
        self.optimizer.zero_grad(); loss.backward(); self.optimizer.step()
        return loss.item()
    def predict(self, state, action):
        import torch
        inp = torch.FloatTensor(np.concatenate([state, action]))
        with torch.no_grad(): out = self.net(inp)
        return out[:-1].numpy(), out[-1].item()  # next_state, reward
Why this matters in AI
The model-free vs model-based distinction explains why RL algorithms have very different sample efficiencies. AlphaGo needed millions of self-play games (model-free component), but also used MCTS which plans ahead using a learned model. In robotics, real robot time is expensive — model-based RL that can simulate thousands of rollouts per real interaction is essential for practical deployment.
Common mistakes
  • Compounding model errors — using a learned model to simulate 100-step trajectories amplifies any model error exponentially. A model with 99% single-step accuracy becomes 0.99^100 ≈ 37% accurate over 100 steps. Limit model-based rollout depth to 1-5 steps and combine with model-free value estimates.
  • Training the model on off-distribution data — the learned model is only accurate for states and actions it has seen. If the policy explores new regions of the state space, the model's predictions there are unreliable. Periodically retrain the model with the latest experience.
  • Not measuring model prediction error — before using a learned model for planning, evaluate its prediction accuracy on a held-out set. A model with high prediction error will produce misleading simulated trajectories that destabilize policy learning.

Reward shaping

Detailed explanation

Reward shaping is the practice of modifying the reward function to provide more informative feedback to the RL agent, accelerating learning. The natural reward signal in many problems is sparse (only at the end of a long episode: +1 for winning, 0 otherwise). An agent learning from sparse rewards must stumble upon the goal by chance before it can start learning — this can take millions of episodes.

Shaped rewards provide intermediate feedback: in a maze navigation task (natural reward: +1 upon reaching the goal), add: -0.01 penalty per step (encourages efficiency), +0.1 when moving closer to the goal (encourages progress), -1 for hitting walls (discourages invalid moves). These intermediate rewards guide the agent toward the goal faster. Potential-based reward shaping F(s, a, s') = γΦ(s') - Φ(s) is theoretically grounded — it does not change the optimal policy, only the speed of learning.

Reward shaping risks: badly shaped rewards can change the optimal behavior. If you reward the robot arm for moving quickly (to encourage exploration), it may learn to move quickly but not accurately. An agent given intermediate rewards for 'trying' may learn to game those rewards (get the intermediate reward repeatedly) rather than achieving the final goal. This is called reward hacking — the agent finds a loophole in the reward function.

Intrinsic motivation is an alternative to hand-crafted reward shaping. Add exploration bonuses based on novelty or uncertainty: curiosity-driven exploration (Pathak et al., 2017) uses prediction error of a self-supervised model as an intrinsic reward — the agent is rewarded for visiting states that are hard to predict. Count-based exploration adds 1/sqrt(visit_count(s)) as a bonus — more for less-visited states. These methods work without domain knowledge.

Curriculum learning arranges training tasks from easy to hard. Start the agent in simple scenarios (short maze, easy physics) and progressively increase difficulty as it improves. This guides exploration toward useful experiences without hand-crafting rewards. Automatic curriculum learning (POET — Paired Open-Ended Trailblazer) generates new tasks automatically calibrated to the current agent's ability level.

Why this matters in AI
Sparse rewards make RL extremely sample-inefficient — learning from natural reward alone may take millions or billions of environment steps. Reward shaping dramatically accelerates learning in complex environments. Understanding reward shaping is essential for applying RL to real-world problems where natural rewards are rare.
Common mistakes
  • Adding intermediate rewards without checking they do not change the optimal policy — always verify that the optimal policy under the shaped reward is the same as under the original reward.
  • Using negative shaping rewards that dominate the original reward — if the step penalty is too large, the agent learns to end episodes quickly (by failing) rather than reaching the goal.
  • Reward hacking — the agent finds a way to maximize shaped rewards without solving the actual task. Monitor both shaped reward and natural reward to detect gaming.

RLHF overview

Detailed explanation

RLHF (Reinforcement Learning from Human Feedback) is the technique used to align large language models with human preferences and values. Without RLHF, a language model trained purely with next-token prediction (SFT on web text) produces outputs that are often unhelpful, verbose, or potentially harmful. RLHF teaches the model what humans actually prefer, not just what appears in web text.

RLHF pipeline: (1) Supervised Fine-Tuning (SFT) — fine-tune the base model on a dataset of high-quality human-written conversations. This teaches the model the general format of helpful responses. (2) Reward Model Training — collect human preference data: show annotators two model responses to the same prompt, ask which is better. Train a reward model on these comparisons (Bradley-Terry model: P(A > B) = sigmoid(R(A) - R(B))). (3) RL Optimization — use PPO to optimize the LM policy to maximize the reward model's score, with a KL penalty to prevent the policy from diverging too far from the SFT model.

The reward model: the reward model takes (prompt, response) as input and outputs a scalar reward. It is initialized from the SFT model (same architecture) with the final layer replaced by a single scalar output. Training: maximize log σ(R(chosen) - R(rejected)) across all human preference pairs. The reward model essentially learns to predict human preferences, generalizing from the labeled comparison data.

Constitutional AI (Anthropic): an extension of RLHF that reduces the amount of human feedback needed by using AI feedback. Instead of human annotators, a set of principles (the 'constitution') guides an AI to critique and revise its own outputs. RLHF from AI Feedback (RLAIF) scales better than pure human feedback because it can generate millions of preference pairs automatically.

Limitations of RLHF: reward hacking — the policy may find ways to score high on the reward model without being genuinely helpful (reward models are imperfect). Annotation biases — human annotators have preferences and biases that the reward model learns. RLHF creates sycophancy — the model learns to say what users want to hear rather than what is true. Direct Preference Optimization (DPO) is a more recent alternative that bypasses the explicit reward model.

RL in games and robotics

Detailed explanation

Reinforcement learning achieved its most famous breakthroughs in games and robotics. Games provide a closed-loop environment with clear reward signals (win/lose/score), perfect simulation, and the ability to run millions of episodes. These properties made games the ideal testbed for developing RL algorithms that have since been applied to robotics, scheduling, and resource management.

Games as RL environments: Atari (DQN, Rainbow — pixels to actions, score as reward), Chess and Go (AlphaGo/AlphaZero — tree search + neural networks, self-play), StarCraft II (AlphaStar — real-time strategy with imperfect information), Dota 2 (OpenAI Five — multi-agent cooperative play). Each game type tests different RL capabilities: Atari tests perception, Chess tests planning, StarCraft tests strategy under partial information.

AlphaZero architecture: Monte Carlo Tree Search (MCTS) + a neural network that estimates: policy π(a|s) (probability of each move), value V(s) (probability of winning from this position). MCTS uses the neural network's policy to guide tree expansion and the value function to evaluate leaf nodes without full rollout. Self-play: the agent trains by playing against previous versions of itself — no human data or domain knowledge required.

Robot learning: the sim-to-real gap is the central challenge. Policies trained in simulation (fast, parallel) often fail when deployed on real robots (sensor noise, actuator lag, contact dynamics differ). Solutions: domain randomization (vary simulation parameters widely so the real world is just another sample), Sim2Real transfer (additional fine-tuning on real-world data), and real-world data collection with safety constraints.

Real-world RL: beyond games and robotics, RL is used in: data center cooling optimization (Google DeepMind reduced cooling energy 40%), chip design (AlphaChip places components on semiconductor chips), healthcare treatment planning, and recommendation systems (treating each user interaction as a sequential decision problem). The key for real-world RL: the reward signal must be well-designed (easy to measure, aligned with the true objective).

# Simple evaluation loop
def evaluate_policy(env, policy_fn, n_episodes=10):
    total_rewards = []
    for _ in range(n_episodes):
        obs, _ = env.reset()
        episode_reward = 0
        for _ in range(1000):
            action = policy_fn(obs)
            obs, reward, terminated, truncated, _ = env.step(action)
            episode_reward += reward
            if terminated or truncated: break
        total_rewards.append(episode_reward)
    return np.mean(total_rewards), np.std(total_rewards)
# Random policy baseline
mean_r, std_r = evaluate_policy(env, lambda obs: env.action_space.sample())
print(f'Random policy: {mean_r:.1f} ± {std_r:.1f}')

The lessons from game RL (reward shaping, curriculum learning, self-play) directly transfer to industrial RL applications.
Common mistakes
  • Underestimating the reward design challenge — a poorly designed reward function produces agents that technically maximize the reward but behave incorrectly.   • Expecting sim-to-real transfer without domain randomization — a policy trained in a perfect simulator will fail on a real robot due to simulator inaccuracies. Use domain randomization: vary friction, mass, contact properties in simulation to build a robust policy.
