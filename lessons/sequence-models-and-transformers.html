Sequence data

Detailed explanation

Sequence data is any data where order matters and the length can vary. Text, audio waveforms, time series (stock prices, sensor readings, weather), DNA sequences, and video frames are all sequence data. The key property is that element N depends on elements before it — you cannot shuffle the sequence and preserve meaning.

Standard feedforward neural networks process fixed-size inputs where each feature is independent. They cannot handle sequences because: (1) input length varies across examples, (2) position matters (word at position 3 has a different role than at position 7), and (3) long-range dependencies exist (the meaning of 'it' in a sentence depends on a noun mentioned 20 words earlier).

Representing sequences as input to neural networks requires a choice: feed the sequence one element at a time (RNNs), process all elements simultaneously with attention to every other element (Transformers), or use fixed-window convolutions that look at N elements at a time (1D CNNs). Each approach has trade-offs in expressiveness, computational cost, and handling of long dependencies.

Padding is required when batching sequences of different lengths. Shorter sequences are padded with a special padding token to match the longest sequence in the batch. The model must be told which positions are real and which are padding — this is done via attention masks (1 for real tokens, 0 for padding). Computing loss or attention over padding positions would corrupt training.

Tokenization for sequences is task-specific. Text is tokenized into subwords or words. Audio is typically discretized into frames (short windows of 10–25ms). Time series is already numeric — the raw values at each timestep are the input features, possibly augmented with lag features and rolling statistics.

Why this matters in AI
Most real-world AI problems involve sequences. Text, logs, events, transactions, and sensor readings are all sequential. Choosing the right architecture (RNN vs Transformer vs 1D CNN) depends on sequence length, dependency range, and latency requirements.
Common mistakes
  • Feeding padded tokens to a model without an attention mask — the model attends to padding positions and learns spurious patterns.
  • Ignoring sequence order by treating tokens as a bag of words — order is the primary signal in NLP and time series.
  • Using RNNs for very long sequences (>500 tokens) — vanishing gradients prevent learning dependencies spanning many steps. Use Transformers instead.

RNN

Detailed explanation

A Recurrent Neural Network (RNN) processes sequences by maintaining a hidden state — a fixed-size vector that carries information from earlier time steps to later ones. At each time step t, the RNN takes the current input x_t and the previous hidden state h_{t-1}, and produces a new hidden state h_t and optionally an output y_t. The same weights (W, U, b) are applied at every time step — this is parameter sharing across time.

The hidden state update equation is: h_t = tanh(W_hx * x_t + W_hh * h_{t-1} + b). W_hx transforms the input, W_hh transforms the previous hidden state, and tanh squashes the result to (-1, 1). The output at each step (for sequence-to-sequence tasks) is y_t = softmax(W_hy * h_t + b_y). For sequence-to-one tasks (classification), only the final hidden state h_T is used.

The fundamental limitation of vanilla RNNs is the vanishing gradient problem over long sequences. When backpropagating through many time steps (Backpropagation Through Time, BPTT), gradients are multiplied by W_hh at each step. If W_hh has eigenvalues < 1, gradients shrink exponentially. If > 1, they explode. For sequences longer than about 10–20 steps, vanilla RNNs effectively cannot learn long-range dependencies.

Despite this limitation, RNNs are valuable for short-range dependencies and for understanding the intuition behind more advanced architectures. Understanding RNNs makes LSTM and GRU immediately clear — they are RNNs with additional gating mechanisms that control what information to keep and what to discard, solving the vanishing gradient problem.

In PyTorch, nn.RNN(input_size, hidden_size, num_layers, batch_first=True) processes a whole sequence at once. Input shape: (batch, seq_len, input_size). Output: (output, h_n) where output has shape (batch, seq_len, hidden_size) — all hidden states — and h_n has shape (num_layers, batch, hidden_size) — final hidden state.

Why this matters in AI
RNNs are the conceptual foundation for LSTM, GRU, and (historically) attention mechanisms. Understanding how hidden state works and why long-range gradients vanish motivates every architectural improvement that follows.
Common mistakes
  • Using vanilla RNN for sequences longer than ~20 steps — gradients will vanish; use LSTM or GRU instead.
  • Forgetting to reset hidden state between unrelated sequences — pass h_0=None to start fresh or explicitly zero the hidden state between sequences.
  • Not using batch_first=True — default PyTorch RNN expects (seq_len, batch, input_size) but most data is (batch, seq_len, features); use batch_first=True to avoid .transpose() calls.

Hidden state

Detailed explanation

The hidden state in a recurrent neural network (RNN, LSTM, GRU) is the network's internal memory — a dense vector that carries information about all previously seen tokens in the sequence. At each timestep t, the hidden state h_t is computed from the previous hidden state h_{t-1} and the current input x_t: h_t = tanh(W_h × h_{t-1} + W_x × x_t + b). The hidden state is both the output (used for prediction) and the memory (passed to the next timestep).

Interpreting hidden states: each dimension of the hidden state corresponds to some learned feature of the sequence history. Hidden state neurons have been shown to encode: sentiment (positive/negative), position in sequence, whether an open parenthesis has been seen, code indent level, and more. Visualizing hidden state activations (coloring tokens by their hidden state value) reveals what information the network is tracking.

Vanishing gradients in hidden states: in standard RNNs, gradients are computed through the chain rule. For sequences of length T, the gradient signal from step T must propagate back T steps, multiplied by the Jacobian matrix W^T. If the largest eigenvalue of W < 1, gradients vanish exponentially; if > 1, they explode. LSTM and GRU solve this with gating mechanisms that provide paths for gradients to flow unchanged.

Hidden states in transformers: transformer models also have 'hidden states' — the vectors at each position after each layer. But these are NOT recurrent (they don't pass from one position to the next) — they are computed in parallel for all positions. The BERT and GPT representations at each layer are often called 'hidden states' or 'activations'. The final layer hidden state at the [CLS] token is used for classification tasks.

Using pretrained hidden states: the hidden states of a pretrained LLM encode rich semantic information. You can extract these states (often called 'features' or 'embeddings') and use them as features for downstream tasks. This is the 'feature extraction' approach to transfer learning: freeze the LLM, extract hidden states, train a small classifier on top. Contrast with fine-tuning (update all LLM weights).

Why this matters in AI
Understanding hidden states is fundamental to debugging recurrent models and extracting information from pretrained models. When a GRU fails to learn long-range dependencies, the hidden state analysis shows the gradient vanishing. When using BERT as a feature extractor, knowing which layer's hidden states to use (typically the last 4 layers averaged) matters for downstream task performance.
Common mistakes
  • Using only the final hidden state for all tasks — for sequence-to-sequence tasks, you need the hidden states at every position, not just the final one. For classification, the final state may miss early parts of the sequence for long inputs.
  • Confusing h_n (final hidden state) with output (hidden states at all positions) in PyTorch — nn.GRU returns (output, h_n). output has shape (batch, seq_len, hidden) — hidden states at every position. h_n has shape (n_layers × n_directions, batch, hidden) — only the final hidden state.
  • Not detaching hidden states when truncating backpropagation through time — for very long sequences, truncated BPTT requires detaching the hidden state between chunks: h = h.detach() prevents gradients from flowing back into previous chunks.

LSTM

Detailed explanation

Long Short-Term Memory (LSTM) networks were introduced by Hochreiter and Schmidhuber in 1997 to solve the vanishing gradient problem in vanilla RNNs. An LSTM cell maintains two state vectors: the hidden state h_t (short-term memory, same as vanilla RNN) and the cell state c_t (long-term memory, a highway for gradients to flow over many time steps without multiplication by recurrent weights).

The LSTM's secret is three gating mechanisms — learnable sigmoid-activated functions that output values between 0 and 1 to control information flow. The forget gate f_t decides what fraction of the previous cell state to discard. The input gate i_t decides what new information to add. The output gate o_t decides what portion of the cell state to output as the hidden state. Each gate is a small neural network of its own, trained end-to-end with the rest of the model.

Cell state update: c_t = f_t ⊙ c_{t-1} + i_t ⊙ ã_t. The forget gate selectively erases old long-term memory. The input gate selectively writes new information. Because c_t is updated by addition (not multiplication by recurrent weights), gradients can flow backward through many time steps along the cell state highway with minimal decay — solving the vanishing gradient problem for sequences of hundreds to thousands of steps.

Hidden state update: h_t = o_t ⊙ tanh(c_t). The output gate selects which parts of the cell state to expose as the output for this time step. This means the LSTM can maintain information in c_t for many steps without exposing it in h_t until it is relevant.

In practice, LSTMs work very well for: language modeling, named entity recognition, part-of-speech tagging, machine translation (pre-Transformer era), speech recognition, and time series forecasting. They handle sequences of 100–1000 steps reliably. For longer sequences or tasks requiring global context, Transformers are preferred.

LSTM equations at time t:
  f_t = σ(W_f · [h_{t-1}, x_t] + b_f)   # forget gate
  i_t = σ(W_i · [h_{t-1}, x_t] + b_i)   # input gate
  ã_t = tanh(W_c · [h_{t-1}, x_t] + b_c) # candidate values
  c_t = f_t ⊙ c_{t-1} + i_t ⊙ ã_t       # cell state update
  o_t = σ(W_o · [h_{t-1}, x_t] + b_o)   # output gate
  h_t = o_t ⊙ tanh(c_t)                  # hidden state
PyTorch: lstm = nn.LSTM(input_size=50, hidden_size=128, num_layers=2, batch_first=True, dropout=0.3)
output, (h_n, c_n) = lstm(x) — note the tuple (h_n, c_n) vs RNN's single h_n.
h_n shape: (num_layers, batch, hidden_size). Use h_n[-1] (last layer's final hidden state) for classification.
Why this matters in AI
LSTMs are still widely used in production for time series forecasting, NLP preprocessing, and sequence labeling. Understanding the gating mechanisms gives you intuition for why they handle long-range dependencies better than RNNs.
Common mistakes
  • Ignoring the cell state c_n when resuming LSTM inference across chunks — for stateful inference, you must pass both h_n and c_n to the next chunk.
  • Using a single-layer LSTM for complex sequences — stack 2–3 layers (num_layers=2) for better performance; add dropout between layers.
  • Not normalizing or truncating very long sequences — even LSTMs struggle with sequences > 1000 steps and benefit from truncated BPTT.

GRU

Detailed explanation

GRU (Gated Recurrent Unit) is a simplified version of LSTM that uses two gates (reset gate, update gate) instead of LSTM's three (forget, input, output) and combines the cell state and hidden state into a single hidden state. GRUs have fewer parameters than LSTMs, train faster, and perform comparably or better on many sequence tasks.

GRU equations: update gate z_t = sigmoid(W_z × [h_{t-1}, x_t]) — how much of the past to keep. Reset gate r_t = sigmoid(W_r × [h_{t-1}, x_t]) — how much of the past to forget when computing candidate state. Candidate state h_hat_t = tanh(W × [r_t ⊙ h_{t-1}, x_t]) — new state based on current input + filtered past. Final state h_t = (1 - z_t) ⊙ h_{t-1} + z_t ⊙ h_hat_t — interpolation between old and new state.

GRU vs LSTM: LSTM: 4 weight matrices (3 gates + cell), can explicitly store long-term information in the cell state, better for very long sequences (100+ steps). GRU: 3 weight matrices (2 gates), simpler, faster to train, fewer parameters (less prone to overfitting on small datasets). Empirically: choose GRU for speed and simplicity, LSTM if you need to capture very long-range dependencies.

Bidirectional GRU: processes the sequence in both forward and backward directions simultaneously. The hidden state at each position = concatenation of the forward and backward GRU hidden states. This allows each position to have context from both past and future tokens — unlike a unidirectional GRU which only has past context. Bidirectional GRUs excel at classification tasks (sentiment, NER) where the full sequence is available at inference.

Modern alternatives: GRUs and LSTMs have largely been replaced by transformers for most NLP tasks, because transformers can process sequences in parallel (much faster training) and have attention mechanisms that directly model long-range dependencies. However, GRUs/LSTMs remain useful for: online learning (processing one token at a time in a streaming setting), time series with long sequences, and resource-constrained inference (GRU inference is faster than transformer for short sequences).

import torch
import torch.nn as nn
# GRU for sentiment classification
class GRUSentiment(nn.Module):
    def __init__(self, vocab_size, embed_dim=128, hidden_dim=256, n_layers=2, dropout=0.3):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim, padding_idx=0)
        self.gru = nn.GRU(embed_dim, hidden_dim, n_layers, bidirectional=True, batch_first=True, dropout=dropout)
        self.fc = nn.Linear(hidden_dim * 2, 1)  # *2 for bidirectional
        self.dropout = nn.Dropout(dropout)
    def forward(self, x):
        embedded = self.dropout(self.embedding(x))  # (batch, seq_len, embed_dim)
        output, hidden = self.gru(embedded)          # output: (batch, seq, hidden*2), hidden: (layers*2, batch, hidden)
        hidden = torch.cat([hidden[-2], hidden[-1]], dim=1)  # last forward + last backward
        return self.fc(self.dropout(hidden)).squeeze(1)

criterion = nn.BCEWithLogitsLoss()

Why this matters in AI
GRU is the lightweight, efficient alternative to LSTM for sequence modeling. Understanding how GRU gates work — which information to carry forward and which to forget — builds deep intuition for how recurrent networks handle memory, which directly informs understanding of more complex attention mechanisms.
Common mistakes
  • Not packing padded sequences — when batching variable-length sequences, you pad shorter sequences to match the longest. Without packing (torch.nn.utils.rnn.pack_padded_sequence), the GRU processes padding tokens, wasting computation and corrupting the hidden state. Always pack before the GRU and unpack after.
  • Using the wrong hidden state for sequence classification — for classification, use the last hidden state. For Bidirectional GRU, concatenate the last forward and last backward hidden states (not the output at the last position, which is the forward-only final state).
  • Not using dropout between GRU layers — multi-layer GRUs are prone to overfitting. Use dropout=0.3-0.5 in the GRU's inter-layer connections (the dropout parameter in nn.GRU applies between layers, not within a layer).

Padding and masking

Detailed explanation

Padding and masking are techniques for handling variable-length sequences in neural networks, especially transformers and RNNs. Neural networks expect fixed-size inputs; real text sequences have different lengths. Padding adds special tokens (typically 0 or [PAD]) to make all sequences in a batch the same length. Masking tells the model to ignore the padded positions.

Why masking is necessary: without masking, the model treats padding tokens as real content. An attention layer would attend to [PAD] tokens, learning spurious relationships. A loss computed over padding positions includes meaningless error signals. Masking fixes both: the attention mask prevents [PAD] tokens from being attended to, and the loss is only computed at non-padding positions.

Attention mask: a binary tensor (1=real token, 0=padding token). In transformer models, the attention score for each position is computed as: softmax((QK^T / sqrt(d)) + attention_bias) where attention_bias = 0 for real tokens, -∞ for padding tokens. After softmax, -∞ becomes 0, effectively preventing any attention to padding positions.

Padding strategies: right padding (default in HuggingFace) — pad at the end. Left padding — pad at the beginning. For decoder models (GPT-style), left padding is often better because the last token's representation is used for generation. For encoder models (BERT-style), right padding is standard. Dynamic padding: instead of padding all sequences to a fixed maximum length, pad each batch to the maximum length in that batch — reduces compute waste.

Dynamic padding with DataCollator: HuggingFace's DataCollatorWithPadding automatically pads each batch to the batch maximum length during training. This is much more compute-efficient than padding all sequences to the model's maximum length (512 or 2048 tokens) — a batch of short sequences (max_len=50) doesn't need to waste computation on 462 padding positions.

from transformers import AutoTokenizer, DataCollatorWithPadding
# Tokenize variable-length sequences
tokenizer = AutoTokenizer.from_pretrained('bert-base-uncased')
texts = ['Hello world', 'This is a longer sentence that needs more tokens', 'Short']
# With padding to same length
encoded = tokenizer(texts, padding=True, truncation=True, max_length=32, return_tensors='pt')
print('input_ids shape:', encoded['input_ids'].shape)  # [3, 32]
print('attention_mask:\n', encoded['attention_mask'])   # 1s for real tokens, 0s for padding
# Visualize which tokens are real vs padding
for i, text in enumerate(texts):
    n_real = encoded['attention_mask'][i].sum().item()
    n_pad = 32 - n_real
    print(f'Text: {text[:30]:30s} | Real: {n_real:3d} | Padding: {n_pad:3d}')
# Dynamic padding with DataCollator (efficient for training)
from torch.utils.data import DataLoader
from datasets import Dataset
dataset = Dataset.from_dict({'text': texts})
tokenized = dataset.map(lambda x: tokenizer(x['text'], truncation=True, max_length=512), batched=True)
data_collator = DataCollatorWithPadding(tokenizer=tokenizer, return_tensors='pt')
loader = DataLoader(tokenized.remove_columns(['text']), batch_size=2, collate_fn=data_collator)
batch = next(iter(loader))
print('Dynamic padding shape:', batch['input_ids'].shape)  # padded to batch max, not 512
Why this matters in AI
Understanding padding and masking is essential for building correct transformer-based models. A model trained without proper attention masking will learn to attend to [PAD] tokens, wasting capacity and potentially learning spurious patterns. A loss computed over padding positions adds noise to gradient updates. These bugs are subtle — the model still trains and achieves some performance — but proper masking improves convergence speed and final accuracy.
Common mistakes
  • Not passing the attention_mask to the model — HuggingFace models accept attention_mask as an argument. Calling `model(input_ids)` without `model(input_ids, attention_mask=mask)` means the model attends to padding. Always pass the attention_mask.
  • Padding to the model's maximum length instead of the batch maximum — padding every batch to 512 tokens when most sequences are 50 tokens wastes 90% of compute. Use DataCollatorWithPadding to pad to the longest sequence in each batch.
  • Not handling causal (decoder) models differently — in causal language models (GPT, Llama), left padding must be used so that the actual tokens appear at the END of the sequence. The model generates from the last non-padding position. Right padding with decoder models causes the model to generate from a padding position — incorrect and nonsensical.

Attention

Detailed explanation

Attention is a mechanism that allows a neural network to dynamically weight the importance of different input elements when producing an output. Instead of relying on a single fixed-size hidden state to summarize all relevant information (as RNNs do), attention computes a weighted combination of all input representations, with the weights determined by how relevant each input is to the current output.

The original attention mechanism was introduced in 2015 for sequence-to-sequence translation. The decoder, at each output step, produces a query vector. Each encoder hidden state is a key-value pair (in early attention, keys and values are the same — the encoder hidden states). The attention score between the query and each key is computed (usually dot product or a learned function), softmax-normalized to get weights, and then the values are averaged according to these weights. The result is a context vector — a soft lookup into the encoder's memory.

Intuitively, attention answers: 'Given what I am trying to generate right now (the query), which parts of the input (keys) are most relevant, and what information should I extract from them (values)?' When translating 'Je suis étudiant' → 'I am a student', when generating the output word 'student', the model learns to attend most strongly to 'étudiant' in the input.

The power of attention over the fixed-size hidden state bottleneck of vanilla encoder-decoder RNNs was dramatic — it allowed the model to access any part of the input sequence at any point during decoding. Longer sentences became translatable because the model did not have to squeeze all source information into one vector.

Attention weights are also interpretable: you can visualize them as a heatmap over input tokens for each output token, providing a rough explanation of which input the model 'focused on' when producing each output. This interpretability, combined with its effectiveness, made attention the cornerstone of modern NLP.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

Encoder outputs H = [h_1, h_2, ..., h_T] — one vector per source token.
Decoder hidden state at step t: s_t (the query).
Attention scores: e_i = dot(s_t, h_i) for each i in 1..T.
Normalize: α_i = softmax(e_i) → all α_i sum to 1.
Context vector: c_t = Σ α_i * h_i — weighted sum of encoder outputs.
Decode: concat [c_t, s_t] → predict next output token.
Why this matters in AI
Attention solved the information bottleneck of seq2seq models and directly led to the Transformer architecture, which powers all modern LLMs. Understanding attention is prerequisite to understanding BERT, GPT, and every modern NLP system.
Common mistakes
  • Confusing attention weights with probabilities of 'correct alignment' — they are soft, differentiable weights used for weighted averaging, not hard decisions.
  • Assuming high attention weight means causal importance — attention weights can be diffuse or concentrated for reasons unrelated to interpretable features.
  • Using additive attention (Bahdanau) vs dot-product attention (Luong) interchangeably — both work but dot-product scales poorly without scaling by sqrt(d_k), leading to the scaled dot-product in Transformers.

Self-attention

Detailed explanation

Self-attention (also called intra-attention) applies the attention mechanism within a single sequence — each element attends to every other element in the same sequence. Unlike the encoder-decoder attention described above, where a decoder query attends to encoder keys/values, self-attention lets each position in a sequence compute how much it should attend to every other position in the same sequence. This enables every token to gather information from the entire context.

In self-attention, for a sequence of N tokens, each token produces three vectors: a query Q, a key K, and a value V. These are computed by multiplying the token embedding by three learned weight matrices W_Q, W_K, W_V. The attention score between token i and token j is computed as the dot product of Q_i and K_j, scaled by sqrt(d_k) where d_k is the key dimension. Scaling prevents dot products from growing very large in high dimensions, which would cause softmax to saturate and gradients to vanish.

The attention output for token i is the weighted sum of all values, where weights are the softmax-normalized attention scores between token i and all tokens. This means every token's output representation depends on the entire sequence — a word at position 1 and a word at position 99 can directly interact in one layer. In contrast, an RNN would need 98 recurrent steps to propagate information from position 1 to position 99.

Self-attention operates in parallel over all positions simultaneously — no sequential dependency between positions, unlike RNNs. This makes it highly parallelizable on GPUs and is a major reason why Transformers train much faster than RNNs on modern hardware. The computational cost is O(N^2 * d) where N is sequence length and d is model dimension — quadratic in sequence length, which becomes expensive for very long sequences (>4096 tokens).

The context captured by self-attention is bidirectional by default — every token sees every other token including future ones. For language understanding (BERT), this is desirable. For language generation (GPT), future tokens must be masked out so the model cannot 'cheat' by looking at tokens it is supposed to predict — this is causal masking (also called autoregressive masking).

Output = weights @ V → shape (3,4). Each token's output is a weighted mix of all values.
Why this matters in AI
Self-attention is the core operation of every Transformer. Understanding it from first principles makes reading BERT, GPT, and ViT source code straightforward — they are all stacks of self-attention + MLP layers.
Common mistakes
  • Forgetting the sqrt(d_k) scaling — without it, dot products grow with dimension, softmax saturates near one-hot, and gradients vanish.
  • Not applying causal mask for generation models — without masking, the model learns to cheat using future token information during training, then fails during inference when future tokens are unavailable.
  • Confusing the QKV projections with the embedding — the query, key, and value are linear projections of the embedding, not the embedding itself.

Query Key Value

Detailed explanation

Query, Key, and Value (Q, K, V) are the three projections that drive the self-attention mechanism in transformers. Each input token is simultaneously projected into three representations: as a Query ('what information am I looking for?'), as a Key ('what information do I contain?'), and as a Value ('what information do I give when attended to?').

Attention computation: for each Query, compute dot products with all Keys: scores = QK^T / sqrt(d_k). This produces attention weights after softmax. The output is a weighted sum of Values: attention_output = softmax(QK^T / sqrt(d_k)) × V. Intuition: the attention weights measure how much each position's Value information should contribute to the current Query position's representation.

Self-attention vs cross-attention: in self-attention (encoder), Q, K, and V all come from the same sequence. Each token attends to all other tokens in the same sequence. In cross-attention (encoder-decoder models like T5, translation models), Q comes from the decoder, while K and V come from the encoder output. This lets the decoder 'look at' the encoded source sequence while generating the target.

Multi-head attention: instead of one Q, K, V projection, use h parallel attention heads, each with their own Q, K, V projections (smaller dimension: d_k = d_model / h). Each head learns to attend to different aspects: head 1 might learn syntactic dependencies, head 2 semantic similarity, head 3 coreference. The outputs of all heads are concatenated and projected.

Grouped Query Attention (GQA): modern efficiency improvement (used in Llama 2/3, Mistral). Instead of h independent Key-Value projections, use fewer KV heads shared across multiple Query heads. This reduces the size of the KV cache (crucial for long sequences) while maintaining most of the representation quality. GQA is why modern models can serve longer contexts efficiently.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import torch
import torch.nn as nn
import torch.nn.functional as F
import math
# Self-attention from scratch
class SelfAttention(nn.Module):
    def __init__(self, d_model, n_heads):
        super().__init__()
        self.d_model = d_model; self.n_heads = n_heads
        self.d_k = d_model // n_heads
        self.W_q = nn.Linear(d_model, d_model)
        self.W_k = nn.Linear(d_model, d_model)
        self.W_v = nn.Linear(d_model, d_model)
        self.W_o = nn.Linear(d_model, d_model)
    def forward(self, x, mask=None):
        B, T, D = x.shape
        Q = self.W_q(x).view(B, T, self.n_heads, self.d_k).transpose(1, 2)
        K = self.W_k(x).view(B, T, self.n_heads, self.d_k).transpose(1, 2)
        V = self.W_v(x).view(B, T, self.n_heads, self.d_k).transpose(1, 2)
        # Scaled dot-product attention
        scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(self.d_k)
        if mask is not None:
            scores = scores.masked_fill(mask == 0, float('-inf'))
        weights = F.softmax(scores, dim=-1)  # [B, n_heads, T, T]
        out = torch.matmul(weights, V)       # [B, n_heads, T, d_k]
        out = out.transpose(1, 2).contiguous().view(B, T, D)
        return self.W_o(out)
# Test
attn = SelfAttention(d_model=256, n_heads=8)
x = torch.randn(4, 16, 256)  # batch=4, seq_len=16, d_model=256
out = attn(x); print(f'Input: {x.shape}, Output: {out.shape}')  # same shape
Why this matters in AI
Understanding Q, K, V is fundamental to understanding how transformers work and why they are so powerful. The attention mechanism is why BERT can capture long-range dependencies in text (any token can directly attend to any other token), unlike RNNs which must pass information through sequential hidden states. This conceptual understanding is also the prerequisite for understanding Flash Attention, GQA, and KV caching — optimizations critical for production LLM deployment.
Common mistakes
  • Confusing the attention weights with the output — the attention weights are the softmax(QK^T/sqrt(dk)) matrix, which shows where each token attends. The actual output is the weighted sum of VALUES, not of Queries or Keys. The Values are what gets mixed; Q and K only determine the mixing weights.
  • Forgetting to scale by sqrt(d_k) — without the scaling factor, for large d_k, the dot products have large magnitudes. After softmax, large-magnitude inputs produce near-one-hot distributions — the model attends almost exclusively to one position. Scaling prevents this and keeps gradients stable.
  • Not masking causal attention in decoder models — decoder models should only attend to previous positions (not future positions). Without a causal mask (upper triangular -∞ mask), the model sees future tokens during training and learns to rely on them — completely wrong for autoregressive generation.

Multi-head attention

Detailed explanation

Multi-head attention runs several independent self-attention computations in parallel, each with its own Q, K, V projection matrices. These parallel computations are called 'heads'. Each head can potentially learn to attend to different aspects of the sequence — one head may attend to syntactic relationships, another to semantic similarity, another to positional proximity. The outputs of all heads are concatenated and projected back to the model dimension.

With h heads and model dimension d_model, each head operates with dimension d_head = d_model/h. For BERT-base: d_model=768, h=12, so each head has d_head=64. All 12 heads are computed in parallel (by splitting the Q, K, V matrices along the head dimension), making multi-head attention computationally similar to single-head attention of the same total dimension.

The intuition: a single attention head produces one weighted combination of values. But a sentence might have multiple simultaneous relationships — 'The bank near the river is closed' needs to track 'bank' relating to 'river' (disambiguation) and 'bank' relating to 'closed' (predicate). Different heads can independently resolve these different relationships without interfering with each other.

After concatenating all head outputs (resulting in shape (batch, seq_len, d_model)), a final linear projection W_O maps this back to d_model. This output projection allows the model to mix information across heads and learn which combinations of head outputs are most useful.

Empirically, not all heads learn distinct useful patterns. Analysis of trained Transformers shows some heads are redundant, some attend mostly to the current position, and a few are highly specialized. Head pruning — removing redundant heads — can compress models with minimal accuracy loss. But in practice, all heads are trained together and useful specialization emerges from the competitive training dynamics.

Why this matters in AI
Multi-head attention is the building block of every Transformer — BERT, GPT, T5, ViT, Whisper, etc. Every time you use a pre-trained model, multi-head attention is the core computation being executed.
Common mistakes
  • Implementing each head separately with separate weight matrices — this is correct conceptually but inefficient. In practice, pack all head projections into one large matrix and split for efficiency.
  • Forgetting the final output projection W_O — concatenating head outputs without projecting loses the ability to mix head information.
  • Expecting every head to have interpretable specialization — some heads are genuinely specialized, but many are not; head analysis requires visualization tools like BERTviz.

Positional encoding

Detailed explanation

Transformers process all sequence positions in parallel — there is no inherent order. A self-attention layer would produce the same output for 'dog bites man' and 'man bites dog' if position information were not added. Positional encoding solves this by injecting a position-dependent signal into each token's embedding before the transformer processes it.

The original Transformer (Vaswani et al., 2017) uses sinusoidal positional encodings: for each position pos and each dimension i, PE(pos, 2i) = sin(pos/10000^{2i/d_model}) and PE(pos, 2i+1) = cos(pos/10000^{2i/d_model}). This produces unique encoding for each position, and the encodings for nearby positions are similar (smoothly varying) while encodings for distant positions are different. Crucially, this formula allows the model to extrapolate to longer sequences than seen in training.

Learned positional embeddings (used in BERT and GPT) treat position as an embedding lookup: there is a learnable matrix of shape (max_seq_len, d_model), and the embedding for position pos is the pos-th row. This is simpler and empirically often works better than sinusoidal encoding on fixed-length tasks, but cannot extrapolate to sequences longer than max_seq_len seen in training.

Relative positional encodings (used in models like T5, Transformer-XL) encode the relative distance between two positions rather than their absolute positions. For attention between positions i and j, the bias term is based on (i-j). This is more flexible — the model learns general patterns about how far apart tokens should attend — and handles variable-length inputs more naturally.

Rotary Position Embedding (RoPE), used in LLaMA and GPT-NeoX, applies position information through rotation of the query and key vectors before the attention dot product. RoPE has become the dominant positional encoding in modern LLMs because it combines relative position sensitivity with good length extrapolation properties.

Why this matters in AI
Without positional encoding, Transformers are permutation invariant — they cannot distinguish word order. The encoding design determines whether the model can generalize to longer sequences than training, which is a critical property for deployment.
Common mistakes
  • Concatenating instead of adding positional encoding to token embeddings — the standard is addition, not concatenation, to keep the embedding dimension fixed.
  • Using learned positional embeddings and testing on sequences longer than max_seq_len — positions beyond the learned range have no valid embedding and the model fails unpredictably.
  • Adding positional encoding after the first attention layer instead of before the input — it must be added at the very beginning so every attention layer knows position from the start.

Transformers

Detailed explanation

The Transformer architecture, introduced in 'Attention Is All You Need' (Vaswani et al., 2017), completely replaced recurrent networks for sequence modeling in NLP. It is built entirely on self-attention and feedforward layers, with no recurrence. This makes it highly parallelizable, allowing training on much larger datasets with much more compute than was possible with RNNs.

A Transformer encoder consists of N identical layers (N=12 for BERT-base, N=24 for BERT-large, N=96 for GPT-4). Each layer has two sublayers: multi-head self-attention and a position-wise feedforward network (two linear layers with a ReLU/GELU in between). Each sublayer uses a residual connection (output = sublayer(x) + x) and layer normalization (Pre-LN normalizes before the sublayer; Post-LN normalizes after — Post-LN is the original, Pre-LN is now more common).

The feedforward network in each Transformer layer operates independently on each position (position-wise), with no interaction between positions. It has an internal dimension 4× the model dimension — for d_model=512, the FFN has hidden size 2048. This FFN is believed to act as a key-value memory, storing factual associations that the model has learned during pre-training.

Residual connections are critical for training depth. Without them, gradients through 12 or 24 layers would vanish. The residual path allows gradients to flow directly from the loss to early layers. Combined with layer normalization, this makes 12–96 layer Transformers trainable. Layer normalization normalizes the feature dimension at each position to have zero mean and unit variance, stabilizing training.

The full encoder-decoder Transformer has an encoder that reads the input sequence and a decoder that generates the output sequence. The decoder has cross-attention layers that attend to the encoder output in addition to self-attention on its own generated sequence. This full architecture is used in translation (original paper) and text-to-text tasks (T5). BERT uses only the encoder. GPT uses only the decoder.

Why this matters in AI
Transformers are the architecture behind every major LLM (GPT, BERT, T5, LLaMA, Claude, Gemini), vision model (ViT, DINO), audio model (Whisper), and multimodal model (CLIP, Flamingo). Everything in modern AI runs on Transformer blocks.
Common mistakes
  • Implementing Post-LN (normalize after residual) when the codebase expects Pre-LN — the two variants have different stability properties and different optimal learning rates.
  • Forgetting residual connections — without them, training collapses in networks deeper than ~4 layers.
  • Confusing the feedforward dimension (4×d_model) with the model dimension — d_model is the token embedding size; the FFN internal dimension is much larger.

Encoder decoder decoder-only

Detailed explanation

Transformer models come in three architectural variants, each suited for different tasks. Encoder-only models (BERT family) use only the Transformer encoder with bidirectional self-attention. Every token sees every other token, making these models excellent at understanding tasks — classification, NER, question answering — where you need a deep contextual representation of the entire input at once.

Decoder-only models (GPT family) use only the Transformer decoder with causal (autoregressive) self-attention. Each token can only attend to previous tokens, never future ones. This makes these models natural generators: given a prefix, they predict the next token, append it to the prefix, and predict again. All modern large language models (GPT-3, GPT-4, LLaMA, Claude) are decoder-only.

Encoder-decoder models (T5, BART, the original Transformer) use both. The encoder reads the input with bidirectional attention, producing rich contextual representations. The decoder generates the output token by token, using causal self-attention within its own generated output AND cross-attention to attend to the encoder's output. These are natural for seq2seq tasks: translation, summarization, document-to-document tasks where input and output are different sequences.

The fundamental difference: encoder models produce one representation per input token — useful for discriminative tasks. Decoder models produce one token at a time from a probability distribution over the vocabulary — useful for generation. Encoder-decoder models do both in series — understand the input fully, then generate the output. The choice depends on your task: understanding → encoder-only; generation → decoder-only; conditional generation (input → different output) → encoder-decoder.

Modern trend: decoder-only models have become dominant even for tasks historically handled by encoder-decoder (translation, summarization) because large decoder-only models trained on web-scale data develop strong generalization. Instruction-tuned decoder-only models (ChatGPT, Claude) handle classification tasks via in-context learning without task-specific fine-tuning.

Why this matters in AI
Choosing the wrong architecture for your task wastes fine-tuning compute and underperforms. BERT for generation or GPT for token classification are mismatched — understanding this distinction is fundamental to applied NLP.
Common mistakes
  • Using BERT for text generation — BERT's bidirectional attention cannot generate because it looks at future tokens that don't exist yet at inference time.
  • Using GPT for masked token prediction tasks — GPT sees only past context, making it inappropriate for tasks that require bidirectional understanding.
  • Treating encoder-decoder as strictly better than decoder-only — decoder-only LLMs fine-tuned with instructions often outperform smaller encoder-decoder models on translation and summarization.
