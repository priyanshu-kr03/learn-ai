AI vs ML vs DL vs Data Science vs GenAI

Detailed explanation

These five terms form a hierarchy. Artificial Intelligence (AI) is the broadest field — any technique that allows machines to mimic human intelligence: rule-based systems, expert systems, planning algorithms, and modern ML. Think of AI as the entire mountain. Machine Learning (ML) is a subset of AI where systems learn from data without being explicitly programmed. Instead of coding rules (if email contains 'urgent wire transfer' → spam), ML learns from examples.

Deep Learning (DL) is a subset of ML that uses neural networks with many layers (deep networks). While classical ML algorithms (random forests, SVMs, logistic regression) work well on structured/tabular data with engineered features, deep learning excels at raw unstructured data: images, audio, text, video. A CNN learns to detect edges → shapes → objects automatically, without hand-engineered features.

Data Science is a broader discipline that combines statistics, data engineering, ML, and domain expertise to extract insights from data and inform decisions. A data scientist might build ML models, but also writes SQL, creates dashboards, designs experiments, and communicates findings to business stakeholders. Not all data science involves ML — descriptive statistics and visualizations are also data science.

Generative AI (GenAI) is the newest sub-field, focusing on models that generate new content: text (GPT-4, Claude), images (Stable Diffusion, DALL-E), code (GitHub Copilot), audio (Suno), and video (Sora). These models are typically large neural networks (LLMs, diffusion models) trained on vast internet-scale datasets. GenAI is a subset of DL, which is a subset of ML, which is a subset of AI.

Practical distinctions: if you want to predict a number (house price), that is ML regression. If you want to classify images, that is DL (CNNs). If you want to generate text or images, that is GenAI. If you want to analyze a dataset and answer business questions, that is Data Science. If you want to build a robot that navigates a room, that is AI (combining computer vision DL + reinforcement learning RL + path planning AI).

Why this matters in AI
Understanding these distinctions prevents confusion in job titles, research papers, and tool selection. A 'machine learning engineer' and a 'data scientist' have different roles. Knowing where each technique fits helps you choose the right tool: classical ML for tabular data, DL for images/text, GenAI for generation tasks.
Common mistakes
  • Thinking AI = GenAI — traditional AI (rule-based systems, game-playing algorithms) predates neural networks by decades.
  • Thinking all ML is deep learning — random forests and gradient boosting outperform neural networks on most tabular/structured data problems.
  • Thinking data science is the same as ML engineering — data scientists focus on insights and analysis; ML engineers focus on building and deploying production models.

Correct learning order

Detailed explanation

Learning machine learning effectively requires building knowledge in the right order — foundations before frameworks, intuition before math, simple models before complex ones. Trying to learn transformers before understanding gradient descent, or fine-tuning GPT before understanding what a neural network is, leads to cargo-cult ML: code that sometimes works but cannot be debugged or improved because the underlying concepts are not understood.

The recommended path: (1) Python programming fundamentals — variables, functions, classes, data structures. Without fluent Python, every ML tutorial becomes twice as hard because you're fighting the language at the same time as the concepts. (2) NumPy and Pandas — the data manipulation layer that all of ML runs on. (3) Basic probability and statistics — mean, variance, distributions, probability, Bayes' theorem. (4) Core ML concepts — supervised/unsupervised learning, train/test split, bias-variance tradeoff. (5) sklearn — the practical ML toolkit. (6) Neural networks — forward pass, backpropagation, gradient descent. (7) Deep learning frameworks (PyTorch). (8) Transformers and LLMs. (9) Production ML — deployment, monitoring, evaluation.

Common mistakes in learning order: jumping to deep learning without understanding sklearn. Using a library without understanding what it computes (using StandardScaler without understanding z-score normalization). Memorizing code patterns without understanding the underlying algorithm. Skipping the math entirely — you don't need to prove theorems, but you need enough math to understand why algorithms work and when they fail.

Projects accelerate learning: implement each concept from scratch before using the library. Implement linear regression with gradient descent in NumPy before using sklearn's LinearRegression. Implement a single attention head before using torch.nn.MultiheadAttention. Building from scratch forces you to understand the mechanism — you can't fake your way through an implementation.

Spaced repetition and active recall: reading tutorials passively doesn't build skill. After reading, close the tutorial and implement the concept from memory. Use spaced repetition (Anki) for key formulas and algorithms. Explain concepts to others (rubber duck debugging, blog posts). Build projects that require combining multiple concepts.

Why this matters in AI
Learning in the right order saves hundreds of hours of confused debugging. A well-ordered curriculum ensures each concept builds on the previous one, making each new concept easier to understand. Skipping steps feels faster initially but creates gaps that cause unexplained failures later.
Common mistakes
  • Skipping NumPy to go straight to PyTorch — PyTorch tensors behave like NumPy arrays. Without NumPy fluency (broadcasting, indexing, vectorization), PyTorch is mysterious.
  • Learning frameworks without understanding what they compute — using adam optimizer without understanding gradient descent, or using LSTM without understanding RNNs. Frameworks hide complexity, which is good for productivity but bad for debugging when things go wrong.
  • Learning only the happy path — tutorials show models that work. Real-world ML involves models that don't work: NaN losses, overfitting, silent bugs. Practice debugging: intentionally break a working model and learn to diagnose the failure.

How to use this website

Detailed explanation

This website is a structured self-learning curriculum for AI and machine learning, organized into 44 chapters covering the full path from Python basics to production AI systems. Each chapter corresponds to a focused topic area, and within each chapter, topics are presented in a logical learning order. The goal is to take you from zero (or wherever you are currently) to being able to build and deploy real AI systems.

How to navigate: the index page shows all 44 chapters grouped by track (Foundations, Core ML, Deep Learning, NLP, LLMs, Production). Click any chapter to open its lesson page. Each lesson page has: an overview explaining what the chapter covers and why it matters, a topics list with links to individual sections, and for each topic: a deep explanation, code examples, key mistakes to avoid, and a hands-on practice task.

Recommended usage pattern: read the explanation deeply (not just skimming), then implement the code examples yourself (type them out, don't copy-paste), then complete the practice task before moving to the next topic. The practice tasks are designed to be substantial — they often take 1-3 hours but build real understanding that passive reading cannot.

Using the search: the search bar on the index page searches across all chapter titles and topic names. If you encounter a concept you don't understand while working on a project, search for it here to find the relevant lesson.

When to revisit topics: the first time you read a concept, you understand it intellectually. The second time you revisit it (after applying it), you understand it deeply. Plan to revisit chapters after you've tried to use the concepts in a project — you'll notice details you missed the first time.

6-month plan

Detailed explanation

A realistic 6-month plan to go from beginner-intermediate Python to building and deploying ML models assumes ~12 hours/week of focused study. The plan is divided into monthly milestones, each building directly on the previous month's work. Progress is measured by projects completed, not hours studied — projects prove understanding that reading cannot fake.

Month 1 — Python and Data Foundations: complete Python basics (data structures, functions, classes, list comprehensions), NumPy (arrays, broadcasting, vectorization), Pandas (DataFrames, groupby, merge), and Matplotlib/Seaborn (basic EDA plots). Project: perform a complete EDA on a real dataset (Titanic, housing prices) and write a 1-page analysis report.

Month 2 — Core Machine Learning: supervised learning (linear regression, logistic regression, decision trees, random forests), unsupervised learning (k-means, PCA), the full sklearn pipeline (preprocessing, cross-validation, hyperparameter tuning). Statistics: probability, distributions, hypothesis testing. Project: build a churn prediction model with full pipeline, cross-validation, and feature importance analysis.

Month 3 — Advanced ML and First Neural Network: gradient boosting (XGBoost, LightGBM), feature engineering, handling imbalanced data, neural network fundamentals (forward pass, backpropagation), PyTorch basics. Project: Kaggle competition — aim for top 30% on a tabular competition using everything from months 1-3.

Month 4 — Deep Learning and NLP: CNNs and transfer learning for images, RNNs and transformers for text, BERT fine-tuning, tokenization. Month 5 — LLMs and RAG: OpenAI/Anthropic APIs, prompt engineering, RAG pipeline (embeddings, FAISS, chunking, retrieval), agents with tools. Month 6 — Production ML: FastAPI serving, Docker, MLflow experiment tracking, model monitoring, CI/CD. Final project: a complete end-to-end application (e.g., a document Q&A chatbot with RAG, deployed to the cloud).

Why this matters in AI
A concrete plan with monthly milestones and project-based checkpoints is 10× more effective than studying topics in whatever order feels interesting. Monthly projects create accountability and force you to integrate concepts rather than just accumulate isolated knowledge.
Common mistakes
  • Planning to study every day — life interrupts. A plan that requires 7 days/week will fail. Plan 3 solid 4-hour sessions per week. Missing a day is not a failure; missing a week needs recovery.
  • Moving forward without completing projects — month 2's churn project forces you to integrate everything from month 1. Skipping it to read month 3 material means you never tested whether you actually understood month 2.
  • Setting unrealistic targets — 12 hours/week is ambitious. If you have less time, extend the plan to 9 or 12 months. A slower plan completed is infinitely better than an abandoned one.

12-month plan

Detailed explanation

A structured 12-month AI learning plan transforms an overwhelming subject into a series of achievable monthly goals. Without a plan, learners jump between topics, never building deep expertise. With a plan, each month builds on the previous, creating a compounding knowledge base.

Months 1-3 — Foundations: Python fluency (pandas, numpy, matplotlib), statistics fundamentals (distributions, hypothesis testing, correlation), and basic ML (linear regression, logistic regression, decision trees). Goal: run a complete supervised learning experiment from raw data to evaluation metric. This phase feels slow but every advanced technique builds on these basics.

Months 4-6 — Core ML: scikit-learn mastery (cross-validation, pipelines, feature engineering), gradient boosting (XGBoost, LightGBM), and deep learning introduction (PyTorch tensors, autograd, simple neural networks). Goal: enter a Kaggle competition and reach the top 40% — this gives calibrated difficulty and feedback.

Months 7-9 — Specialization: pick one domain to go deep (NLP, computer vision, or tabular ML). For NLP: transformers, fine-tuning, RAG. For CV: CNNs, object detection, image classification. For tabular: feature engineering, ensembles, AutoML. Goal: build a real project with your chosen domain and deploy it publicly.

Months 10-12 — Production and Portfolio: MLOps (experiment tracking with MLflow, model serving with FastAPI, Docker containers), LLM applications (LangChain, RAG pipeline, agents), and portfolio completion (3 well-documented GitHub projects, a technical blog post, a capstone project). Goal: demonstrate production-readiness to a potential employer.

Why this matters in AI
A 12-month plan works because it front-loads the fundamentals (the boring but essential parts) and progressively increases complexity. The most common mistake in AI learning is skipping to LLMs before understanding gradient descent and cross-validation — these learners get stuck at the first bug because they lack the mental models to debug.
Common mistakes
  • Planning to learn everything at once — trying to cover Python, statistics, linear algebra, ML, and deep learning in month 1 leads to burnout. Focus each month on one area until you can apply it independently, then move on.
  • Following the plan without building projects — reading books and watching tutorials without writing code is passive learning. The plan's deliverables (competition entries, deployed models) are non-negotiable: they force active application.
  • Not adjusting the plan based on what interests you — if after month 6 you discover you love NLP more than computer vision, change the specialization. A rigid plan that doesn't account for your interests creates resentment.

Project-based learning

Detailed explanation

Project-based learning is the most effective way to develop practical AI skills. Tutorials teach syntax; projects teach judgment — which tool to use, how to debug when nothing is documented, how to structure code others can read. Every hour spent building a complete project (from data to deployed model) is worth 5 hours of passive tutorial consumption.

Choosing the right project: the ideal first project is something you care about. If you follow soccer, build a match outcome predictor. If you're interested in health, build a symptom checker. Personal interest sustains motivation through the debugging phases that kill tutorial-only learners. The topic matters less than the commitment to seeing it through to deployment.

Project structure: every AI project should have the same structure: (1) problem definition and success metric, (2) data collection and EDA, (3) baseline model (simple, interpretable), (4) improved model, (5) evaluation and error analysis, (6) deployment (notebook → API → public URL), (7) README and documentation. Projects that stop at step 4 don't demonstrate production capability.

Showcasing on GitHub: a GitHub project that demonstrates real skill has: a clear README with problem statement, methodology, and results, a requirements.txt or environment.yml for reproducibility, clean code organized into functions or classes (not a 500-line notebook), model performance metrics with baseline comparison, and a deployment URL where others can test it live.

Learning by teaching: writing a blog post or tutorial about your project forces you to understand it deeply enough to explain it. When you attempt to explain why a feature was important or why one model beat another, you discover gaps in your understanding. This metacognitive exercise cements knowledge better than any other learning method.

Why this matters in AI
Employers and clients evaluate AI skills through projects, not certificates. A well-documented GitHub project that solves a real problem is more convincing than 10 completed courses. Project-based learning also reveals the skills that courses don't teach: debugging production errors, handling messy real-world data, making tradeoffs between model quality and latency, and writing code that others can maintain.
Common mistakes
  • Abandoning projects when they get hard — every ML project hits a phase where nothing works (the model doesn't improve, the data has a bug you can't find). This phase is the most valuable learning. Learners who push through develop genuine debugging skills. Treat the 'stuck' phase as the core learning experience, not a signal to stop.
  • Choosing projects that are too complex too early — a first project should be completable in 2-4 weeks. End-to-end time matters more than technical complexity. Build a simple text classifier that works and is deployed before attempting a multi-modal model.
  • Not documenting the project properly — a GitHub repo with 50 files and no README is invisible to employers. Always write: what problem this solves, why it matters, how to run it, and what the results were. Add your evaluation metrics comparison with the baseline.

Portfolio mindset

Detailed explanation

A portfolio mindset means treating every project, side experiment, and learning exercise as a potential addition to your public professional portfolio. Instead of accumulating private notebooks that no one can see, you document and share your work incrementally — each project building a public track record of your capabilities and interests.

What makes a strong AI portfolio: breadth AND depth. Breadth: show you can handle different problem types (classification, generation, retrieval, time series). Depth: at least one project where you go beyond basic tutorials — custom architecture, novel dataset, end-to-end deployment, or rigorous evaluation. Three deep projects are more impressive than ten shallow ones.

HuggingFace Hub as portfolio: HuggingFace Hub lets you publish fine-tuned models and datasets. A fine-tuned model with a model card describing training data, evaluation metrics, and intended use is searchable and usable by others. Having a HuggingFace profile with models others have downloaded is a concrete signal of technical contribution.

Open source contributions: contributing to AI open source projects (LangChain, transformers, sklearn) demonstrates both technical skill and the ability to work within a large codebase. A single merged pull request fixing a bug or adding documentation is valuable portfolio material. Start with issues labeled 'good first issue.'

Writing and sharing: blog posts, Twitter/X threads, and LinkedIn posts about what you've learned create a discoverable public record. Technical writing demonstrates communication skills (underrated in ML). Writing 'how I built X' posts for each project creates a permanent, searchable record of your work. Medium, Substack, or a personal site all work.

Why this matters in AI
A portfolio mindset changes what you learn and how you learn it. Instead of 'I'll understand this later,' you think 'how would I explain this to someone reading my README?' This shift from passive to active learning dramatically improves retention. More practically: in a competitive job market, a strong portfolio of public work differentiates you from candidates with the same educational background.
Common mistakes
  • Waiting until projects are 'perfect' to share them — 'perfect' never comes. Ship a working v1 with a clear README. Iterations on a public project (v1 → v2 → v3) demonstrate growth and commitment more than a single perfectly polished project.
  • Copying tutorial projects exactly — a portfolio filled with Titanic survival, MNIST digit classification, and sentiment analysis from the same tutorials everyone does adds no differentiation. Apply the same techniques to a domain you care about using data you found yourself.
  • Not measuring and reporting results — a project without evaluation metrics is a science experiment with no conclusion. Always include: your metric, the baseline performance, your model's performance, and whether the improvement is statistically significant.
