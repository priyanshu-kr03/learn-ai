LLM tokens

Detailed explanation

A token is the basic unit of text that a language model processes. Tokens are not always words — they are subword units produced by a tokenizer. Common English words like 'the', 'is', 'cat' are single tokens. Rare or long words like 'uncharacteristically' may be split into 3–4 tokens. Numbers, punctuation, and code are often tokenized character by character or in small chunks.

The most widely used tokenization algorithm for modern LLMs is Byte Pair Encoding (BPE) or its variants (SentencePiece, Unigram). BPE starts with a vocabulary of individual bytes, then repeatedly merges the most frequent adjacent pair into a single token. This process is run until the vocabulary reaches a target size (GPT-4 uses ~100,000 tokens). The result is a vocabulary where common words are single tokens and rare words are split into pieces.

Tokens directly determine cost and context. OpenAI charges per token. GPT-4 can process 128,000 tokens in its context window. Claude can handle 200,000. A rule of thumb: 1 token ≈ 0.75 English words. So a 128K context window holds roughly 96,000 words — about the length of a novel. Non-English text often tokenizes less efficiently: Japanese or Chinese characters may be 1–2 tokens each, but some scripts (emoji, rare scripts) may need 2–5 tokens per character.

Tokenization explains why LLMs sometimes struggle with character-level tasks like counting letters or reversing words. The model never sees individual characters — it sees tokens. When asked 'how many e's are in strawberry?', the model sees 'strawberry' as a single token and cannot introspect its internal letters. This is a fundamental limitation of subword tokenization.

Token limits shape how you architect LLM applications. Long documents must be chunked, retrieved (RAG), or summarized before being fed to the model. System prompts consume tokens from the context window. Multi-turn conversations accumulate tokens each turn. In production, you track token usage to control latency, cost, and whether the conversation history will overflow the context window.

Token counts determine LLM cost, latency, and context limits. Miscounting tokens leads to truncated prompts, unexpected charges, and application failures in production.
Common mistakes
  • Assuming 1 word = 1 token — a 2000-word document is actually ~2700 tokens. Always count explicitly for cost and context planning.
  • Ignoring system prompt tokens — a 500-token system prompt reduces the usable context for user conversation by 500 tokens every turn.
  • Not checking if a document fits in context before sending it — sending text longer than the context window silently truncates the input.

Context window

Detailed explanation

The context window is the maximum number of tokens an LLM can process in a single forward pass — including the system prompt, all conversation history, retrieved documents, and the model's own response. Everything in the context window is what the model 'knows' and can reference when generating its next token. Anything outside it is invisible to the model.

Context window sizes have grown dramatically. GPT-2 had 1,024 tokens. GPT-3 had 4,096. GPT-4-turbo has 128,000. Claude has up to 200,000. Gemini 1.5 Pro has 1 million. This growth enables processing entire books, codebases, or long conversations in a single call, eliminating many chunking hacks necessary for shorter contexts.

Long contexts have two practical challenges. First, cost and latency increase approximately linearly with context length — a 128K context call is much slower and more expensive than a 4K context call. Second, the 'lost in the middle' problem: research has shown that LLMs are less reliable at attending to information in the middle of very long contexts compared to beginning or end.

Context management is a key skill in LLM engineering. You must track how many tokens are being used (system prompt + all message history + retrieved chunks + reserve for response) and proactively truncate, summarize, or re-chunk when approaching the limit. Conversation summarization — replacing old messages with a short summary — is a common technique for long-running agents.

The context window also determines what few-shot examples and retrieved documents can fit. A RAG system that retrieves 5 documents of 1,000 tokens each uses 5,000 tokens of context just for retrieved content. Careful management of how many and how long retrieved chunks are directly impacts response quality and cost.

Why this matters in AI
Context window overflows cause silent truncation — the model receives an incomplete prompt and generates wrong answers without any error. Proactive context management is essential for production LLM applications.
Common mistakes
  • Assuming the model will warn you when context is exceeded — most APIs silently truncate from the beginning, causing the model to lose the system prompt or early context.
  • Not reserving tokens for the response — if you fill the context window with input, the model's response gets cut off mid-sentence.
  • Keeping every message in history forever for long-running agents — implement rolling window or summary strategy from the start.

Pretraining

Detailed explanation

Pretraining is the process of training a language model from scratch on a massive corpus of text — web pages, books, code, scientific papers, Wikipedia, and more. The pretraining objective for decoder-only LLMs (GPT family) is next-token prediction: given a sequence of tokens, predict the next one. This seemingly simple task forces the model to learn grammar, facts, reasoning patterns, and world knowledge across hundreds of billions of tokens.

The pretraining corpus for modern LLMs is enormous. GPT-3 was trained on ~300 billion tokens from CommonCrawl (filtered web), books, and Wikipedia. LLaMA-3 was trained on 15 trillion tokens. Training data quality matters more than quantity — web-scraped data contains noise, propaganda, and harmful content, so extensive filtering and deduplication are applied before training.

Pretraining is computationally expensive to an extreme degree. GPT-3 required roughly 3,640 petaflop-days of compute — approximately 10,000 NVIDIA V100 GPUs running for 15 days, costing ~$5 million. Training a frontier LLM today costs hundreds of millions of dollars. This cost is amortized across all downstream use cases, which is why fine-tuning a pre-trained model is vastly more practical than pretraining from scratch.

What does the model actually learn during pretraining? By predicting the next token across diverse text, it implicitly learns: language syntax and grammar, factual associations, causal relationships, common reasoning patterns, programming languages, mathematical notation, and stylistic conventions for different genres. These capabilities emerge from the objective without being explicitly programmed.

The Chinchilla scaling laws (Hoffmann et al., 2022) showed that for a given compute budget, the optimal strategy is to train a model of size N on 20N tokens. This corrected earlier practice of training very large models on too little data. Models like LLaMA follow Chinchilla-optimal training even at smaller scales.

Why this matters in AI
Pretraining is why LLMs are so capable out of the box. The foundation model learned from trillions of tokens of human-generated text. Fine-tuning and prompting layer task-specific behavior on top. Understanding pretraining explains what LLMs know, why they hallucinate, and what they cannot do.
Common mistakes
  • Confusing pretraining and fine-tuning — pretraining trains from scratch on trillions of tokens; fine-tuning adapts a pre-trained model on thousands of task-specific examples.
  • Expecting a pretrained model to follow instructions — a raw pretrained model continues text; instruction tuning is required to make it conversational.
  • Underestimating data quality — training on noisy or biased data produces a model that reproduces those biases.

Instruction tuning

Detailed explanation

A pretrained language model completes text. Instruction tuning (supervised fine-tuning, SFT) teaches the model to follow instructions. The model is fine-tuned on a dataset of (instruction, response) pairs, where instructions include tasks like 'Summarize this', 'Write a Python function that...', 'Translate to French'. This shifts the model from text completer to instruction follower.

The instruction tuning dataset typically contains tens of thousands to millions of examples. Early datasets (Alpaca, Dolly) used GPT-3/4 to generate instruction-response pairs automatically. Higher quality datasets use human demonstration — InstructGPT used 13,000 human-written examples. The quality and diversity of instructions matters enormously.

Multi-turn instruction tuning teaches the model to maintain a conversation across multiple turns. The training format is a sequence of [System][User][Assistant][User][Assistant]... turns with special separator tokens. The model is trained to generate only the assistant turns; user turns are given as context. The loss is applied only on assistant turn tokens.

Instruction tuning is relatively cheap compared to pretraining. Fine-tuning LLaMA-7B on Alpaca (52K examples) required ~3 hours on 8 A100 GPUs. The result is a model that follows instructions well without losing the knowledge acquired during pretraining.

Chat templates standardize how instruction-tuning data is formatted. Llama uses [INST]/[/INST] delimiters. ChatML uses <|im_start|>/<|im_end|>. Using the wrong chat template for a model produces poor results because the model was trained to expect a specific input format.

Why this matters in AI
Instruction tuning is the step that makes a raw LLM useful as an assistant. All chat models (ChatGPT, Claude, Gemini) underwent instruction tuning before deployment.
Common mistakes
  • Training with only one instruction type — the model becomes good at Q&A but loses capability on coding or creative tasks.
  • Not masking the instruction from the loss — use loss_mask=0 on instruction tokens; the model should predict only the response.
  • Over-training on instruction data — catastrophic forgetting overwrites pre-trained knowledge. Use low learning rate (1e-5) and few epochs (1–3).

RLHF

Detailed explanation

Reinforcement Learning from Human Feedback (RLHF) aligns language models with human preferences beyond what supervised instruction tuning achieves. After SFT, RLHF collects human preference data (raters compare model response pairs and indicate which they prefer), trains a reward model on these preferences, then uses RL to fine-tune the LLM to produce responses that score highly according to the reward model.

RLHF has three stages. Stage 1: SFT. Stage 2: reward model training — human raters compare SFT-model response pairs; a reward model is trained to predict which response is preferred. Stage 3: RL optimization — the SFT model is further trained with PPO using the reward model as the scoring function, with a KL divergence penalty to stay close to SFT.

The KL divergence penalty is critical — without it, the model reward-hacks by finding responses that score high on the reward model without being genuinely good. Common reward hacking: very long responses, sycophantic responses, or confusing the reward model with surface-level patterns.

Direct Preference Optimization (DPO) is a simpler RLHF alternative that was introduced in 2023. DPO skips the separate reward model entirely and directly fine-tunes the LLM on preference pairs using a loss function derived from the reward maximization objective. DPO is more stable, cheaper, and often matches RLHF performance.

RLHF is why ChatGPT felt dramatically better than GPT-3: the InstructGPT paper showed that a 1.3B-parameter RLHF model was preferred over the 175B pretrained GPT-3 by human raters — alignment matters more than raw scale for user satisfaction.

Why this matters in AI
RLHF is how all major LLMs become aligned with human values and preferences. Without RLHF, LLMs produce harmful or unhelpful outputs. Understanding RLHF gives insight into why modern LLMs behave the way they do and enables building aligned custom models.
Common mistakes
  • Collecting preference data from raters who do not understand the domain — unreliable reward signal for specialized tasks.
  • Omitting the KL penalty — the policy reward-hacks immediately with very long repetitive responses.
  • Using DPO and PPO interchangeably — they optimize the same objective but DPO is simpler and PPO allows more flexible reward shaping.

Temperature

Detailed explanation

Temperature is a parameter that controls the randomness of an LLM's output during text generation. It modifies the probability distribution over the vocabulary before sampling the next token. At each step, the model produces logits for every token in the vocabulary. Softmax converts these to probabilities. Temperature T divides all logits by T before softmax: p_i = exp(logit_i / T) / Σ exp(logit_j / T).

Temperature=1.0 leaves probabilities unchanged. Temperature < 1 (e.g., 0.2) sharpens the distribution — the highest-probability token becomes dominant, making the model more deterministic. Temperature > 1 (e.g., 1.5) flattens the distribution — probability gaps narrow, making the model more random. Temperature=0 approaches greedy decoding (always pick the most likely token).

Choosing temperature depends on the use case. For factual Q&A, code generation, structured extraction: use temperature 0–0.2 for consistent, predictable outputs. For creative writing, brainstorming: use temperature 0.7–1.0 for diversity. For poetry: temperature 1.0+ introduces interesting variety. In production, temperature is one of the most important parameters to tune per use case.

Temperature interacts with top-p (nucleus) sampling. Top-p=0.9 limits sampling to the smallest set of tokens whose cumulative probability ≥ 0.9. Combined with temperature, this removes very low-probability tokens while still allowing diversity among plausible ones.

A common misconception: lowering temperature does not make the model more accurate — it makes it more consistent. If the model has learned wrong information, temperature 0 will reproduce that wrong information more reliably. Temperature only controls randomness, not factual accuracy.

Logits for 4 tokens: [3.0, 1.5, 0.5, 0.1].
Temperature=1.0: softmax → [0.57, 0.28, 0.10, 0.05]. Most likely token has 57% chance.
Temperature=0.5: divide by 0.5 → [6.0, 3.0, 1.0, 0.2]. Softmax → [0.84, 0.14, 0.02, 0.00]. Much sharper.
Temperature=2.0: divide by 2 → [1.5, 0.75, 0.25, 0.05]. Softmax → [0.42, 0.33, 0.19, 0.06]. Flatter.
Sample: draw a random number uniformly in [0,1]. Cumulative sum over probabilities to find which token is selected.
Why this matters in AI
Temperature is the primary knob for balancing determinism and creativity in LLM outputs. Getting this right per use case is a fundamental production LLM engineering skill.
Common mistakes
  • Using temperature=1.0 for factual tasks — model samples less-likely but coherent wrong answers. Use 0.0–0.2 for facts.
  • Using temperature=0 for all tasks — creative tasks become repetitive and formulaic.
  • Confusing temperature with model confidence — a low-temperature output is not more certain about facts, just more consistently wrong if the model has bad knowledge.

top-p

Detailed explanation

Top-p sampling (also called nucleus sampling) is a text generation strategy that samples from the smallest set of tokens whose cumulative probability exceeds p. At each generation step, the model computes a probability distribution over all vocabulary tokens. Top-p sampling: sort tokens by probability (highest first), compute the cumulative sum, take the smallest prefix of tokens whose cumulative probability ≥ p, and sample from this 'nucleus' of tokens.

Intuition: for p=0.9, the nucleus contains enough tokens to account for 90% of the probability mass. High-probability tokens are always in the nucleus; low-probability tokens are included only if needed to reach p. The nucleus size varies dynamically — if the model is very confident (one token has probability 0.95), the nucleus may contain just that one token. If the model is uncertain (50 tokens each with probability 0.02), the nucleus may contain 45 tokens.

Top-p vs top-k: top-k always considers exactly k tokens. If the next token is almost certain (p=0.99 for 'Paris' in 'The capital of France is ___'), top-k=50 still considers 49 other low-probability tokens. Top-p adapts: when the model is certain, the nucleus is small (few tokens); when uncertain, the nucleus is large. Top-p generally produces better quality text than top-k.

Temperature interaction: temperature is applied BEFORE top-p. Temperature < 1 sharpens the distribution (fewer tokens in the nucleus). Temperature > 1 flattens the distribution (more tokens in the nucleus). Typical production settings: temperature=0 for factual tasks (greedy decoding, deterministic), temperature=0.7 + top_p=0.9 for creative writing, temperature=0 for code generation. Anthropic's recommended defaults for Claude: temperature=1 for most tasks.

Combined sampling: modern LLMs typically use: temperature + top_p (most common combination), sometimes also top_k as an additional constraint. The order: (1) apply temperature to logits, (2) convert to probabilities with softmax, (3) apply top_k filter (zero out all but top-k), (4) apply top_p filter (zero out beyond nucleus), (5) renormalize, (6) sample.

# Implement top-p sampling from scratch
import torch
import torch.nn.functional as F
def top_p_sample(logits, temperature=1.0, top_p=0.9):
    # Apply temperature
    logits = logits / temperature
    probs = F.softmax(logits, dim=-1)
    # Sort by probability (descending)
    sorted_probs, sorted_indices = torch.sort(probs, descending=True)
    # Cumulative sum
    cumulative_probs = torch.cumsum(sorted_probs, dim=-1)
    # Remove tokens after nucleus
    sorted_probs[cumulative_probs - sorted_probs > top_p] = 0
    sorted_probs = sorted_probs / sorted_probs.sum()  # renormalize
    # Sample
    idx = torch.multinomial(sorted_probs, 1)
    return sorted_indices[idx]
Why this matters in AI
Top-p sampling is the standard decoding strategy for creative, open-ended text generation. Understanding it explains why the same model prompt gives different outputs on different runs (stochastic sampling) and why temperature = 0 gives deterministic outputs (greedy decoding). Choosing the right temperature and top-p settings for your use case significantly affects output quality.
Common mistakes
  • Using temperature > 0 for factual or code generation tasks — high temperature introduces randomness that causes factual errors and code bugs. Use temperature=0 (greedy) for deterministic, accurate outputs.
  • Setting top_p very close to 1.0 — top_p=0.99 is almost the same as no nucleus filtering. Use top_p=0.9 to 0.95 to meaningfully restrict to the high-probability tokens.
  • Confusing temperature and top_p effects — temperature changes the shape of the entire distribution (scales all logits). Top-p truncates the tail (removes low-probability tokens). Both affect output diversity but through different mechanisms.

max tokens

Detailed explanation

Max tokens (max_tokens or max_new_tokens) is the parameter that limits how many tokens the model generates in a single response. Setting it appropriately is critical for cost control (more tokens = more cost), latency management (more tokens = longer wait), and user experience (response should be complete but not verbose).

Tokens vs words: tokens are not words. GPT-4 and Claude use BPE tokenization where common words are 1 token, less common words are 2-3 tokens, and code/special characters may be many tokens. Rule of thumb: 1 token ≈ 0.75 words in English. 100 tokens ≈ 75 words ≈ a short paragraph. 1000 tokens ≈ 750 words ≈ 1-2 pages. 4000 tokens ≈ 3000 words ≈ a short article.

Setting max_tokens correctly: for classification tasks (output is a category): max_tokens=10 is sufficient (the answer is 'positive' or 'negative'). For summarization: max_tokens = max acceptable summary length (200-500 for paragraph summary). For open-ended generation: set a generous limit based on expected output length, or use streaming so users see the response building up. For production APIs: always set an explicit max_tokens to prevent runaway generation.

Context window: max_tokens for the OUTPUT is separate from the context window limit (max total tokens in the conversation). Claude's 200K context window means the TOTAL conversation (system prompt + all messages + response) can be up to 200K tokens. The output max_tokens must fit within: context_window - input_tokens. If you have a 100K token document as input, you can only generate up to 100K tokens of output.

Truncation and incomplete responses: if the model reaches max_tokens before finishing its response, the output is cut off mid-sentence. Check finish_reason: 'end_turn' means the model naturally finished; 'max_tokens' means it was cut off. Handle cut-off responses by increasing max_tokens, summarizing the input first, or splitting the request into smaller pieces.

Why this matters in AI
Max tokens is a practical production parameter that affects cost, latency, and user experience. Setting it too low truncates responses mid-thought; setting it too high wastes compute and money. Measuring actual token usage across your request distribution informs the right value for each endpoint.
Common mistakes
  • Not setting max_tokens in production — without a max_tokens limit, a model generating a very long response can time out your request, accumulate unexpected costs, and violate latency SLAs.
  • Setting max_tokens the same for all endpoints — a classification endpoint needs max_tokens=5; a document generation endpoint needs max_tokens=2000. Tune per endpoint based on expected output length.
  • Not checking stop_reason — if responses are sometimes getting cut off (stop_reason='max_tokens'), increase max_tokens or summarize the input. Serving truncated responses to users degrades quality silently.

Hallucination

Detailed explanation

Hallucination refers to when an LLM generates text that is factually incorrect, fabricated, or inconsistent with provided context — but presented with the same confident fluency as correct information. The model makes things up: inventing citations, names, dates, statistics, and events that do not exist. The name comes from the model perceiving something that isn't there.

Why do LLMs hallucinate? The fundamental cause is that LLMs are trained to produce fluent, coherent text that fits statistical patterns. The pretraining objective (next-token prediction) does not reward truthfulness explicitly — it rewards text that looks like plausible continuations. When the model encounters a prompt about a rare fact not in training, it generates something that sounds plausible rather than admitting uncertainty.

There are two types. Closed-domain hallucination: the model's response contradicts information explicitly provided in the prompt or retrieved documents. Open-domain hallucination: the model makes up facts not in the prompt — fabricated citations, wrong dates, invented people. Both are serious but have different mitigations.

Mitigation strategies: for factual Q&A, use Retrieval-Augmented Generation (RAG) — provide source documents in context and instruct the model to answer only from those sources. For structured data tasks, use function calling to look up facts from databases. For high-stakes applications, always include human review. Prompt techniques like 'if you don't know, say I don't know' help but are unreliable.

Evaluation of hallucination uses FactScore (checks atomic claims against Wikipedia), RAGAS (checks faithfulness to retrieved documents), or human evaluation. Automated hallucination detection using another LLM as judge is common but imperfect.

Grounding

Detailed explanation

Grounding in AI refers to connecting AI-generated content to verifiable external sources, preventing the model from generating information that cannot be traced back to a real source. A grounded response cites specific retrieved documents; an ungrounded response draws purely from the model's training data, which may be outdated, incorrect, or hallucinated. Grounding is the antidote to hallucination for factual queries.

Why models hallucinate without grounding: LLMs are trained to predict plausible next tokens based on statistical patterns in training data. They generate responses that 'sound right' based on context, even when the specific fact is incorrect or unknown. Without external grounding, an LLM completing 'The CEO of Acme Corp in 2024 is...' will generate a plausible-sounding name even if it doesn't know the answer.

Grounding mechanisms: (1) RAG (Retrieval Augmented Generation) — retrieve relevant documents and provide as context; instruct the model to only use provided context. (2) Tool use / web search — the model calls a search tool to fetch current information before answering. (3) Structured data access — the model queries a database or knowledge graph for specific facts. (4) Code execution — for mathematical or computational questions, execute code to verify the answer.

Grounding vs knowledge-based answering: a model can answer from training knowledge (ungrounded) or from retrieved documents (grounded). Grounded answers are traceable and verifiable; knowledge-based answers cannot be verified without knowing the training data. For factual queries (current events, specific company information, product specifications), always ground responses in retrieved documents.

Evaluating groundedness: RAGAS faithfulness metric measures what fraction of claims in the response are supported by the retrieved context. A faithfulness score of 1.0 means every claim is grounded. Score < 0.8 indicates significant hallucination — the model is generating claims not present in the retrieved context.

import anthropic
client = anthropic.Anthropic()
# Grounded generation: model can ONLY use provided context
grounded_system = 'You must answer ONLY using the provided context. If the context does not contain the answer, say exactly: "This information is not available in the provided context." Never use knowledge from outside the context.'
def grounded_qa(question, context_docs):
    context = '\n\n'.join([f'[Doc {i+1}]\n{doc}' for i, doc in enumerate(context_docs)])
    resp = client.messages.create(
        model='claude-opus-4-8', max_tokens=500,
        system=grounded_system,
        messages=[{'role': 'user', 'content': f'Context:\n{context}\n\nQuestion: {question}'}]
    )
    return resp.content[0].text
# Test: provide context that contains the answer
context = ['Acme Corp was founded in 1995. The current CEO is Sarah Johnson, appointed in 2022.']
print(grounded_qa('Who is the CEO of Acme Corp?', context))
# Test: ask something NOT in context
print(grounded_qa('What is Acme Corp\'s annual revenue?', context))
Why this matters in AI
Grounding is the technical solution to the hallucination problem for factual queries. In high-stakes domains — customer support (product specifications), legal research (case law), medical information (drug interactions) — an ungrounded response that sounds correct but is wrong can cause real harm. Grounding with retrieved documents makes AI responses verifiable and trustworthy.
Common mistakes
  • Weak grounding instructions — 'use the context' is weaker than 'ONLY use the context; never use outside knowledge'. LLMs default to helpful completion and will supplement context with training knowledge unless explicitly prohibited.
  • Not testing the 'cannot answer' case — a grounding system that always generates an answer (even for questions not in the context) is dangerous. Test that it correctly says 'not available' for questions outside the knowledge base.
  • Conflating grounding with accuracy — a response grounded in an incorrect document is still grounded but wrong. Grounding ensures traceability, not accuracy. The accuracy of the source document is a separate concern.

Model limitations

Detailed explanation

LLMs have a knowledge cutoff date — the date after which training data was not collected. Events, people, papers, and products that emerged after the cutoff are unknown to the model. The model may confidently extrapolate stale information when asked about recent events, making knowledge cutoff a hard limitation for time-sensitive applications.

Mathematical and precise reasoning is a weakness for LLMs without tool use. LLMs tokenize numbers and compute on subword patterns — they have no arithmetic unit. Multi-step arithmetic, algebra, and precise computation often fail. The solution is not better prompting but tool use: call a Python interpreter for math, a symbolic solver for algebra.

Context length limitations mean the model cannot process arbitrarily long inputs. Even with 128K-token context windows, very long documents may be poorly attended to (the 'lost in the middle' problem). For documents longer than the context window, chunking with overlap and retrieval is necessary.

LLMs can struggle with formal logic, multi-step deductive reasoning, and problems requiring systematic search. They may produce reasoning that sounds valid but has subtle errors. Chain-of-thought helps but does not guarantee correctness. For critical reasoning tasks, use structured tools (SMT solvers, symbolic math) alongside LLMs.

LLMs are not grounded in the world — they cannot browse the internet, call APIs, or observe real-time data unless explicitly given those tools. A pure LLM is a pattern matcher over text. Its knowledge is entirely from its frozen training data.

Cost and latency

Detailed explanation

Running LLMs in production involves two key resource metrics: cost (how much you pay) and latency (how long a user waits). These trade off against each other and against capability. Larger models are more capable but more expensive and slower. Finding the Pareto-efficient model for each task — 90% of the quality at 5% of the cost — is a core LLM engineering skill.

Cost is primarily determined by token count (input + output) and model tier. GPT-4 costs roughly 30× more per token than GPT-3.5-Turbo. For the same task, a smaller capable model may provide near-equivalent quality at dramatically lower cost. Techniques to reduce cost: use smaller models for classification and routing, cache responses for identical prompts, reduce prompt verbosity, and use batch APIs for non-latency-sensitive workloads.

Latency is primarily determined by model size, output token count (each token requires one forward pass), and infrastructure (API provider throughput, server proximity, current load). Time to First Token (TTFT) is how long until the user sees the first response token — this determines perceived responsiveness more than total generation time.

Streaming responses (receiving tokens as they are generated rather than waiting for the complete response) dramatically improves perceived latency. A 10-second response feels much faster if the user sees the first words within 500ms. Always implement streaming for user-facing LLM applications.

Caching is a powerful latency and cost reduction technique. Exact caching handles identical prompts with Redis or database lookup. Semantic caching stores past (prompt, response) pairs and retrieves them when a similar prompt arrives using embedding similarity. Providers like OpenAI offer prompt caching — identical system prompt prefixes are computed once and reused.

  • Not implementing streaming for user-facing features — users abandon slow-feeling responses.
  • Forgetting output tokens in cost estimation — for tasks generating long outputs, output token cost often exceeds input token cost.
