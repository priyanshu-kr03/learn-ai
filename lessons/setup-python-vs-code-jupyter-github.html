Python installation

Detailed explanation

Python installation is the first step in your ML journey and getting it right saves hours of future debugging. The recommended approach for ML is to install Python via Miniconda or Anaconda rather than the system Python — this gives you an isolated package manager (conda) that handles complex binary dependencies (numpy, scipy, CUDA drivers) much better than pip alone. Go to docs.anaconda.com/miniconda and download the installer for your OS.

After installing Miniconda, always create a separate conda environment for each project: 'conda create -n ml-project python=3.11'. This isolates dependencies — project A requires numpy 1.24 while project B requires numpy 2.0. Without environments, upgrading one package breaks another. Activate with 'conda activate ml-project'. List environments with 'conda env list'.

Python version matters: Python 3.10 or 3.11 is recommended as of 2024. Python 3.12 is newer but some ML libraries (PyTorch, TensorFlow) lag behind in support. Avoid Python 2 — it is dead. Check your Python version: 'python --version' or 'python3 --version'. In conda environments, always use 'python' not 'python3'.

Installing ML packages: after activating your environment, install the core ML stack: 'pip install numpy pandas matplotlib scikit-learn jupyter'. For deep learning: 'pip install torch torchvision' (for CPU) or visit pytorch.org for the CUDA version command. For transformers: 'pip install transformers datasets'. Always install packages inside an active environment, never globally.

Verifying the installation: open Python and run 'import numpy as np; print(np.__version__)'. If it imports without error, the installation succeeded. A common issue: installing packages globally and then opening Jupyter in a different environment — the packages are not found. Solution: install Jupyter inside the environment ('pip install jupyter') and launch from within it.

Why this matters in AI
Correctly setting up Python with isolated environments prevents 'dependency hell' — the state where upgrading one package breaks another. Every ML practitioner must know how to manage environments — it is the foundation that everything else builds on.
Common mistakes
  • Installing packages globally (without an active environment) — this causes conflicts across projects and makes it impossible to recreate the environment on another machine.
  • Using pip and conda interchangeably without care — conda manages both packages and binary dependencies; mixing pip and conda can corrupt the environment. Prefer pip after conda activation for ML packages not available on conda-forge.
  • Not pinning Python version when creating the environment — 'conda create -n ml' uses the latest Python which may not be compatible with all ML libraries yet.

VS Code setup

Detailed explanation

VS Code is the most widely used editor for Python and ML development. A well-configured VS Code setup dramatically improves productivity: IntelliSense for auto-completion, Python debugger for interactive debugging, integrated terminal, and Jupyter notebook support. Setting it up correctly takes 30 minutes but saves hours every week.

Essential extensions: Python (Microsoft) — the core Python extension with IntelliSense, debugging, formatting. Pylance — fast, accurate type checking and IntelliSense. Jupyter — native Jupyter notebook support (run .ipynb files in VS Code). GitLens — enhanced git integration. Ruff — fast Python linter and formatter. autoDocstring — auto-generates docstrings. Even Better TOML — for pyproject.toml files.

Python environment selection: VS Code must know which Python environment to use. Bottom status bar shows the Python version — click to select from available environments. For each project: create a virtual environment (python -m venv .venv), activate it (.venv/bin/activate), install requirements, and select the .venv interpreter in VS Code. This ensures VS Code IntelliSense uses the correct packages.

Settings for ML development: editor.formatOnSave: true (auto-format with Ruff on save), python.formatting.provider: 'ruff', editor.rulers: [88] (show a line at 88 chars for PEP8), files.autoSave: afterDelay (prevent losing work), terminal.integrated.env.osx/linux: set PYTHONPATH to your project root (so imports from src/ work in the integrated terminal).

Debugging Python and notebooks: set a breakpoint by clicking left of the line number. Press F5 to start debugging. The debugger shows variable values, the call stack, and allows step-through execution. For notebooks: click the debugger icon in the cell toolbar. Use 'Run Cell and Inspect' to see all variable values without adding print statements everywhere. Learning to use the debugger instead of print statements saves hours.

    'files.autoSaveDelay': 3000,
    'terminal.integrated.env.osx': {'PYTHONPATH': '${workspaceFolder}/src'},
}
import json; print(json.dumps(settings, indent=2))
# .vscode/launch.json for debugging ML scripts

    'version': '0.2.0',
    'configurations': [{
        'name': 'Python: Current File',
        'type': 'python',
        'request': 'launch',
        'program': '${file}',
        'console': 'integratedTerminal',
        'env': {'PYTHONPATH': '${workspaceFolder}/src'}
    }]
}
print(json.dumps(debug_config, indent=2))
Why this matters in AI
A well-configured development environment is infrastructure for learning. Poor tooling means you spend time fighting your editor instead of learning ML. VS Code with proper Python setup provides IntelliSense (stops you from making typos in API calls), debugging (stops you from using print statements for debugging), and formatting (enforces consistent style automatically).
Common mistakes
  • Not setting PYTHONPATH — without PYTHONPATH including your project root, 'from src.features import FeatureEngineer' fails in both the terminal and Jupyter notebooks. Add PYTHONPATH to settings.json.
  • Using the wrong Python interpreter — if VS Code is using the system Python but your packages are in a virtualenv, IntelliSense won't find your packages. Always verify the interpreter in the status bar.
  • Not using the debugger — using print() everywhere for debugging is 5× slower than setting a breakpoint and inspecting variables interactively in the debugger. Learn F9 (set breakpoint), F5 (start debug), F10 (step over), F11 (step into).

Jupyter Notebook

Detailed explanation

Jupyter Notebook is the standard interactive computing environment for data science and ML. It combines executable code cells, narrative markdown text, mathematical equations (LaTeX), and rich visualizations (plots, tables, images) in a single document. This literate programming format is ideal for exploratory data analysis, prototyping models, and sharing reproducible research.

Cells in Jupyter: (1) code cells — run Python (or R, Julia, others) code and display output (print statements, errors, plots). (2) markdown cells — formatted text with headers, bullet points, bold/italic, links, and LaTeX math ($E = mc^2$). (3) raw cells — unformatted content. Keyboard shortcuts: Shift+Enter runs the current cell and moves to the next. Ctrl+Enter runs in place. Esc then B inserts a new cell below. Esc then M converts to markdown. Esc then Y converts to code.

JupyterLab: the next-generation interface (jupyter lab command) with a file browser, multiple notebooks side-by-side, a terminal, CSV viewer, and extension system. JupyterLab is replacing the classic Notebook interface. Key extensions: jupyterlab-git (version control integration), jupyterlab-lsp (autocomplete, go-to-definition), jupyter-widgets (interactive widgets).

Magic commands: special commands starting with % (line magic) or %% (cell magic). %timeit — time a single expression (averages over multiple runs). %%time — time an entire cell. %matplotlib inline — display plots in the notebook. %who / %whos — list variables. %load_ext autoreload; %autoreload 2 — automatically reload modules when they change. !command — run shell commands: !pip install package, !ls -la.

Best practices for notebooks: (1) restart and run all before sharing — ensures reproducibility (no hidden state from running cells out of order). (2) Don't use notebooks for production code — extract reusable code into .py modules, import them into notebooks. (3) Clear output before committing to git — large outputs inflate repository size. Use nbstripout. (4) Number sections clearly — Chapter 1, 2, 3... makes notebooks easy to navigate.

Why this matters in AI
Jupyter Notebooks are the primary development environment for data science and ML prototyping. Virtually every ML tutorial, research paper reproduction, and Kaggle notebook uses Jupyter. Mastering Jupyter — keyboard shortcuts, magic commands, debugging — is essential for productive ML development.
Common mistakes
  • Running cells out of order and getting confused — Jupyter allows running any cell at any time, creating hidden state (a variable set in cell 10 affects cell 3 if cell 10 is run first). Always restart kernel and run all cells in order before sharing.
  • Putting all code in notebooks rather than .py modules — notebooks are for exploration and communication. Reusable code (data loaders, model classes, training loops) belongs in .py files that are imported into notebooks.
  • Committing notebooks with large output (embedding matrices, full DataFrames) to git — this inflates the repo significantly. Use nbstripout (git hook that automatically strips output before committing) or clear output manually.

Google Colab

Detailed explanation

Google Colab (Colaboratory) is a free cloud-based Jupyter notebook environment with GPU access. It eliminates the need for local GPU hardware for most ML experiments. Key features: free T4 GPU (15GB VRAM), free A100 GPU with Colab Pro+, 12 hours runtime (free), integration with Google Drive (mount your drive to access datasets), and pre-installed ML libraries (PyTorch, TensorFlow, Hugging Face, sklearn, pandas).

Runtime types: CPU (free, unlimited), T4 GPU (free, up to 12 hours per session — disconnects after idle), A100 GPU (Colab Pro+, $10/month), TPU (Tensor Processing Unit, for very large models). Connect a GPU runtime: Runtime → Change Runtime Type → GPU → T4. Check GPU availability: !nvidia-smi shows GPU type and VRAM. Free Colab disconnects after 90 minutes of browser inactivity.

Google Drive integration: mount your Drive to access large datasets and save model checkpoints without re-uploading each session. from google.colab import drive; drive.mount('/content/drive'). Store datasets in Drive/MyDrive/datasets/. Access: df = pd.read_csv('/content/drive/MyDrive/datasets/titanic.csv'). Save models: torch.save(model.state_dict(), '/content/drive/MyDrive/models/checkpoint.pt').

Installing packages: most packages are pre-installed, but some need to be installed each session: !pip install package. These installations persist for the session but are lost on reconnect. For frequently used packages, put the pip install cell at the top of the notebook and run it first.

Colab vs local Jupyter: Colab advantages: free GPU, no setup, easy sharing (share URL like Google Docs), collaborative editing. Colab disadvantages: session timeout (lose progress), limited persistent storage (Drive mount is the workaround), internet required, slower for data loading (data must be in Drive or downloaded). Use Colab for: training models that require GPU, sharing demos, quick prototyping. Use local Jupyter for: long-running experiments, sensitive data, complex custom environments.

Google Colab democratizes access to GPU compute for ML practitioners. For students and early-career practitioners who don't have access to expensive GPU hardware, Colab is the primary way to train deep learning models. Understanding how to work effectively in Colab is a practical skill for anyone starting out in ML.

Virtual environments

Detailed explanation

A virtual environment is an isolated Python installation that contains its own interpreter, libraries, and pip. When you activate a virtual environment, 'python' and 'pip' refer to that environment's versions — not the system-wide Python. This isolation means: project A's packages don't interfere with project B's, you can have numpy 1.24 in one env and numpy 2.0 in another, and you can easily recreate the exact environment on another machine.

venv (built-in): Python 3.3+ includes the venv module. 'python -m venv myenv' creates a directory called myenv/ containing a Python interpreter and empty site-packages. Activate with 'source myenv/bin/activate' (Mac/Linux) or 'myenv\Scripts\activate' (Windows). Deactivate with 'deactivate'. The environment prompt shows '(myenv)' to indicate activation.

conda environments: conda create -n myenv python=3.11 creates an environment in ~/miniconda3/envs/myenv/. Conda environments can also install non-Python packages (CUDA libraries, HDF5, OpenBLAS). conda activate myenv activates it. conda deactivate deactivates. 'conda env list' shows all environments. Conda is preferred for ML because it handles binary dependencies (compiled C extensions) better than pip alone.

Freezing dependencies: once your project works, freeze the exact package versions. For pip: 'pip freeze > requirements.txt'. Recreate on another machine: 'pip install -r requirements.txt'. For conda: 'conda env export > environment.yml'. Recreate: 'conda env create -f environment.yml'. This makes your project reproducible — a collaborator can set up the exact same environment.

Best practices: (1) one environment per project, (2) never install ML packages in the base/system Python, (3) always activate the environment before installing packages or running code, (4) add the environment directory (myenv/) to .gitignore — commit requirements.txt/environment.yml instead, (5) give environments descriptive names ('fraud-detection-model' rather than 'env1').

Why this matters in AI
Virtual environments are the difference between a reproducible ML project and one that 'works on my machine.' They prevent the frustrating experience of a working project breaking after upgrading a package for a different project, and they make it easy to share and deploy projects.
Common mistakes
  • Committing the environment directory to git — the compiled binaries are OS-specific and the directory can be hundreds of MB. Commit requirements.txt instead.
  • Installing into the wrong environment — always check the prompt shows the environment name before running pip install.
  • Using pip without upgrading it first — 'pip install --upgrade pip' before installing packages prevents cryptic installation errors.

pip and requirements.txt

Detailed explanation

pip is Python's package installer. 'pip install package-name' downloads from PyPI (Python Package Index — pypi.org), resolves dependencies, and installs into the current environment. 'pip install numpy==1.24.0' installs a specific version. 'pip install numpy>=1.24,<2.0' installs any version in the range. Understanding pip and requirements.txt is essential for reproducible ML projects.

requirements.txt: a plain text file listing packages and their versions, one per line. Format: 'numpy==1.24.0' (exact version), 'pandas>=2.0' (minimum version), 'scikit-learn' (latest). Best practice: use exact versions (==) for production and approximate versions (>=) for development. Generate with 'pip freeze > requirements.txt'. Install all: 'pip install -r requirements.txt'.

pip freeze vs pip list: 'pip freeze' shows all installed packages in requirements.txt format (includes transitive dependencies). 'pip list' shows a human-readable table. For requirements.txt, use pip freeze. Note: pip freeze includes EVERY package including indirect dependencies — this is good for perfect reproducibility but results in long files. Tools like 'pip-tools' (pip-compile) can separate direct from transitive dependencies.

Common pip commands: 'pip install package' (install latest), 'pip install package==version' (exact version), 'pip uninstall package' (remove), 'pip show package' (show info, including location), 'pip list --outdated' (find outdated packages), 'pip install --upgrade package' (upgrade to latest). For speed: 'pip install --no-cache-dir' avoids using stale cached downloads.

PyPI alternatives: some packages are on conda-forge rather than PyPI (complex binary extensions). For GPU-related packages (CUDA, cuDNN), always follow the official installation guide (e.g., pytorch.org) rather than plain pip install — the GPU version requires matching CUDA version. The command 'pip install torch' installs CPU-only torch; the GPU version needs a specific index URL.

Why this matters in AI
pip and requirements.txt are the foundation of Python dependency management. Without requirements.txt, your ML project cannot be reproduced by a colleague or deployed to a server. Understanding how to use pip correctly prevents hours of debugging environment issues.
Common mistakes
  • Running pip without activating the virtual environment — this installs packages globally, polluting the system Python and potentially breaking other projects.
  • Using requirements.txt with only approximate version constraints (>=) for deployment — deployment environments should use exact versions (==) to prevent surprise breakage when a package updates.
  • Forgetting to regenerate requirements.txt after adding new packages — always run 'pip freeze > requirements.txt' after any 'pip install' to keep requirements.txt current.

Git and GitHub

Detailed explanation

Git is a distributed version control system that tracks changes to files over time. For ML projects, Git solves: 'I changed the model and broke everything' (you can revert to the last working version), 'which model configuration produced that result?' (git log shows what code generated what output), and 'how do I share my work?' (push to GitHub). Every ML project should be a git repository from day one.

Core git workflow: (1) 'git init' — initialize a repository, (2) 'git add file.py' — stage changes for commit, (3) 'git commit -m "add feature"' — save a snapshot of staged changes, (4) 'git push origin main' — upload to GitHub, (5) 'git pull origin main' — download latest changes. 'git status' shows what has changed. 'git log' shows commit history. 'git diff' shows uncommitted changes.

Branching: branches let you work on a feature without affecting the main codebase. 'git checkout -b experiment/bert-fine-tune' creates a new branch. Work, commit, then merge: 'git checkout main && git merge experiment/bert-fine-tune'. For ML: use branches for experiments — 'experiment/bert-vs-roberta', 'feature/add-attention'. This keeps main stable while experiments proceed.

.gitignore for ML projects: never commit large files (datasets, model weights), secrets (API keys), or environment directories. ML .gitignore should exclude: __pycache__/, .ipynb_checkpoints/, *.pyc, .env, venv/, data/raw/, models/weights/, *.h5, *.pt. Use github.com/github/gitignore for templates. Store data in S3/GCS or use DVC (Data Version Control).

GitHub for ML: GitHub stores code history, enables collaboration, and provides CI/CD. GitHub Actions can run tests automatically on every commit. GitHub Releases can tag model versions. Keep your ML repo organized: src/ (code), notebooks/ (exploration), configs/ (hyperparameters), tests/ (validation), README.md (what the project is, how to run it, results).

Why this matters in AI
Git is the foundation of professional software and ML engineering. Without it, you cannot safely experiment, cannot collaborate, and cannot reproduce results. Git is non-negotiable for ML practice.
Common mistakes
  • Committing large datasets or model weights to git — git is not a data store. Large files slow down the repo and may exceed GitHub's file size limit. Use DVC, Git LFS, or cloud storage.
  • Using vague commit messages like 'update' or 'fix bug' — good ML commit messages: 'feat: add LoRA fine-tuning', 'fix: correct data leakage in feature engineering'.
  • Not using .gitignore from the start — accidentally committing .env files with API keys is a critical security risk. Create .gitignore before the first commit.

Project folder structure

Detailed explanation

A well-organized project folder structure makes ML projects reproducible, maintainable, and collaborative. Without structure, ML projects accumulate in Jupyter notebooks named 'final_v2_ACTUALLY_FINAL.ipynb', with undocumented intermediate results, hardcoded paths, and no way to reproduce the original model. A standard structure enforces good practices through convention.

Recommended structure: data/ (raw/, processed/, external/), notebooks/ (exploratory/, reports/), src/ (data/, features/, models/, visualization/), models/ (trained model files), configs/ (hyperparameter YAML files), tests/ (unit tests), docs/ (documentation), and root-level files: requirements.txt, .env, README.md, Makefile or dvc.yaml. The data/ directory should not be committed to git (add to .gitignore) — use DVC or cloud storage for data versioning.

src/ as an installable package: create src/__init__.py and src/setup.py so you can install your code with pip install -e . (editable install). This means you can import from src.features import FeatureEngineer in any notebook or script without sys.path hacks. This is the most important structural decision — it makes your code reusable across notebooks and scripts.

Configuration management: don't hardcode model hyperparameters, file paths, or API keys in scripts. Put hyperparameters in config.yaml (loaded with PyYAML or Hydra), file paths in .env (loaded with python-dotenv), and API keys in environment variables. This makes experiments reproducible (version config YAML alongside code) and prevents secrets from being committed to git.

Reproducibility: a project is reproducible if someone else (or you in 6 months) can run a single command and reproduce your results. Requirements: requirements.txt with pinned versions (pip freeze > requirements.txt), random seed management (set seeds at the start of every training script), and data versioning (DVC tracks which version of the data produced which model). Cookiecutter Data Science provides a good starting template.

Why this matters in AI
Project structure is the invisible infrastructure of reproducible ML. Every ML project with bad structure eventually reaches a point where no one (including the author) knows which script produced the final model, what data it was trained on, or what hyperparameters were used. Good structure makes ML projects maintainable for their entire lifetime.
Common mistakes
  • Keeping all code in notebooks — notebooks are great for exploration but terrible for production code: they execute out of order, have hidden state, and are hard to test. Move reusable code to src/ early.
  • Committing large data files to git — binary files in git balloon the repository size and slow down operations. Use .gitignore for data/ and models/, and use DVC or cloud storage (S3, GCS) for data versioning.
  • Not pinning dependency versions — requirements.txt without pinned versions (numpy>=1.20 instead of numpy==1.24.3) means your project breaks when packages update. Use pip freeze > requirements.txt to capture exact versions.
