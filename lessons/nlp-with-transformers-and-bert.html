BERT architecture

Detailed explanation

BERT (Bidirectional Encoder Representations from Transformers) is a transformer-based language model that reads text in both directions simultaneously (bidirectional) to understand context. Unlike GPT (which reads left-to-right, autoregressive), BERT processes the entire sequence at once, allowing each token's representation to incorporate context from all other tokens in the input. This bidirectionality makes BERT exceptionally good at understanding tasks (classification, NER, QA) but prevents it from generating text.

BERT's input representation: each token is embedded as the sum of three embeddings: (1) token embedding (the meaning of the word), (2) segment embedding (which sentence does this token belong to — A or B, for sentence-pair tasks like NLI), (3) positional embedding (the position of the token in the sequence). Special tokens: [CLS] at the start (its final hidden state is used as the sequence representation for classification tasks) and [SEP] separating sentences.

BERT is pretrained with two objectives: Masked Language Modeling (MLM) — randomly mask 15% of tokens and predict the masked words (this forces the model to use bidirectional context to fill in blanks) and Next Sentence Prediction (NSP) — given two sentences, predict whether B follows A in the original document. MLM is what gives BERT its powerful contextual representations; NSP was later found to have minimal benefit (RoBERTa drops NSP and improves performance).

Fine-tuning BERT: add a task-specific head on top of the pretrained BERT encoder. For classification: a linear layer on the [CLS] representation. For token classification (NER): a linear layer on each token's representation. For question answering: linear layers on each token to predict start and end positions of the answer span. Fine-tuning updates all BERT weights plus the task head, typically for 3–10 epochs on the downstream task.

BERT variants: RoBERTa (larger pretraining, no NSP, dynamic masking — significantly better than BERT). DistilBERT (60% of BERT's parameters, 40% faster, 97% of BERT's performance — uses knowledge distillation). ALBERT (parameter sharing across layers — much smaller, slightly lower performance). DeBERTa (disentangled attention — SOTA on many NLU benchmarks). For most practical tasks, use DeBERTa-base or RoBERTa-base as your starting point.

Why this matters in AI
BERT and its variants are the backbone of modern NLP. They achieve state-of-the-art performance on classification, NER, and question answering with surprisingly little task-specific data. Understanding BERT's architecture explains why bidirectional transformers dominate NLU tasks.

Masked language modeling

Detailed explanation

Masked language modeling (MLM) is the pre-training objective used by BERT. During training, 15% of input tokens are randomly selected: 80% are replaced with a [MASK] token, 10% are replaced with a random token, and 10% are left unchanged. The model must predict the original tokens for the masked positions. This self-supervised task forces the model to learn deep bidirectional context — to predict 'bank' in 'I went to the [MASK] to deposit money', the model must read BOTH the words before AND after the mask.

Why bidirectional context matters: GPT predicts the next token from previous tokens (left-to-right only — unidirectional). BERT's MLM reads the full sentence bidirectionally, learning richer contextual representations. This makes BERT much better for tasks that require understanding full sentence context: named entity recognition, question answering, sentiment classification. GPT's unidirectional training makes it better for generation tasks (each token is generated from the past context).

The 80/10/10 masking strategy: replacing 100% with [MASK] would cause a mismatch at fine-tuning time (no [MASK] tokens appear in real text). Replacing 10% with random tokens encourages the model to maintain useful representations for unmasked tokens (it cannot just pass through tokens that look like real words). The 10% unchanged keeps the model accurate for normal (unmasked) tokens.

Masked span prediction: RoBERTa, SpanBERT, and DeBERTa use variants of MLM that mask contiguous spans (multiple consecutive tokens) rather than individual tokens. Span masking is harder — the model must predict several related tokens simultaneously and learns better representations for multi-word entities and phrases. SpanBERT was particularly effective for span-extraction tasks like SQuAD.

MLM vs other pre-training objectives: Next Sentence Prediction (NSP, also in BERT) predicts whether two sentences appear consecutively in a document. RoBERTa removed NSP and found it unhelpful. T5 uses a span masking variant called 'salient span masking'. XLNet uses permutation language modeling (predict tokens in a random order given all other positions). All these are self-supervised learning objectives that create labels from the data itself — no human annotation needed.

Why this matters in AI
Understanding MLM is essential for understanding BERT and all BERT-family models (RoBERTa, DeBERTa, ALBERT, DistilBERT). These models form the backbone of modern NLP — used for classification, NER, QA, and as the encoder in RAG systems. The MLM pre-training objective explains why BERT understands context so well.
Common mistakes
  • Using BERT for text generation — BERT is a masked language model trained for understanding, not generation. For generation, use GPT-2, GPT-4, T5, or BART (an encoder-decoder). BERT's bidirectional attention doesn't support autoregressive generation.
  • Fine-tuning BERT without the [CLS] token strategy — for classification, use the [CLS] token embedding (the first token's representation) as the input to the classification head. It is designed to capture the aggregate sentence representation.
  • Using BERT with sequences longer than 512 tokens — BERT's positional embeddings only go up to 512 tokens. For long documents, use chunking, sliding windows, or Longformer/BigBird models.

Next sentence prediction

Detailed explanation

Next Sentence Prediction (NSP) is a pre-training task used in the original BERT model alongside Masked Language Modeling (MLM). In NSP, the model is given two sentences A and B and must predict whether B actually follows A in the original document (IsNext) or is a random sentence from the corpus (NotNext). This teaches the model to understand relationships between sentences, not just within individual sentences.

NSP training data construction: (1) take a document corpus, (2) for each training example, 50% of the time select sentence A and its actual following sentence B (label: IsNext), (3) 50% of the time select sentence A and a random sentence from the corpus as B (label: NotNext). Input format: [CLS] Sentence A [SEP] Sentence B [SEP]. The [CLS] token embedding is trained to encode the IsNext/NotNext judgment.

NSP input encoding: token embeddings (which token) + position embeddings (where in the sequence) + segment embeddings (segment A=0 or segment B=1). Segment embeddings distinguish which sentence each token belongs to. The final hidden state of the [CLS] token is passed to a binary classification head to predict IsNext vs NotNext.

Criticism and successors: subsequent research found that NSP was too easy and did not improve downstream performance. The RoBERTa paper showed that removing NSP (training on longer sequences without NSP) improved performance on most benchmarks. ALBERT replaced NSP with Sentence Order Prediction (SOP): given two consecutive sentences, predict which one came first — a harder and more useful task. BERT-variants like RoBERTa, DeBERTa, ALBERT have mostly moved away from NSP.

Why NSP enabled cross-sentence understanding: before BERT, pre-trained models processed one sentence at a time. NSP forced BERT to capture discourse coherence — understanding that 'I went to the store. I bought milk.' are related, while 'I went to the store. The quantum field collapsed.' are not. This enables BERT to excel at tasks requiring multi-sentence understanding: question answering (Q and A are two sentences), natural language inference, document similarity.

Worked example

from transformers import BertTokenizer, BertForNextSentencePrediction
import torch
tokenizer = BertTokenizer.from_pretrained('bert-base-uncased')
model = BertForNextSentencePrediction.from_pretrained('bert-base-uncased')
# Is sentence B the next sentence after A?
sentence_A = 'The cat sat on the mat.'
sentence_B_next = 'It was warm and comfortable.'
sentence_B_random = 'Quantum mechanics describes atomic behavior.'
for s_b, label in [(sentence_B_next, 'IsNext'), (sentence_B_random, 'NotNext')]:
    inputs = tokenizer(sentence_A, s_b, return_tensors='pt')
    with torch.no_grad():
        outputs = model(**inputs)
    probs = torch.softmax(outputs.logits, dim=-1)
    print(f'{label}: P(IsNext)={probs[0,0]:.3f}, P(NotNext)={probs[0,1]:.3f}')
# Sentence pair similarity with BERT CLS token
from transformers import BertModel
bert = BertModel.from_pretrained('bert-base-uncased')
def sentence_similarity(s1, s2):
    inputs = tokenizer(s1, s2, return_tensors='pt', padding=True, truncation=True)
    with torch.no_grad():
        out = bert(**inputs)
    cls_embedding = out.last_hidden_state[:,0,:]  # [CLS] token
    return cls_embedding

Why this matters in AI
NSP shows how pre-training tasks shape the representations a model learns. Understanding NSP explains why BERT's [CLS] token embedding is useful for sentence pair classification tasks (NLI, QA, document ranking) — it was explicitly trained to encode cross-sentence relationships.
Common mistakes
  • Using NSP BERT for sentence similarity — BERT's [CLS] token is trained for NSP (IsNext vs NotNext), not semantic similarity. Using CLS for similarity gives poor results. Use sentence-transformers (SBERT) instead.
  • Assuming all BERT variants support NSP — RoBERTa, DeBERTa, and most modern BERT-like models do NOT have an NSP head. Only the original BERT and Albert (with SOP) have cross-sentence prediction heads.
  • Using NSP for long documents — BERT's maximum sequence length is 512 tokens. For longer documents, use Longformer, BigBird, or chunking strategies.

Subword tokenization

Detailed explanation

Subword tokenization splits words into smaller units (subwords) rather than treating each word as a single token. Why: (1) vocabulary efficiency — with whole-word tokenization, a 50,000-word vocabulary misses rare words ('bioaugmentation', 'COVID-19' → [UNK]). With subword tokens, rare words are split into known subwords ('bio', '##aug', '##ment', '##ation'). (2) Cross-lingual transfer — subwords are shared across languages (English and French share 'nation', 'tion', 'al').

Byte Pair Encoding (BPE): start with individual characters. Iteratively merge the most frequent adjacent pair. Repeat until the vocabulary reaches the target size (e.g., 50,000). GPT-2, GPT-3, GPT-4, and RoBERTa use BPE. Example: 'low', 'lower', 'lowest' might produce tokens 'low', '##er', '##est'. BPE is greedy and deterministic once the merge rules are learned.

WordPiece: similar to BPE but uses a different merge criterion — instead of frequency, merges the pair that maximizes the likelihood of the corpus under the language model. Used by BERT (## prefix indicates a continuation token). 'unbelievable' → 'un', '##believ', '##able'. The ## prefix distinguishes word-initial tokens from word-continuation tokens.

SentencePiece: a language-agnostic subword tokenizer that treats the input as a raw byte stream (no whitespace tokenization). Handles any language including those without word boundaries (Chinese, Japanese). Used by XLNet, T5, LLaMA, Mistral, and most modern open-source LLMs. SentencePiece trains on text to find the subword vocabulary using BPE or unigram language model.

Tokenization in practice: tokenizer.encode('Hello world') returns a list of integer token IDs. tokenizer.decode([15496, 995]) returns the original text. tokenizer(text, return_tensors='pt', truncation=True, max_length=512, padding='max_length') prepares a batch for a transformer model. Understanding tokenization is crucial for: (1) understanding model input length limits, (2) counting tokens for API cost estimation (GPT-4 costs per token), (3) debugging model behavior on OOD text.

Why this matters in AI
Subword tokenization is the foundation of modern NLP. Understanding it explains: why model costs scale with tokens (not characters or words), why rare words and OOV terms are handled correctly, why different models have different vocabulary sizes, and why tokenization affects model performance on specialized domains.
Common mistakes
  • Assuming one token = one word — 'tokenization' might be 4 tokens: 'token', '##iza', '##tion'. A 100-word sentence might be 120–150 tokens. Always count tokens, not words, for context length and API cost estimates.
  • Using the wrong tokenizer for a model — each model has its own tokenizer vocabulary. BERT's tokenizer will produce garbage if used with RoBERTa's model. Always use AutoTokenizer.from_pretrained(model_name).
  • Not handling truncation for long documents — tokenizer(text) with a 10,000-character text raises an error or silently truncates. Always set truncation=True and max_length explicitly.

BPE and WordPiece

Detailed explanation

BPE (Byte Pair Encoding) and WordPiece are the two dominant subword tokenization algorithms in NLP. Both start from characters and iteratively merge subwords to build a vocabulary. Their difference: BPE merges based on pair frequency (the most common adjacent pair is merged), while WordPiece merges based on the mutual information of the pair (the merge that most increases the likelihood of the corpus).

BPE algorithm step by step: (1) initialize vocabulary with all characters plus an end-of-word symbol. (2) Count all adjacent symbol pairs in the corpus. (3) Merge the most frequent pair into a new symbol: 'l o w' → 'lo w'. (4) Update the corpus with the merged symbol. (5) Repeat from step 2 until vocabulary size reaches K. For K=50,000, this runs ~50,000 merge operations, each adding one new token to the vocabulary.

WordPiece algorithm: similar to BPE but the merge criterion is different. Instead of frequency, WordPiece maximizes: log P(corpus with merge) - log P(corpus without merge). This is equivalent to selecting the pair where the merged symbol probability exceeds the product of the individual symbol probabilities — a mutual information criterion. The ## prefix is added to all tokens that are word-internal (not word-starting) in the learned vocabulary.

In practice: for most applications, you do not implement BPE or WordPiece yourself — you use a pre-trained tokenizer. The critical skill is loading the correct tokenizer for your model, understanding the vocabulary, and handling special tokens ([CLS], [SEP], [PAD], [MASK] for BERT; <s>, </s>, <pad>, <unk> for others). Special tokens have fixed IDs and specific roles in the model's attention patterns.

Unigram language model (SentencePiece): an alternative to BPE/WordPiece. Start with a large vocabulary, iteratively REMOVE symbols whose removal minimally decreases the language model likelihood. The result is a vocabulary where each token has a probability score. During tokenization, the segmentation with the highest probability is chosen. More principled than BPE but more complex. Used by T5, ALBERT, XLM-RoBERTa.

Why this matters in AI
BPE and WordPiece explain why different models have different token boundaries and vocabulary sizes. This directly affects: domain adaptation (is medical terminology in the vocabulary?), API cost (token count varies across tokenizers), and model behavior (models with smaller vocabularies that split words more aggressively have different representations for rare words).
Common mistakes

Fine-tuning BERT

Detailed explanation

Fine-tuning BERT (or any pre-trained language model) means taking the pre-trained encoder and adapting it for a specific downstream task by continuing training on task-specific labeled data. BERT was pre-trained on masked language modeling and next sentence prediction — tasks that gave it rich contextual language understanding. Fine-tuning redirects this understanding toward a specific task.

The fine-tuning architecture adds a task-specific head on top of BERT's [CLS] token representation. For classification: a linear layer Dense([CLS_embedding → num_classes]). For token classification (NER): a linear layer at every token position Dense(token_embedding → num_tags). For question answering: two linear layers predicting the start and end positions of the answer span. The entire model (BERT encoder + task head) is trained end-to-end.

Fine-tuning hyperparameters are different from pre-training. Learning rate: very small (1e-5 to 5e-5) to avoid catastrophic forgetting. Batch size: 16–32 (larger batches improve stability). Epochs: 3–5 only (more epochs cause overfitting on small datasets). Warmup steps: linearly increase lr from 0 to target for the first 10% of steps, then use cosine decay. These recommendations come from the original BERT paper.

Fine-tuning requires relatively little labeled data compared to training from scratch. BERT fine-tuned on 1,000 labeled examples typically outperforms a traditional ML model trained on 10,000+ examples because BERT already understands language. For specialized domains with very little data (< 100 examples), few-shot or prompt-based approaches may work better.

Evaluation during fine-tuning should use the same metrics as production. For classification: F1-macro (balanced across classes, good for imbalanced datasets). For NER: entity-level F1 (strict matching of entity span and type). For QA: exact match and F1. Always evaluate on a held-out test set, never on the validation set used for early stopping.

Why this matters in AI
Fine-tuning BERT is the standard approach for NLP classification, NER, and QA with moderate amounts of labeled data. Understanding how to fine-tune a pre-trained model is the most practical NLP engineering skill after prompting.
Common mistakes
  • Using a learning rate too high (e.g., 1e-3) — catastrophic forgetting destroys all pre-trained knowledge in the first few steps.
  • Training for too many epochs (>5) on small datasets — the model memorizes the training set and validation performance degrades.
  • Using [CLS] token representation for token-level tasks like NER — use all token representations, not just [CLS].

Sentence-BERT

Detailed explanation

Sentence-BERT (SBERT) is a modification of BERT that produces fixed-size sentence embeddings optimized for semantic similarity comparison. Regular BERT requires running a forward pass over pairs of sentences to compute similarity — O(N^2) for a corpus of N sentences. SBERT independently encodes each sentence, making similarity search O(N) and enabling efficient semantic search over millions of sentences.

SBERT is trained with a siamese network architecture: two identical BERT models with shared weights process two sentences independently. The [CLS] tokens or mean-pooled representations are compared using cosine similarity. The model is fine-tuned on natural language inference (NLI) data and semantic textual similarity (STS) benchmarks using a contrastive loss that pulls similar sentence embeddings together and pushes dissimilar ones apart.

The resulting embeddings capture semantic similarity much better than averaging BERT token embeddings or using BERT's [CLS] token without fine-tuning. For the task 'find the most similar sentence to query Q from 10,000 candidates', SBERT can embed all 10,000 sentences once (during indexing) and embed Q once at query time — then cosine similarity finds the nearest neighbors in milliseconds. BERT would require 10,000 forward passes per query.

Multiple SBERT models are available for different use cases: all-mpnet-base-v2 (best quality, 768-dim), all-MiniLM-L6-v2 (fast, good quality, 384-dim), multi-qa-mpnet-base-dot-v1 (optimized for question-answer retrieval), and multilingual models (paraphrase-multilingual-mpnet-base-v2) for cross-lingual search.

SBERT is the backbone of most semantic search and RAG retrieval systems. When you see sentence_transformers in a RAG pipeline, it is producing SBERT-style embeddings. The quality of retrieval in RAG directly depends on embedding quality — upgrading from a basic SBERT model to a task-specific fine-tuned version is often the highest-leverage improvement for RAG system quality.

Why this matters in AI
SBERT enables efficient semantic search at scale. Without it, semantic search requires O(N^2) BERT computations. With it, you can search millions of documents in milliseconds. SBERT is why RAG systems and semantic search are practical.
Common mistakes
  • Using BERT's raw [CLS] token without SBERT fine-tuning — raw [CLS] embeddings have poor semantic similarity properties; use SBERT which was specifically trained for this.
  • Encoding very long documents as a single SBERT embedding — SBERT has a 256-512 token limit; longer inputs are truncated. Chunk long documents first.
  • Mixing embedding models — never compare embeddings from different models; the vector spaces are incompatible.

Cross-encoder vs bi-encoder

Detailed explanation

Bi-encoders and cross-encoders are two architectures for computing sentence similarity or relevance scores, with fundamentally different accuracy/speed trade-offs. Understanding this distinction is essential for building efficient retrieval systems.

Bi-encoder (also called dual encoder): encodes each sentence independently with a shared (or separate) encoder. f(sentence_A) → embedding_A; f(sentence_B) → embedding_B. Similarity = cosine(embedding_A, embedding_B). Key advantage: precompute and cache all embeddings. For 1 million documents, compute 1 million embeddings once and store them. At query time, embed the query (one inference) and compute cosine similarity against all stored embeddings — extremely fast with FAISS approximate nearest neighbor search. Key limitation: the encoder cannot see both sentences simultaneously, so it misses cross-sentence attention (interactions between specific tokens in A and B).

Cross-encoder: takes both sentences as input simultaneously: f([CLS] sentence_A [SEP] sentence_B [SEP]) → relevance score. The encoder sees both sentences at once — all attention heads can attend to tokens across both sentences. This allows the model to detect fine-grained relevance (e.g., 'the model achieves state-of-the-art on GLUE' is relevant to 'GLUE benchmark results', which requires understanding that 'state-of-the-art on GLUE' relates to 'GLUE benchmark results'). Much more accurate than bi-encoders but extremely slow for retrieval: for Q against 1 million documents, need 1 million forward passes.

Bi-encoder + cross-encoder pipeline (retrieve then rerank): retrieve the top-100 most similar documents using fast bi-encoder + FAISS, then rerank those 100 with the accurate cross-encoder. This combines the speed of bi-encoders (retrieve 100 from 1M in milliseconds) with the accuracy of cross-encoders (rerank 100 precisely). This is the standard production architecture for dense retrieval systems.

Training: bi-encoders are trained with contrastive loss (e.g., multiple negatives ranking loss: given a batch of (query, positive) pairs, all other positives in the batch are negatives). Cross-encoders are trained with binary/regression cross-entropy on labeled (query, document, relevance) triples. Sentence-transformers library provides both: SentenceTransformer (bi-encoder) and CrossEncoder classes.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from sentence_transformers import SentenceTransformer, CrossEncoder, util
import numpy as np
# Bi-encoder: fast similarity
bi_enc = SentenceTransformer('all-MiniLM-L6-v2')
query = 'What is BERT used for?'
docs = ['BERT is used for NLP tasks.', 'Python is a programming language.', 'BERT excels at text classification and QA.']
q_emb = bi_enc.encode(query, normalize_embeddings=True)
d_embs = bi_enc.encode(docs, normalize_embeddings=True)
bi_scores = (q_emb @ d_embs.T)
print('Bi-encoder scores:', bi_scores)
# Cross-encoder: accurate reranking
cross_enc = CrossEncoder('cross-encoder/ms-marco-MiniLM-L-6-v2')
cross_scores = cross_enc.predict([(query, doc) for doc in docs])
print('Cross-encoder scores:', cross_scores)
# Full pipeline: bi-encoder retrieve + cross-encoder rerank
top_k_indices = np.argsort(bi_scores)[::-1][:5]  # top 5 from bi-encoder
top_k_docs = [docs[i] for i in top_k_indices]
reranked_scores = cross_enc.predict([(query, doc) for doc in top_k_docs])
reranked = sorted(zip(top_k_docs, reranked_scores), key=lambda x: x[1], reverse=True)
print('Reranked:', [(d[:40], round(s,3)) for d,s in reranked])
Why this matters in AI
Bi-encoder vs cross-encoder is the central trade-off in modern neural retrieval systems. Every production search or RAG system must choose where on the speed-accuracy spectrum to operate. Knowing this distinction lets you design scalable retrieval systems that are both fast (bi-encoder for candidate generation) and accurate (cross-encoder for reranking).
Common mistakes
  • Using a cross-encoder for retrieval over large corpora — cross-encoder inference is O(n_docs) per query. For 1M documents, this is 1M forward passes per query — completely infeasible. Use bi-encoder + FAISS for retrieval.
  • Using a bi-encoder for reranking without a cross-encoder — bi-encoders are fast but miss cross-sentence interactions. For a top-10 result list where precision matters, use a cross-encoder to rerank.
  • Training a bi-encoder on labeled binary relevance pairs only — in-batch negatives training (treating other batch items as negatives) is much more efficient and effective. Use sentence-transformers' MultipleNegativesRankingLoss.

Zero-shot NLP classification

Detailed explanation

Zero-shot classification in NLP uses natural language inference (NLI) models to classify text into arbitrary classes without any task-specific labeled training data. Given a text and a list of candidate class labels, the model predicts which label best describes the text by formulating the classification as an entailment problem: does this text 'entail' that it is about topic X?

NLI models (like Facebook's bart-large-mnli or roberta-large-mnli) were trained to predict if a hypothesis is entailed, neutral, or contradicted by a premise. Zero-shot classification repurposes this: premise = your text, hypothesis = 'This text is about {label}'. The model's entailment score for each label hypothesis serves as the classification probability.

HuggingFace's pipeline(task='zero-shot-classification') makes this straightforward. You provide text and a list of candidate labels — no training required. This is extremely useful for prototyping and for classification tasks with insufficient labeled data, changing label sets (new product categories, emerging topics), or rare categories where collecting training data is infeasible.

Zero-shot classification is less accurate than a fine-tuned classifier trained on task-specific labeled data. A fine-tuned BERT classifier typically achieves 90%+ accuracy on standard benchmarks; zero-shot NLI typically achieves 70–85% on the same benchmarks. But zero-shot requires zero labeled examples while fine-tuning requires hundreds to thousands.

Large instruction-tuned LLMs (GPT-4, Claude) are extremely effective at zero-shot classification via prompting — often better than NLI-based zero-shot. 'Classify the following text as one of: {labels}. Return only the label.' works reliably. This is the recommended approach when cost is not a concern. NLI-based zero-shot is preferable for high-volume, cost-sensitive applications.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from transformers import pipeline
classifier = pipeline('zero-shot-classification', model='facebook/bart-large-mnli')
result = classifier('This movie was absolutely terrible. The acting was wooden and the plot made no sense.', candidate_labels=['positive review', 'negative review', 'neutral review'])
Output: {'sequence': '...', 'labels': ['negative review', 'neutral review', 'positive review'], 'scores': [0.92, 0.05, 0.03]}
Multi-label: result = classifier(text, candidate_labels, multi_label=True) → allows multiple labels to be true.
Why this matters in AI
Zero-shot classification enables applying NLP classification to any domain or category without collecting labeled data. For new products or emerging categories where you cannot wait for data collection and labeling, zero-shot gets you working immediately.
Common mistakes
  • Using generic labels — 'good' and 'bad' are harder for NLI than 'positive customer review' and 'negative customer review'. Make labels more descriptive.
  • Expecting fine-tuned accuracy from zero-shot — zero-shot is a starting point; if accuracy needs to exceed 85%, collect labels and fine-tune.
  • Not using multi_label=True for multi-label tasks — the default assumes mutually exclusive classes.

NER with transformers

Detailed explanation

Named Entity Recognition (NER) is the task of identifying and classifying named entities in text: persons (PER), organizations (ORG), locations (LOC), dates, money amounts, and custom domain entities. Transformer-based NER uses a BERT-like encoder to generate contextual token representations, then a classification head assigns a BIO tag to each token.

BIO tagging scheme: B-TYPE = beginning of entity, I-TYPE = continuation of entity, O = outside any entity. 'Apple Inc. is based in Cupertino' → Apple=B-ORG, Inc.=I-ORG, .=O, is=O, based=O, in=O, Cupertino=B-LOC. The BIO scheme allows entities that span multiple tokens. BIOES adds E (end) and S (single token) tags for more expressiveness.

Token-level classification: the NER model passes the input tokens through a transformer encoder. Each token gets a contextual embedding. A linear classification head maps each token's embedding to a probability vector over BIO tags. Training loss: cross-entropy computed for each token. The key challenge: tokenization. BERT's WordPiece tokenizer may split 'Cupertino' into ['Cu', '##per', '##tino'] — only the first subword token should be labeled; others are typically masked in the loss.

Fine-tuning approach: start from a pretrained encoder (BERT, RoBERTa, or domain-specific BERT like BioBERT for medical). Add a token classification head. Fine-tune on annotated NER data. With 1,000-5,000 annotated sentences, transformer NER achieves near state-of-the-art. The HuggingFace `pipeline('ner')` provides a ready-to-use interface for common entity types.

spaCy for NER: spaCy provides fast, production-ready NER with pretrained models for common entities. `nlp = spacy.load('en_core_web_trf')` (transformer-based). `doc = nlp('Apple Inc. is in Cupertino')`. `doc.ents` returns entity spans with type and confidence. For custom entities, train a custom NER component with spaCy's training CLI.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from transformers import pipeline
# Zero-shot NER with transformers pipeline
ner_pipeline = pipeline('ner', model='dbmdz/bert-large-cased-finetuned-conll03-english', aggregation_strategy='simple')
text = 'Apple Inc. was founded by Steve Jobs in Cupertino, California in 1976.'
entities = ner_pipeline(text)
for ent in entities:
    print(f'Entity: {ent["word"]:20s} | Type: {ent["entity_group"]:4s} | Score: {ent["score"]:.3f}')
# → Entity: Apple Inc.    | Type: ORG  | Score: 0.998
# → Entity: Steve Jobs    | Type: PER  | Score: 0.999
# → Entity: Cupertino     | Type: LOC  | Score: 0.997
# → Entity: California    | Type: LOC  | Score: 0.994
# Fine-tune NER model
from transformers import AutoModelForTokenClassification, AutoTokenizer, TrainingArguments, Trainer
from datasets import load_dataset
# Load CoNLL-2003 benchmark dataset
dataset = load_dataset('conll2003')
# Label mapping
label_list = dataset['train'].features['ner_tags'].feature.names
model = AutoModelForTokenClassification.from_pretrained('bert-base-cased', num_labels=len(label_list))
tokenizer = AutoTokenizer.from_pretrained('bert-base-cased')
# tokenize_and_align_labels handles subword alignment
def tokenize_and_align_labels(examples):
    tokenized = tokenizer(examples['tokens'], truncation=True, is_split_into_words=True)
    labels = []
    for i, label in enumerate(examples['ner_tags']):
        word_ids = tokenized.word_ids(batch_index=i)
        labels.append([-100 if w is None or w != word_ids[max(0,j-1)] else label[w] for j,w in enumerate(word_ids)])
    tokenized['labels'] = labels
    return tokenized
Why this matters in AI
NER is a foundational NLP task used in information extraction, knowledge graph construction, document processing, and compliance systems (extracting PII, company names, dollar amounts from contracts). Understanding how transformers handle token-level classification — especially the subword alignment problem — is essential for anyone building NLP pipelines.
Common mistakes
  • Not handling subword token alignment — BERT tokenizes 'Cupertino' as ['Cu', '##per', '##tino']. If you label all subwords with the entity tag, the loss is inflated and the model trains poorly. Assign the label only to the first subword token; mask (-100) the rest so they're excluded from the loss computation.
  • Using aggregation_strategy='none' when you need grouped entities — 'none' returns one entry per subword token ('Ap', '##ple', 'Inc', '.') instead of the grouped entity 'Apple Inc.'. Use aggregation_strategy='simple' or 'first' for human-readable output.
  • Not evaluating with seqeval — NER is evaluated at the entity level (did the model correctly identify the full span), not the token level. Use the `seqeval` metric from HuggingFace — it computes exact entity match precision, recall, and F1.
