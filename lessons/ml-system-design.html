Problem framing

Detailed explanation

Problem framing is the process of translating a vague business goal into a concrete ML problem definition. This is the most important step in any ML project — a wrong framing leads to a model that is technically correct but solves the wrong problem. Getting framing right requires understanding both the business domain and the constraints of ML. The questions to answer: what is the prediction target? what data is available? what is success? how will the model be used?

The ML decision: is ML even the right approach? If the problem has clear rules that cover all cases, write code (if price > threshold, flag it). If the problem has complex patterns that humans cannot articulate explicitly, ML is appropriate. If there is no labeled data and collecting it is prohibitively expensive, ML may be premature. Many business problems that seem to need ML can be solved effectively with simple rules or SQL queries — validate that assumption first.

Output type determines model type. Single number → regression (predict house price). Binary outcome → binary classification (spam/not spam). Multiple categories → multi-class classification (sentiment: positive/negative/neutral). Probability distribution → generative models. A ranking problem (what to show user next?) looks like a recommendation problem but is actually a pairwise ranking problem. Getting the output type wrong means building the wrong model entirely.

Label definition is where most ML projects go wrong. 'Predict customer churn' is not a ML problem yet — when is a customer churned? 30 days of inactivity? Cancellation? Failure to renew subscription? Different definitions produce different labels, different models, and different business interventions. Each definition requires agreement with business stakeholders before data collection begins.

Formulating success metrics: the business metric (revenue impact, user retention rate, NPS) and the ML metric (AUC, F1, MAE) must be aligned. Define the minimum ML metric that achieves the business goal. Add constraints: latency < 100ms, cost < $0.001/prediction, explainability required. These constraints eliminate model architectures before you start. Document all assumptions and validate them with stakeholders before building.

Why this matters in AI
The best model solving the wrong problem is worse than no model. Framing errors are discovered late, expensive to fix, and frustrating for everyone. 30 minutes of careful problem framing saves weeks of wasted modeling effort.
Common mistakes
  • Starting with data exploration before defining the problem — without a clear target variable, EDA has no focus and any finding seems relevant.
  • Using model performance as the success metric without connecting it to business impact — 90% accuracy on a test set means nothing if the model adds no value in production.
  • Not checking whether the label is actually measurable — 'customer satisfaction' is a business goal, not a label. 'Rating given in post-interaction survey' is a measurable label.

ML vs rules

Detailed explanation

One of the most important system design decisions is whether to use an ML model or hand-coded rules to solve a problem. Rules are deterministic, interpretable, and cheap to implement for well-understood problems. ML is better when: the rules would be too complex to write and maintain, patterns exist in data that are hard to articulate explicitly, or the problem distribution changes over time and the model can be retrained.

When rules win: (1) the problem is simple and well-defined: 'if transaction amount > $10,000, flag for review' — clear, auditable, legally defensible. (2) data is scarce — ML requires training data; if you have < 100 labeled examples, rules will likely outperform. (3) interpretability is mandatory — in medical diagnosis, credit decisions, or legal contexts, you may need to explain exactly why a decision was made. (4) the cost of a wrong ML prediction is catastrophic and reversible errors are impossible.

When ML wins: (1) the rules would be too complex — spam detection has thousands of features and patterns. Writing rules for all cases is infeasible. (2) the problem distribution changes — ML can be retrained as spam tactics evolve; rules become stale. (3) you have abundant labeled data — ML learns from millions of examples, capturing patterns no human would articulate. (4) non-linear patterns — email spam depends on interactions between hundreds of features that rules can't capture.

Hybrid approach: start with rules for high-confidence cases, use ML for the uncertain middle. Example: fraud detection — rules block obvious fraud (known stolen card numbers, geo-impossible transactions), ML scores ambiguous cases, rules allow obvious legitimate transactions. The rules handle the easy cases cheaply and quickly; ML handles the hard cases where the data shows nuanced patterns.

Production implications: rules are version-controlled, testable, auditable, and change only when explicitly modified. ML models require monitoring (concept drift, data quality), retraining pipelines, and can change behavior subtly even without explicit changes (different data distribution). Before building an ML model, ask: 'Could a smart engineer write the rules for this?' If yes, start with rules and only add ML if rules don't suffice.

# Decision framework: ML vs rules
def should_use_ml(labeled_examples, rule_complexity, distribution_change, interpretability_required):
    score = 0
    if labeled_examples > 1000: score += 2
    if labeled_examples > 10000: score += 1
    if rule_complexity == 'high': score += 3
    if distribution_change == 'frequent': score += 2
    if interpretability_required: score -= 3
    return 'ML' if score >= 3 else 'Rules'

Why this matters in AI
Choosing ML over rules prematurely is one of the most common and expensive mistakes in production AI systems. ML models require: training data collection, labeling, model development, monitoring, retraining pipelines — weeks to months of work. Rules can be deployed in hours. Always establish a rule-based baseline and measure how much ML improves before committing to the ML path.
Common mistakes
  • Jumping to ML without trying rules first — 'we need ML for this classification problem' before even checking if 5 rules handle 80% of cases. Build and measure the rule baseline first.
  • Replacing working rules with ML to 'modernize' — if the rules work and the distribution is stable, replacing with ML introduces uncertainty and maintenance overhead without benefit.
  • Not documenting why a rule was written — rules encode domain knowledge. Without documentation, they become mysterious and teams are afraid to change them. Document: what the rule does, why it exists, when it was added, and what data supports it.

Data requirements

Detailed explanation

Data requirements specification is the discipline of defining, before data collection or annotation begins, exactly what data you need to build a working ML model. Poor data requirements lead to: collecting the wrong data, insufficient data quantity, wrong label definitions, or mismatched data distribution between training and deployment.

Quantity estimation: 'how much labeled data do I need?' — depends on: problem complexity (binary vs 100-class), model type (logistic regression needs less than ResNet-50), expected accuracy (99% needs 10x more than 90%), and class balance. Rules of thumb: simple text classification with BERT fine-tuning — 500-1,000 labeled examples per class for 85%+ accuracy. Complex NER with 20 entity types — 5,000-10,000 sentences. Tabular gradient boosting — 10,000 rows minimum.

Label definition: ambiguous labels produce inconsistent annotations that hurt model quality. Before annotating: write a detailed label guide with examples and edge cases for each class. Test with 2+ annotators on 100 examples and compute inter-annotator agreement (Cohen's kappa). Target kappa > 0.8 (strong agreement). If kappa < 0.6, the task is ambiguous — revise the label guide before proceeding.

Data distribution: training data must match deployment data. Key questions: what is the distribution of input lengths/sizes in production? What languages/dialects will the model see? What time periods does production data span? What edge cases will occur? Mismatched distribution is the most common cause of unexpected model failure in production.

Active learning for efficient annotation: instead of randomly labeling, use the model's uncertainty to identify the most informative examples to label. Start with 100 labeled examples, train a model, find the 100 most uncertain predictions, label those, retrain. Iterate. This can reduce the total labeled data needed by 50-80% compared to random labeling.

Why this matters in AI
Data requirements specification is the ML equivalent of software requirements specification. Without it, data collection efforts go in wrong directions: you collect 10,000 examples of one class and 100 of another, or you annotate with inconsistent labels, or you collect data from a different distribution than deployment. Clear requirements save weeks of wasted annotation effort.
Common mistakes
  • Not defining edge cases in the label guide — edge cases (an email that's both a complaint AND a support request) will be labeled inconsistently by annotators without explicit guidance. Every label guide needs an 'edge cases and how to handle them' section.
  • Estimating data quantity without a pilot experiment — data quantity estimates are rough. Run a pilot: collect 200 labeled examples, train, evaluate. The learning curve (accuracy vs training set size) tells you whether you need 1,000 or 10,000 examples to reach your accuracy target.
  • Ignoring the cost of annotation — annotating 10,000 medical records at $5/record = $50,000. Unexpected annotation costs have killed more ML projects than technical failures. Define annotation budget before starting and design the labeling task to fit within budget.

Model selection

Detailed explanation

Model selection is the process of choosing which ML algorithm and architecture to use for a given problem. It depends on: the problem type (classification, regression, generation), data characteristics (size, dimensionality, feature types), constraints (latency, memory, interpretability), and empirical performance comparison.

Default starting points: for tabular data with < 1M rows and moderate features: gradient boosting (XGBoost/LightGBM) is the default. For tabular data with > 10M rows: add neural network (TabNet, FT-Transformer). For NLP tasks: pretrained transformer (BERT, RoBERTa) fine-tuned on your data. For image classification: ResNet or EfficientNet fine-tuned from ImageNet. For generation/completion/agents: LLMs (Claude, GPT).

Algorithm comparison framework: define a baseline first (simplest possible model: logistic regression, constant predictor). Compare to more complex models. Use statistical significance testing (paired t-test on cross-validation scores) to confirm differences are real and not noise. Plot: model complexity vs CV score to find the knee point where complexity stops providing returns.

Interpretability requirements: if the model's decisions must be explained to regulators or customers, linear models and decision trees are more appropriate than gradient boosting or neural networks — even if they underperform by a few percentage points. Interpretability is a constraint that narrows model selection.

Practical sequence: (1) build the simplest possible baseline. (2) evaluate gradient boosting (XGBoost or LightGBM) with default hyperparameters. (3) evaluate neural network (if data size and feature types warrant). (4) fine-tune the best algorithm. (5) ensemble if ensemble > best individual by > 1 percentage point on the metric that matters.

Why this matters in AI
Systematic model selection prevents premature optimization on the wrong algorithm. Many practitioners jump directly to neural networks or the latest architecture without first verifying that a well-tuned gradient boosting model doesn't already achieve 95% of the potential performance at 10% of the complexity. Benchmarking with a comparison framework gives you confident, data-driven model choice justification.
Common mistakes
  • Not establishing a baseline first — without a baseline, you don't know if your complex model is actually better than 'predict the majority class always'. A baseline makes improvement claims meaningful.
  • Comparing models with different preprocessing — comparing logistic regression (needs StandardScaler) against XGBoost (doesn't need scaling) without using pipelines means one model is disadvantaged. Always use pipelines for fair comparison.
  • Not considering serving constraints — a model that achieves the highest CV score but requires 2GB of memory and 500ms inference time may be unusable in production. Always define serving constraints (max latency, max memory) as elimination criteria before model comparison.

Offline inference

Detailed explanation

Offline inference (batch inference) processes a large set of inputs all at once, outside of real-time serving. Examples: score all customers for churn risk every night, generate embeddings for all new product listings in a batch, classify all support tickets from the last week in a single run. Offline inference optimizes for throughput rather than latency.

Architecture: a scheduled job (cron, Airflow, Prefect) triggers the inference pipeline. Data is loaded in batches from a database or data warehouse. The model processes batches in parallel (GPU vectorized inference). Results are written back to the database for downstream use. For LLM inference: use the messages batching API (Anthropic, OpenAI) which processes requests asynchronously at 50% lower cost.

Batch size optimization: larger batches utilize GPU more efficiently (higher GPU utilization → better throughput). For a neural network, batch_size=128 typically processes 10-20x more samples/second than batch_size=1. The optimal batch size is the largest that fits in GPU memory without memory errors: `torch.cuda.OutOfMemoryError` signals to reduce batch size.

Monitoring and error handling: batch inference over millions of samples will encounter some errors (malformed input, memory spikes, model failures). Implement: per-sample error catching (don't fail the full batch on one bad sample), progress logging (how many of 1M samples processed?), checkpointing (save progress so you can resume after failure), and a dead-letter queue for samples that fail repeatedly.

Fresh data and staleness: offline inference results are stale as soon as they're computed. Churn scores computed last night may not reflect today's activity. For decisions requiring fresh predictions (real-time fraud), use online inference. For decisions where yesterday's prediction is acceptable (personalized email recommendations), offline is more cost-effective.

import torch
import numpy as np
from torch.utils.data import DataLoader, TensorDataset
# Batch inference with PyTorch
def batch_inference(model, data: np.ndarray, batch_size: int = 256, device: str = 'cpu') -> np.ndarray:
    model.eval()
    model.to(device)
    dataset = TensorDataset(torch.FloatTensor(data))
    loader = DataLoader(dataset, batch_size=batch_size, num_workers=4, pin_memory=(device != 'cpu'))
    all_predictions = []
    with torch.no_grad():
        for batch_idx, (inputs,) in enumerate(loader):
            inputs = inputs.to(device)
            outputs = model(inputs)
            predictions = torch.sigmoid(outputs).cpu().numpy()
            all_predictions.append(predictions)
            if batch_idx % 100 == 0:
                processed = (batch_idx + 1) * batch_size
                print(f'Processed {processed:,}/{len(data):,} samples')
    return np.concatenate(all_predictions, axis=0)
# Anthropic batch API for LLM inference
import anthropic
client = anthropic.Anthropic()
def batch_llm_inference(prompts: list) -> list:
    requests = [{'custom_id': f'req_{i}', 'params': {'model': 'claude-haiku-4-5-20251001', 'max_tokens': 200, 'messages': [{'role': 'user', 'content': p}]}} for i, p in enumerate(prompts)]
    batch = client.messages.batches.create(requests=requests)
    print(f'Batch ID: {batch.id}. Poll for results...')
    return batch.id
Why this matters in AI
Offline inference is how ML models generate value for the majority of business decisions. Most decisions (which customers to target for a campaign, which products to recommend in an email, which accounts to review for fraud) are made in batch mode — not in real-time. Knowing how to run efficient, fault-tolerant batch inference at scale (millions of predictions per run) is essential for deploying ML in production.
Common mistakes
  • Running inference one sample at a time in a loop — a Python loop calling model.predict(single_sample) is 10-100x slower than batched inference. Always batch inputs for GPU-accelerated models.
  • Not implementing checkpointing for large batches — running inference on 10M samples in one job can take hours. If the job fails after 8 hours (at sample 9M), you restart from zero. Checkpoint every 100K samples: save progress to disk, resume from the last checkpoint.
  • Not validating output distribution in batch inference — after generating predictions for 1M samples, check the output distribution: does the fraction of positive predictions match historical rates? A sudden change indicates a data distribution shift or a bug.

Online inference

Detailed explanation

Online inference (also called real-time inference) produces predictions in response to individual requests, typically requiring a response within 100-500ms for user-facing applications. Unlike batch inference (precomputed, stored), online inference requires the model to be loaded in memory, ready to process any input at any time. The architecture: user request → API endpoint → preprocessing → model inference → postprocessing → response.

Serving infrastructure: ML models are served via REST APIs (FastAPI, Flask) or gRPC servers (TensorFlow Serving, Triton Inference Server). The model is loaded into memory at server startup and kept resident for the lifetime of the server. Multiple server instances run behind a load balancer for scalability. Stateless design: each request is independent, servers can be scaled horizontally.

Latency components: total request latency = network round trip + preprocessing + model inference + postprocessing. For a typical tabular model: preprocessing (10ms) + model inference (1-5ms) + serialization (1ms) ≈ 15ms. For a BERT model: tokenization (5ms) + inference on 1 GPU (20-50ms) ≈ 30-60ms. For an LLM: time-to-first-token (200-1000ms) + streaming generation (50 tokens/second) ≈ 1-10 seconds.

GPU serving for deep learning: inference on GPU is 10-100× faster than CPU for neural networks. But GPU memory is limited (24GB for an RTX 4090) and GPUs are expensive. Optimization strategies: model quantization (4-bit/8-bit reduces memory 4-8×), ONNX Runtime (2-3× faster than PyTorch for inference), TensorRT (up to 10× faster than PyTorch, NVIDIA GPU only), vLLM (state-of-the-art LLM serving with continuous batching and PagedAttention).

Autoscaling: production serving systems must scale up when traffic spikes and scale down when traffic decreases. Container orchestration (Kubernetes, AWS ECS) monitors request queue depth and CPU/GPU utilization, adding more server instances when load is high. For bursty traffic patterns, prewarming (keeping some instances running in advance) prevents cold start latency.

Why this matters in AI
Online inference is the architecture for all user-facing ML applications: search ranking, product recommendations, content moderation, autocomplete, fraud detection. Understanding how to build, deploy, and scale online inference systems is the bridge between an ML model and a production AI product.
Common mistakes
  • Loading the model on every request — loading a 500MB model takes 2-5 seconds. Load the model once at server startup and keep it in memory. Never load inside the request handler.

Latency vs accuracy

Detailed explanation

Latency (response time) and accuracy (prediction quality) are in tension in most ML systems. Higher accuracy often requires larger models, more complex inference, or ensemble methods — all of which increase latency. System design requires explicitly choosing where on the accuracy-latency frontier to operate based on the application's requirements.

Sources of ML inference latency: (1) model loading time (amortized over many requests), (2) preprocessing (tokenization, feature extraction), (3) forward pass (matrix multiplications through model layers), (4) postprocessing (decoding, softmax, thresholding), (5) data serialization and network round trips (for API calls). For LLM inference, token generation time is the dominant cost — measured in tokens/second.

Latency budgets for different applications: real-time recommendations (e-commerce): < 100ms total, ML inference budget ~30ms. Voice assistants: first token in < 500ms (time-to-first-token is most important, streaming hides generation latency). Batch processing: no real-time constraint — maximize throughput (queries per second) rather than minimize latency. Offline analytics: minutes to hours acceptable.

Accuracy-latency trade-offs: model distillation (student model trained to match teacher) gives 4-10× speedup with 1-5% accuracy loss. Quantization (float32 → int8) gives 2-4× speedup with < 1% accuracy loss. ONNX export gives 1.5-2× speedup for free. Cascading models: use a fast model for easy cases, slow accurate model for hard cases — overall high accuracy with average latency close to the fast model.

Measuring and profiling: never guess where latency comes from. Profile with torch.profiler, cProfile, or simply timing each step. For LLM inference: time-to-first-token (TTFT, affects user perception of responsiveness) and tokens-per-second (TPS, affects time to complete the full response) are the key metrics. vLLM, TensorRT-LLM, and llama.cpp are optimized inference runtimes that improve both TTFT and TPS.

    start = time.perf_counter()
    with torch.no_grad():
        for _ in range(n_runs): output = model(inputs)
    torch.cuda.synchronize() if inputs.is_cuda else None
    elapsed = (time.perf_counter() - start) / n_runs
    print(f'Latency: {elapsed*1000:.2f}ms, Throughput: {1/elapsed:.1f} req/s')
    return elapsed
# Quantization for 2-4x speedup
from torch.quantization import quantize_dynamic
quantized_model = quantize_dynamic(model, {torch.nn.Linear}, dtype=torch.qint8)
print('Original size:', sum(p.numel() for p in model.parameters()) * 4 / 1e6, 'MB')
print('Quantized size:', sum(p.numel() for p in quantized_model.parameters()) * 1 / 1e6, 'MB')
# Compare latency and accuracy
cpu_inputs = torch.randn(1, 128)

Scalability

Detailed explanation

Scalability is the ability of a system to handle growing workloads by adding resources. An ML serving system that works for 100 requests/day may fail at 100,000 requests/day if not designed for scale. Scalability planning asks: how does this system behave as users 10×, 100×, 1000× increase? Where are the bottlenecks? What needs to change to handle 10× more traffic?

Horizontal vs vertical scaling: vertical scaling (scale up) = add more powerful hardware to each server (more RAM, faster GPU). Has a ceiling — you can't add infinite RAM to one machine, and it doesn't improve availability. Horizontal scaling (scale out) = add more server instances. No ceiling in principle. Requires stateless services (each request can go to any server). ML serving: for CPU models, horizontal scaling is standard (spin up more containers). For GPU models, vertical scaling (better GPU) is often preferred for smaller scales.

Stateless service design: ML serving services should be stateless — each request is independent and can be handled by any server instance. State (user sessions, model versions, feature caches) should be stored in external systems (Redis for session state, S3 for model artifacts, database for predictions). This enables horizontal scaling: add 10 more servers behind a load balancer, and the system handles 10× the traffic.

Bottleneck analysis: profile your system under load to find where time is spent. Common bottlenecks: (1) database reads/writes (use connection pooling, read replicas, caching), (2) model inference (GPU saturation, request queuing), (3) network I/O (large feature payloads, slow upstream services), (4) CPU preprocessing (vectorize transformations, parallelize). Always profile before optimizing — guessing the bottleneck is wrong 70% of the time.

Auto-scaling: cloud platforms (AWS, GCP, Azure) can automatically add or remove server instances based on metrics (CPU utilization, queue depth, custom metrics). Configure: min_instances (always-on), max_instances (budget cap), scale-out trigger (CPU > 70% for 2 minutes), scale-in delay (wait 5 minutes before removing instances to avoid flapping). For ML workloads with GPU requirements, prewarming (spin up instances before traffic hits) prevents cold-start latency.

Reliability

Detailed explanation

Reliability in AI systems means consistently delivering correct, available, and performant results. A system is reliable when it behaves predictably under normal load, degrades gracefully under stress, and recovers quickly from failures. Reliability engineering for AI adds AI-specific concerns (model drift, hallucination rates, prompt injection) to standard SRE practices.

SLOs (Service Level Objectives): measurable targets for reliability. 'API availability > 99.9%' (< 8.76 hours downtime/year). 'P99 latency < 2 seconds'. 'Correct response rate > 95%'. Define SLOs before deployment so you know when the system is meeting its reliability targets. Error budgets: with 99.9% availability, you have 43.8 minutes/month of allowed downtime before burning the error budget.

Health checks and monitoring: endpoint-level health checks detect when a model serving instance is unresponsive. Application-level monitoring tracks: request rate, error rate, latency distribution (p50, p90, p99), and AI-specific metrics (average response length, refusal rate, safety classifier score). Alert when metrics exceed thresholds.

Circuit breaker pattern: if an external service (LLM API, database) becomes slow or fails, stop sending requests to it until it recovers. A circuit breaker transitions between: CLOSED (normal operation), OPEN (all requests immediately fail with fallback), and HALF-OPEN (test traffic to check recovery). This prevents cascading failures when a dependency is overwhelmed.

Chaos engineering: proactively inject failures (kill a model serving instance, introduce network latency, return errors from the LLM API) to verify that the system's fallback and recovery mechanisms work correctly. Tools: chaostoolkit, AWS Fault Injection Simulator. Netflix's Chaos Monkey concept — random instance termination in production — is the extreme version.

import time
import anthropic
from enum import Enum
from dataclasses import dataclass, field
client = anthropic.Anthropic()
# Circuit breaker implementation
class CircuitState(Enum): CLOSED = 'closed'; OPEN = 'open'; HALF_OPEN = 'half_open'
@dataclass
class CircuitBreaker:
    failure_threshold: int = 5; reset_timeout: float = 60.0
    failure_count: int = 0; last_failure_time: float = 0.0; state: CircuitState = CircuitState.CLOSED
    def call(self, fn, *args, **kwargs):
        now = time.time()
        if self.state == CircuitState.OPEN:
            if now - self.last_failure_time >= self.reset_timeout: self.state = CircuitState.HALF_OPEN
            else: raise RuntimeError('Circuit OPEN — service unavailable')
        try:
            result = fn(*args, **kwargs)
            if self.state == CircuitState.HALF_OPEN: self.state = CircuitState.CLOSED; self.failure_count = 0
            return result
        except Exception as e:
            self.failure_count += 1; self.last_failure_time = time.time()
            if self.failure_count >= self.failure_threshold: self.state = CircuitState.OPEN
            raise
llm_circuit = CircuitBreaker(failure_threshold=5, reset_timeout=30)
def reliable_llm_call(prompt: str) -> str:
    def fn(): return client.messages.create(model='claude-opus-4-8', max_tokens=100, messages=[{'role':'user','content':prompt}]).content[0].text
    try: return llm_circuit.call(fn)
    except RuntimeError: return 'Service temporarily unavailable. Please try again in a moment.'
Why this matters in AI

Building reliability in — health checks, circuit breakers, fallbacks, SLOs — from the beginning is much easier than retrofitting reliability after users are experiencing outages. Every hour of downtime in a customer-facing AI system erodes trust and drives users to competitors.
Common mistakes
  • Not defining SLOs before deployment — without SLOs, there's no objective measure of whether the system is 'reliable enough.' Define them based on user impact: 'users will retry after a 3-second timeout' suggests p99 < 2s is acceptable.
  • Not testing failure scenarios — writing fallback code that you never test means discovering bugs in the fallback path during a real outage. Test every failure scenario before deployment: API timeout, database unavailable, model error, rate limit exceeded.
  • Treating AI-specific failures as novel — a model that starts hallucinating more (due to drift) is a reliability failure just like an API timeout. Include model quality metrics (hallucination rate, refusal rate, answer correctness on a canary dataset) in your SLO monitoring alongside traditional availability metrics.

Fallbacks

Detailed explanation

Fallbacks are alternative paths taken when the primary approach fails — a model call times out, returns an error, produces an invalid response, or the confidence is too low. Robust production systems always have fallbacks; brittle systems fail completely when any component fails.

Model fallback chain: if the primary model (e.g., Claude Opus) is unavailable or times out, fall back to a faster/cheaper model (Claude Haiku). If that also fails, fall back to a cached response, a rules-based response, or a graceful 'I cannot answer this right now' message. Each level of fallback is less capable but more reliable.

Tool fallback: if the primary search tool returns empty results, try an alternative search strategy (broader query, different data source). If OCR fails to parse a document, try a different OCR engine. If a translation API fails, try a different translation service or skip translation. The key: always have at least one backup path that cannot fail (even if the backup is 'return None and log the failure').

Confidence-based fallback: use model confidence (prediction probability) to route requests. High confidence (>0.9): auto-approve the response. Medium confidence (0.5-0.9): add a disclaimer. Low confidence (<0.5): fallback to a human reviewer or a more capable model. This approach maximizes automation for confident cases while ensuring quality for uncertain cases.

LangChain fallback chains: LangChain provides `.with_fallbacks([fallback_chain])` syntax for automatic fallback. If the primary chain raises an exception, the fallback chain is tried. You can chain multiple fallbacks: primary_model → backup_model → rules_based_responder → 'service unavailable' message.

Human-in-the-loop

Detailed explanation

Human-in-the-loop (HITL) is a system design pattern where a human reviewer is embedded in the automated workflow for high-stakes or uncertain decisions. Instead of full automation, the system runs autonomously for routine cases and escalates to a human for cases that are uncertain, novel, or high-risk. This combines the speed of automation with the judgment of human expertise.

When HITL is necessary: medical diagnosis (AI suggests, doctor confirms), legal document review (AI flags issues, lawyer reviews), financial fraud decisions (AI scores risk, analyst confirms before blocking), content moderation (AI classifier, human reviewer for borderline cases), AI agent actions that are irreversible (send an email, execute a trade, delete data).

HITL in LLM agents: use LangGraph's interrupt mechanism to pause agent execution before critical actions. The agent plans an action (e.g., 'send email to customer'), raises an interrupt, a human reviews the planned action, approves or modifies it, and the agent continues. This prevents autonomous agents from taking irreversible wrong actions.

Active learning: HITL is not just for deployment — it's also for training data creation. Active learning uses model uncertainty to identify which examples would be most valuable for a human to label. Instead of labeling randomly, a human labels the examples the model is most uncertain about — dramatically reducing the number of labels needed to achieve target performance.

Confidence-based routing: a practical HITL implementation — route requests to human review when model confidence falls below a threshold. For a classification model: if max(softmax_output) < 0.7, escalate to human. Tune the threshold to balance automation rate (cost) vs accuracy (quality). Track: automation rate, human review queue depth, and error rate on auto-approved vs human-reviewed cases.

Feedback loop

Detailed explanation

A feedback loop in ML systems is the cycle where model predictions influence future training data. This is the mechanism by which deployed models improve over time — but also how they can degrade through systematic bias amplification. Understanding feedback loops is essential for designing ML systems that improve with use rather than becoming stuck in self-reinforcing errors.

Positive feedback loops (virtuous or vicious): a recommendation system suggests item A, users click it (because it was suggested), the system records this as 'users like A', and suggests A even more. Good case: discovering genuinely popular items. Bad case: popular items become more popular (filter bubble), suppressing diverse recommendations. This feedback loop requires active mitigation: exploration (occasionally recommend non-popular items), diversity constraints, and offline evaluation on historical data before the recommendation influenced behavior.

Implicit feedback collection: every time a user acts on (or ignores) a model's prediction, they generate implicit feedback. Click = positive signal, scroll past = weak negative, return and rephrase query = dissatisfaction signal. Collecting and using this implicit feedback creates a continual learning loop without requiring explicit labels.

Data drift from feedback loops: if a spam filter becomes very effective, legitimate email patterns drift as spammers adapt their tactics to avoid detection. The model's training distribution shifts because its predictions changed the world. ML systems in adversarial domains (spam, fraud, content moderation) must account for this — continuously update the model as adversaries adapt.

Measuring the feedback loop health: compare model performance on data labeled BEFORE the model was deployed vs data labeled AFTER. If performance on pre-deployment data is higher, the model may be performing well on the data it influenced but poorly on organic behavior. Use randomized holdout (randomly serve the baseline/no-prediction condition to a small fraction of users) to get unbiased organic behavior data.

Evaluation pipeline

Detailed explanation

An evaluation pipeline is an automated system that regularly measures the quality of an AI model or system using a defined set of test cases. Unlike running evaluation once during development, a pipeline runs continuously (daily, weekly, or triggered by code changes) to detect regressions before they affect users.

Pipeline components: test dataset (a fixed, representative set of inputs with expected outputs or rubrics for judgment), evaluation function (compute the score for each test case), aggregation (compute overall metrics), comparison to baseline (did quality go up or down?), reporting (format results for human review), and gating (optionally block deployment if quality drops below threshold).

LLM-based evaluation: for tasks without ground truth answers (summarization, chat, creative writing), use an LLM judge. The judge receives: the input, the model's output, and an evaluation rubric. The judge scores the output on dimensions like helpfulness, accuracy, format compliance. Meta-evaluate the judge: compute agreement with human ratings on a calibration set to quantify the judge's reliability.

Regression testing: when you update the model, prompt, or retrieval system, compare the new scores to the previous run on the SAME test set. A score decrease > 3% is a regression that requires investigation before deployment. Track: mean score, score distribution, per-category scores, and worst-scoring cases (the inputs that consistently score lowest reveal systematic weaknesses).

Continuous evaluation with evals frameworks: RAGAS for RAG system evaluation (faithfulness, answer relevance, context recall). BrainBench or EleutherAI lm-evaluation-harness for LLM capability benchmarks. Promptfoo for prompt engineering evaluation across model versions. These frameworks standardize evaluation so results are comparable across teams and time.

import anthropic
import pandas as pd
import json
from typing import Callable
client = anthropic.Anthropic()
# Evaluation pipeline
def run_evaluation_pipeline(model_fn: Callable, test_cases: list, judge_fn: Callable = None) -> dict:
    results = []
    for case in test_cases:
        output = model_fn(case['input'])
        if case.get('expected'):  # Exact or reference-based eval
            score = 1.0 if output.strip() == case['expected'].strip() else 0.0
        elif judge_fn:  # LLM judge
            score = judge_fn(case['input'], output, case.get('rubric', 'helpful and accurate'))
        else:
            score = None
        results.append({'input': case['input'][:50], 'score': score, 'output': output[:100]})
    df = pd.DataFrame(results)
    return {'mean_score': df['score'].mean(), 'min_score': df['score'].min(), 'n_failures': (df['score'] < 0.5).sum(), 'results': df}

Why this matters in AI
Without an evaluation pipeline, you deploy changes blindly and discover quality regressions when users complain. With an evaluation pipeline, you detect regressions before deployment and can make confident, evidence-based decisions about when to release new model versions. This is the ML equivalent of CI/CD tests in software engineering — it makes AI development disciplined and reliable.
Common mistakes
  • Not using a fixed test set — if you update the test set with each evaluation run, you cannot compare results over time. Use a frozen evaluation set for regression tracking; add new test cases to a separate discovery set.
  • Using only aggregate metrics — mean_score can hide regressions in specific categories. Track per-category scores: if 'math questions' score drops from 0.9 to 0.6 while 'general questions' stays at 0.85, the aggregate (0.77 → 0.73) hides the severity of the math regression.
  • Not calibrating the LLM judge — if your LLM judge gives 4.5/5 to everything (too lenient) or 2/5 to everything (too strict), the evaluation signal is useless. Run the judge on 50 human-rated examples and measure correlation. Calibrate the judge's prompt until correlation > 0.8.

Which categories are most affected? (6) implement automatic alerting: if mean score drops > 5% from baseline, print an alert with the affected categories.
