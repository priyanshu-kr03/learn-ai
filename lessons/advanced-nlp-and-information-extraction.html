Coreference resolution

Detailed explanation

Coreference resolution identifies all expressions in a text that refer to the same entity. 'Sarah said she would help John. He thanked her.' — coreference resolution links 'she' → 'Sarah', 'He' → 'John', 'her' → 'Sarah'. This enables understanding multi-sentence text and is a prerequisite for relation extraction and knowledge graph construction.

Mention detection: the first step is identifying all mentions — noun phrases, pronouns, and named entities that potentially refer to an entity. 'The company', 'Apple', 'the tech giant', and 'it' may all refer to the same entity. Neural mention detection uses a span representation model (encode all possible spans and classify which are mentions).

Coreference models: span-based neural models (e2e-coref, SpanBERT) represent each candidate span with its BERT contextual embedding + span length embedding, then compute compatibility scores between pairs of spans. The span with the highest compatibility score to a previous mention is its antecedent.

spaCy and neuralcoref: spaCy has a built-in coreference component in spaCy v3.5+ (experimental). neuralcoref is the popular third-party library for spaCy v2. `import spacy; nlp = spacy.load('en_coreref_web_trf'); doc = nlp(text); doc._.coref_clusters` returns a list of clusters, each containing all coreferent mentions.

Applications: information extraction (extracting all statements about 'Apple' across a long document, even when referred to as 'the company' or 'it'), question answering (resolving 'What did he say?' requires knowing who 'he' refers to), summarization (a good summary must replace pronouns with their antecedents for clarity).

# FastCoref - efficient coreference resolution
# pip install fastcoref spacy
import spacy
from fastcoref import FCoref
nlp = spacy.load('en_core_web_sm')
coref_model = FCoref(device='cpu')
text = 'Sarah noticed that John was struggling. She decided to help him with the project. They worked together for hours.'
preds = coref_model.predict(texts=[text])
# Get cluster representations
for cluster in preds[0].get_clusters(as_strings=True):
    print(f'Cluster: {cluster}')
# For each mention, find its antecedent
for cluster in preds[0].get_clusters():
    spans = [(start, end, text[start:end]) for start, end in cluster]
    print('Coreferent mentions:', [(s, e, t) for s, e, t in spans])
Why this matters in AI
Coreference resolution is necessary for understanding any multi-sentence text. An NLP model that can't resolve pronouns misses most of the semantic content of a document. For information extraction from long documents (medical records, legal contracts, news articles), coreference resolution determines whether 'it' in sentence 50 refers to the same entity as 'the company' in sentence 3.
Common mistakes
  • Applying coreference to single sentences — coreference is a multi-sentence phenomenon. Running it on individual sentences from a longer document misses cross-sentence pronoun resolution. Always run on full paragraphs or documents.
  • Conflating coreference with NER — NER identifies entity types ('Apple' is ORG). Coreference links mentions ('Apple', 'the company', 'it') that all refer to the same entity across the document. They are complementary: use NER to identify what type an entity is, coreference to track where it's mentioned.
  • Expecting high accuracy for complex pronoun resolution — 'they' in 'The managers told the workers that they would get a raise' is ambiguous (managers or workers?). Modern models still struggle with these semantically complex cases. Always validate output quality on your specific domain.
Practice task
Apply coreference resolution to document analysis: (1) take a 500-word news article, (2) run FastCoref or spaCy coreference, (3) for each coreferent cluster, identify: what entity does this cluster refer to? How many mentions does it have?, (4) implement coreference-enhanced entity tracking: for all mentions of 'Apple' and all pronouns/mentions in the same cluster, extract every sentence about Apple from the article, (5) compare entity tracking with vs without coreference — how many relevant sentences does coreference resolution surface that NER alone would miss?

Dependency parsing

Detailed explanation

Dependency parsing analyzes the grammatical structure of a sentence by identifying relationships between words. Each word (except the root) has exactly one head (governor) and the relationship is labeled (subject, object, modifier, etc.). The result is a dependency tree. 'The quick brown fox jumps' → 'jumps' is the root, 'fox' is its nsubj (nominal subject), 'The', 'quick', 'brown' are modifiers of 'fox'.

Dependency labels: nsubj (nominal subject), dobj/obj (direct object), amod (adjectival modifier), prep (preposition), nmod (noun modifier), compound (compound noun), conj (conjunction). The Universal Dependencies (UD) project standardizes these labels across 100+ languages, enabling multilingual parsing.

spaCy dependency parsing: `doc = nlp('Apple acquired Shazam for $400 million'); doc[2].dep_` gives 'ROOT' (acquired). `doc[0].dep_` gives 'nsubj'. `doc[2].head` is the head of each token. Iterating over `token.children` gives all tokens that depend on it.

Applications: information extraction — extract subject-verb-object triples ('Apple acquired Shazam'). Question answering — identify what the question is asking about. Semantic role labeling — who did what to whom. Text simplification — identify main clauses vs subordinate clauses.

Constituency vs dependency parsing: constituency parsing (NLTK, CoreNLP) produces a hierarchical phrase structure tree (NP, VP, PP, etc.). Dependency parsing (spaCy, stanza) produces a flat tree of word-to-word relationships. Dependency parsing is faster, requires less annotation, and works better for free-word-order languages. For English NLP tasks, dependency parsing is preferred.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import spacy
from spacy import displacy
nlp = spacy.load('en_core_web_sm')
doc = nlp('Apple acquired Shazam for $400 million in 2018.')
# Print dependency tree
for token in doc:
    print(f'{token.text:12s} | dep: {token.dep_:10s} | head: {token.head.text:12s} | pos: {token.pos_}')
# Extract subject-verb-object triples
def extract_svo(doc):
    triples = []
    for token in doc:
        if token.dep_ == 'ROOT' and token.pos_ == 'VERB':
            subj = [t.text for t in token.children if t.dep_ in ('nsubj', 'nsubjpass')]
            obj = [t.text for t in token.children if t.dep_ in ('dobj', 'obj', 'pobj')]
            if subj and obj:
                triples.append((subj[0], token.text, obj[0]))
    return triples
triples = extract_svo(doc)
for s, v, o in triples:
    print(f'{s} → {v} → {o}')  # Apple → acquired → Shazam
# Visualize (in Jupyter)
# displacy.render(doc, style='dep', jupyter=True)
Why this matters in AI
Dependency parsing enables structured information extraction from free text without manual pattern writing. Extracting 'who acquired what' from thousands of news articles requires a systematic way to find the subject and object of acquisition verbs — dependency parsing provides this systematically. This is the foundation of knowledge graph construction and open information extraction.
Common mistakes
  • Assuming the ROOT is always the main verb — in questions ('What did Apple acquire?'), the auxiliary verb may be ROOT. In some complex sentences, a noun phrase is ROOT. Always check the POS tag of the ROOT, not just its dependency label.
  • Not handling passives — in passive constructions ('Shazam was acquired by Apple'), the subject of 'acquired' is 'Shazam' (the patient, not the agent). Use nsubjpass (passive subject) to detect passives and swap the semantic roles accordingly.
  • Using spaCy's small model (en_core_web_sm) for accuracy-critical applications — en_core_web_sm is fast but less accurate than en_core_web_trf (transformer-based). For production information extraction, use the transformer model.

Relation extraction

Detailed explanation

Relation extraction identifies semantic relationships between entities in text. 'Apple, headquartered in Cupertino, was founded by Steve Jobs.' Relations: (Apple, headquarters, Cupertino), (Apple, foundedBy, Steve_Jobs). Relation extraction converts unstructured text into structured triples that populate knowledge graphs.

Approaches: rule-based (fast, high precision, low recall — patterns like 'X, headquartered in Y'), supervised classification (train a classifier: is there a relation between entity pair X, Y? What type?), and generative (use an LLM to generate all relations from a passage in JSON format — highest quality, highest cost).

REBEL: a generative relation extraction model (BARTbased) that extracts (subject, relation, object) triples from any text. Fine-tuned on WebNLG and NYT datasets. `model = AutoModelForSeq2SeqLM.from_pretrained('Babelscape/rebel-large')`. Input: text. Output: '<triplet> Apple <subj> Cupertino <rel> headquarters'.

OpenIE vs closed-domain RE: Open Information Extraction (OpenIE) extracts relations without a predefined ontology — any (subject, relation, object) where the relation is expressed explicitly. Closed-domain RE uses a predefined set of relation types (headquarters, foundedBy, employedBy) and trains a classifier for each type. For knowledge graph population: use closed-domain RE. For free-form extraction: use OpenIE or LLMs.

LLM-based relation extraction: with few-shot prompting, an LLM can extract relations for any predefined ontology without training data. Prompt: 'Extract all (entity1, relation, entity2) triples from the text. Relation types: headquarters, foundedBy, productOf, employedAt.' Output as JSON array. This approach achieves high accuracy with minimal setup but at LLM API cost.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

# LLM-based relation extraction
import anthropic
import json
client = anthropic.Anthropic()
def extract_relations(text: str, relation_types: list) -> list:
    prompt = f"""Extract all entity relationships from the text.
Relation types allowed: {', '.join(relation_types)}
Output format: JSON array of {{"subject": str, "relation": str, "object": str}}
Text: {text}
"""
    response = client.messages.create(
        model='claude-opus-4-8', max_tokens=500,
        messages=[{'role': 'user', 'content': prompt}]
    )
    text_response = response.content[0].text
    # Parse JSON from response
    import re
    json_match = re.search(r'\[.*?\]', text_response, re.DOTALL)
    if json_match:
        return json.loads(json_match.group())
    return []
relations = extract_relations(
    'Google, founded by Larry Page and Sergey Brin, is headquartered in Mountain View. The company acquired YouTube in 2006.',
    ['foundedBy', 'headquarters', 'acquired']
)
for rel in relations:
    print(f'{rel["subject"]} → [{rel["relation"]}] → {rel["object"]}')
Why this matters in AI
Relation extraction powers knowledge graphs that make information queryable. Instead of searching text, you can query a structured database: 'which companies acquired companies worth > $1B?'. This is used for financial analysis (acquisition networks), medical research (drug-disease interactions), and enterprise search. LLM-based relation extraction makes it accessible without a labeled training dataset.
Common mistakes
  • Treating relation extraction as pure NLP without domain knowledge — the relation 'X acquired Y' requires knowing whether X and Y are both companies. Always combine NER (entity type) with relation extraction (relationship type) for accurate extraction.
  • Not handling negated relations — 'Apple did not acquire Beats Music' contains 'Apple', 'acquire', 'Beats Music' — a rule-based extractor might incorrectly extract (Apple, acquired, Beats Music). Always check for negation markers before asserting a relation.
  • Over-relying on syntactic patterns for complex sentences — 'Apple's 2014 acquisition of Beats Music by its senior team' is complex. Simple subject-verb-object extraction fails. Use a generative approach (REBEL or LLM) for complex sentences.

Event detection

Detailed explanation

Event detection identifies occurrences of specific events in text: 'Apple announced Q3 earnings' (EARNINGS_RELEASE), 'CEO resigned effective Friday' (LEADERSHIP_CHANGE), 'the earthquake struck the region' (NATURAL_DISASTER). Event detection is a classification task: given a sentence, which events are present, and what are their triggers (the word that indicates the event)?

ACE event types: the ACE (Automatic Content Extraction) benchmark defines event types like CONFLICT (Attack, Demonstrate), MOVEMENT (Transport), TRANSACTION (Transfer-Money), PERSONNEL (Elect, Start-Position, End-Position). Fine-tuning BERT on ACE produces state-of-the-art event detection.

Trigger identification: the trigger is the word that signals the event. In 'Apple announced earnings', 'announced' is the trigger for a COMMUNICATION event. In 'Steve Jobs died in 2011', 'died' is the trigger for a LIFE:DIE event. Event detection models identify both the trigger and its event type.

Argument extraction: beyond detecting that an event occurred, extract who/what/where/when. 'Microsoft acquired Activision Blizzard for $69B in 2023.' Event: TRANSACTION:Transfer-Money. Trigger: 'acquired'. Buyer: 'Microsoft' (BUYER role), Target: 'Activision Blizzard' (ARTIFACT role), Price: '$69B' (PRICE role), Date: '2023' (TIME role). Argument extraction is the hardest part of event extraction.

LLM-based event extraction: LLMs can extract events in a structured format with few-shot prompting, without training on labeled ACE data. Prompt: 'Identify all events in the text. For each event, provide: event_type, trigger, and all arguments with their roles.' Output as JSON. This approach is flexible (works for any event ontology) but more expensive than fine-tuned models.

Worked example

# LLM-based event extraction
import anthropic
import json
client = anthropic.Anthropic()
def extract_events(text: str) -> list:
    prompt = f"""Extract all events from this text.
For each event provide: event_type, trigger_word, and a dictionary of arguments (role: entity_or_value).
Focus on: business events (acquisition, earnings, leadership change, product launch, bankruptcy).
Output as JSON array.
Text: {text}"""
    response = client.messages.create(
        model='claude-opus-4-8', max_tokens=800,
        messages=[{'role': 'user', 'content': prompt}]
    )
    import re
    json_match = re.search(r'\[.*?\]', response.content[0].text, re.DOTALL)
    if json_match: return json.loads(json_match.group())
    return []
events = extract_events('Microsoft acquired Activision Blizzard for $69 billion. Following the announcement, CEO Phil Spencer said the deal would close in 2023.')
for event in events:
    print(f'Event: {event.get("event_type")} | Trigger: {event.get("trigger_word")}')
    print(f'Arguments: {event.get("arguments", {})}')
Why this matters in AI
Event detection is how intelligent systems monitor what happens in the world from text streams. Financial firms use event detection to track earnings releases, M&A activity, and leadership changes for trading signals. News aggregators use it to categorize articles by event type. Crisis response systems detect disaster events from social media to coordinate aid. Automating event detection from text streams provides real-time situational awareness at scale.
Common mistakes
  • Conflating event detection with topic classification — topic classification says 'this article is about business'. Event detection says 'this article contains an ACQUISITION event triggered by acquired with arguments Microsoft (buyer), Activision (target), $69B (price)'. Event detection is more structured and specific.
  • Ignoring implicit events — some events are implied, not explicitly stated. 'After the crash, three people were hospitalized' implies a CONFLICT:Attack or TRANSPORTATION:Crash event even though 'crash' is the only trigger. Implicit event detection is harder and requires commonsense reasoning.
  • Not evaluating argument extraction separately from trigger detection — a model can correctly detect 'there is an acquisition event' but incorrectly extract the buyer and target. Report trigger F1 and argument F1 separately.
Practice task
Build an event monitoring system: (1) collect 200 financial news articles, (2) define 5 event types: ACQUISITION, EARNINGS, LEADERSHIP_CHANGE, PRODUCT_LAUNCH, BANKRUPTCY, (3) use LLM extraction to identify events and their arguments in each article, (4) build a timeline: events sorted by date — what happened when?, (5) for ACQUISITION events: build a graph of acquirer → target relationships, (6) evaluate accuracy: manually label 30 articles and compare to LLM extraction — what is the precision and recall?

Text clustering

Detailed explanation

Text clustering groups documents by topic, theme, or style without labeled training data. It enables: discovering major themes in a large corpus (1M customer reviews → 15 topic clusters), grouping similar support tickets for routing, and finding duplicate or near-duplicate content. The core approach: embed documents → cluster embeddings.

Embedding for clustering: the embedding model determines how similar documents look to the clustering algorithm. A TF-IDF vector will cluster documents sharing exact keywords. A sentence embedding (all-MiniLM-L6-v2) will cluster documents sharing semantic meaning (even if they use different words). For topic-based clustering, sentence embeddings are superior.

K-means for text: the simplest approach. Embed all documents, run K-means on embeddings. Works well for well-separated clusters. Limitation: must specify K in advance, assumes spherical clusters. For exploratory analysis where K is unknown, use HDBSCAN or BERTopic.

BERTopic: the state-of-the-art approach for topic modeling with transformer embeddings. Pipeline: BERT embeddings → UMAP dimensionality reduction (20D) → HDBSCAN clustering → c-TF-IDF to extract representative keywords per cluster. BERTopic automatically determines the number of topics (unlike K-means), handles noise (HDBSCAN marks outliers), and provides interpretable topic labels.

Evaluating text clusters: topic coherence (are the words in each topic semantically related?), topic diversity (are topics distinct from each other?), human evaluation (do domain experts agree that the clusters make sense?). For quantitative metrics: silhouette score on embeddings, perplexity (for LDA), and NPMI (Normalized Pointwise Mutual Information) for topic word coherence.

Why this matters in AI
Text clustering is one of the most valuable unsupervised NLP techniques for business. When you have 500,000 customer support tickets and want to understand what customers are struggling with, you can't read them all. Text clustering with BERTopic gives you the 20 major themes in 5 minutes. This drives product decisions, documentation priorities, and support team organization.
Common mistakes
  • Clustering raw text without embeddings — TF-IDF + K-means works but misses semantic groupings. 'Cannot login' and 'password reset failure' would be in different clusters with TF-IDF but the same cluster with sentence embeddings. Always use semantic embeddings for meaningful topic clustering.
  • Not reducing dimensionality before clustering — clustering in 384-dimensional embedding space is slow and suffers from the curse of dimensionality. UMAP to 10-50 dimensions before K-means or HDBSCAN dramatically improves cluster quality and speed.
  • Forcing a fixed K without validation — K=20 might produce 15 meaningful clusters and 5 garbage clusters that capture noise. Use silhouette score or HDBSCAN (which automatically determines K) to avoid forcing the data into artificial K clusters.

Topic modeling

Detailed explanation

Topic modeling is an unsupervised NLP technique that discovers latent themes (topics) in a collection of documents. Each document is represented as a mixture of topics, and each topic is a distribution over words. Given 10,000 news articles, topic modeling might discover 20 topics: one about sports (football, player, team, goal), one about politics (election, vote, senate), one about technology (AI, software, startup). This is useful for organizing and understanding large text corpora without labels.

LDA (Latent Dirichlet Allocation) is the classic topic model. It assumes: (1) each topic is a distribution over words (topic 'sports' has high probability for 'football', 'player', etc.), (2) each document is a distribution over topics (a sports article is 80% sports topic, 10% business topic, 10% other), (3) each word in a document is generated by first sampling a topic from the document's topic distribution, then sampling a word from that topic's word distribution. The model is fit using variational inference or Gibbs sampling.

Choosing the number of topics K is the main challenge in LDA. Too few topics: distinct themes are merged. Too many topics: topics become redundant or nonsensical. Evaluation metrics: coherence score (how often do the top words in a topic co-occur in the corpus? Higher = more interpretable topics — typically c_v or u_mass coherence). Perplexity (how well the model fits held-out documents — lower = better, but doesn't correlate well with interpretability). In practice, try K = 5, 10, 20, 50 and manually inspect the top words per topic.

Modern alternatives to LDA: BERTopic uses BERT embeddings + UMAP + HDBSCAN clustering to discover topics in embedding space, then extracts representative keywords using TF-IDF within each cluster. BERTopic produces more coherent, semantically meaningful topics than LDA, handles hierarchical topics, and works well on short texts (tweets, reviews) where LDA struggles. NMF (Non-negative Matrix Factorization) is another alternative — faster than LDA, good for short texts.

Applications: document organization and search (tag articles by topic for navigation), customer feedback analysis (discover what customers complain about without predefined categories), content recommendation (recommend articles from the same topic), trend analysis (how has the prevalence of each topic changed over time). In business intelligence, topic modeling on customer reviews and support tickets surfaces actionable themes without manual labeling.

Topic modeling is a powerful unsupervised technique for making sense of large text corpora. When you receive a million customer reviews, topic modeling reveals what customers care about without requiring any labels. It is the first analysis step for text dataset exploration.

LDA

Detailed explanation

LDA (Latent Dirichlet Allocation) is the classical probabilistic topic model. Each document is modeled as a mixture of topics, and each topic is a distribution over words. LDA assumes a generative process: (1) choose topic proportions θ for the document, (2) for each word, choose a topic z from θ, (3) choose the word from the topic distribution φ_z. The model learns θ (document-topic) and φ (topic-word) by inverting this process on observed documents.

LDA vs neural topic models (BERTopic): LDA uses bag-of-words and ignores word order/semantics. 'Machine learning' and 'learning machine' have the same LDA representation. BERTopic uses contextual embeddings and captures semantic meaning. In practice: BERTopic typically produces more coherent and interpretable topics. LDA is still used for: interpretability requirements, when compute is limited, when you have very clean domain-specific text, and when you need a probabilistic document-topic distribution.

Key hyperparameters: n_topics (number of topics — requires the same K selection as K-means), alpha (controls topic diversity per document: low alpha = each document focuses on few topics, high alpha = topics spread evenly), beta (controls word diversity per topic: low beta = each topic focuses on few words, high beta = words spread evenly). Typical values: alpha = 50/n_topics, beta = 0.01.

Coherence score: measures how semantically similar the top words in each topic are. Higher is better. Computed as the average NPMI (Normalized Pointwise Mutual Information) between word pairs in each topic. Use gensim's CoherenceModel with metric='c_v' for the standard coherence measure. Tune n_topics by maximizing coherence score.

pyLDAvis: the standard LDA visualization tool. Creates an interactive 2D map of topics (by inter-topic distance) plus word frequency charts for each topic. Topics that appear as distinct bubbles in the visualization are well-separated. Topics that overlap share vocabulary and could be merged.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from gensim import corpora, models
from gensim.models import CoherenceModel
from gensim.parsing.preprocessing import remove_stopwords, preprocess_string
# Preprocessing
preprocessed_docs = [preprocess_string(doc) for doc in documents]
# Create dictionary and corpus
dictionary = corpora.Dictionary(preprocessed_docs)
dictionary.filter_extremes(no_below=5, no_above=0.5)  # remove rare and common words
corpus = [dictionary.doc2bow(doc) for doc in preprocessed_docs]
# Train LDA
print('Training LDA...')
lda_model = models.LdaModel(
    corpus=corpus, id2word=dictionary,
    num_topics=10, random_state=42,
    passes=10, alpha='auto', eta='auto'
)
# Print topics
for i, topic in lda_model.print_topics(num_topics=10, num_words=8):
    print(f'Topic {i}: {topic}')
# Coherence score
coherence = CoherenceModel(model=lda_model, texts=preprocessed_docs, dictionary=dictionary, coherence='c_v').get_coherence()
print(f'Coherence score: {coherence:.4f}')
Why this matters in AI
LDA is foundational to understanding how probabilistic topic models work and is still widely used in the NLP field. Even though BERTopic often outperforms LDA in practice, LDA's probabilistic interpretation (each document is a mixture of topics with known proportions) is valuable for downstream analysis (how much of document X is about topic Y?). Understanding LDA also helps you understand the limitations of bag-of-words approaches that motivate transformer-based methods.
Common mistakes
  • Not tuning n_topics — LDA with n_topics=5 on a diverse corpus will produce 5 overly broad topics. Tune n_topics by training with n_topics in [5, 10, 20, 50] and selecting the value with the highest coherence score.
  • Not filtering stopwords and very common words — 'the', 'a', 'is' appear in every document. Without removing them, they dominate all topics. Use dictionary.filter_extremes(no_above=0.5) to remove words that appear in more than 50% of documents.
  • Using LDA on very short documents (tweets, titles) — LDA needs enough words per document to estimate topic mixtures. For tweets (< 30 words), LDA often fails. Use a tweet-specific variant (Twitter-LDA) or aggregate similar short documents before applying LDA.

Language identification

Detailed explanation

Language identification (language detection) determines which natural language a text is written in. It is the first step in any multilingual NLP pipeline: route text to language-specific models, apply language-appropriate preprocessing, or filter by language. Modern language identification uses character n-grams and short-text models that work with as few as 10-20 characters.

langdetect: port of Google's language-detect library. Supports 55 languages. `detect(text)` returns the language code. `detect_langs(text)` returns a list of (language, probability) pairs. Fast and accurate for clean text. Unreliable for: code (Python, SQL), mixed-language text, text with many proper nouns, and very short text.

lingua: newer library designed for accuracy on short text (10-15 characters) and ambiguous cases. Supports 75 languages. Uses a combination of statistical n-gram models and rule-based heuristics. More accurate than langdetect on edge cases at the cost of slightly more compute. `from lingua import LanguageDetectorBuilder; detector = LanguageDetectorBuilder.from_all_languages().build()`.

CLD3 (Compact Language Detector): Google's fast language detector. The chromium-based detector used in production for web content. Available as pycld3 Python binding. Very fast and optimized for short web text. Outputs ISO 639-1 language codes with probabilities.

Handling edge cases: mixed-language text (English with Spanish phrases) — no single language detector handles this well. Custom solution: sliding window over sentences, detect each independently. Code and technical text — all detectors fail on SQL/Python. Solution: detect code language separately (pygments), fallback to english for undetected. Very short text (< 20 chars) — any detector will be unreliable. Set a minimum text length threshold before detection.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

# Multiple language detection libraries
# pip install langdetect lingua-language-detector
from langdetect import detect, detect_langs, DetectorFactory
DetectorFactory.seed = 42  # for reproducibility
# langdetect
texts = ['This is English text.', 'Ceci est du texte francais.', 'Das ist deutscher Text.', 'Este es texto espanol.', 'これは日本語のテキストです。']
for text in texts:
    try:
        lang = detect(text)
        langs = detect_langs(text)
        print(f'{text[:30]:30s} → {lang} ({langs[0]})')
    except Exception as e:
        print(f'Error: {e}')
# lingua (more accurate on short text)
from lingua import LanguageDetectorBuilder, Language
detector = LanguageDetectorBuilder.from_all_languages().build()
for text in texts:
    lang = detector.detect_language_of(text)
    conf = detector.compute_language_confidence(text, lang) if lang else 0
    print(f'{text[:30]:30s} → {lang.iso_code_639_1.name if lang else "UNKNOWN"} (conf: {conf:.3f})')
Why this matters in AI
Language identification is a critical preprocessing step for any global NLP system. Sending French text to an English-only model, applying English stopwords to Spanish text, or routing German support tickets to an English-only team are all failures caused by missing or incorrect language detection. Adding language identification is typically one line of code but prevents these silent failures in multilingual pipelines.
Common mistakes
  • Not handling detection failures — both langdetect and lingua raise exceptions on empty strings and sometimes on very short text. Always wrap in try-except and fall back to a default language or 'unknown' flag.
  • Using langdetect without fixing the seed — langdetect uses random initialization, so the same text can return different results across runs. Fix with `DetectorFactory.seed = 0` for reproducible results.
  • Not validating language distribution in your dataset — after language detection, always check: what fraction of your dataset is in each language? If 5% is in an unexpected language (Tagalog, Arabic), you may need to handle these differently or filter them.

On what types of text does detection fail? (5) implement a routing function: detected language → appropriate sentiment model (or log language and route to English model if unsupported).

Text augmentation

Detailed explanation

Text augmentation creates additional training examples by applying transformations to existing text. It's used when labeled data is scarce (adding augmented examples increases the effective training set) and when robustness is needed (augmented examples with word substitutions train the model to handle synonyms and paraphrases). Text augmentation is more constrained than image augmentation — it must preserve meaning while changing surface form.

Synonym replacement: replace random non-stopwords with synonyms from WordNet. 'The movie was great' → 'The film was excellent.' Implementation: nlpaug or textaugment libraries handle this. Limit to 10-30% of words to preserve meaning. Synonyms from WordNet are sometimes odd ('movie' → 'film' is good; 'movie' → 'video recording' is too weird).

Back-translation: translate to an intermediate language and back. 'The model performs well' → French: 'Le modèle fonctionne bien' → English: 'The model works well.' Produces natural-sounding paraphrases that preserve meaning. High quality but slow (requires translation calls). Use Google Translate, DeepL, or Helsinki-NLP transformers for offline translation.

EDA (Easy Data Augmentation): four simple operations: (1) SR (Synonym Replacement): replace n words with synonyms. (2) RI (Random Insertion): insert a random synonym of a random word. (3) RS (Random Swap): swap two words. (4) RD (Random Deletion): randomly delete words with probability p. EDA paper showed these simple augmentations improve text classification accuracy by 2-3% on small datasets.

LLM-based augmentation: generate paraphrases using an LLM. 'Rephrase the following while preserving meaning: [text]'. LLM paraphrases are the highest quality augmentation — semantically identical but surface-form varied. The trade-off: cost and latency. For 10,000 training examples, generating 3 paraphrases each = 30,000 LLM API calls.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

# pip install nlpaug
import nlpaug.augmenter.word as naw
import nlpaug.augmenter.sentence as nas
# Synonym replacement
aug_syn = naw.SynonymAug(aug_src='wordnet', aug_max=3)  # replace up to 3 words
original = 'The quick brown fox jumps over the lazy dog'
augmented = [aug_syn.augment(original)[0] for _ in range(3)]
print('Original:', original)
for a in augmented: print('Augmented:', a)
# Back-translation (using Helsinki-NLP)
from transformers import pipeline
translate_en_to_fr = pipeline('translation', model='Helsinki-NLP/opus-mt-en-fr')
translate_fr_to_en = pipeline('translation', model='Helsinki-NLP/opus-mt-fr-en')
def back_translate(text, src_lang='en', pivot_lang='fr'):
    to_pivot = translate_en_to_fr(text)[0]['translation_text']
    back = translate_fr_to_en(to_pivot)[0]['translation_text']
    return back
print('Back-translated:', back_translate(original))
Why this matters in AI
Text augmentation is essential when you have few labeled examples (50-500 per class) and need to improve model performance without collecting more labels. Synonym replacement and back-translation are standard augmentation techniques in NLP papers and production systems. For low-resource languages and niche domains, augmentation can provide 5-15% improvement in classification accuracy, which can mean the difference between a usable and unusable model.
Common mistakes
  • Augmenting too aggressively — replacing 50% of words with synonyms often produces text with wrong meaning or grammatically odd phrasing. Limit augmentation to 10-20% of words per sentence. Test augmented examples manually to ensure meaning is preserved.
  • Applying augmentation to validation and test sets — augmentation is only for the training set. The validation and test sets must contain original, unmodified text to give an honest estimate of performance on real data.
  • Using synonym replacement for domain-specific text — WordNet synonyms are general-purpose. For medical text, 'hypertension' synonyms are not 'blood pressure disorder' — the clinical synonym is wrong. Use domain-specific resources or back-translation, which preserves meaning more reliably than synonym substitution.

Multilingual NLP

Detailed explanation

Multilingual NLP models process text in multiple languages within a single model. Rather than training separate models for each language, multilingual models learn shared representations across languages. The key insight: many linguistic structures (syntax, semantics) are universal across languages, and a model trained on enough multilingual data learns to represent them in a shared space.

Multilingual BERT (mBERT): trained on Wikipedia text from 104 languages with a shared vocabulary (119,547 WordPiece tokens) and shared model weights. Languages share a single embedding space — similar sentences in different languages map to similar vectors. Fine-tuning on one language often generalizes to related languages (cross-lingual transfer).

XLM-RoBERTa (XLM-R): the state-of-the-art multilingual model — RoBERTa architecture trained on 2.5TB of web text from 100 languages. Significantly outperforms mBERT, especially for low-resource languages. XLM-R-large (550M parameters) achieves near monolingual performance on many tasks.

Zero-shot cross-lingual transfer: fine-tune an XLM-R model on an English task (NER, sentiment, QA), then evaluate directly on the same task in French, Chinese, or German without any target-language labeled data. This works remarkably well for structurally similar languages (French, German, Spanish) but degrades for distant languages (Japanese, Arabic).

Practical considerations: tokenization — multilingual models use a shared SentencePiece vocabulary. Less common languages get fewer vocabulary tokens and are over-segmented (more subword splits per word), which hurts performance. Language identification — langdetect or lingua are fast language detection libraries. Always identify the language and route to the appropriate model (or multilingual model) in production.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from transformers import pipeline
# Multilingual sentiment analysis
sentiment = pipeline('sentiment-analysis', model='cardiffnlp/twitter-xlm-roberta-base-sentiment')
texts = [
    'This is amazing!',           # English
    'C est magnifique!',          # French
    'Das ist fantastisch!',       # German
    'Esto es increible!',         # Spanish
]
for text in texts:
    result = sentiment(text)[0]
    print(f'{text[:30]:30s} → {result["label"]} ({result["score"]:.3f})')
# Cross-lingual NER: fine-tuned on English, tested on Spanish
ner = pipeline('ner', model='Babelscape/wikineural-multilingual-ner', aggregation_strategy='simple')
spanish_text = 'Google fue fundado por Larry Page en California.'
print(ner(spanish_text))
Why this matters in AI
Multilingual NLP is essential for global products. A sentiment analyzer that only works in English cannot handle customer reviews from France or Germany. Building separate per-language models is expensive and inconsistent. Multilingual models like XLM-R provide a single model that handles 100 languages, with minimal additional annotation required for each language through cross-lingual transfer.
Common mistakes
  • Expecting equal performance across all 100 languages — XLM-R achieves 80-90% of English performance for high-resource languages (French, German, Spanish) but may only achieve 60-70% for very low-resource languages. Always evaluate on the target language.
  • Not accounting for code-switching — multilingual text often mixes languages ('I love la vida loca!'). Standard monolingual tokenizers fail here. Use XLM-R which handles this naturally, or detect and segment by language first.
  • Ignoring right-to-left languages — Arabic and Hebrew are right-to-left. UI rendering, text direction, and tokenization all behave differently. Test multilingual models explicitly on RTL text.
