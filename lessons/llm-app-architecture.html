Frontend backend model provider

Detailed explanation

The frontend-backend-model provider architecture is the standard three-tier design for production AI applications. Each tier has a single responsibility: the frontend handles user interface and interaction, the backend handles business logic and orchestration, and the model provider supplies AI capabilities via API.

Frontend: the user-facing interface. Technologies: React/Next.js for web apps, React Native/Swift/Kotlin for mobile, voice interfaces for smart speakers. The frontend sends user inputs to the backend API and renders AI responses. The frontend should NEVER call model provider APIs directly — API keys would be exposed to users who can inspect network requests.

Backend: the application server. Technologies: FastAPI (Python), Express.js (Node), Spring Boot (Java). Responsibilities: authenticate users, validate inputs, call the model provider API, apply business logic to the response, log usage, apply rate limits, and return formatted responses to the frontend. The backend holds API keys securely in environment variables.

Model provider: the AI capability layer. Commercial providers: Anthropic (Claude API), OpenAI (GPT-4), Google (Gemini). Self-hosted: vLLM, Ollama, TGI (Text Generation Inference). The backend calls the model provider, which returns completions, embeddings, or structured outputs. Switching providers (from OpenAI to Anthropic) only requires changing the backend's model provider integration, not the frontend.

Separation of concerns in practice: the frontend sends 'please summarize this document.' The backend: authenticates the request, retrieves the document from storage, chunks it, calls Anthropic's API, assembles the summary, logs the request and response, and returns the summary. The model provider only sees 'please summarize [text]' — it has no knowledge of the user's identity or the document's storage location.

# Backend API that calls Anthropic securely
from fastapi import FastAPI, Depends, HTTPException
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials
from pydantic import BaseModel
import anthropic
import os
app = FastAPI()
client = anthropic.Anthropic(api_key=os.environ['ANTHROPIC_API_KEY'])  # Never hardcode
security = HTTPBearer()
USERS_DB = {'user_token_123': {'id': 'user1', 'credits': 100}}  # Simplified
def get_current_user(credentials: HTTPAuthorizationCredentials = Depends(security)):
    user = USERS_DB.get(credentials.credentials)
    if not user: raise HTTPException(401, 'Invalid token')
    return user
class ChatRequest(BaseModel): message: str; max_tokens: int = 500
@app.post('/chat')
async def chat(request: ChatRequest, user: dict = Depends(get_current_user)):
    if user['credits'] <= 0: raise HTTPException(402, 'Insufficient credits')
    response = client.messages.create(
        model='claude-opus-4-8', max_tokens=request.max_tokens,
        messages=[{'role': 'user', 'content': request.message}]
    )
    # Deduct credits based on tokens used
    user['credits'] -= response.usage.output_tokens / 1000
    return {'response': response.content[0].text, 'credits_remaining': user['credits']}
Why this matters in AI
The three-tier architecture is not just best practice — it's a security requirement. Exposing model provider API keys in the frontend means any user can extract the key from browser dev tools and make unlimited API calls at your expense. The backend-as-proxy pattern is the only way to safely call commercial AI APIs in user-facing applications.
Common mistakes
  • Calling the LLM API from the frontend — the API key is visible in browser network requests. Never call Anthropic, OpenAI, or any other API with secrets from client-side code. All such calls must go through your backend.
  • Not adding user authentication before AI calls — without authentication, anyone can call your backend AI endpoints. Even a simple API key system prevents anonymous abuse. For production: JWT tokens, OAuth, or session cookies.
  • Not implementing caching at the backend layer — the backend is the right place to implement response caching (semantic caching, exact match). The frontend should not cache LLM responses — that belongs in the application layer that can access the database.

Prompt versioning

Detailed explanation

Prompt versioning is the practice of tracking changes to LLM prompts over time, enabling rollback, A/B testing, and systematic improvement. As prompts evolve (you add examples, refine instructions, change the format), prompt versioning ensures you can always return to a working version and understand what changed between versions.

Why version prompts: prompt changes can unexpectedly break downstream behavior. 'Added a formatting instruction' seems harmless but might reduce the model's accuracy on edge cases. Without versioning, you can't compare the new prompt's performance to the old one, or roll back if quality drops.

Version control approaches: simple text files in Git (each prompt is a .txt or .md file, changes tracked by git history). Langfuse or PromptLayer (purpose-built prompt management platforms with version history, performance tracking, and A/B testing). Environment-specific prompts (different prompt versions for dev/staging/prod in a config file). The simplest approach: a JSON file mapping {prompt_name: version: prompt_text} committed to Git.

Prompt testing before deployment: before promoting a new prompt version to production, run it through your evaluation pipeline. Compare: mean quality score, per-category scores, failure modes. A new prompt is only deployed if it performs > current version on the evaluation set without regressions on any category.

Prompt engineering best practices under versioning: small, targeted changes (one change per version), clear commit messages ('v3: added 2 few-shot examples to improve JSON formatting'), and semantic versioning (v1 = major behavior change, v1.1 = minor tweak, v1.0.1 = fix edge case).

import json
import hashlib
from datetime import datetime
import os
# Simple prompt version manager
class PromptVersionManager:
    def __init__(self, storage_path='prompts.json'):
        self.path = storage_path
        self.prompts = self._load()
    def _load(self): return json.load(open(self.path)) if os.path.exists(self.path) else {}
    def _save(self): json.dump(self.prompts, open(self.path, 'w'), indent=2)
    def save_version(self, name: str, content: str, description: str = '') -> str:
        hash_id = hashlib.md5(content.encode()).hexdigest()[:8]
        if name not in self.prompts: self.prompts[name] = {'versions': [], 'current': None}
        version = {'id': hash_id, 'content': content, 'description': description, 'created_at': datetime.now().isoformat()}
        self.prompts[name]['versions'].append(version)
        self.prompts[name]['current'] = hash_id
        self._save()
        return hash_id
    def get(self, name: str, version_id: str = None) -> str:
        p = self.prompts.get(name)
        if not p: raise ValueError(f'Prompt {name} not found')
        vid = version_id or p['current']
        return next(v['content'] for v in p['versions'] if v['id'] == vid)
    def rollback(self, name: str, version_id: str): self.prompts[name]['current'] = version_id; self._save()
pmgr = PromptVersionManager()
pmgr.save_version('system_prompt', 'You are a helpful assistant.', 'Initial version')
pmgr.save_version('system_prompt', 'You are a helpful, concise assistant. Always respond in bullet points.', 'Added formatting requirement')
print(pmgr.get('system_prompt'))  # Gets current version
Task: Classify the customer message.
Return JSON with keys: label, reason.
Message: "I want to cancel my plan"
Why this matters in AI
Prompt versioning is the foundation of systematic prompt engineering. Without it, 'the model used to work better' is just an observation you can't act on — you don't know what the previous prompt was. With versioning, you can: reproduce any historical behavior, A/B test prompt variants, and make evidence-based decisions about which prompt to use in production.
Common mistakes
  • Storing prompts as string literals in code — when a prompt is a string constant in a Python file, changing it is a code deployment. Using a prompt management system (or even a JSON file) separates prompt engineering from code deployment, allowing faster iteration.
  • Not evaluating before promoting to production — every prompt change should go through the evaluation pipeline. A prompt that improves instruction following might degrade safety behaviors or format compliance in ways that aren't obvious without systematic testing.
  • Over-versioning trivial changes — not every prompt tweak needs a version. Version at meaningful milestones: when you change the behavior, add examples, or modify the task framing. Don't create v1.0.1, v1.0.2, v1.0.3 for typo fixes — fix typos in-place and record the final state.

RAG architecture

Detailed explanation

RAG (Retrieval-Augmented Generation) is an architecture that combines a vector retrieval system with a generative language model. Instead of relying solely on the LLM's parametric knowledge (baked in during pretraining), RAG retrieves relevant context from an external knowledge base at query time and provides it to the LLM as part of the prompt. This enables the LLM to answer questions about information it was not trained on — private documents, real-time data, and domain-specific knowledge.

The RAG pipeline has four stages: (1) Ingestion — chunk documents into passages (typically 256–512 tokens with 50-token overlap), encode each chunk into a dense vector using a text embedding model (e.g., text-embedding-3-small, BAAI/bge-large-en), and store vectors in a vector database (FAISS, Chroma, Pinecone, pgvector). (2) Retrieval — encode the query, find the top-K most similar document chunks by cosine similarity. (3) Augmentation — insert retrieved chunks into the prompt as context. (4) Generation — the LLM generates an answer conditioned on the query and retrieved context.

Chunking strategy is critical and often underestimated. Fixed-size chunking (512 tokens, 50-token overlap) is simple but splits sentences mid-thought. Semantic chunking uses embedding similarity to find natural break points. Hierarchical chunking creates document-level and paragraph-level chunks and retrieves both. Sentence-window retrieval retrieves individual sentences but passes their surrounding window to the LLM for context. Poor chunking is the most common source of RAG failures.

Hybrid search combines dense retrieval (semantic similarity via embeddings) with sparse retrieval (keyword matching via BM25). Dense retrieval finds semantically similar passages even when they use different words. Sparse retrieval (BM25) is better at exact keyword matches (product codes, names, specific terminology). Combining them with a reranker (a cross-encoder that scores query-passage pairs) consistently outperforms either alone. The Reciprocal Rank Fusion (RRF) algorithm combines rankings from multiple retrieval systems without score normalization.

Evaluating RAG systems requires domain-specific test sets with 50–200 question-answer pairs covering the document collection. Metrics: context relevance (are retrieved chunks actually relevant to the query?), groundedness (is the LLM's answer supported by the retrieved context?), answer relevance (does the answer address the question?). RAGAS is a popular framework that automates these evaluations using an LLM judge. Human evaluation of a random sample is the gold standard.

Why this matters in AI
RAG is the most important LLM application pattern in enterprise AI. It enables LLMs to work with private, current, and domain-specific information — the use cases companies actually need. Understanding RAG architecture, chunking, retrieval, and evaluation is essential for anyone building LLM-powered applications.
Common mistakes
  • Not overlapping chunks — if a relevant sentence spans a chunk boundary, it will not be fully retrieved. Always use chunk_overlap of 10–20% of chunk_size.
  • Retrieving too few chunks (K=1–2) or too many (K=20+) — too few misses relevant context; too many fills the context window with noise. K=3–7 is typically optimal.
  • Not evaluating retrieval separately from generation — a RAG system can fail at retrieval (relevant docs not retrieved) or generation (retrieved correctly but answer is wrong). Diagnose each separately.

Tool layer

Detailed explanation

The tool layer in an LLM agent system is the collection of tools available to the agent, their definitions, implementation, validation, and the dispatch mechanism that executes tool calls. A well-designed tool layer enables the agent to take meaningful actions; a poorly designed one is a source of errors, security vulnerabilities, and agent failures.

Tool design principles: each tool should do exactly one thing. Tools should be idempotent where possible (calling a tool twice should not cause double effects). Tools should have clear, informative descriptions (the model decides which tool to call based on the description). Tools should return structured, parseable output (JSON > free text).

Tool categories: read tools (safe, no side effects — web search, database query, file read, calculator), write tools (side effects — send email, database write, file create), system tools (dangerous — execute code, shell commands). Apply permission levels: read tools always allowed, write tools require user-tier authentication, system tools require explicit human approval.

Tool registration and dispatch: maintain a registry of available tools. When the model returns a tool call, the dispatcher: (1) validates the tool exists, (2) validates the tool call format against the tool's schema, (3) checks permissions, (4) executes the tool, (5) handles errors gracefully, (6) returns the result. A clean dispatcher abstracts these concerns from both the tool implementations and the agent loop.

Tool observability: log all tool calls: tool name, inputs, outputs, execution time, success/failure. This log enables: debugging agent failures, auditing what actions were taken, measuring tool reliability (which tools fail most often?), and improving tool descriptions (which tools are called incorrectly most often?).

from typing import Callable, Any
from pydantic import BaseModel
import json
# Tool registry
class Tool:
    def __init__(self, name: str, description: str, fn: Callable, param_model: type):
        self.name = name; self.description = description; self.fn = fn; self.param_model = param_model
    @property
    def schema(self) -> dict:
        return {'name': self.name, 'description': self.description, 'input_schema': self.param_model.model_json_schema()}
class ToolRegistry:
    def __init__(self): self.tools = {}
    def register(self, tool: Tool): self.tools[tool.name] = tool
    def get_schemas(self) -> list: return [t.schema for t in self.tools.values()]
    def dispatch(self, tool_name: str, tool_input: dict) -> dict:
        if tool_name not in self.tools: return {'error': f'Tool {tool_name} not found'}
        tool = self.tools[tool_name]
        try:
            params = tool.param_model(**tool_input)  # Pydantic validation
            result = tool.fn(**params.model_dump())
            return {'result': result, 'success': True}
        except Exception as e: return {'error': str(e), 'success': False}
# Register tools
class SearchParams(BaseModel): query: str; max_results: int = 5
registry = ToolRegistry()
registry.register(Tool('search_web', 'Search the web for current information', lambda query, max_results: f'Results for {query}', SearchParams))
print(json.dumps(registry.get_schemas(), indent=2))
Why this matters in AI
A well-designed tool layer is what makes agent systems reliable and maintainable. Without a proper tool layer, every agent has ad-hoc tool dispatch, scattered validation, and no consistent error handling. With a tool registry and dispatcher, tool management is centralized: adding a new tool means registering it in one place; changing validation logic updates all tools at once; adding logging or rate limiting to all tools requires one change.
Common mistakes
  • Not returning structured output from tools — a tool that returns 'The weather in London is 15 degrees' forces the model to parse prose to extract the temperature. A tool that returns {'location': 'London', 'temperature': 15, 'unit': 'celsius'} is unambiguous. Always return structured JSON from tools.
  • Not rate-limiting expensive tools — a web search tool with no rate limiting can be called 100 times per agent run at your API cost. Add rate limiting at the tool dispatch level for tools that have external costs.
  • Exposing too many tools at once — giving an agent 20 tools increases the chance of the model calling the wrong one. Present only the tools relevant to the current task. Use a dynamic tool selection mechanism: given the user's intent, which 3-5 tools are most likely needed?

Streaming

Detailed explanation

Streaming in the context of LLM APIs means sending partial responses back to the client token by token as they are generated, rather than waiting for the entire response to complete. Without streaming, a user sending a query to an LLM must wait the full response generation time (10-60 seconds for long responses) before seeing anything. With streaming, the first token appears within milliseconds of inference starting, and the response accumulates in real time.

How LLM streaming works: LLMs generate text autoregressively — one token at a time. Without streaming: generate all N tokens, then send. With streaming: send each token as it is generated using Server-Sent Events (SSE) or WebSockets. The user sees the response build up progressively, which dramatically improves perceived responsiveness even though total generation time is the same.

Streaming with the Anthropic SDK: client.messages.stream() returns a stream context manager. In a with block, iterate over stream.text_stream (yields string chunks) or stream.content_stream (yields typed content blocks). stream.get_final_message() waits for completion and returns the full message object. Use streaming for any response that might be more than a few sentences — it fundamentally changes the user experience.

Streaming in FastAPI (Server-Sent Events): use StreamingResponse with an async generator. The generator yields text chunks as they are produced by the LLM. The client receives a server-sent event for each chunk and can display it immediately. For streaming chat interfaces (like ChatGPT's typing effect), this is the standard implementation.

Streaming for non-LLM cases: streaming is also useful for: model inference that takes > 500ms (stream intermediate results), real-time data processing pipelines (stream prediction results as data arrives), and large file processing (stream output rows to avoid holding all results in memory).

Worked example

import anthropic
client = anthropic.Anthropic()
# Streaming response
with client.messages.stream(
    model='claude-opus-4-8', max_tokens=1024,
    messages=[{'role': 'user', 'content': 'Explain quantum entanglement in simple terms.'}]
) as stream:
    for text in stream.text_stream:
        print(text, end='', flush=True)  # print each chunk as it arrives
    message = stream.get_final_message()
print(f'\nTotal tokens: {message.usage.input_tokens + message.usage.output_tokens}')
# FastAPI streaming endpoint
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
app = FastAPI()
@app.get('/generate')
async def generate(prompt: str):
    async def event_generator():
        with client.messages.stream(
            model='claude-opus-4-8', max_tokens=512,
            messages=[{'role': 'user', 'content': prompt}]
        ) as stream:
            for text in stream.text_stream:
                yield f'data: {text}\n\n'  # SSE format
        yield 'data: [DONE]\n\n'
    return StreamingResponse(event_generator(), media_type='text/event-stream')
Why this matters in AI
Streaming is what makes LLM-powered applications feel responsive. A 5-second response that streams feels instantaneous; a 5-second response without streaming feels broken. For any consumer-facing LLM application, streaming is a requirement, not an optimization.
Common mistakes
  • Buffering the entire stream before rendering — using ''.join([c for c in stream.text_stream]) loads all chunks into memory before rendering. Render each chunk as it arrives.
  • Not handling stream interruption — network timeouts, client disconnections, or API errors can interrupt a stream mid-generation. Implement try/except around the stream and return a graceful error message if the stream is interrupted.
  • Using streaming for short, fast responses — for responses that take < 500ms to generate, the overhead of streaming infrastructure outweighs the benefit. Use streaming only when response generation takes > 1 second.

Caching

Detailed explanation

Caching stores the results of expensive computations so that repeated requests return the cached result instantly, without re-running the computation. In ML systems, caching is used for: LLM responses (same prompt returns same response), embedding computations (same text returns same embedding), preprocessing pipelines (same raw input returns same features), and model predictions (same input returns same prediction). Caching dramatically reduces cost and latency for workloads with repeated inputs.

Cache strategies: (1) exact match caching — cache based on exact input hash. Fast lookup, but two semantically identical queries with different wording are treated as different. (2) Semantic caching — cache based on embedding similarity. 'What is the capital of France?' and 'Which city is France's capital?' return the same cached response. (3) Prefix caching — cache the KV cache for common prompt prefixes (system prompt). Anthropic and OpenAI offer this natively — saves 90% of token costs for a long system prompt reused across many requests.

Redis for distributed caching: Redis is the standard in-memory cache for production systems. redis-py provides Python access: redis_client.get(key) → cached value, redis_client.setex(key, ttl_seconds, value) → set with expiration. TTL (time-to-live): how long a cached value remains valid. For deterministic computations (embeddings), TTL can be very long (days). For dynamic content (LLM responses that depend on current date), TTL should be short (hours).

Cache invalidation: when the underlying data changes (model is updated, documents are reindexed), cached values become stale. Strategies: (1) TTL expiration (automatic, no explicit invalidation needed), (2) key versioning (include model version in cache key — old version's cache entries are naturally orphaned), (3) explicit invalidation (when data changes, delete affected cache entries). The hardest problem in computer science: knowing when to invalidate.

LLM prompt caching: Anthropic's prompt caching feature caches the transformer KV state for a fixed system prompt prefix. On the first request, the full prompt is processed (full token cost). On subsequent requests that reuse the same system prompt prefix, only new tokens are processed (90% cost reduction). Set cache_control: {'type': 'ephemeral'} on the system prompt content block to enable it.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import redis
import hashlib
import json
redis_client = redis.Redis(host='localhost', port=6379, db=0)
def get_embedding_cached(text: str, embed_fn, ttl=86400) -> list:
    # Create cache key from text
    cache_key = f'embed:{hashlib.sha256(text.encode()).hexdigest()[:16]}'
    # Check cache
    cached = redis_client.get(cache_key)
    if cached:
        return json.loads(cached)
    # Compute and cache
    embedding = embed_fn(text)
    redis_client.setex(cache_key, ttl, json.dumps(embedding))
    return embedding
# Anthropic prompt caching
response = client.messages.create(
    model='claude-opus-4-8', max_tokens=1024,
    system=[{
        'type': 'text',
        'text': 'You are a helpful assistant.' + '(very long system prompt...)' * 500,
        'cache_control': {'type': 'ephemeral'}  # cache this prefix
    }],
    messages=[{'role': 'user', 'content': 'What is machine learning?'}]
)
print(f'Cache hit: {response.usage.cache_read_input_tokens} tokens from cache')
print(f'Cache miss: {response.usage.cache_creation_input_tokens} tokens written to cache')
Why this matters in AI
Caching is the highest ROI optimization for LLM applications in production. Prompt caching alone can reduce LLM API costs by 50-90% for applications with repeated system prompts. Embedding caching eliminates redundant computation for static documents. Understanding cache design is essential for cost-efficient production ML systems.
Common mistakes
  • Not including the model version in the cache key — if you update the model, the old cached predictions/embeddings are stale but will still be served. Always include the model version in the cache key.
  • Caching non-deterministic or time-sensitive outputs — caching a response that includes the current date and time, or a response generated with temperature > 0, gives different results each time. Only cache outputs that are deterministic and time-stable.
  • No cache size limit — unbounded caches grow indefinitely, eventually consuming all available memory. Set a max memory limit (Redis: maxmemory 2gb, maxmemory-policy allkeys-lru) and use LRU eviction to remove least-recently-used items when the limit is reached.

Rate limiting

Detailed explanation

Rate limiting controls how many requests an API or service can receive per unit time, protecting it from overload and ensuring fair usage. In AI API contexts, you encounter rate limits as a consumer (OpenAI limits your requests per minute) and must implement rate limiting as a provider (your FastAPI app limits how often users can call your LLM endpoint).

Common rate limit types: requests per minute (RPM) — total API calls per minute. Tokens per minute (TPM) — total tokens (input + output) consumed per minute, relevant for LLM APIs where a single request might consume 2,000 tokens. Requests per day — daily budget, often for free tiers. Concurrent requests — how many simultaneous requests you can have in flight.

Handling rate limits as a consumer: exponential backoff — when you receive a 429 (Too Many Requests), wait 2^n seconds before retry, where n is the retry attempt number. Add jitter (random noise) to prevent thundering herd: `wait = (2 ** attempt + random.uniform(0, 1))`. The `tenacity` library provides retry decorators. Token budgeting: track your token usage per minute and proactively slow down before hitting the limit.

Implementing rate limiting as a provider: token bucket algorithm — tokens refill at a constant rate, each request consumes tokens, requests are rejected when the bucket is empty. Sliding window counter — count requests in the last N seconds using a Redis sorted set. Fixed window counter — count requests in the current minute, reset at the minute boundary. Token bucket is most common for LLM rate limiting.

Practical tools: for FastAPI applications, `slowapi` adds rate limiting middleware based on IP address, user ID, or API key. For Redis-backed rate limiting, `redis-py` with a Lua script implements atomic token bucket. For distributed rate limiting (multiple servers), always use a shared store (Redis) rather than in-memory counters.

# Handling API rate limits as consumer
import time
import random
def call_api_with_backoff(fn, max_retries=5, *args, **kwargs):
    for attempt in range(max_retries):
        try:
            return fn(*args, **kwargs)
        except Exception as e:
            if '429' in str(e) or 'rate_limit' in str(e).lower():
                wait = (2 ** attempt) + random.uniform(0, 1)  # exponential backoff + jitter
                print(f'Rate limited. Retry {attempt+1}/{max_retries} in {wait:.1f}s...')
                time.sleep(wait)
            else:
                raise  # non-rate-limit error: don't retry
    raise RuntimeError(f'Max retries ({max_retries}) exceeded')

Why this matters in AI
Rate limiting is the defense against two common problems: being blocked by third-party APIs (when you hit their rate limits, your pipeline fails), and being overwhelmed by your own users (when your LLM endpoint gets traffic spikes, costs explode and quality degrades for everyone). Building rate-limit-aware clients and rate-limited APIs is essential for production AI applications.
Common mistakes
  • Not implementing jitter in retry backoff — if 100 requests all hit the rate limit simultaneously and all wait exactly 2 seconds, they all retry simultaneously and hit the limit again. Jitter (random delay) spreads the retries over a time window.
  • Using in-memory rate limiting in a multi-process deployment — if your API runs 4 workers, each worker tracks its own counter. A user can make 40 requests/minute (10 per worker) while appearing to have made only 10 per worker. Use Redis for shared rate limiting across all workers.
  • Not communicating rate limit metadata to API consumers — include headers: `Retry-After`, `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`. Well-designed APIs let clients proactively avoid hitting limits rather than discovering them through 429 errors.

Observability

Detailed explanation

Observability in ML systems means having visibility into what the system is doing at every layer: data, models, and infrastructure. The three pillars of observability are: metrics (numerical measurements over time), logs (structured event records), and traces (request journeys across services). Without observability, ML system failures are invisible until they become catastrophic.

ML-specific metrics: prediction distribution (histogram of model output scores — shifts indicate concept drift or data quality issues), input feature distribution (monitor for training-serving skew — if production features differ from training features, model performance degrades), request latency (p50, p95, p99), error rate, throughput (requests/second), and business metrics (conversion rate, click-through rate — the ultimate measure of model value).

Concept drift detection: production model performance degrades when the data distribution changes. Monitor: prediction drift (KL divergence between training and production output distributions), feature drift (PSI — Population Stability Index — measures how much feature distributions have shifted), label drift (if you have delayed labels). Evidentlyai and Whylogs provide drift detection dashboards.

Structured logging for ML: log every prediction with: request_id, timestamp, model_version, input features (or input hash), prediction, confidence score, latency. This enables post-hoc debugging (why did the model predict X for this user?), drift detection, and error analysis. Use correlation IDs to trace a request through preprocessing, model inference, and postprocessing.

Alerting: set up alerts for: accuracy drop > 5% (measured on labeled feedback), prediction distribution shift (PSI > 0.2), p99 latency > SLA, error rate > 1%. Use Grafana, Prometheus, or CloudWatch for ML monitoring dashboards. Alert on trends (gradual degradation) not just thresholds (sudden failures).

Evals

Detailed explanation

Evals (evaluations) are systematic tests that measure whether an AI system behaves as intended. For LLM-based applications, automated tests replace traditional unit tests — instead of testing exact outputs, evals measure properties like accuracy, tone, safety, format compliance, and task completion. Building a comprehensive eval suite is the foundational practice of responsible LLM deployment.

Types of evals: (1) exact match — does the output exactly match the expected string? (low utility for LLMs, only for structured outputs), (2) reference-free — does the output have the right format/length/language? (regex, length checks), (3) model-based — use a stronger LLM to judge the output quality (LLM-as-a-judge), (4) human — human annotators rate outputs on quality, safety, and helpfulness, (5) task-specific — for code generation: does the code execute and pass tests? For summarization: ROUGE score vs reference summary.

LLM-as-a-judge: use a capable model (GPT-4, Claude) to evaluate another model's outputs. Provide the judge with a rubric: 'Rate this response 1-5 on helpfulness (does it directly answer the question?), accuracy (are facts correct?), and safety (does it avoid harmful content?)'. Return structured JSON. Meta-issue: the judge model may have biases — validate judge ratings against human ratings on a calibration set.

Eval dataset design: representative coverage (include all task types and difficulty levels), adversarial cases (edge cases, tricky inputs, jailbreak attempts), regression tests (cases where previous versions failed — if fixed, track that they stay fixed), and held-out test set (never use for prompt tuning, only for final evaluation). A good eval set for an LLM application has 100-500 examples covering the full range of use cases.

Running evals in CI/CD: run the eval suite automatically on every code change (prompt updates, model version changes, system prompt changes). Set quality gates: if eval scores drop > 5%, block the change. This prevents regressions and makes prompt engineering systematic rather than informal. Use frameworks like promptfoo, LangSmith, or Braintrust for automated eval pipelines.

Why this matters in AI
Evals are what makes LLM application development rigorous rather than intuitive. Without evals, prompt changes are guesswork — you change the prompt and hope it gets better. With evals, you measure before and after: 'changing from 300 to 500-word system prompt improved task completion from 78% to 89%, with no regression in safety'. Evals make AI engineering a discipline.

Cost optimization

Detailed explanation

Cost optimization for LLM applications reduces API costs while maintaining quality. LLM API costs can grow linearly with usage — a feature that costs $100/day at 1,000 users costs $10,000/day at 100,000 users. Strategic optimization can reduce costs by 60-90% without significant quality degradation.

Model routing (cascade): use a cheap, fast model (Claude Haiku) for simple requests and route only complex requests to the expensive model (Claude Opus). Classification: before calling the LLM, use a fast classifier to estimate request complexity. Or use a two-stage approach: try Haiku first, if it returns 'I'm not sure' or a low-confidence response, retry with Opus.

Prompt compression: system prompts that grew organically often contain redundant instructions. Audit your system prompt: is each sentence doing work? Removing 500 tokens from a system prompt that's called 1M times/day saves 500M input tokens = $7,500/day at Claude Opus pricing. Use LLMLingua (Microsoft) for automated prompt compression.

Caching: identical or near-identical queries should return cached results rather than making a new API call. Exact caching: hash the full prompt and cache the response. Semantic caching: embed the query and retrieve cached responses for similar queries (using a vector database). Anthropic's prompt caching feature reduces the cost of repeated system prompts by 90%.

Batching and async: for non-real-time workloads (offline analysis, batch classification), use the messages batching API (50% cost reduction). For real-time workloads, use async requests to maximize throughput without parallel API instances. Reduce output tokens by instructing the model to be concise: 'Respond in maximum 2 sentences.'

Security design

Detailed explanation

Security design for AI systems addresses the unique threats that emerge when ML models and LLMs are deployed in production: adversarial attacks on models, data poisoning, model stealing, and the agent-specific threats of prompt injection and tool misuse. Standard web application security (OWASP Top 10) applies, plus AI-specific threats.

OWASP LLM Top 10: the OWASP working group maintains a list of the most critical LLM security risks: LLM01 — Prompt Injection (untrusted input hijacks LLM behavior), LLM02 — Insecure Output Handling (LLM output used in security-sensitive contexts without sanitization), LLM03 — Training Data Poisoning, LLM04 — Model Denial of Service, LLM05 — Supply Chain Vulnerabilities (third-party model or dataset risks).

Adversarial inputs for ML models: carefully crafted inputs that cause ML models to misclassify. For image classifiers: imperceptible pixel perturbations that cause a model to classify a cat as a dog with high confidence. For text classifiers: adding specific tokens that flip the classification. Defenses: adversarial training (include adversarial examples in training), input pre-processing (smoothing, denoising), and ensemble approaches.

API security for LLM services: authentication (API keys, OAuth), authorization (which users can access which models and features), input validation (maximum input length, content type checking), rate limiting (per user, per key), output sanitization (HTML encoding for web display, SQL parameterization for database queries from LLM output).

Secure system prompt practices: the system prompt is the most powerful security control in an LLM system. It defines the model's role and constraints. Protect it: don't expose it in error messages, don't include credentials in it (use environment variables), test against prompt injection, and update it when new attack patterns emerge.

# Secure LLM API wrapper
import anthropic
import hashlib
import time
from typing import Optional
client = anthropic.Anthropic()
# Input validation
def validate_user_input(text: str, max_length: int = 10000) -> tuple[bool, str]:
    if not text or not text.strip(): return False, 'Empty input'
    if len(text) > max_length: return False, f'Input too long (max {max_length} chars)'
    # Check for obvious injection patterns
    injection_patterns = ['ignore previous', 'ignore all', 'new instructions', 'system:', 'assistant:']
    text_lower = text.lower()
    for pattern in injection_patterns:
        if pattern in text_lower: return False, f'Potential injection detected: {pattern}'
    return True, 'OK'
# Rate limiting tracker
request_log = {}  # {user_id: [(timestamp, tokens), ...]}
def check_rate_limit(user_id: str, max_rpm: int = 10) -> bool:
    now = time.time(); cutoff = now - 60
    user_requests = [t for t in request_log.get(user_id, []) if t > cutoff]
    request_log[user_id] = user_requests + [now]
    return len(user_requests) < max_rpm
# Secure call
def secure_llm_call(user_id: str, user_input: str) -> Optional[str]:
    valid, reason = validate_user_input(user_input)
    if not valid: return f'Invalid input: {reason}'
    if not check_rate_limit(user_id): return 'Rate limit exceeded. Please try again later.'
    response = client.messages.create(model='claude-opus-4-8', max_tokens=1024, messages=[{'role': 'user', 'content': user_input}])
    return response.content[0].text

Understanding OWASP LLM Top 10 and implementing defense-in-depth protects your AI application and the users who trust it.
