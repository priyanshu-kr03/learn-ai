Features and labels

Detailed explanation

In supervised ML, features (inputs, predictors, X) are the observable attributes used to make predictions. Labels (targets, outputs, y) are what we predict. Example: predicting house prices — features = [size_sqft, num_bedrooms, location, age], label = price. The model learns f(X) → y.

Feature types: numeric continuous (income, temperature), categorical (city, color), ordinal (rating 1–5, education level), boolean (has_internet), text (review, description), datetime (timestamps), complex (images, audio). Each type requires different preprocessing.

Label types: binary (fraud/not-fraud: 0 or 1), multi-class (which of K classes: digit 0–9), multi-label (multiple tags: a news article as politics AND economy), regression (continuous: house price), sequence (translation output). The label type determines the model's output layer and loss function.

Feature matrix X: in sklearn convention, X is a 2D array of shape (n_samples, n_features). Each row is one sample, each column is one feature. y is 1D of shape (n_samples,). Access: X[i] is the feature vector for sample i, X[:, j] is all values of feature j.

Feature engineering: raw features are rarely the most useful form. Engineer better representations: extract hour-of-day from timestamp, compute age from birthdate, create interaction features (income / debt = debt_income_ratio), discretize into buckets (age → age_group). Good features make models significantly better — often more than changing the algorithm.

X['price_per_sqft'] = X['size_sqft'] / X['num_rooms']

Why this matters in AI
The distinction between features and labels is the foundation of supervised learning. Feature quality is often the most important factor in model performance — better features matter more than better models. Understanding what makes a good feature is one of the most valuable ML skills.
Common mistakes
  • Including the label as a feature — leaving 'price' in X causes perfect training accuracy but completely invalid predictions (the model just reads the answer).
  • Treating all columns as features without domain analysis — IDs, raw timestamps, and irrelevant columns add noise without signal.
  • Not analyzing feature distributions before training — missing values, outliers, and wrong dtypes in features cause silent degraded model performance.

Supervised learning

Detailed explanation

Supervised learning is the most common ML paradigm: train a model using labeled examples (input-output pairs), then use it to predict outputs for new unseen inputs. The 'supervision' comes from the labels — the correct answers guide training. During training, the model's predictions are compared to the true labels, and parameters are updated to reduce the error. Examples: predict house price from features (regression), classify email as spam or not-spam (classification).

Training in supervised learning: the model sees (X_train, y_train) where X_train are features and y_train are labels. It learns a function f: X → y that minimizes a loss function L(f(x), y). For regression, the loss is typically mean squared error (MSE). For binary classification, it is binary cross-entropy. The optimization algorithm (gradient descent for neural networks, closed-form solution for linear regression) minimizes the loss.

Generalization: the goal is not to minimize training loss but to minimize test loss (on unseen data). A model that perfectly memorizes training data (training loss = 0) but fails on test data has overfit. Cross-validation provides an unbiased estimate of test performance without using the test set. The test set is reserved for the final evaluation only.

Supervised learning requires labeled data — often the most expensive part of an ML project. Collecting labels requires domain expertise (radiologists to label X-rays, lawyers to label legal documents) or user behavior (clicks, purchases, ratings). The quality and quantity of labels determine the ceiling of model performance.

Types of supervised learning: binary classification (is this fraud?), multi-class classification (which of 1000 product categories?), multi-label classification (a document can have multiple tags), regression (predict a continuous value), ordinal regression (predict a rank), structured prediction (predict a sequence, tree, or graph — machine translation, dependency parsing).

Why this matters in AI
Supervised learning is the workhorse of applied ML. Most high-value business problems — fraud detection, churn prediction, demand forecasting, recommendation — are supervised learning tasks. Understanding the full workflow (data → features → model → evaluation → deployment) is the foundation of ML practice.
Common mistakes
  • Evaluating on training data — training accuracy is always higher than test accuracy due to overfitting. Always hold out validation and test sets.
  • Starting with the most complex model — start with a simple baseline (logistic regression, linear regression). Then add complexity only if needed.
  • Ignoring label quality — garbage labels → garbage model. Spend time validating label quality before training.

Unsupervised learning

Detailed explanation

Unsupervised learning finds patterns in data without labeled examples. The model discovers structure — clusters, components, associations, anomalies — purely from the input features. No supervision: there is no 'correct answer' to guide training. Applications: customer segmentation (discover natural customer groups), dimensionality reduction (PCA, UMAP), anomaly detection, topic modeling, and feature learning (autoencoders).

Clustering: partition data into groups (clusters) where items within a group are more similar to each other than to items in other groups. K-Means assigns each point to the nearest centroid and iterates. DBSCAN finds dense regions. Hierarchical clustering builds a tree of nested clusters. The challenge: evaluation is hard without ground truth labels. Use silhouette score, Davies-Bouldin index, or domain expert judgment.

Dimensionality reduction: find a lower-dimensional representation of high-dimensional data. PCA (linear) finds directions of maximum variance. UMAP and t-SNE (non-linear) preserve local neighborhood structure for visualization. Autoencoders (neural networks) learn a compressed latent representation. Use cases: visualization (reduce to 2–3 dimensions for plotting), preprocessing (reduce noise and dimensions before supervised learning), and feature learning.

Association rule mining: find items that frequently occur together. Apriori algorithm finds rules like 'customers who buy bread and butter also buy jam' (support, confidence, lift metrics). Used in market basket analysis, recommender systems, and clickstream analysis. Tools: mlxtend library in Python.

Evaluation of unsupervised learning: without labels, evaluation is indirect. For clustering: (a) intrinsic metrics (silhouette score — how well each point fits its cluster vs neighboring clusters, +1 is perfect, -1 is wrong cluster), (b) if labels exist for comparison: adjusted rand index (ARI) compares predicted cluster assignments to true labels. For dimensionality reduction: reconstruction error (how well the original data can be recovered from the low-d representation).

Why this matters in AI
Unsupervised learning is used when labels are unavailable (most of the world's data is unlabeled). It discovers hidden structure: customer segments that can be targeted differently, anomalies that flag fraud, and latent features that improve downstream supervised models.
Common mistakes
  • Expecting unsupervised learning to give 'the right answer' — there is no single correct clustering. Different algorithms and parameters give different results. Interpret results with domain knowledge.
  • Using K-Means on data with very non-spherical clusters — K-Means assumes spherical clusters. Use DBSCAN or Gaussian mixture models for irregular shapes.
  • Not scaling before K-Means or PCA — both are distance/variance based and are dominated by features with large scales.

Semi-supervised learning

Detailed explanation

Semi-supervised learning (SSL) trains a model using a small amount of labeled data and a large amount of unlabeled data. This is practically important because labeled data is expensive and time-consuming to collect (requires domain expertise), while unlabeled data is abundant and cheap. SSL sits between supervised learning (all data labeled) and unsupervised learning (no data labeled).

Core assumption: the structure of the unlabeled data provides useful information about the decision boundary. If unlabeled data points cluster together, a natural decision boundary runs through the low-density region between clusters. SSL algorithms exploit this cluster assumption, manifold assumption (data lies on a low-dimensional manifold), or smoothness assumption (nearby points likely have the same label).

Common SSL approaches: (1) self-training — train a model on labeled data, predict on unlabeled data, add high-confidence predictions to the labeled set, retrain. Iterates until convergence. (2) Label propagation — propagate labels from labeled to unlabeled points based on graph connectivity (connected points share labels). (3) Pseudo-labeling — similar to self-training; used in deep learning (FixMatch, UDA). (4) Co-training — train two models on different feature views; each model labels examples for the other.

Self-supervised learning (contrast with SSL): in self-supervised learning, the model creates its own labels from the data structure — no human labels needed. Examples: BERT's masked language modeling (predict masked words from context — the mask position is the 'label'), SimCLR and contrastive learning (augmented views of the same image should have similar representations). Self-supervised learning pre-trains powerful representations that are then fine-tuned on small labeled datasets.

When to use SSL: when you have a small labeled set (100–1000 examples) but large unlabeled set (10,000–1M examples). Examples: medical imaging (few labeled scans, many unlabeled), NLP (few labeled sentences, large corpus), web content moderation (few manually reviewed examples, vast unlabeled content). SSL is a key technique for reducing labeling costs in production ML systems.

# Self-training with sklearn
from sklearn.semi_supervised import SelfTrainingClassifier
from sklearn.ensemble import RandomForestClassifier
# -1 labels mean unlabeled
y_mixed = y.copy()  # y_mixed[unlabeled_indices] = -1
base = RandomForestClassifier(n_estimators=100)

self_training.fit(X, y_mixed)
print(f'Val acc: {self_training.score(X_val, y_val):.4f}')
# Label propagation

lp.fit(X, y_mixed)
print(lp.score(X_val, y_val))
Why this matters in AI
Semi-supervised learning is essential in production ML where labeled data is scarce and expensive. BERT, GPT, and most modern LLMs are trained with self-supervised learning on massive unlabeled text corpora, then fine-tuned on small labeled datasets. SSL is why LLMs can be adapted to specialized tasks with just a few hundred examples.
Common mistakes
  • Using SSL when the labeled data is insufficient to capture the class structure — self-training propagates errors. If the initial model is very inaccurate, high-confidence wrong predictions pollute the training data.
  • Setting the confidence threshold too low in self-training — using predictions with 60% confidence as training labels introduces many wrong labels. Use threshold >= 0.9.
  • Confusing self-supervised learning with semi-supervised learning — self-supervised creates labels from data structure (BERT's masking), semi-supervised uses a small set of human labels plus unlabeled data.

Reinforcement learning overview

Detailed explanation

Reinforcement Learning (RL) is a learning paradigm where an agent learns by interacting with an environment to maximize cumulative reward. Unlike supervised learning (learn from labeled examples) and unsupervised learning (find patterns in unlabeled data), RL learns from trial and error through experience. The agent takes actions, observes the resulting state and reward, and updates its policy to collect more reward in the future.

Core RL loop: (1) agent observes current state s, (2) agent selects action a based on its policy π(a|s), (3) environment transitions to new state s' and emits reward r, (4) agent updates its policy based on (s, a, r, s'). This loop repeats for thousands or millions of steps. The agent starts with a random policy and gradually learns which actions lead to high cumulative reward.

Key RL applications: game playing (AlphaGo, Atari — agent learns to play by reward = game score), robotics (agent learns to walk, grasp objects — reward = distance traveled, object grasped), dialog systems (agent learns to conduct conversations — reward from user ratings), recommendation systems (reward = user engagement), autonomous driving (reward = reaching destination safely), and RLHF (Reinforcement Learning from Human Feedback — used to train GPT-4, Claude: reward = human preference ratings).

RL vs supervised learning: in supervised learning, the correct action (label) is given for every input. In RL, there is no correct action — only a delayed, sparse reward signal. The agent must infer from experience which actions contributed to the reward. This credit assignment problem is what makes RL hard. The reward may come many steps after the action that caused it (sparse reward).

Model-free vs model-based RL: model-free RL learns directly from experience (Q-learning, policy gradients) — it does not build an explicit model of the environment. Model-based RL builds a model of the environment and plans ahead using it (AlphaGo's tree search uses a learned model). Model-based is more sample-efficient but requires an accurate model. Model-free is more general but requires more environment interactions.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import gym
import numpy as np
# CartPole: balance a pole on a cart
env = gym.make('CartPole-v1')
obs, _ = env.reset()
for _ in range(200):
    action = env.action_space.sample()  # random policy
    obs, reward, done, truncated, info = env.step(action)
    if done: break
print('Random policy survived', _, 'steps')
# Q-learning on FrozenLake
env = gym.make('FrozenLake-v1', is_slippery=False)
Q = np.zeros((16, 4))  # 16 states, 4 actions
for ep in range(5000):
    s, _ = env.reset()
    for _ in range(100):
        a = np.random.randint(4) if np.random.rand() < 0.1 else Q[s].argmax()
        s2, r, done, _, _ = env.step(a)
        Q[s,a] += 0.1*(r + 0.99*Q[s2].max() - Q[s,a])
        s = s2
        if done: break
print(f'Learned Q-table:\n{Q.reshape(4,4,4).mean(axis=2).round(2)}')
Why this matters in AI
Reinforcement learning is the training paradigm behind the most impressive AI achievements: AlphaGo defeating world chess champions, robotics arms learning to grasp, and RLHF making ChatGPT helpful and safe. Understanding RL opens the door to building agents that learn from interaction — the next frontier beyond classification and regression.
Common mistakes
  • Using RL when supervised learning suffices — RL is much harder to train (sparse rewards, high variance, instability) than supervised learning. If you have labeled examples of correct behavior, use supervised learning (imitation learning or behavioral cloning) instead.
  • Setting a reward that causes unintended behavior — reward hacking: the agent maximizes the reward in unexpected ways. Example: a boat racing agent learns to spin in circles collecting reward tokens rather than finishing the race. Carefully design rewards to align with the actual objective.
  • Not using vectorized environments — RL training requires millions of environment steps. Running 8 environments in parallel (gym's SubprocVecEnv or stable-baselines3 VecEnv) gives an 8× speedup with no additional complexity.

Classification

Detailed explanation

Classification is a supervised learning task where the output (label) is a discrete category. Examples: spam detection (spam/not-spam), medical diagnosis (disease A / disease B / healthy), handwriting recognition (digit 0–9), sentiment analysis (positive/negative/neutral). The model assigns inputs to one of K predefined categories. Binary (K=2) and multi-class (K>2) are the main variants.

Classification algorithms: logistic regression (linear decision boundary, outputs probabilities), decision tree (piecewise constant, threshold-based rules), random forest (ensemble of trees, robust and accurate), gradient boosting XGBoost/LightGBM (state of the art for tabular data), SVM (maximum margin), naive Bayes (probabilistic, excellent for text), neural networks (images, text, complex patterns).

Classification output: models typically output class probabilities P(y=k | x) for each class k, summing to 1. The predicted class is argmax(probabilities). The 0.5 threshold for binary classification can be tuned — a fraud detector might use 0.3 to flag more potential fraud (higher recall at lower precision).

Classification metrics: accuracy (misleading for imbalanced classes), precision (of all predicted positives, what fraction were actually positive?), recall (of all actual positives, what fraction were detected?), F1 (harmonic mean of precision and recall), AUC-ROC (discriminative ability regardless of threshold), AUC-PR (better for imbalanced problems).

Multi-class strategies: one-vs-rest (OvR) trains K binary classifiers (each class vs all others). One-vs-one (OvO) trains K(K-1)/2 binary classifiers (each pair). These are used for algorithms that support only binary classification. Most modern algorithms (decision trees, random forest, XGBoost) natively support multi-class.

Why this matters in AI
Classification is the most common ML task in industry — spam filtering, fraud detection, content moderation, medical diagnosis, churn prediction. Understanding classification thoroughly — from algorithm selection to threshold tuning to evaluation — is the foundation of applied ML.
Common mistakes
  • Using accuracy as the sole metric for imbalanced classification — 95% accuracy sounds great until you realize the model just predicts 'negative' for everything (on a 95% negative dataset).
  • Using default 0.5 threshold without tuning — the optimal threshold depends on business cost of false positives vs false negatives. Plot the precision-recall curve.
  • Evaluating with predict() rather than predict_proba() — AUC-ROC requires probability scores, not hard predictions.

Regression

Detailed explanation

Regression is a supervised learning task where the output is a continuous numeric value. Examples: predicting house prices ($450,000), demand forecasting (2,340 units), temperature forecast (23.5°C). Unlike classification (discrete categories), regression predicts a value on a continuous scale.

Regression algorithms: linear regression (linear relationship), polynomial regression (non-linear via polynomial features), decision tree regression (piecewise constant), random forest regression (ensemble), gradient boosting XGBoost/LightGBM (state of the art for tabular regression), neural networks (complex non-linear), SVR (support vector regression).

Linear regression: y = β₀ + β₁x₁ + β₂x₂ + ... + βₙxₙ + ε. Coefficients fit by minimizing MSE. OLS closed-form: β = (X^T X)^{-1} X^T y. For correlated features (multicollinearity), OLS is unstable — use Ridge (L2 regularization) or Lasso (L1, which zeros some coefficients for feature selection).

Regression metrics: MAE (mean absolute error — interpretable, robust to outliers), MSE (penalizes large errors heavily, differentiable), RMSE (same units as target, sensitive to outliers), R² (proportion of variance explained: 1 is perfect, 0 means predicting the mean), MAPE (percentage error — breaks when target is near 0).

Residual analysis: plot residuals (y_actual - y_predicted) vs predicted values. Good model: residuals randomly scattered around 0. Patterns indicate: systematic bias (trend in residuals), heteroscedasticity (residuals fan out — log-transform the target), non-linearity (U-shaped pattern — add polynomial features).

Why this matters in AI
Regression is fundamental to ML — price prediction, demand forecasting, click-through rate prediction, and many other business-critical applications are regression problems. Understanding regression deeply, including assumptions and diagnostic plots, leads to more reliable models.
Common mistakes
  • Using MSE as the evaluation metric when the target has outliers — MSE penalizes outliers heavily, making the model seem worse than it is. Use MAE or evaluate both.
  • Not checking residuals — a model with good RMSE can still have systematic biases that make it unreliable for subgroups.
  • Not transforming skewed targets — house prices have right-skewed distribution. Log-transforming (predict log(price)) often gives better RMSE and symmetric residuals. Exponentiate predictions: np.exp(y_pred_log).

Clustering

Detailed explanation

Clustering is an unsupervised learning task that groups similar data points together without predefined labels. Unlike classification (known categories), clustering discovers the natural group structure in data. Applications: customer segmentation (discover behavioral groups), document clustering (group similar articles), image compression (group similar colors), anomaly detection (outliers don't fit any cluster), gene expression analysis.

K-Means: the most common clustering algorithm. Partition n points into K clusters by iteratively: (1) assigning each point to the nearest centroid, (2) recomputing centroids as the mean of assigned points. Repeats until assignments stabilize. Requires K to be specified in advance. Assumes spherical clusters of similar size — fails on non-spherical or very different sized clusters.

Cluster evaluation without labels: silhouette score (measures how well each point fits its cluster vs neighbors — higher is better, +1 ideal, -1 wrong cluster), elbow method (plot WCSS — within-cluster sum of squares — vs K, look for the 'elbow' where WCSS decreases more slowly), Davies-Bouldin index (lower is better), Calinski-Harabasz index (higher is better).

Choosing K: run K-Means for K=2..20 and plot silhouette score for each K. Choose K with the highest silhouette score. Additionally, domain knowledge constrains K — if you know there are 5 product categories, K=5 is natural. The elbow method and silhouette method often agree on the best K.

Beyond K-Means: DBSCAN (density-based, finds non-spherical clusters, handles noise), hierarchical clustering (builds a dendrogram of nested clusters, does not require K in advance), Gaussian Mixture Models (soft clustering — each point has a probability of belonging to each cluster), BIRCH (scalable hierarchical for large datasets), HDBSCAN (robust density-based, handles varying density).

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from sklearn.cluster import KMeans
from sklearn.metrics import silhouette_score
scores = []
for k in range(2, 15):
    km = KMeans(n_clusters=k, random_state=42, n_init=10)
    labels = km.fit_predict(X_scaled)
    scores.append(silhouette_score(X_scaled, labels))
best_k = np.argmax(scores) + 2
print(f'Best K: {best_k}')
km = KMeans(n_clusters=best_k, random_state=42, n_init=10)
labels = km.fit_predict(X_scaled)
centroids = km.cluster_centers_
plt.scatter(X_2d[:,0], X_2d[:,1], c=labels, cmap='tab10', alpha=0.7)
plt.scatter(centroids[:,0], centroids[:,1], marker='X', s=200, c='red')
Why this matters in AI
Clustering is the entry point to unsupervised learning. Customer segmentation alone drives billions of dollars in targeted marketing spend. Understanding clustering — including when K-Means fails and which alternatives to use — is essential for any ML practitioner working with unlabeled data.
Common mistakes
  • Not scaling features before K-Means — K-Means is distance-based. Features with large scales (income) dominate the clustering. Always scale first.
  • Using K-Means on non-spherical clusters — K-Means fails on crescent or ring shapes. Use DBSCAN or Gaussian mixture models.
  • Interpreting K-Means cluster numbers as meaningful — cluster labels (0, 1, 2) are arbitrary and change across runs. Interpret clusters by their centroid feature values, not their number.

Recommendation problem

Detailed explanation

A recommendation system predicts what items a user will like or find useful, given their history and context. Examples: Netflix recommends movies, Amazon recommends products, Spotify recommends songs, LinkedIn recommends jobs. The core challenge: given millions of users and millions of items, efficiently find the few items each user will like. Recommendations are a critical business driver — Netflix attributes 80% of watched content to recommendations.

Types of recommendation: (1) collaborative filtering — based on similar users ('users who liked these items also liked...'). (2) Content-based filtering — based on item features ('you liked action movies, here are more action movies'). (3) Matrix factorization — decomposes the user-item interaction matrix into user and item embedding matrices, capturing latent factors. (4) Two-tower models — separate neural networks for user and item, dot product for relevance score. (5) Large Language Model recommendations — use conversation history and item descriptions.

Implicit vs explicit feedback: explicit feedback (ratings 1–5) is scarce — users rarely rate items. Implicit feedback (clicks, views, purchases, listens) is abundant but noisy — a click doesn't mean the user liked the item. Most production systems use implicit feedback. Positive signals: purchase, long watch time, repeat interaction. Negative signals: skipped, short watch time, explicit 'not interested' feedback.

The cold start problem: new users have no history (user cold start) — cannot use collaborative filtering. New items have no interactions (item cold start) — cannot use CF. Solutions: for new users, use demographic features, show popular items, ask onboarding questions. For new items, use content-based features (description, category, tags). After a few interactions, switch to collaborative filtering.

Two-stage pipeline: retrieval (candidate generation) → ranking. Retrieval: efficiently retrieve 100–1000 candidate items from millions using approximate nearest neighbor search (FAISS on user/item embeddings). Ranking: score the 100–1000 candidates with a more complex model (features + LightGBM or neural network) to select the top 10–20. This two-stage approach balances speed (retrieval is fast) with accuracy (ranking is accurate).

Why this matters in AI
Recommendation systems are the highest-revenue ML application — responsible for driving purchases, engagement, and retention across every major platform. Building a recommendation project demonstrates: matrix factorization, embedding models, evaluation with offline and online metrics, and the cold start problem.
Common mistakes
  • Evaluating with RMSE on explicit ratings only — most real recommendation systems use implicit feedback. RMSE on explicit ratings measures prediction accuracy, not recommendation quality. Use ranking metrics: precision@K, recall@K, NDCG@K.
  • Not handling the popularity bias — collaborative filtering recommends popular items more frequently (popular items have more interactions). This creates a popularity feedback loop and disadvantages long-tail items. Use popularity debiasing in training.
  • Evaluating offline without considering temporal order — splitting train/test randomly means the model can use future interactions to predict past ones. Always use chronological splits for recommendation evaluation.

Train validation test split

Detailed explanation

Data must be split into three non-overlapping sets. Training set: used to fit model parameters (weights, tree structure). Validation set: used to tune hyperparameters, select features, and make early stopping decisions — the model never trains on this data but it influences model selection. Test set: used ONCE for the final honest performance estimate — the model never sees this during development. If you evaluate on the test set multiple times, it becomes a de facto validation set.

Why three sets? Hyperparameter tuning on the validation set optimistically selects the configuration that happened to work best on that particular validation sample. The test set provides an independent, unbiased estimate of how the final model generalizes. With only train/test: you'd need to tune hyperparameters on the test set (leakage) or have no unbiased final estimate.

Typical split ratios: 60/20/20 for medium datasets (1000–100K samples). 80/10/10 for large datasets (100K+ samples). For small datasets (< 1000 samples), use k-fold cross-validation instead of a held-out validation set — k-fold gives more reliable estimates with less data wasted. The test set should be at least 1000 samples for reliable performance estimates.

Stratified splitting: for classification with imbalanced classes, use stratify=y in train_test_split to ensure each split has the same class ratio as the full dataset. Without stratification, a 95%/5% class split may place almost all minority class examples in one split by chance, giving misleading evaluation.

Nested cross-validation for hyperparameter search: outer loop estimates model performance (k-fold CV), inner loop selects hyperparameters (another k-fold CV on each outer training fold). This gives an unbiased estimate of the best-hyperparameter-selected model's performance, without test set contamination. Computationally expensive but the most statistically sound approach.

Why this matters in AI
The train/validation/test split is the foundational experimental design for all of applied ML. Violating it — using the test set for hyperparameter selection, preprocessing before splitting, or evaluating multiple times — invalidates the performance estimate and creates models that fail in production.
Common mistakes
  • Using the test set more than once — every time you evaluate on the test set and adjust the model based on the results, you are using the test set for model selection. Reserve it for the single final evaluation.
  • Splitting before preprocessing — StandardScaler().fit_transform(X) before split leaks. Always split first, then fit all preprocessing on X_train only.
  • Using random split for time series — chronological order must be preserved. Split so that training data is earlier than validation which is earlier than test data.

Data leakage

Detailed explanation

Data leakage occurs when information that would not be available at prediction time is used during model training, causing the model to appear artificially better than it actually is in production. Leakage is insidious because it does not cause errors — it causes models that look great during development but fail in production. It is one of the most common and costly mistakes in applied ML.

Target leakage: using features that are derived from or correlated with the target, in a way that would not be available at prediction time. Example: predicting loan default using 'account_closed' (which happens after default). The feature reveals the label. Another example: predicting hospital readmission using 'discharge_summary' (written after the outcome is known). These features perfectly predict the target but are unavailable when the model is deployed.

Preprocessing leakage: fitting preprocessing steps (StandardScaler, PCA, TF-IDF) on the full dataset (train + test) before splitting. Example: StandardScaler().fit_transform(X) before train_test_split leaks test set statistics (mean, std) into training. The scaler uses information from the test set to preprocess the training set. Fix: use sklearn Pipelines — they automatically fit only on training data.

Time leakage: using future information to predict past events when using time-series data. Example: in a churn prediction model with a time-based split (train on users from 2023, test on users from 2024), using features computed from the full history (including 2024 data) for the 2023 training set. Fix: use a point-in-time feature computation — features for each user are computed only from data available at that user's training time.

Detection: the most reliable signal of leakage is suspiciously high performance (AUC > 0.99 on a hard problem), large feature importance for counterintuitive features, and significant performance degradation after deployment. To find leakage: for each feature, ask 'is this feature available at prediction time for a new unseen user/sample?' If not, exclude it. Time-ordered splits reveal temporal leakage that random splits mask.

Why this matters in AI
Data leakage is the single most common cause of ML models that work great in development but fail in production. Detecting and preventing leakage is the difference between a model that is genuinely predictive and one that is gaming the evaluation.
Common mistakes
  • Random train/test split on time-series data — this leaks future information to the training set. Always use time-based splits for temporal data.
  • Fitting imputers, scalers, or encoders on the full dataset — use Pipelines or manually fit on training data only.
  • Using 'post-event' features — features that are recorded after the event being predicted are always leaky. Check feature timestamps vs label timestamps.

Baseline model

Detailed explanation

A baseline model is the simplest possible model that solves the problem, against which all more complex models are measured. Without a baseline, you cannot know if a model is good — is 75% accuracy impressive or terrible? A majority-class baseline (always predict the most frequent class) achieves the base rate. If your model barely beats random guessing, it adds little value. Baselines set the performance floor.

Types of baselines: (1) trivial baseline — always predict majority class (classification) or mean (regression), (2) rule-based baseline — simple if-then rules using domain knowledge, (3) linear model baseline — logistic regression or linear regression (often harder to beat than expected), (4) previous model baseline — the model currently in production, (5) human performance baseline — how well domain experts do on the same task.

Evaluation order: start with the trivial baseline, then the linear model. Only add complexity if the linear model is insufficient. Many Kaggle top performers say a properly tuned XGBoost beats 90% of deep learning models on tabular data. Complexity should be justified by meaningful performance improvement — not just because a more complex model is available.

sklearn's DummyClassifier implements common baselines: DummyClassifier(strategy='most_frequent') always predicts the majority class. DummyClassifier(strategy='stratified') samples proportionally to class frequency. DummyClassifier(strategy='uniform') predicts each class with equal probability. Always compare your model against these to understand how much lift it provides.

Why baselines matter: they prevent over-engineering. If logistic regression achieves AUC 0.87 and a neural network achieves AUC 0.89, the 2% improvement may not justify the deployment complexity, latency, and maintenance cost. The business case must justify the jump from baseline to complex model.

Why this matters in AI
Baselines calibrate our sense of model quality. Without baselines, it is impossible to know if additional model complexity is worthwhile. Strong baselines also reveal problems: if a trivial model performs nearly as well as your ML model, the features or data quality may be insufficient for the task.
Common mistakes
  • Skipping the baseline — jumping directly to complex models wastes time if a simple model already solves the problem.
  • Comparing models to each other without comparing to baselines — model A (AUC 0.72) beats model B (AUC 0.70), but if the majority-class baseline is AUC 0.68, neither model is very impressive.
  • Using the wrong metric for the baseline — for imbalanced classification, majority class achieves high accuracy. Always use the same metric for baseline and models.

Scikit-learn workflow

Detailed explanation

The scikit-learn workflow follows a consistent pattern: instantiate → fit → transform/predict → evaluate. Every scikit-learn estimator (model, preprocessor, transformer) implements fit() and one or more of predict(), transform(), fit_transform(). This consistent API means you can swap any model for another with minimal code changes — a huge advantage for experimentation.

Pipelines are the cornerstone of clean scikit-learn code. A Pipeline chains preprocessors and a final estimator into one object. pipe = Pipeline([('scaler', StandardScaler()), ('pca', PCA(n_components=50)), ('clf', SVC())]). pipe.fit(X_train, y_train) applies each step in order. pipe.predict(X_test) applies the same transformations to test data automatically. Pipelines prevent the most common data leakage bug: fitting a preprocessor on both train and test data.

Column transformers handle heterogeneous data (some columns are numerical, some categorical). ColumnTransformer applies different preprocessing to different column subsets: num_pipe = Pipeline([('imputer', SimpleImputer()), ('scaler', StandardScaler())]); cat_pipe = Pipeline([('imputer', SimpleImputer(strategy='most_frequent')), ('encoder', OneHotEncoder())]); preprocessor = ColumnTransformer([('num', num_pipe, num_cols), ('cat', cat_pipe, cat_cols)]). This is the production-standard pattern for preprocessing heterogeneous tabular data.

Cross-validation with pipelines: cross_val_score(pipe, X_train, y_train, cv=5, scoring='roc_auc') fits and transforms the entire pipeline k times, each time fitting the preprocessor only on the training fold. This is the correct way to estimate out-of-sample performance — no leakage. GridSearchCV also works natively with pipelines: parameters are specified as 'step__param' (e.g., 'clf__C': [0.1, 1, 10]).

Model persistence: after training, save the fitted pipeline with joblib.dump(pipe, 'model.pkl'). Load in production: pipe = joblib.load('model.pkl'); predictions = pipe.predict(X_new). The fitted pipeline includes all fitted preprocessors (scalers, encoders) — you get the same transformations applied at training. Never save only the model without the preprocessors — they must be applied identically at inference.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer

pre = ColumnTransformer([('num', num_pipe, num_cols), ('cat', cat_pipe, cat_cols)])
full_pipe = Pipeline([('preprocess', pre), ('model', RandomForestClassifier(n_estimators=100))])
full_pipe.fit(X_train, y_train)
scores = cross_val_score(full_pipe, X_train, y_train, cv=5, scoring='roc_auc')
Grid search: param_grid = {'model__n_estimators': [100,200], 'model__max_depth': [5,10,None]}
gs = GridSearchCV(full_pipe, param_grid, cv=5, n_jobs=-1)
gs.fit(X_train, y_train)
Save: import joblib; joblib.dump(full_pipe, 'pipeline.pkl')

Why this matters in AI
The scikit-learn pipeline pattern is the production standard for tabular ML. It prevents data leakage, ensures consistent preprocessing at train and inference time, enables clean cross-validation, and makes models portable. Every professional data scientist must know this pattern.
Common mistakes
  • Fitting the scaler on all data before splitting — StandardScaler().fit_transform(X) before train_test_split leaks test set statistics into training. Always fit preprocessors inside a pipeline on training data only.
  • Not saving the full pipeline — saving only the model (not the preprocessors) means manually reapplying the same scaling/encoding at inference, which is fragile and error-prone.
  • Forgetting handle_unknown='ignore' in OneHotEncoder — at inference, categorical columns may contain values unseen during training. Without this parameter, the encoder throws an error.
