Multimodal AI

Detailed explanation

Multimodal AI processes and generates content across multiple modalities — text, images, audio, video, and structured data. Humans communicate multimodally: we read (text), look at pictures (images), listen (audio), and watch videos (video + audio). Early AI was unimodal: separate models for text, image, audio. Modern multimodal AI unifies these into single models that can understand and generate across all modalities within the same context.

Vision-language models (VLMs): combine computer vision and NLP. They can describe images (image captioning), answer questions about images (VQA — visual question answering), reason about scenes ('Is there a dog in this image and what is it doing?'), perform optical character recognition (OCR), and ground text references to image regions. Models: GPT-4V, Claude 3, Gemini 1.5, LLaVA (open-source), PaliGemma.

Architecture of VLMs: (1) vision encoder (CLIP ViT, SigLIP) converts the image to a sequence of visual tokens, (2) a projector layer maps visual tokens to the LLM's embedding space, (3) the LLM processes interleaved text and visual tokens. The LLM sees visual tokens as if they were word tokens — the architecture is identical to the text-only LLM except the input now includes image-derived tokens.

Audio-language models: Whisper (OpenAI) converts speech to text (ASR — automatic speech recognition). Audio-language models go further: understanding tone, detecting speaker identity (speaker diarization), performing audio captioning ('A piano is playing in a crowded restaurant'). GPT-4o processes audio natively, including real-time speech-to-speech conversation.

Multimodal RAG: retrieval-augmented generation with multiple modalities. A user uploads a PDF with text and charts. The system extracts: text chunks (for text RAG), images of charts (for visual RAG), and table data (for structured RAG). The retriever finds relevant context across all modalities. The VLM generates answers grounded in the combined text + image context. This enables 'chat with your documents' applications for complex technical documents, medical reports, and scientific papers.

Why this matters in AI
Multimodal AI is the next frontier in AI applications. Enterprise documents, medical reports, e-commerce product listings, and social media are inherently multimodal. Building applications that can understand and generate content across text, image, and audio — rather than just text — dramatically expands the scope of what AI can do.
Common mistakes
  • Resizing images too aggressively before passing to VLMs — GPT-4V and Claude 3 have optimal resolutions. Very small images (< 512×512) lose critical detail (text, small objects). Check the model's recommended image size.
  • Not chunking large documents before multimodal RAG — VLMs have context limits. A 100-page PDF needs intelligent chunking (by page, by section) before processing. Passing the entire PDF as one context window is not feasible.
  • Treating VLM image descriptions as ground truth — VLMs can hallucinate image details, especially for small text, complex charts, and unusual objects. Validate VLM outputs for factual claims.

Vision-language models

Detailed explanation

Vision-language models (VLMs) are neural networks that process both images and text as unified input. The key innovation: learning joint representations where image and text features occupy the same embedding space. CLIP (Contrastive Language-Image Pre-training, OpenAI) first demonstrated this at scale — trained on 400 million image-text pairs from the web to align image and text embeddings via contrastive learning.

CLIP architecture: a vision encoder (ViT — Vision Transformer, or ResNet) produces image embeddings. A text encoder (Transformer) produces text embeddings. Both are projected to the same d-dimensional space. Training: for a batch of (image, text) pairs, maximize similarity between matching pairs and minimize similarity between non-matching pairs (InfoNCE contrastive loss). This forces the model to learn that 'a photo of a cat' has an embedding close to an actual cat photo.

Zero-shot classification with CLIP: without any task-specific fine-tuning, CLIP can classify images by comparing the image embedding to text embeddings of class descriptions. For CIFAR-10: compute text embeddings for ['a photo of a airplane', 'a photo of a automobile', ..., 'a photo of a truck'], compare the image embedding to all 10, predict the closest class. This achieves ~76% accuracy on CIFAR-10 with NO training examples — zero-shot learning.

LLaVA (Large Language and Vision Assistant): extends VLMs to conversational question-answering. Architecture: CLIP ViT vision encoder → MLP projection → LLaMA/Mistral/Vicuna LLM. The projection maps visual tokens to the LLM's token embedding space — the LLM then processes interleaved image and text tokens. Training: instruction-following data pairs like ('Describe this image', image) fine-tune the model for instruction-following behavior.

Applications: image captioning (auto-generate alt text for accessibility), visual question answering (chatbots that can analyze product photos), chart understanding (extract data from graphs), document processing (understand invoices, receipts, forms), medical imaging (assist radiologists in analyzing X-rays and MRIs), robot perception (describe the scene for a robot arm to pick objects).

Why this matters in AI
VLMs are enabling a new generation of AI applications that understand the visual world as naturally as they understand text. Product search by image description, automated accessibility captioning, medical imaging analysis, and autonomous navigation all rely on VLMs. CLIP embeddings are widely used for multimodal RAG systems.
Common mistakes
  • Using CLIP for detailed image understanding — CLIP produces a single embedding per image (global representation). It cannot answer 'What is in the top-left corner?' For spatial/detailed questions, use a VLM like LLaVA or GPT-4V.
  • Not normalizing CLIP embeddings before similarity search — cosine similarity requires unit-norm vectors. Always normalize: embeddings /= embeddings.norm(dim=-1, keepdim=True).
  • Training a VLM from scratch — large VLMs require millions of GPU hours to train. Always start from a pre-trained model (CLIP, LLaVA) and fine-tune on your domain data.

Image captioning

Detailed explanation

Image captioning generates natural language descriptions of images. A caption model takes an image as input and outputs a sentence like 'A golden retriever runs through a field of yellow flowers.' This requires both visual understanding (what is in the image) and language generation (how to describe it naturally).

Encoder-decoder architecture: image encoder (CNN or Vision Transformer) extracts visual features → cross-attention mechanism attends to visual features while generating text → text decoder (language model) generates captions token by token. BLIP-2 and LLaVA use a more powerful approach: align a visual encoder with a frozen LLM via a lightweight adapter (Q-Former or MLP projection).

BLIP-2: Bootstrapping Language-Image Pre-training. Uses a Q-Former (Querying Transformer) that connects a frozen image encoder (EVA-CLIP) to a frozen LLM (Flan-T5 or OPT). The Q-Former learns to extract the most language-relevant features from the image. Fine-tuning only the Q-Former while keeping both the vision encoder and LLM frozen is extremely efficient.

Evaluation metrics: BLEU, METEOR, ROUGE measure n-gram overlap between generated and reference captions. CIDEr (Consensus-based Image Description Evaluation) specifically designed for captions — measures TF-IDF weighted n-gram similarity, highly correlated with human judgment. CLIP score measures semantic alignment between image and generated caption using CLIP embeddings.

Practical applications: image search (index images by their captions), accessibility (describe images for visually impaired users), content moderation (describe what's in a flagged image for human review), e-commerce (auto-generate product descriptions from product images).

from transformers import BlipProcessor, BlipForConditionalGeneration
from PIL import Image
import requests
# BLIP for image captioning
processor = BlipProcessor.from_pretrained('Salesforce/blip-image-captioning-base')
model = BlipForConditionalGeneration.from_pretrained('Salesforce/blip-image-captioning-base')
# Load image
img_url = 'https://storage.googleapis.com/sfr-vision-language-research/BLIP/demo.jpg'
raw_image = Image.open(requests.get(img_url, stream=True).raw).convert('RGB')
# Unconditional captioning
inputs = processor(raw_image, return_tensors='pt')
out = model.generate(**inputs, max_new_tokens=50)
caption = processor.decode(out[0], skip_special_tokens=True)
print(f'Caption: {caption}')
# Conditional captioning (describe specific aspect)
cond_text = 'a photography of'
inputs = processor(raw_image, cond_text, return_tensors='pt')
out = model.generate(**inputs, max_new_tokens=50)
print(f'Conditional: {processor.decode(out[0], skip_special_tokens=True)}')
Why this matters in AI
Image captioning is the bridge between visual and language understanding. It enables: image search without manual tagging, auto-generated alt text for accessibility (legal requirement in many jurisdictions), and is the foundation of visual Q&A, image-to-story generation, and multimodal document processing. BLIP-2's efficient adapter training approach is also a template for other multimodal fusion tasks.
Common mistakes
  • Not evaluating with CLIP score in addition to n-gram metrics — BLEU/ROUGE measure surface similarity to reference captions. A correct but differently phrased caption ('a dog runs' vs 'a canine is running') scores low on BLEU. CLIP score measures semantic alignment between image and caption regardless of exact phrasing.
  • Using the base model for domain-specific captioning — BLIP captioning models are trained on general images. For medical images (X-rays, pathology), fine-tune on domain-specific caption pairs for accurate technical descriptions.
  • Not handling image preprocessing — low-resolution images, very dark images, or unusual aspect ratios can degrade caption quality. Preprocess: resize to the model's expected resolution, enhance contrast for dark images.

OCR

Detailed explanation

OCR (Optical Character Recognition) converts images containing text into machine-readable text. Modern OCR combines deep learning for text detection (locating where text is in the image) and text recognition (reading the text at each location). OCR is a prerequisite for Document AI: processing scanned documents, PDFs, receipts, and forms.

Tesseract: the most widely used open-source OCR engine (maintained by Google). Works well for high-quality scanned text with clear fonts. pytesseract provides a Python wrapper. Best performance requires preprocessing: convert to grayscale, apply adaptive thresholding, deskew (correct rotation). Struggles with: handwritten text, complex layouts, low-resolution images, tables.

EasyOCR: a Python library supporting 80+ languages. Better than Tesseract on many real-world documents. Uses CRAFT (text detection) + CRNN (text recognition). `reader = easyocr.Reader(['en', 'fr']); results = reader.readtext(image_path)` returns a list of (bounding_box, text, confidence) tuples.

PaddleOCR: the state-of-the-art open-source OCR from Baidu. PP-OCR v4 is extremely accurate on complex document layouts, mixed languages, and rotated text. Pipeline: text detection (DBNet) → text direction classification → text recognition (SVTR). Significantly outperforms Tesseract and EasyOCR on challenging documents.

Cloud OCR: Google Cloud Vision API, AWS Textract, and Azure Computer Vision provide managed OCR. Textract additionally extracts structured data from forms and tables (key-value pairs, table cells) with high accuracy on standard business documents. Use cloud OCR for production when: high accuracy is required, documents have complex layouts, or you need structured extraction (forms, invoices) with minimal custom training.

# Tesseract with preprocessing
import pytesseract
import cv2
img = cv2.imread('document.png')
# Preprocessing for better OCR accuracy
gray = cv2.cvtColor(img, cv2.COLOR_BGR2GRAY)
# Adaptive thresholding (handles uneven lighting)
thresh = cv2.adaptiveThreshold(gray, 255, cv2.ADAPTIVE_THRESH_GAUSSIAN_C, cv2.THRESH_BINARY, 11, 2)
# Deskew (straighten tilted scans)
def deskew(image):
    coords = np.column_stack(np.where(image > 0))
    angle = cv2.minAreaRect(coords)[-1]
    if angle < -45: angle = 90 + angle
    (h, w) = image.shape[:2]
    M = cv2.getRotationMatrix2D((w//2, h//2), -angle, 1.0)
    return cv2.warpAffine(image, M, (w, h), flags=cv2.INTER_CUBIC)
deskewed = deskew(thresh)
# OCR
text = pytesseract.image_to_string(deskewed, config='--psm 6')  # psm 6: uniform block of text
print(text)
Why this matters in AI
OCR is the entry point for processing the vast majority of business documents — contracts, invoices, medical records, and reports exist as scanned PDFs or images. A Document AI pipeline that can OCR any document and extract structured information from it can automate enormous amounts of manual data entry. Understanding the OCR pipeline from preprocessing to structured extraction is a high-value practical skill.
Common mistakes
  • Not preprocessing images before OCR — contrast enhancement, thresholding, and deskewing improve OCR accuracy significantly on real-world documents. Don't send a raw low-contrast scan to OCR and expect good results.
  • Using Tesseract on complex layouts — Tesseract assumes a uniform text layout. For documents with columns, tables, and mixed text+graphics, use PaddleOCR or a cloud OCR service that handles complex layouts.
  • Not validating OCR output — OCR always makes some errors. For production pipelines, validate extracted values against expected formats (dates, amounts, IDs) and flag low-confidence extractions for human review.

Document AI

Detailed explanation

Document AI (intelligent document processing) extracts structured information from unstructured documents: invoices, contracts, medical records, forms, and reports. It combines OCR (text extraction), layout understanding (where is the text on the page?), and NLP (what does the text mean?) to convert document images or PDFs into structured data.

Document types and their challenges: structured forms (tax forms, medical intake forms) have fixed layouts — field extraction is straightforward with template matching. Semi-structured documents (invoices, receipts) have variable layouts but consistent entities (vendor, date, total). Unstructured documents (contracts, research papers) require deep NLP to extract entities and relationships.

Layout-aware models: standard NLP models treat text as a sequence of tokens without positional information. Layout-aware models (LayoutLM, LayoutLMv2, LayoutLMv3) incorporate the 2D position of each token (its bounding box coordinates) as additional input. This allows the model to understand: 'the number to the right of the label TOTAL is the total amount.'

LayoutLMv3 architecture: combines text embeddings, image patches (from the document image), and layout positions (normalized bounding box coordinates). Pre-trained on large document datasets. Fine-tuned on specific document types (invoices: extract vendor/date/amount; contracts: extract parties/dates/key clauses). State-of-the-art for form understanding (FUNSD benchmark) and document classification.

AWS Textract and Google Document AI: managed cloud services for Document AI. Textract extracts: text, tables (as structured CSV-like data), and forms (key-value pairs). Google Document AI provides specialized processors: invoice processor, W-2 form processor, identity document processor. These cloud services are often the fastest path to production for standard document types.

Why this matters in AI
Document AI is one of the highest-value real-world AI applications. Enterprises process millions of invoices, contracts, and forms manually — automating even 80% of this with accurate extraction saves enormous labor costs and reduces errors. Understanding the full Document AI pipeline (OCR → layout parsing → entity extraction → structured output) positions you for applied ML work in finance, healthcare, legal, and logistics.
Common mistakes
  • Using a text-only NLP model for documents with complex layouts — BERT sees the text of an invoice without knowing which numbers are totals vs quantities vs prices. LayoutLM understands that a number directly to the right of a label 'Total Due:' is the total amount. Always use layout-aware models for document information extraction.
  • Skipping document quality assessment — LayoutLM and OCR performance degrades dramatically on low-resolution scans (< 150 DPI), tilted documents, and documents with poor contrast. Add a quality check step: reject documents below a quality threshold and request a better scan.
  • Over-relying on template matching — invoices from different vendors have different layouts. A template matcher trained on Vendor A's invoices fails completely on Vendor B's. Use layout-aware models that generalize across layouts, not template matching that only works for one specific layout.

PDF parsing

Detailed explanation

PDF parsing extracts text, tables, images, and metadata from PDF files. PDFs come in two types: text-based (PDF with embedded text fonts — easy to parse) and image-based (scanned documents stored as images — requires OCR). A production document processing pipeline must handle both.

pdfplumber: the best Python library for extracting text AND tables from text-based PDFs. It exposes each page's text, bounding boxes, and table structure. `page.extract_text()` gets plain text. `page.extract_table()` extracts a table as a list of rows. `page.extract_tables()` extracts all tables on the page. Preserves reading order better than PyPDF2.

PyPDF2 / pypdf: simpler library for basic text extraction. `PdfReader(file).pages[i].extract_text()`. Good for text-only PDFs without complex layouts. Fails on: two-column layouts (interleaves columns incorrectly), text in textboxes overlaid on images, and scanned PDFs.

Camelot: specialized for table extraction from text-based PDFs. Two modes: lattice (detects tables by cell borders — accurate for PDF tables with visible borders), stream (infers table structure from whitespace — for borderless tables). Better than pdfplumber for complex tables.

Handling scanned PDFs: if `page.extract_text()` returns empty or garbled text, the PDF is image-based. Use Pillow to convert each page to an image (`pdf2image.convert_from_path()`), then run OCR (EasyOCR, Tesseract, PaddleOCR) on each page image. The result is a text representation that can be processed by NLP models.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import pdfplumber
from pathlib import Path
# Extract text and tables from a PDF
def extract_pdf_content(pdf_path: str) -> dict:
    result = {'pages': [], 'tables': []}
    with pdfplumber.open(pdf_path) as pdf:
        for page_num, page in enumerate(pdf.pages, 1):
            text = page.extract_text()
            tables = page.extract_tables()
            result['pages'].append({'page': page_num, 'text': text, 'n_chars': len(text or '')})
            for table in tables:
                result['tables'].append({'page': page_num, 'rows': len(table), 'cols': len(table[0]) if table else 0, 'data': table})
    return result
content = extract_pdf_content('annual_report.pdf')
print(f'Pages: {len(content["pages"])}')
print(f'Tables: {len(content["tables"])}')
# Check if text was extracted (vs scanned PDF)
for page in content['pages'][:3]:
    print(f'Page {page["page"]}: {page["n_chars"]} chars')
    if page['n_chars'] < 50: print('  WARNING: possibly scanned — run OCR')
# Convert scanned PDF pages to images for OCR
from pdf2image import convert_from_path
import easyocr
def extract_scanned_pdf(pdf_path: str) -> list:
    images = convert_from_path(pdf_path, dpi=300)  # 300 DPI for good OCR quality
    reader = easyocr.Reader(['en'])
    all_text = []
    for i, image in enumerate(images):
        # Convert PIL Image to numpy array for easyocr
        import numpy as np
        img_array = np.array(image)
        results = reader.readtext(img_array)
        page_text = ' '.join([text for (_, text, conf) in results if conf > 0.3])
        all_text.append({'page': i+1, 'text': page_text})
        print(f'Page {i+1}: {len(page_text)} chars extracted')
    return all_text
Why this matters in AI
PDF parsing is a foundational skill for any Document AI or RAG application. Most enterprise knowledge lives in PDFs: contracts, financial reports, research papers, technical documentation. A RAG system that only handles plain text misses the majority of enterprise documents. Knowing how to extract text and tables from both text-based and scanned PDFs unlocks the ability to process almost any document type.
Common mistakes
  • Not detecting whether a PDF is text-based or scanned before parsing — pdfplumber.extract_text() returns None or an empty string for scanned PDFs. Check the character count per page; if it's < 50 for a non-empty page, run OCR.
  • Not preserving page structure for RAG chunking — extracting one long string from a 100-page PDF and then chunking it loses page boundaries. Chunk by page, preserving page numbers in metadata so retrieved chunks can be cited correctly.
  • Not handling multi-column layouts — pdfplumber reads text in page order (left-to-right, top-to-bottom within the page). In two-column layouts, it interleaves left and right column content. Use `page.extract_text(x_tolerance=3)` or column detection to handle multi-column PDFs correctly.

Layout understanding

Detailed explanation

Layout understanding is the ability to interpret the spatial structure of a document — not just the text content, but where that text is on the page and what its position means. A table cell in row 3 column 2 has different semantics from the same text in a paragraph. A label to the left of a field tells you the field's name. A bold heading signals a section boundary.

Why layout matters: the same word 'Total' can mean different things based on its position on a page. In an invoice, 'Total' in the bottom-right, followed by a dollar amount, signals the total invoice amount. In a contract, 'Total' may be part of a legal clause. Layout-aware models (LayoutLM, LayoutLMv3) incorporate bounding box coordinates alongside text to disambiguate.

Document layout analysis: before extracting text, analyze the page structure. Identify: text regions (paragraphs, headings), table regions (rows and columns), figure regions (charts, images), and form fields (label-value pairs). Tools: Detectron2 with the DocLayNet dataset (trained to segment documents into these regions), PaddlePaddle's PP-Structure.

Table structure recognition: detecting that a table exists is step one. Recognizing its structure (number of rows/columns, which cells are headers, which cells span multiple rows/columns) is harder. TATR (Table Transformer) from Microsoft is the state-of-the-art model for table structure recognition. After structure recognition, populate a 2D array with the cell text and export as CSV/DataFrame.

Form understanding: forms have a regular structure: label → value pairs. 'First Name: John', 'Date of Birth: 1990-01-15'. LayoutLMv3 fine-tuned on FUNSD (Form Understanding in Noisy Scanned Documents) achieves near-human accuracy on linking form labels to their values. Speech-to-text

Detailed explanation

Speech-to-text (STT), also called automatic speech recognition (ASR), converts audio waveforms to text. The audio signal is a time series of pressure values. STT systems preprocess this into spectrograms (frequency-time representations), extract acoustic features (Mel-frequency cepstral coefficients, filterbank features), and use a sequence model to decode the most likely text sequence.

Modern STT: Whisper (OpenAI) is a Transformer-based sequence-to-sequence model trained on 680,000 hours of multilingual audio from the web. It takes mel-spectrogram patches as input and outputs text tokens autoregressively. Whisper achieves near-human accuracy on English transcription and works in 99 languages. It handles accents, background noise, and technical terminology surprisingly well. Whisper is open-source (Apache 2.0), runs locally, and is the current standard for high-quality STT.

The mel spectrogram: raw audio is first divided into 25ms frames with 10ms hop. Each frame is converted to a frequency spectrum (FFT). Frequency bins are mapped to mel scale (logarithmic, matching human hearing) — 80 mel bins for Whisper. Time is the x-axis, mel frequency is the y-axis, and color intensity is energy at that frequency/time. The spectrogram captures the 'shape' of audio in a form amenable to CNNs and Transformers.

Challenges: code-switching (speaker switches languages mid-sentence), background noise, overlapping speakers (diarization problem), domain-specific vocabulary (medical, legal terminology), strong accents, and low-resource languages. Whisper handles many of these with its large training data. For highly specialized vocabulary, Whisper can be fine-tuned on domain-specific audio data using the HuggingFace transformers library.

Speaker diarization (who spoke when?) is a separate problem from transcription. It identifies speech segments and attributes each to a speaker identity. Tools: pyannote.audio provides state-of-the-art diarization. Combining Whisper (what was said) + pyannote (who said it) gives a speaker-attributed transcript — used for meeting notes, call center analysis, and interview processing.

Text-to-speech

Detailed explanation

Text-to-speech (TTS) converts written text to spoken audio. Modern neural TTS (Tacotron 2, VITS, Bark, Parler-TTS) produces natural-sounding speech indistinguishable from human voice in many cases. TTS is used in: voice assistants, audiobook generation, accessibility tools, and any application that needs to speak to users.

TTS pipeline: text normalization (convert '2024' → 'two thousand twenty-four', '$5.99' → 'five dollars and ninety-nine cents') → phoneme conversion (text → phoneme sequence) → acoustic model (phonemes → mel spectrogram) → vocoder (mel spectrogram → audio waveform). End-to-end models (VITS) combine these stages into a single model.

gTTS and pyttsx3: the simplest Python TTS libraries. gTTS uses Google Translate's TTS API (requires internet, good quality). pyttsx3 uses the operating system's built-in TTS (offline, lower quality). For quick prototyping, `from gtts import gTTS; tts = gTTS('Hello world'); tts.save('output.mp3')` works in 3 lines.

Coqui TTS (XTTS): open-source, high-quality, offline TTS. Supports 17 languages, voice cloning (provide a 6-second audio sample → TTS in that voice), and fine-tuning for custom voices. `from TTS.api import TTS; tts = TTS('tts_models/multilingual/multi-dataset/xtts_v2'); tts.tts_to_file('Hello world', speaker_wav='reference.wav', file_path='out.wav')`.

Streaming TTS: for voice interfaces, latency matters. Don't generate the full audio before speaking — stream audio as it's generated. Eleven Labs, Play.ht, and the Cartesia API support streaming TTS via WebSocket or Server-Sent Events. First chunk latency (time to first audio) is the key metric for voice interfaces, not total generation time.

Speaker diarization

Detailed explanation

Speaker diarization answers the question: 'who spoke when?' Given a multi-speaker audio file, diarization segments the audio by speaker identity and assigns each segment a speaker label (SPEAKER_00, SPEAKER_01, ...). This is a prerequisite for meeting transcription, interview analysis, and call center recording processing.

Diarization pipeline: voice activity detection (VAD) → speaker segmentation (find where speakers change) → speaker embedding extraction (embed each segment into a speaker representation space) → clustering (group segments by speaker identity). pyannote.audio is the state-of-the-art Python library for this pipeline.

pyannote.audio: a comprehensive audio analysis library. `Pipeline.from_pretrained('pyannote/speaker-diarization-3.1')` provides an end-to-end diarization pipeline. Input: an audio file. Output: an annotation object with speaker labels and time segments. When combined with Whisper transcription: each transcript segment can be attributed to a specific speaker.

Challenges: overlapping speech (two speakers talking simultaneously — diarization typically assigns to the dominant speaker), similar-sounding speakers (hard to distinguish), short speaker segments (insufficient audio to identify the speaker), and varying audio quality (background noise, compression artifacts).

Evaluation: Diarization Error Rate (DER) is the standard metric. DER = (missed speech + false alarm + speaker confusion) / total speech duration. Lower is better. State-of-the-art systems achieve DER < 5% on clean two-speaker conversations and DER 10-20% on noisy multi-speaker meetings.

Audio classification

Detailed explanation

Audio classification categorizes audio clips into predefined classes. Examples: environmental sound classification (bird, car, music, speech), music genre classification, speech command recognition (yes/no/stop/go), emotion recognition from voice, and acoustic event detection. The input is a raw audio waveform; the output is a class label.

Audio representations: raw waveform (1D time series, processed by WaveNet-style models), mel spectrogram (2D image of frequency vs time, processed by CNNs), and Mel-frequency cepstral coefficients (MFCCs, traditional feature extraction for classical ML). Modern approaches: fine-tune audio transformers (Wav2Vec2, Whisper, HuBERT) on the classification task.

Wav2Vec2 for audio classification: a transformer pretrained on raw audio with self-supervised learning (similar to BERT for text). Fine-tuning on a classification dataset with ~1,000 labeled examples achieves strong performance. HuggingFace pipeline: `pipe = pipeline('audio-classification', model='ehcalabres/wav2vec2-lg-xlsr-en-speech-emotion-recognition')`.

Librosa for feature extraction: the standard Python library for audio analysis. `librosa.load(file, sr=22050)` loads audio as a numpy array. `librosa.feature.mfcc(y, sr)` computes 20 MFCC features per frame. `librosa.feature.melspectrogram(y, sr)` computes the mel spectrogram. These features are inputs to classical ML classifiers or lightweight CNNs.

Data augmentation for audio: time stretching (speed up/slow down), pitch shifting (raise/lower pitch without changing duration), adding background noise, room impulse response convolution (simulate different acoustic environments). `audiomentations` library handles all of these: `augment = Compose([AddGaussianNoise(p=0.5), TimeStretch(p=0.5), PitchShift(p=0.5)])`.

# MFCC-based audio classification
def extract_features(audio_path, sr=22050, n_mfcc=40):
    y, sr = librosa.load(audio_path, sr=sr)
    mfccs = librosa.feature.mfcc(y=y, sr=sr, n_mfcc=n_mfcc)  # [40, time_frames]
    mfccs_mean = mfccs.mean(axis=1)  # [40]

    mel = librosa.feature.melspectrogram(y=y, sr=sr).mean(axis=1)[:32]  # [32]
    return np.concatenate([mfccs_mean, mfccs_std, chroma, mel])  # 124 features

Image generation

Detailed explanation

Image generation models synthesize new images from text descriptions or other images. The current state-of-the-art uses diffusion models — a class of generative models that learn to denoise images. During training, clean images are progressively noised (a forward process adding Gaussian noise over T steps until the image is pure noise). The model learns to reverse this process — predict the clean image from a noisy version (a reverse denoising process). At inference, start from pure noise and iteratively denoise to generate a new image.

Stable Diffusion is an open-source latent diffusion model. The key insight: instead of diffusing in pixel space (expensive for 512×512 images — 786,432 dimensions), Stable Diffusion encodes images into a compressed latent space (64×64 with 4 channels — 16,384 dimensions) using a VAE, performs diffusion in this latent space (64× cheaper), then decodes back to pixel space. This enables running on consumer GPUs.

Text conditioning: to guide generation from a text prompt, the model uses CLIP (Contrastive Language-Image Pretraining) to encode the text prompt into an embedding. At each denoising step, the model conditions on this text embedding via cross-attention. The text embedding steers denoising toward images matching the description. Classifier-free guidance (CFG) scales the influence of the text condition — higher CFG scale (7–15) makes the output more faithful to the prompt but less diverse.

ControlNet adds fine-grained spatial control to diffusion models by conditioning on structural inputs: pose estimation (generate an image of a person in a specific pose), edge maps (generate a photo from a sketch), depth maps (maintain scene structure), or segmentation masks (maintain object layouts). This is used for: photo editing (keep composition, change style), product visualization (place a product in different scenes), and design ideation.

DALL-E 3, Midjourney, and Imagen are proprietary image generation models with higher aesthetic quality than open-source models but less flexibility. For applications requiring commercial use, data privacy, or fine-tuning on custom styles, Stable Diffusion (SDXL, Stable Diffusion 3) is the appropriate choice. Fine-tuning techniques: DreamBooth (personalize a model to generate a specific subject from 3–20 images), LoRA (fine-tune a new style efficiently).

Diffusion models

Detailed explanation

Diffusion models are a class of generative models that learn to generate images by learning to reverse a noise-addition process. The forward process gradually adds Gaussian noise to an image over T steps until it becomes pure noise. The reverse process (denoising) learns to predict the noise added at each step, allowing recovery of the original image. By starting with pure noise and reversively denoising, the model generates new images.

DDPM (Denoising Diffusion Probabilistic Models): the noise schedule adds small amounts of Gaussian noise at each of T=1000 timesteps. The forward process: x_t = sqrt(αt) × x_{t-1} + sqrt(1-αt) × ε where ε ~ N(0, I). The reverse process: a U-Net neural network ε_θ(x_t, t) predicts the noise at each timestep. Training: minimize E[||ε - ε_θ(x_t, t)||²] — predict the added noise. Inference: start from x_T ~ N(0, I), run the U-Net T times to generate x_0.

Stable Diffusion: the dominant open-source text-to-image model. Key innovation: latent diffusion — instead of operating in pixel space (expensive: 512×512×3 = 786K dimensions), Stable Diffusion operates in a compressed latent space (64×64×4 = 16K dimensions). A VAE encodes/decodes between pixel and latent space. CLIP encodes the text prompt. The U-Net denoises in latent space conditioned on CLIP text embeddings. Much faster than pixel-space diffusion.

Classifier-free guidance (CFG): during training, the text conditioning is randomly dropped (replaced with empty text). The U-Net learns both conditional and unconditional score estimates. During inference, guidance scale g controls how strongly the text prompt influences generation: ε_final = ε_unconditional + g × (ε_conditional - ε_unconditional). Higher g (7–15) produces images more aligned with the prompt but less diverse. Lower g produces more varied but less prompt-adherent images.

ControlNet: an extension that adds spatial conditioning to Stable Diffusion. Instead of just a text prompt, ControlNet accepts additional control signals: edge maps (generate an image matching the edge structure), depth maps (generate an image matching depth), human pose skeletons, segmentation masks. This enables precise control over image composition — a person's pose is preserved while generating a completely different style.

Video understanding

Detailed explanation

Video understanding involves extracting information from video: action recognition, video captioning, temporal localization (when does an event occur?), and video question answering. Video extends image understanding with a temporal dimension — the sequence of frames carries information about motion, causation, and storyline.

Key challenges: computational cost (a 1-minute video at 30fps = 1,800 frames), temporal modeling (understanding that frame 200 depends on frame 1), long-range dependencies (an action that starts in the first 10 seconds completes in the last 10 seconds), and audio-visual alignment (speech and visual action should be synchronized).

Frame sampling strategies: processing every frame is expensive and redundant. Uniform sampling (every Nth frame), key frame extraction (frames where scene changes), and scene detection (segment video by scene). PySceneDetect detects scene changes. Dense optical flow captures motion between consecutive frames — a compact representation of motion without needing all frames.

Video transformers: TimeSformer (divided space-time attention), VideoMAE (masked video autoencoder), and InternVideo 2 are the state-of-the-art for video understanding. These models extend image transformers to process sequences of frames. The HuggingFace video-text models (VideoLLM, Video-LLaMA) support video Q&A.

Practical applications: sports analytics (detecting player actions, tracking ball position), security surveillance (detecting unusual events), content moderation (flagging violent or inappropriate video content), and e-learning (automatically generating summaries and chapters from lecture videos).

import cv2
import numpy as np
# Frame extraction from video
def extract_frames(video_path: str, n_frames: int = 8) -> list:
    cap = cv2.VideoCapture(video_path)
    total_frames = int(cap.get(cv2.CAP_PROP_FRAME_COUNT))
    fps = cap.get(cv2.CAP_PROP_FPS)
    duration = total_frames / fps
    print(f'Video: {total_frames} frames, {fps:.1f} fps, {duration:.1f}s')
    # Uniform sampling
    frame_indices = np.linspace(0, total_frames-1, n_frames, dtype=int)
    frames = []
    for idx in frame_indices:
        cap.set(cv2.CAP_PROP_POS_FRAMES, idx)
        ret, frame = cap.read()
        if ret: frames.append(cv2.cvtColor(frame, cv2.COLOR_BGR2RGB))
    cap.release()
    return frames
# Video captioning with transformers
from transformers import pipeline
video_captioner = pipeline('image-to-text', model='nlpconnect/vit-gpt2-image-captioning')
frames = extract_frames('video.mp4', n_frames=4)
for i, frame in enumerate(frames):
    from PIL import Image
    caption = video_captioner(Image.fromarray(frame))[0]['generated_text']
    print(f'Frame {i}: {caption}')
Why this matters in AI
Video is the dominant form of content on the internet — YouTube has 500 hours of video uploaded every minute. Video understanding systems that can automatically: generate captions, detect key moments, extract action information, and answer questions about video content are transforming media, security, healthcare, and sports industries. Understanding the pipeline from frame sampling to temporal modeling is essential for any practitioner in this space.
Common mistakes
  • Processing every frame of a high-FPS video — 60fps × 60 seconds = 3,600 frames, each taking 50ms to process = 3 minutes of inference for 1 minute of video. Sample uniformly (8-16 frames per video) or use scene detection to find the most informative frames.
  • Ignoring audio in video understanding — audio provides critical context (speech tells you what's happening, sound effects signal events). Use a multimodal model that processes both audio and video, or separately transcribe audio with Whisper and fuse the text with video features.
  • Not considering memory constraints for long videos — loading an entire 10-minute video into GPU memory is infeasible. Process videos in sliding windows (30s clips with 10s overlap), extract features per clip, and aggregate temporal representations.

Multimodal RAG

Detailed explanation

Multimodal RAG (Retrieval Augmented Generation) extends standard text-based RAG to retrieve and reason over multiple modalities: text, images, tables, charts, and PDFs. Instead of only querying a text corpus, multimodal RAG can retrieve: a relevant image, a specific table from a PDF, a chart from a financial report, and text paragraphs — then feed all of these to a multimodal LLM for grounded response generation.

Architecture options: (1) late fusion — embed text and images with separate encoders, store in the same vector database, query returns mixed-modality chunks. (2) caption-based — convert all non-text content to text (OCR tables, BLIP-caption images, pandas-table CSV) and use standard text RAG. (3) CLIP-based — use CLIP embeddings for both text queries and image retrieval (shared embedding space).

Multimodal document processing: for a PDF with tables and figures: extract text with pdfplumber, extract tables with camelot and convert to markdown/CSV, detect figure regions with LayoutLM and caption with BLIP. Each chunk is stored with its type (text, table, image) and source metadata.

LLaVA and GPT-4V for multimodal generation: once relevant chunks (including images) are retrieved, pass them to a multimodal LLM. GPT-4V and Claude Vision can process images directly. LLaVA (Large Language and Vision Assistant) is the open-source alternative. The model receives: the text query, retrieved text passages, and retrieved images — and generates a grounded response.

Evaluation: multimodal RAG evaluation requires checking both retrieval (did we retrieve the relevant image/table?) and generation (is the answer grounded in the retrieved visual content?). RAGAS extensions for multimodal RAG: visual_faithfulness (does the answer match the retrieved image?), visual_context_recall (did we retrieve the relevant visual content?).

import anthropic
from pathlib import Path
import base64
client = anthropic.Anthropic()
# Multimodal RAG with image context
def answer_with_image_context(question: str, text_context: str, image_paths: list) -> str:
    # Build message with text + images
    content = [{'type': 'text', 'text': f'Question: {question}\n\nText context:\n{text_context}\n\nAlso consider the images provided. Answer based only on the provided context.'}]
    # Add images
    for img_path in image_paths[:4]:  # Claude accepts up to 20 images
        with open(img_path, 'rb') as f:
            img_data = base64.standard_b64encode(f.read()).decode('utf-8')
        media_type = 'image/png' if img_path.endswith('.png') else 'image/jpeg'
        content.append({'type': 'image', 'source': {'type': 'base64', 'media_type': media_type, 'data': img_data}})
    response = client.messages.create(
        model='claude-opus-4-8', max_tokens=1024,
        messages=[{'role': 'user', 'content': content}]
    )
    return response.content[0].text

def embed_image_for_retrieval(image): return clip_model.get_image_features(**clip_processor(images=image, return_tensors='pt')).detach().numpy()
def embed_text_for_retrieval(text): return clip_model.get_text_features(**clip_processor(text=[text], return_tensors='pt')).detach().numpy()
Why this matters in AI
Multimodal RAG is necessary for processing real-world business documents. A financial report has text, tables, and charts; a medical report has text and scan images; a technical manual has text and diagrams. A text-only RAG system retrieves the surrounding text but misses the information encoded in the table or chart. Multimodal RAG retrieves and reasons over all content types.
Common mistakes
  • Caption-only approach loses visual precision — a chart showing a 40% revenue drop is better described as an image than by its auto-generated caption ('a bar chart with blue bars'). For high-precision use cases, pass images directly to a multimodal model rather than relying on captions.
  • Not chunking tables correctly for retrieval — a 50-row table as a single chunk may retrieve only one row's context. Split large tables by column group (related columns together) or mark each row as its own chunk with table metadata for more precise retrieval.
  • Not labeling the type of each retrieved chunk — if the model receives text+image chunks without knowing which is which, it can't use them appropriately. Always tag chunks with their type and source: {type: 'table', source: 'Q3_report.pdf:page5', content: '...'}.
