File handling

Detailed explanation

File handling in Python covers reading and writing text files, binary files, and structured formats (CSV, JSON). The open() function opens a file: f = open('data.txt', 'r') opens for reading, 'w' for writing (overwrites), 'a' for appending, 'rb'/'wb' for binary. Always use the context manager (with open(...) as f:) which automatically closes the file even if an exception occurs.

Reading files: f.read() reads the entire file as a string. f.readlines() returns a list of lines (with \n). for line in f: iterates line by line (memory-efficient for large files). f.readline() reads one line at a time. For text files: always specify encoding='utf-8' to handle international characters.

Writing files: f.write(string) writes a string (no automatic newline). f.writelines(list_of_strings) writes multiple strings. For ML: save model weights, training logs, evaluation results, and configuration files. Common pattern: save results as JSON (structured) or CSV (tabular).

Pathlib: the modern, object-oriented way to handle file paths. from pathlib import Path. Path('data/train.csv').exists() checks if file exists. Path('outputs').mkdir(parents=True, exist_ok=True) creates directories. Path('data/train.csv').stem returns 'train' (filename without extension). Path('data/train.csv').parent returns Path('data'). Use pathlib instead of os.path for all file operations.

File I/O for ML workflows: save/load model config as JSON, training metrics as CSV, model weights as .pt (PyTorch) or .pkl (sklearn with joblib). Template: save_dir = Path(f'experiments/{experiment_name}'); save_dir.mkdir(parents=True, exist_ok=True). Then: json.dump(config, open(save_dir/'config.json', 'w')), pd.DataFrame(metrics).to_csv(save_dir/'metrics.csv'). This creates a self-contained experiment directory.

Why this matters in AI
File handling is essential for: saving trained models (persistence), loading datasets (data pipelines), writing experiment logs (reproducibility), and reading configuration files (hyperparameter management). Every ML project involves reading and writing files extensively.
Common mistakes
  • Not using context managers (with open()) — open files consume system resources. If the code crashes before f.close(), the file remains locked. Context managers guarantee proper cleanup.
  • Hardcoding absolute file paths — '/Users/john/data/train.csv' breaks on any other machine. Use relative paths or Path objects with configurable root directories.
  • Not specifying encoding when reading text — open('file.txt') uses the system default encoding (often ASCII on Windows). Text with accented characters or emoji raises UnicodeDecodeError. Always: open('file.txt', encoding='utf-8').

CSV and JSON

Detailed explanation

CSV (Comma-Separated Values) and JSON (JavaScript Object Notation) are the two most common data file formats in ML. CSV stores tabular data — rows of values separated by commas (or tabs, semicolons). JSON stores structured data — nested dictionaries and lists. Understanding both formats and their Python equivalents (pandas for CSV, json module for JSON) is foundational for data ingestion.

Reading CSV with pandas: pd.read_csv('data.csv') loads a CSV into a DataFrame. Common parameters: sep=',' (or '\t' for TSV), header=0 (row number for column names — header=None if no header), index_col=0 (column to use as row index), usecols=['col1', 'col2'] (load only specific columns), dtype={'price': float, 'id': str} (specify column types), na_values=['NA', 'None', '-'] (additional strings to treat as NaN).

Writing CSV: df.to_csv('output.csv', index=False) writes without the row index. index=False is almost always what you want. sep='\t' writes as tab-separated. For large files: chunksize parameter in read_csv processes in chunks to avoid memory issues. For very large files (> RAM): use pd.read_csv() with iterator=True or chunksize=10000 to process in batches.

JSON in Python: import json. json.loads(string) parses a JSON string to a Python dict/list. json.dumps(obj) converts a Python dict/list to a JSON string. json.load(file) reads from a file. json.dump(obj, file, indent=2) writes to a file with pretty-printing. JSON supports: strings, numbers, booleans, null, arrays, objects (nested). JSON does NOT support: numpy arrays (convert to list first), Python tuples (become arrays), dates (convert to ISO string).

Common data formats in ML: JSONL (JSON Lines) — one JSON object per line, common for LLM training data (easy to stream). Parquet — columnar binary format, much faster to read than CSV for analytical queries. Feather — fast binary format for DataFrame interchange. Pickle — Python-specific binary format (security risk for untrusted files). For ML datasets: prefer Parquet for large datasets (10GB+) over CSV for speed and compression.

Why this matters in AI
CSV and JSON are the entry point for 90% of ML data ingestion. Almost every dataset comes as CSV (tabular data) or JSON (API responses, configuration files, model predictions). Knowing the edge cases (encoding, special characters, nested JSON, large files) prevents hours of data loading bugs.
Common mistakes
  • Reading large CSV files without usecols — loading a 50-column CSV when you only need 10 columns wastes memory. Specify usecols=['needed_col1', 'needed_col2'] to load only required columns.
  • Not handling JSON values that can't be serialized by default — numpy arrays, datetime objects, and custom classes raise TypeError with json.dumps. Use a custom encoder: json.dumps(obj, default=lambda x: x.tolist() if hasattr(x, 'tolist') else str(x)).
  • Trusting CSV column dtypes — pandas infers dtypes from the first few rows. A column of integers may load as float (due to NaN values). Always verify dtypes with df.dtypes and cast explicitly.

Datetime

Detailed explanation

Datetime handling is a foundational Python skill for ML practitioners working with time-series data, logs, event streams, or any dataset with temporal information. Python's datetime module and pandas' datetime support handle the conversion, arithmetic, and formatting of dates and times.

Python's datetime module: `datetime.datetime` represents a specific moment (year, month, day, hour, minute, second, microsecond). `datetime.date` represents only a date. `datetime.timedelta` represents a duration — the difference between two datetimes. Key operations: creating datetimes, parsing strings to datetimes (strptime), formatting datetimes as strings (strftime), and computing date arithmetic.

Common format codes for strptime/strftime: %Y=4-digit year, %m=2-digit month, %d=2-digit day, %H=24-hour, %M=minute, %S=second, %f=microseconds. Example: `datetime.strptime('2024-03-15 14:30', '%Y-%m-%d %H:%M')` parses that string. `datetime.strftime('%B %d, %Y')` formats as 'March 15, 2024'.

Pandas datetime support: `pd.to_datetime()` converts a column to datetime dtype. Once converted, you can access components via `.dt`: `df['date'].dt.year`, `.dt.month`, `.dt.dayofweek` (0=Monday), `.dt.hour`, `.dt.date`. Date arithmetic in pandas: `df['date'] + pd.Timedelta(days=7)` adds 7 days. `pd.DateOffset(months=1)` adds exactly one month (handles month-end edge cases).

Timezone handling: naive datetimes have no timezone information. Timezone-aware datetimes include UTC offset. `pytz.timezone('America/New_York')` creates a timezone object. `datetime.now(tz=pytz.utc)` creates a UTC-aware datetime. `dt.astimezone(tz)` converts between timezones. For ML features, always convert to UTC first to avoid issues with daylight saving time.

Why this matters in AI
Datetime handling is unavoidable in data science. Transaction logs, user events, sensor readings, and financial data all have timestamps. Failing to handle timezones correctly introduces subtle bugs (events appear to happen before midnight when they actually happened after midnight in another timezone). Proper datetime feature engineering (day of week, hour of day, month) adds significant predictive power to models.
Common mistakes
  • Treating datetime columns as strings — comparing dates as strings ('2024-10-01' > '2024-09-30') works but sorting and arithmetic don't. Always convert to datetime dtype with pd.to_datetime() at the start of your pipeline.
  • Ignoring timezones — storing timestamps without timezone information causes bugs when users are in different timezones. Normalize all timestamps to UTC on ingestion and convert to local time only for display.

Regular expressions

Detailed explanation

Regular expressions (regex) are patterns for matching, searching, and replacing text. In ML and data science, regex is used for: text cleaning (remove HTML tags, URLs, special characters), feature extraction (extract dates, phone numbers, prices from text), data validation (check email format, credit card format), and NLP preprocessing.

Python's re module: re.match(pattern, string) matches at the START of string. re.search(pattern, string) finds the FIRST match anywhere in string. re.findall(pattern, string) returns a list of ALL matches. re.sub(pattern, replacement, string) replaces matches. re.compile(pattern) compiles a pattern for repeated use (10× faster for repeated calls).

Regex syntax essentials: . (any char), \d (digit 0-9), \w (word char: a-z, A-Z, 0-9, _), \s (whitespace: space, tab, newline), + (1 or more), * (0 or more), ? (0 or 1), {3} (exactly 3), {2,5} (2 to 5), ^ (start of string), $ (end of string), [] (character class: [aeiou] matches any vowel), [^] (negated class: [^\d] matches non-digit), () (capturing group), | (or: cat|dog).

Common ML text cleaning with regex: remove HTML tags: re.sub('<[^>]+>', '', text). Remove URLs: re.sub(r'https?://\S+', '', text). Extract prices: re.findall(r'\$[\d,.]+', text). Remove non-alphabetic: re.sub('[^a-zA-Z\s]', '', text). Normalize whitespace: re.sub('\s+', ' ', text).strip(). Extract dates: re.findall(r'\d{1,2}[/-]\d{1,2}[/-]\d{2,4}', text).

Named capture groups: (?P<name>pattern) creates a named group. re.search(r'(?P<year>\d{4})-(?P<month>\d{2})-(?P<day>\d{2})', '2024-01-15').group('year') returns '2024'. Named groups make complex extractions readable. Use in pandas: df['text'].str.extract(r'(?P<city>[A-Z][a-z]+), (?P<state>[A-Z]{2})') extracts city and state into separate columns.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import re
# Text cleaning for NLP
def clean_text(text):
    text = re.sub(r'<[^>]+>', '', text)          # remove HTML tags

    text = re.sub(r'\s+', ' ', text).strip()     # normalize whitespace
    return text.lower()

prices = price_pattern.findall(text)  # ['1,299.99']
dates = date_pattern.findall(text)    # [('12', '31', '2024')]
# Pandas string methods with regex
import pandas as pd
df['price_clean'] = df['price_str'].str.replace(r'[\$,]', '', regex=True).astype(float)

Why this matters in AI
Regular expressions are the fastest way to extract structured information from messy text data. Feature extraction from product descriptions, parsing addresses, cleaning HTML-laden text — all are much faster with regex than with string methods. Every data scientist who works with text needs regex skills.
Common mistakes
  • Not using raw strings for regex patterns — r'\d+' is a raw string. Without r, '\d+' becomes the two-character string 'd+' (\d becomes just d). Always use r'' for regex patterns.
  • Greedy vs lazy matching — .* is greedy: it matches as much as possible. .*? is lazy: matches as little as possible. '<.*>' matches the entire '<b>text</b>' string; '<.*?>' matches just '<b>'. Use .*? for HTML/XML patterns.
  • Using regex when string methods suffice — ',' in text is faster than re.search(',', text). Reserve regex for patterns that string methods cannot express (variable-length patterns, alternation).

APIs with requests

Detailed explanation

The `requests` library is the standard Python tool for making HTTP API calls. Most AI APIs (OpenAI, Anthropic, HuggingFace, weather, financial data) are accessible via HTTP REST endpoints. `requests` handles the low-level HTTP details: building headers, encoding request bodies, handling responses, and managing sessions.

HTTP verbs and when to use them: GET retrieves data (idempotent — calling multiple times has same result). POST creates new resources or sends data. PUT replaces an existing resource. PATCH partially updates. DELETE removes. Most ML data APIs use GET (fetch data by ID) and POST (submit analysis jobs).

Authentication: most APIs require authentication. Bearer token: `headers = {'Authorization': f'Bearer {api_key}'}`. API key in URL parameters: `params = {'api_key': api_key}`. Basic auth: `requests.get(url, auth=(username, password))`. Never hardcode API keys in code — load from environment variables: `api_key = os.environ['OPENAI_API_KEY']`.

Error handling: always check the response status code. `response.raise_for_status()` raises an exception for 4xx/5xx codes. Implement retry logic for transient failures (network timeouts, 429 rate limit, 503 service unavailable) using exponential backoff. The `tenacity` library provides clean retry decorators.

Pagination: APIs that return many results use pagination. Common patterns: cursor-based (next_cursor in response, pass as parameter), offset-based (page=1, page=2, ...), and link-based (response includes next page URL in Link header). Always implement pagination loops — relying on default page size (often 20-100 items) silently truncates results.

import time
import requests
# Retry with exponential backoff
def api_call_with_retry(url, max_retries=3, **kwargs):
    for attempt in range(max_retries):
        try:
            response = requests.get(url, timeout=10, **kwargs)
            if response.status_code == 429:  # rate limited
                wait = int(response.headers.get('Retry-After', 2 ** attempt))
                print(f'Rate limited. Waiting {wait}s...'); time.sleep(wait)
                continue
            response.raise_for_status()
            return response.json()
        except requests.exceptions.Timeout:
            if attempt == max_retries - 1: raise
            time.sleep(2 ** attempt)
    raise RuntimeError(f'Failed after {max_retries} retries')
# Pagination
def fetch_all_pages(base_url, headers):
    results, page = [], 1
    while True:
        data = api_call_with_retry(base_url, headers=headers, params={'page': page, 'per_page': 100})
        if not data: break
        results.extend(data)
        if len(data) < 100: break  # last page
        page += 1
    return results
Why this matters in AI
ML practitioners constantly work with APIs: fetching training data, calling external ML services, and deploying models as APIs. Robust API handling — authentication, error handling, retries, pagination — determines whether your data collection pipeline runs reliably overnight or requires constant babysitting.
Common mistakes
  • Not handling rate limits — most APIs limit requests per minute/second. Without retry logic for 429 responses, the script fails immediately when hitting the rate limit. Always implement retry with exponential backoff.
  • Hardcoding API keys in code — API keys committed to git are exposed publicly. Always use environment variables: `os.environ['API_KEY']` or `.env` files with python-dotenv.
  • Not using sessions for multiple requests to the same host — each `requests.get()` opens and closes a new TCP connection. For many requests to the same server, use `requests.Session()` which reuses connections (connection pooling) — can be 3-10x faster.

Exception handling

Detailed explanation

Exception handling with try/except/finally allows programs to gracefully respond to errors rather than crashing. In production ML systems, exceptions occur constantly: API calls time out, files are missing, model inputs have unexpected shapes, external services fail. Without handling, one bad input crashes the entire service. With handling, bad inputs are logged, and the service continues.

Python exception hierarchy: all exceptions inherit from BaseException. The most important subclass is Exception, the base for user-facing errors. Common exceptions: ValueError (wrong value type or range), TypeError (wrong argument type), KeyError (missing dict key), IndexError (list index out of range), FileNotFoundError (file doesn't exist), RuntimeError (generic runtime error), torch.cuda.OutOfMemoryError (GPU OOM).

try/except/else/finally: the try block runs the code. If an exception occurs, the matching except block runs. The else block runs if no exception was raised. The finally block always runs (useful for cleanup: closing files, releasing GPU memory). Catch specific exceptions, not bare except — catching everything hides bugs and makes debugging impossible.

Raising exceptions: raise ValueError(f'Expected input shape (N, 784), got {x.shape}'). Custom exceptions: class InvalidPredictionError(Exception): pass. Raise with context: raise RuntimeError('Training failed') from original_error — this preserves the original traceback in the error chain.

Exception logging: always log exceptions with traceback, not just the error message. import logging; logging.exception('Prediction failed') logs the full traceback automatically. In a web serving context, unhandled exceptions should return HTTP 500 with a structured error response, while the full traceback goes to the server logs for debugging.

Why this matters in AI
Production ML services handle millions of requests. Even 0.01% bad inputs means thousands of exceptions per day. Proper exception handling makes services resilient, logs actionable information for debugging, and prevents cascading failures.
Common mistakes
  • Using bare except: — this catches everything including KeyboardInterrupt and SystemExit, making it impossible to Ctrl+C the program. Always catch specific exceptions.
  • Silently swallowing exceptions — except Exception: pass hides bugs. At minimum, log the exception.
  • Not including the original exception when re-raising — raise NewError('message') loses the original traceback. Use raise NewError('message') from original_error.

Logging

Detailed explanation

Python's logging module provides a production-grade logging system that is far superior to print() for debugging and monitoring ML code. Unlike print(), logging lets you: filter by severity level (DEBUG, INFO, WARNING, ERROR, CRITICAL), route logs to files and console simultaneously, include timestamps and line numbers automatically, and disable verbose logging without removing the logging calls.

Log levels: DEBUG (detailed diagnostic information — variable values, loop iterations), INFO (high-level confirmation that things are working — training started, epoch completed, model saved), WARNING (something unexpected but not fatal — missing features, class imbalance), ERROR (a serious problem — file not found, API timeout), CRITICAL (the program cannot continue — database connection failed). In production, set level to INFO or WARNING; for debugging, set to DEBUG.

Logger hierarchy: loggers are named objects. logging.getLogger('my_module') creates a logger named 'my_module'. logging.getLogger('my_module.submodule') creates a child logger. Log levels propagate up the hierarchy. In ML projects, use a logger named after the module: logger = logging.getLogger(__name__). Configure the root logger once in main.py; all module loggers inherit the configuration.

Structured logging for ML pipelines: use structured formats (JSON) for logs that will be processed by log aggregation systems (Elasticsearch, Splunk, Datadog). Log important events as structured dictionaries: {'event': 'epoch_completed', 'epoch': 5, 'train_loss': 0.234, 'val_loss': 0.267, 'elapsed_s': 45.2}. This enables searching and aggregating over log events.

Logging in long-running ML jobs: log at the start of each major phase (data loading, preprocessing, training, evaluation), log progress every N steps (not every step — avoid overwhelming log files), log final results and metrics, log any warnings or anomalies (NaN values, unexpected class distributions). Use tqdm for visual progress bars alongside logging for audit-trail logs.

import logging
import json
from datetime import datetime
# Configure logging once at program start
def setup_logging(log_file='training.log', level=logging.INFO):
    logger = logging.getLogger()
    logger.setLevel(level)
    formatter = logging.Formatter('%(asctime)s [%(levelname)s] %(name)s: %(message)s')
    # File handler
    fh = logging.FileHandler(log_file)
    fh.setFormatter(formatter)
    logger.addHandler(fh)
    # Console handler
    ch = logging.StreamHandler()
    ch.setFormatter(formatter)
    logger.addHandler(ch)
    return logging.getLogger(__name__)
logger = setup_logging()
logger.info('Training started')
logger.debug('Feature matrix shape: %s', X_train.shape)  # %s lazy formatting
logger.warning('Class imbalance: positive ratio = %.3f', y_train.mean())
# Structured logging for ML
class StructuredLogger:
    def __init__(self, name):
        self.logger = logging.getLogger(name)
    def log_event(self, event, **kwargs):
        entry = {'timestamp': datetime.utcnow().isoformat(), 'event': event, **kwargs}
        self.logger.info(json.dumps(entry))
ml_logger = StructuredLogger('training')
for epoch in range(50):
    ml_logger.log_event('epoch_completed', epoch=epoch, train_loss=0.3/(epoch+1), val_loss=0.35/(epoch+1))
Why this matters in AI
Logging is the most underused debugging and monitoring tool in ML projects. When a training job runs for 12 hours and then fails, print() statements are gone but log files persist. When a deployed model starts returning unexpected predictions, structured logs let you trace exactly what input was processed and what the model returned.
Common mistakes
  • Using print() instead of logging in production code — print() has no timestamps, no levels, no filtering, and goes to stdout only. Switching from print() to logging.info() is a 2-minute change that pays dividends forever.
  • Logging inside tight loops at DEBUG level without checking — logging every iteration of a 1M-row preprocessing loop creates massive log files and slows execution. Check: 'if loop_iter % 10000 == 0: logger.debug(...)'
  • Configuring logging in module files — configure logging once at the entry point (main.py or __main__). Configuring in module files causes duplicate handlers if the module is imported multiple times.

Typing basics

Detailed explanation

Python type hints (introduced in Python 3.5+) are annotations that specify the expected types of function arguments and return values. They are optional and not enforced at runtime — Python ignores them during execution. However, type hints enable: static analysis with tools like mypy and Pylance (catch type errors before running the code), better IDE IntelliSense (VS Code can suggest methods for the correct type), and self-documenting code (types communicate intent as clearly as comments).

Basic type hints: def add(x: int, y: int) -> int: means add takes two integers and returns an integer. def process(data: pd.DataFrame, n_features: int = 10) -> np.ndarray: specifies more complex types. Optional[str] means the argument can be a string or None (equivalent to Union[str, None]). Use from typing import Optional, List, Dict, Tuple, Union for complex types.

Type hints for ML functions: def train_model(X: np.ndarray, y: np.ndarray, config: dict) -> Any. For sklearn-like typing: def fit(self, X: 'ArrayLike', y: 'ArrayLike') -> 'Self'. TypeVar for generic functions: T = TypeVar('T'); def repeat(item: T, n: int) -> list[T]. Python 3.9+ allows list[int] and dict[str, float] without importing from typing.

Dataclasses with type hints: @dataclass is the most useful application of type hints for ML. from dataclasses import dataclass; @dataclass class ModelConfig: n_estimators: int = 100; learning_rate: float = 0.01; max_depth: int = 6. Creates a class with __init__, __repr__, __eq__ automatically based on the typed fields. Replacing dict configs with typed dataclasses catches mistakes ('learnig_rate' typo → AttributeError instead of silently using the wrong key).

Running mypy: mypy script.py runs static type checking. mypy --strict script.py is the strictest mode, requiring all functions to have type annotations. In practice, annotate all function signatures and the most important variables. Full type coverage is a goal for large production codebases; for scripts and notebooks, annotate only public APIs.

from typing import Optional, Union, List, Dict, Tuple
import numpy as np
import pandas as pd
from dataclasses import dataclass, field
# Typed function signatures
def preprocess(df: pd.DataFrame, target_col: str, categorical_cols: Optional[List[str]] = None) -> Tuple[pd.DataFrame, pd.Series]:
    X = df.drop(columns=[target_col])
    y = df[target_col]
    return X, y

@dataclass

    model_type: str = 'xgboost'
    n_estimators: int = 500
    learning_rate: float = 0.05

Why this matters in AI
Type hints transform Python's dynamic typing from a liability into an asset: you get the flexibility of dynamic typing while catching a whole class of bugs statically. For ML code that processes tensors and arrays with specific shapes and dtypes, type hints communicate constraints that are otherwise only discovered at runtime.
Common mistakes
  • Using Any everywhere — from typing import Any; def process(data: Any) -> Any: defeats the purpose of type hints. Use specific types, even if they're Union or Optional.
  • Not using field() for mutable default arguments in dataclasses — @dataclass with features: list = [] shares the SAME list across all instances (a Python gotcha). Use features: List[str] = field(default_factory=list) instead.
  • Conflating type hints with runtime validation — type hints are not enforced at runtime. def add(x: int, y: int): add('a', 'b') runs without error. For runtime validation, use pydantic's BaseModel, which validates types when instances are created.

Clean code

Detailed explanation

Clean code in the context of ML means code that is readable, maintainable, and testable. ML code is especially prone to becoming messy: analysis code from Jupyter notebooks gets copy-pasted into production, model training scripts grow to 500 lines without functions, and magic numbers appear everywhere. Clean code practices prevent this decay.

Functions and single responsibility: every function should do exactly one thing and be named after that thing. `process_data()` is a bad function name — it's too vague. `normalize_features()`, `encode_categorical_columns()`, `remove_outliers()` are clear. A function longer than 20-30 lines is usually doing multiple things — split it. Functions with more than 4-5 parameters need a refactor — group related parameters into a dataclass or dict.

Naming conventions: variable names should describe their content. `df` is acceptable for a temporary variable; use `customers_df` or `train_features_df` for variables that persist. Avoid: `x`, `data`, `temp`, `result`, `output` without context. In ML: use descriptive names: `X_train_scaled` (not `X`), `model_predictions` (not `preds`), `precision_at_10` (not `p10`).

Don't repeat yourself (DRY): if you write the same code twice, extract it into a function. If you preprocess train data with 15 lines of code and then repeat those 15 lines for test data, you've created a maintenance nightmare (changing one often forgets the other). The sklearn Pipeline class is the ML solution to DRY: wrap all preprocessing steps so the same transformations apply consistently to any data split.

Code reviews and tests: write tests for your core logic — not test.py with a single assert, but meaningful unit tests for your feature engineering functions (given this input, expect this output). In ML, test edge cases: what happens with a single sample? An empty DataFrame? A DataFrame with missing values? Tests catch regressions when you refactor.

# Bad code example → clean code example
# BAD: one massive function, magic numbers, bad names
def process(df):
    df2 = df.copy()
    df2 = df2[df2['a'] < 99999]  # magic number
    df2['b'] = (df2['b'] - df2['b'].mean()) / df2['b'].std()
    df2['c'] = df2['c'].fillna('unknown')
    return df2

# GOOD: each step is named and documented
MAX_SALARY = 99_999  # Named constant, not magic number
def remove_outliers(df: 'pd.DataFrame', salary_col: str = 'salary') -> 'pd.DataFrame':
    return df[df[salary_col] < MAX_SALARY].copy()
def standardize_numerical(df: 'pd.DataFrame', col: str) -> 'pd.DataFrame':
    mean, std = df[col].mean(), df[col].std()
    df = df.copy(); df[col] = (df[col] - mean) / std
    return df
def fill_categorical_missing(df: 'pd.DataFrame', col: str) -> 'pd.DataFrame':
    return df.assign(**{col: df[col].fillna('unknown')})
def preprocess(df: 'pd.DataFrame') -> 'pd.DataFrame':
    df = remove_outliers(df)
    df = standardize_numerical(df, 'age')
    df = fill_categorical_missing(df, 'department')
    return df
Why this matters in AI
Clean code in ML is not just aesthetics — it's how you catch bugs. Messy data preprocessing code is the leading cause of training-serving skew (the model performs well in experiments but fails in production because the preprocessing was slightly different). When code is clean, each transformation is visible, testable, and consistent across environments.
Common mistakes
  • Jupyter notebook code going directly to production — notebook code works great for exploration but is terrible for production: no functions, no tests, no type hints, global state everywhere. Refactor to proper Python modules before deploying.
  • Not using type hints — `def train_model(X, y, params)` gives no information about expected types. `def train_model(X: np.ndarray, y: np.ndarray, params: Dict[str, Any]) -> sklearn.base.BaseEstimator` documents the contract. Type hints also enable static analysis with mypy.
  • Ignoring code style — inconsistent indentation, random spacing, and inconsistent naming make code harder to read. Use `black` for automatic formatting and `flake8` or `ruff` for linting: `pip install black ruff && black . && ruff check .`

Basic time and space complexity

Detailed explanation

Time complexity describes how the execution time of an algorithm grows as input size increases. Space complexity describes how memory usage grows. Understanding these concepts explains why some ML operations are fast and others are slow — and helps you choose the right algorithm for the job.

Big-O notation: O(1) — constant time (access a dict by key, regardless of dict size). O(log n) — logarithmic (binary search: each step halves the search space). O(n) — linear (iterating over a list: double the size, double the time). O(n log n) — typical sort algorithms. O(n²) — nested loops (comparing each element to every other element). O(2^n) — exponential (brute force all subsets).

ML algorithm complexities: training linear regression — O(n × p²) where n=samples, p=features (due to matrix operations). Training K-nearest neighbors — training O(1), prediction O(n × p) per query (must check every training point). Training decision tree — O(n × p × log n). Training naive Bayes — O(n × p). Inference is typically much faster than training for neural networks but can be slow for tree ensembles with many trees.

Space complexity in ML: storing a model takes memory proportional to its number of parameters. A logistic regression on p features: O(p) weights. A decision tree with n_nodes: O(n_nodes). A neural network with L layers: O(sum of layer sizes). For inference with batch size B: O(B × model_size). This determines whether a model can fit on GPU VRAM.

Practical implications: pandas `.apply()` on a column is O(n) but with Python-level overhead — 10-100x slower than vectorized numpy operations. SQL GROUP BY is O(n log n) (sorting). A full-table scan is O(n). KNN on 1 million points takes 1 million distance computations per query — use approximate nearest neighbor (FAISS) which reduces to O(log n) with some accuracy tradeoff.

import time
import numpy as np
# Demonstrate O(n) vs O(n^2)
def linear_search(lst, target):
    for i, x in enumerate(lst):
        if x == target: return i
    return -1
def has_duplicate_n_squared(lst):
    for i in range(len(lst)):
        for j in range(i+1, len(lst)):
            if lst[i] == lst[j]: return True
    return False
def has_duplicate_linear(lst):
    seen = set()
    for x in lst:
        if x in seen: return True
        seen.add(x)
    return False
# Measure performance
for n in [1000, 10000, 100000]:
    data = list(range(n))
    t0 = time.time(); has_duplicate_n_squared(data[:1000]); print(f'O(n^2) n={1000}: {time.time()-t0:.4f}s')
    t0 = time.time(); has_duplicate_linear(data); print(f'O(n) n={n}: {time.time()-t0:.6f}s')
Why this matters in AI
Time and space complexity determine whether your algorithm can run on real data at all. A working KNN classifier that takes 10ms per query on 1,000 training examples takes 10 SECONDS per query on 1,000,000 training examples. Understanding complexity tells you when to switch from KNN to approximate nearest neighbor, from full-table scan to indexed lookup, from pandas to Spark.
Common mistakes
  • Not considering the constant factors — O(n²) in C++ can be faster than O(n) in Python for small n. The Big-O hides constant factors. Profile your actual code on real data sizes before optimizing.
  • Confusing training and inference complexity — random forests are fast to train relative to neural networks, but inference requires running the input through all N trees. For large N and many predictions, this matters. Know both training and inference complexities.
  • Using pandas .apply() when vectorized operations are available — .apply() is O(n) but calls Python for each row. Vectorized pandas/numpy operations do the same computation but at C speed, often 10-100x faster.
