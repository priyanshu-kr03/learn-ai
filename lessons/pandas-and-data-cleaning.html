Pandas DataFrame

Detailed explanation

Pandas is Python's primary library for tabular data — spreadsheet-like data with rows and columns. Its core data structure is the DataFrame: a 2D table where each column can have a different data type (string, int, float, datetime, bool). DataFrames are the starting point for virtually every data science and ML project: load your CSV, explore the data, clean it, engineer features, and export to NumPy arrays for model training.

A DataFrame is indexed by both rows (the index, default 0..N-1) and columns (column names). You access a single column with df['column_name'] (returns a Series). You access multiple columns with df[['col1', 'col2']] (returns a DataFrame). You access rows by label with df.loc[row_label] or by position with df.iloc[row_position]. Boolean indexing df[df['age'] > 30] filters rows based on a condition.

Essential DataFrame operations: df.head(n) shows the first n rows. df.info() shows column names, non-null counts, and dtypes. df.describe() gives summary statistics (mean, std, quartiles) for numeric columns. df.shape gives (num_rows, num_cols). df.dtypes shows the dtype of each column. These 5 commands are the first thing you run on any new dataset.

Modifying DataFrames: df['new_col'] = expression creates a new column. df.drop('col', axis=1) removes a column (axis=1 means column; axis=0 means row). df.rename(columns={'old': 'new'}) renames columns. df.sort_values('col', ascending=False) sorts rows. df.reset_index(drop=True) resets the row index after filtering. Always use inplace=False (the default) and reassign: df = df.drop(...) rather than relying on inplace=True.

The pandas-scikit-learn bridge: after cleaning and engineering features in pandas, convert to NumPy with df.values or df.to_numpy() for scikit-learn. Alternatively, many modern sklearn functions accept DataFrames directly. Keep feature names as column names throughout the pandas pipeline so that feature importance and error analysis remain interpretable.

Why this matters in AI
DataFrames are the universal interface for tabular data in Python. Every ML project that touches structured data (CSVs, SQL exports, Excel files) starts with pandas. Fluency with DataFrame operations is prerequisite to any data science or ML work.
Common mistakes
  • Chaining operations without assignment: df.dropna().fillna(0) — the result is lost if not assigned back to df.
  • Mixing .loc and .iloc — .loc uses label-based indexing; .iloc uses position-based. After filtering and resetting index, label and position may not match.
  • Setting values on a slice: df[df['age'] > 30]['salary'] = 0 — this sets values on a copy, not the original. Use df.loc[df['age'] > 30, 'salary'] = 0.

Series

Detailed explanation

A pandas Series is a one-dimensional labeled array. Like a NumPy array but with an associated index (labels for each element). s = pd.Series([1.5, 2.3, 0.7], index=['a', 'b', 'c']). Access by label: s['b'] → 2.3. Access by position: s.iloc[1] → 2.3. A DataFrame is a collection of Series sharing an index — each column is a Series.

Creating a Series: from a list (pd.Series([1,2,3])), from a dict (pd.Series({'age': 25, 'income': 50000})), from a NumPy array (pd.Series(np.random.randn(100))). The index defaults to 0, 1, 2, ... but can be any labels: dates, strings, or integers.

Series operations: vectorized math (s * 2, s + 10, s ** 2), aggregations (s.sum(), s.mean(), s.std(), s.min(), s.max(), s.median()), string operations via s.str (s.str.lower(), s.str.contains('pattern')), datetime operations via s.dt (s.dt.year, s.dt.dayofweek).

Alignment: pandas automatically aligns Series on their index before operations. s1 = pd.Series([1,2,3], index=['a','b','c']); s2 = pd.Series([4,5,6], index=['b','c','d']); s1 + s2 → a=NaN, b=6, c=8, d=NaN. This index-alignment prevents common bugs when combining data from different sources — mismatched rows are explicitly NaN rather than silently misaligned.

Series in ML context: a column of a DataFrame (df['age']) is a Series. Model predictions (pd.Series(model.predict(X_test), index=test_df.index)) can be joined back to the test DataFrame using the shared index. Series.value_counts() is the fastest way to examine class distribution in a target column.

Why this matters in AI
pandas Series is the basic unit of tabular data in Python ML. Every DataFrame column is a Series. Understanding Series operations — especially alignment, string operations, and datetime operations — is prerequisite to working with real-world datasets.
Common mistakes
  • Using .ix[] (deprecated) — use .loc[] for label-based access or .iloc[] for position-based access.
  • Chained assignment (df['col'][mask] = value) — this may not modify the original DataFrame. Use df.loc[mask, 'col'] = value instead.
  • Ignoring index alignment — if two Series have different indices, operations produce NaN for non-matching labels. Always check that indices align before combining.

Reading CSV Excel JSON

Detailed explanation

Reading data into a pandas DataFrame is the first step in any ML project. pd.read_csv('data.csv') reads a CSV file — the most common tabular data format. Key parameters: sep='\t' for tab-separated, header=0 for first row as column names, index_col='id' uses a column as the row index, dtype={'age': int, 'price': float} enforces column types, na_values=['N/A', '-', 'NULL'] treats these strings as NaN, parse_dates=['date'] parses date columns, nrows=1000 reads only the first 1000 rows (for quick testing).

pd.read_excel('data.xlsx') reads Excel files. sheet_name='Sheet1' or sheet_name=0 for the first sheet. Reading multiple sheets: pd.read_excel(path, sheet_name=None) returns a dict of DataFrames. Excel files often contain formatting, merged cells, and blank rows — use skiprows and header carefully. xlrd (for .xls) and openpyxl (for .xlsx) are the backend libraries.

pd.read_json('data.json') handles JSON files. orient parameter specifies the JSON structure: 'records' (list of dicts), 'split', 'index', 'columns'. For nested JSON (APIs often return nested structures), normalize with pd.json_normalize(data, record_path=['items'], meta=['id', 'name']). This handles multiple levels of nesting into a flat DataFrame.

Reading from databases: pd.read_sql('SELECT * FROM table WHERE date > 2024-01-01', connection) reads SQL query results directly into a DataFrame. Use with SQLAlchemy for multiple database types (PostgreSQL, MySQL, SQLite). This is the standard way to get training data from production databases.

Chunked reading for large files: pd.read_csv(path, chunksize=10000) returns an iterator of DataFrames of 10000 rows each. Process each chunk without loading the full file into memory: for chunk in pd.read_csv(path, chunksize=10000): process(chunk). This is essential for files larger than available RAM.

Why this matters in AI
Data rarely comes in a perfect format. Real datasets are CSVs with messy headers, Excel files with merged cells, JSON from REST APIs, or SQL tables. Knowing how to read each format cleanly — handling missing values, dates, and dtypes correctly — prevents downstream preprocessing errors.
Common mistakes
  • Not specifying parse_dates for date columns — pandas reads dates as strings by default. Specify parse_dates=['date_col'] to get datetime dtype, enabling date arithmetic.
  • Not specifying dtypes — pandas infers dtypes, sometimes incorrectly. A column of integers with one missing value becomes float64. Specify dtype explicitly for critical columns.
  • Reading the entire file when you need only a sample — pd.read_csv(path, nrows=1000) for quick exploration. Don't load 10GB into memory just to check the schema.

Inspecting data

Detailed explanation

Data inspection is the first step of any ML project — before training any model, you must understand what your data looks like, what's in it, what's missing, and what's wrong. Skipping data inspection is the leading cause of wasted ML experiments: you spend weeks training models on data with subtle bugs that invalidate all results.

Shape and types: `df.shape` (rows, columns), `df.dtypes` (dtype per column — int64, float64, object, bool, datetime), `df.info()` (dtype + non-null count per column), `df.memory_usage(deep=True)` (memory usage in bytes per column). These catch the most common early bugs: columns read as object when they should be numeric, or a datetime column stored as string.

Summary statistics: `df.describe()` for numerical columns shows count, mean, std, min, 25th, median, 75th percentile, max. For categorical: `df.describe(include='object')` shows count, unique, top (mode), freq. Anomalies to look for: min=0 when 0 is invalid, max values that are far from the 75th percentile (outliers), std close to 0 (constant feature with low variance).

Missing values: `df.isnull().sum()` counts NaN per column. `df.isnull().mean()` gives fraction missing. Visualize with: `import missingno as msno; msno.matrix(df)` — shows patterns of missingness. Critical: not all missing values appear as NaN — check for sentinel values (99, -1, 999, 'N/A', 'none', 'unknown') that indicate missingness.

Value distributions: `df['col'].value_counts()` for categorical (shows class imbalance immediately). `df['col'].hist()` for numerical (shows shape: normal, skewed, bimodal). Cross-tabulations: `pd.crosstab(df['col1'], df['col2'])` shows joint distributions. These catch: extreme class imbalance (1000:1 ratio), non-numeric outliers encoded as strings, categories with very few samples.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import pandas as pd
import numpy as np
# Comprehensive data inspection function
def inspect_dataframe(df, target_col=None):
    print('=== Shape =='); print(df.shape)
    print('\n=== Dtypes ==='); print(df.dtypes)
    print('\n=== Missing values ==='); print(df.isnull().sum()[df.isnull().sum() > 0])
    print('\n=== Numerical summary ==='); print(df.describe())
    print('\n=== Duplicate rows ==='); print(df.duplicated().sum())
    # Check for hidden sentinels
    for col in df.select_dtypes(include='number').columns:
        pct_zero = (df[col] == 0).mean()
        if pct_zero > 0.3: print(f'WARNING: {col} has {pct_zero:.1%} zeros — potential sentinel')
        pct_neg = (df[col] < 0).mean()
        if pct_neg > 0: print(f'INFO: {col} has {pct_neg:.1%} negative values')
    # Value counts for categoricals
    for col in df.select_dtypes(include='object').columns:
        n_unique = df[col].nunique()
        if n_unique < 50: print(f'\n{col} ({n_unique} unique):'); print(df[col].value_counts().head())
    # Target distribution
    if target_col and target_col in df.columns:
        print(f'\n=== Target distribution ({target_col}) ==='); print(df[target_col].value_counts(normalize=True))
inspect_dataframe(df, target_col='label')
Why this matters in AI
Data inspection catches bugs that cannot be caught later. If a column has 40% zeros because of a data pipeline bug, your model will learn to predict 'zero case' separately — but you won't know why. Inspection finds this before training. The hour spent inspecting data saves days of debugging unexpected model behavior.
Common mistakes
  • Relying on df.describe() alone — describe() only shows numerical columns and only their first and last quartile. It misses: string columns with numeric values, datetime columns stored as objects, columns with high cardinality categoricals.
  • Not checking for duplicate rows — ML models trained on duplicate rows overfit to those specific examples. `df.duplicated().sum()` takes one line — always check.
  • Treating missing values as a single problem — MCAR (missing completely at random) can be safely imputed. MNAR (missing not at random) requires domain understanding. For example, income=NaN might mean income is very high (people refuse to disclose). Imputing with the mean would introduce significant bias.

Missing values

Detailed explanation

Missing values (NaN — Not a Number, or None) appear in almost every real-world dataset. They arise from: data collection failures (sensor outage, form field skipped), data merging (joining tables where some rows have no match), data truncation (values beyond a range set to missing), or truly unknown information (a field that did not apply to that record). Ignoring missing values causes most ML libraries to crash or silently produce wrong results.

Before handling missing values, understand WHY they are missing. Missing Completely At Random (MCAR): missingness has no relationship to any variable — randomly corrupted sensor readings. Missing At Random (MAR): missingness depends on observed variables — richer customers skip income question. Missing Not At Random (MNAR): missingness depends on the value itself — people with very high income are more likely to hide income. MCAR and MAR can be handled with imputation; MNAR is harder and may require modeling the missingness as a feature.

Imputation strategies: for numeric columns, impute with median (robust to outliers) or mean (sensitive to outliers). For categorical columns, impute with mode or a new 'Unknown' category. For time series, forward-fill (propagate last valid value) or interpolation. KNN imputation uses the values of the k most similar rows (measured by non-missing features) to impute missing values — more accurate but slower.

Adding a missingness indicator column alongside imputation is often valuable: df['age_was_missing'] = df['age'].isnull().astype(int); df['age'] = df['age'].fillna(df['age'].median()). This lets the model learn different patterns for rows where age was originally missing vs. rows with observed ages. The missingness itself can be predictive of the target.

Never impute before splitting train and test — fit the imputer on training data only, then transform both. If you fit on all data, you leak information from the test set into the imputer's learned statistics (mean, median, k-nearest neighbors). Always use sklearn's Pipeline or fit_transform on train followed by transform on test.

Why this matters in AI
Missing value handling is one of the most impactful preprocessing decisions. Wrong handling (dropping rows, leaking test statistics) causes data loss and evaluation inflation. Correct handling requires understanding why values are missing and using the pipeline pattern to prevent leakage.
Common mistakes
  • Dropping all rows with any missing value — if 40% of rows have at least one missing value, this destroys your dataset. Impute strategically.
  • Fitting the imputer on all data before train/test split — leaks test set statistics into training. Always fit only on training data.
  • Using mean imputation for skewed distributions — an outlier-heavy income column's mean is pulled by outliers. Use median for skewed numeric features.

Duplicates

Detailed explanation

Duplicate data occurs when the same observation appears multiple times in the dataset. Types: exact duplicates (all columns identical), near-duplicates (most columns identical, minor differences), and key duplicates (same ID but different feature values — data inconsistency). Duplicates in training data cause: over-representation bias (duplicated samples have more weight in training), inflated evaluation metrics (if duplicates exist in both train and test splits), and increased training time.

Detecting duplicates: df.duplicated() returns a boolean series — True for rows that are exact duplicates of a previous row. df.duplicated().sum() counts duplicates. df[df.duplicated()] shows all duplicate rows. df.duplicated(subset=['user_id', 'date']) checks duplicates on specific columns (for time-series data: a user can appear multiple times but not on the same date). df.duplicated(keep=False) marks ALL occurrences of duplicates, not just the second.

Removing duplicates: df.drop_duplicates() removes all but the first occurrence. df.drop_duplicates(keep='last') keeps the last occurrence. df.drop_duplicates(subset=['user_id']) drops duplicate user_id rows (keeps first). df.drop_duplicates(keep=False) removes ALL copies of any row that has duplicates. After dropping: reset_index(drop=True) resets the row index.

Near-duplicate detection: exact matching misses near-duplicates (same content, slightly different formatting). For text data: use MinHash or TF-IDF cosine similarity to find near-duplicate documents. For tabular data: Levenshtein distance for string columns, fuzzy matching with fuzzywuzzy. For images: perceptual hash (dhash, phash) detects visually identical images even with minor resizing or watermarks.

Handling duplicates in train/test splits: even if you remove exact duplicates from the dataset, if you split AFTER feature engineering, your feature engineering may create identical rows for different original samples. The safest approach: deduplicate BEFORE feature engineering, use stratified train_test_split, and check if any test rows appear in the training set (especially critical for NLP datasets where the same sentence might appear in multiple examples).

Why this matters in AI
Duplicate data is extremely common in real-world datasets — from data collection bugs, system errors, or joining multiple data sources. Blindly training on duplicated data can inflate performance metrics, waste compute, and introduce subtle bias. Deduplication is a standard step in data cleaning.
Common mistakes
  • Not checking for duplicates after joining tables — SQL joins can create duplicate rows if the join key is not unique. After every join, verify the expected row count.
  • Removing ALL duplicates when near-duplicates should be kept — sometimes duplicates are legitimate (the same user placing two identical orders). Understand WHY duplicates exist before removing them.
  • Not deduplicating before train/test split — if you split first, duplicates may appear in both train and test sets, inflating test performance. Always deduplicate before splitting.

Filtering sorting

Detailed explanation

Filtering selects rows from a DataFrame based on conditions. The standard method: boolean indexing with df[condition]. Examples: df[df['age'] > 18] (rows where age > 18), df[df['status'] == 'active'] (string match), df[df['score'].between(0.5, 0.9)] (range), df[df['category'].isin(['A', 'B', 'C'])] (set membership). Combine conditions: df[(df['age'] > 18) & (df['income'] > 30000)] (AND, use & not 'and'), df[(df['age'] < 18) | (df['age'] > 65)] (OR, use | not 'or').

df.query() provides a more readable string-based filtering syntax: df.query('age > 18 and income > 30000'). It handles variable injection: threshold = 30000; df.query('income > @threshold'). Query is more readable for complex conditions but slightly slower than boolean indexing for large DataFrames.

Sorting: df.sort_values('column') sorts ascending. df.sort_values(['col1', 'col2'], ascending=[True, False]) sorts by multiple columns. df.sort_index() sorts by index. Sorting is essential for: time-series data (sort by date before any analysis), ranking (sort by score to find top-K), and viewing the most/least extreme values in a column.

nlargest and nsmallest: df.nlargest(10, 'revenue') returns the 10 rows with highest revenue — much faster than sorting the full DataFrame and taking the head. Similarly df.nsmallest(10, 'error_count') finds the least error-prone records.

Filtering with loc and iloc: df.loc[mask, 'column'] selects specific rows and columns simultaneously. df.iloc[0:100, [0, 3, 5]] selects the first 100 rows and columns at positions 0, 3, 5. This combined row/column selection is the standard way to extract a subset of a DataFrame for analysis or modeling.

Why this matters in AI
Filtering and sorting are the most basic data manipulation operations. Every EDA, feature engineering, and evaluation step requires selecting specific subsets of data. Writing clean, efficient boolean conditions prevents data leakage and ensures correct model evaluation.
Common mistakes
  • Using 'and'/'or' instead of '&'/'|' for boolean indexing — df[df['a'] > 0 and df['b'] > 0] raises a ValueError. Use & and | with parentheses: df[(df['a'] > 0) & (df['b'] > 0)].
  • Filtering the target variable from training data — never filter training samples based on the target variable (e.g., remove difficult cases). This biases the model.
  • Not resetting the index after filtering — df.loc[mask].index may have gaps (e.g., [0, 3, 7, 15]). Use df[mask].reset_index(drop=True) for a clean 0-based index.

GroupBy and merge

Detailed explanation

GroupBy performs split-apply-combine: split a DataFrame into groups based on a key column, apply a function to each group, combine the results. df.groupby('category')['revenue'].mean() computes the mean revenue per category. The groupby object is lazy — the function call triggers computation. Common aggregation functions: mean(), sum(), count(), max(), min(), std(), nunique() (unique count).

Multiple aggregations per group: df.groupby('category').agg({'revenue': ['mean', 'sum', 'count'], 'price': 'median'}). This produces a hierarchical column index. Or use named aggregation: df.groupby('category').agg(avg_revenue=('revenue', 'mean'), total_revenue=('revenue', 'sum')). The named syntax is cleaner and produces flat column names.

merge (join) combines two DataFrames based on shared keys. pd.merge(df_orders, df_customers, on='customer_id', how='left') — a left join keeps all orders and attaches customer information where available. How parameter: 'inner' (only matching keys), 'left' (all left, matched right), 'right', 'outer' (all keys). This is the SQL JOIN equivalent in pandas, used to enrich training data with features from multiple tables.

Groupby transform: df.groupby('country')['salary'].transform('mean') returns a Series with the same length as the original DataFrame where each element is replaced by its group's mean. This is used for mean-encoding (target encoding) and adding group-level features without losing the row-level granularity: df['salary_vs_country_mean'] = df['salary'] - df.groupby('country')['salary'].transform('mean').

Pivot tables: df.pivot_table(index='category', columns='region', values='revenue', aggfunc='sum') creates a 2D summary table. This is the Excel-style analysis tool in pandas — useful for EDA and business reporting. pivot_table handles missing combinations with fill_value=0.

Why this matters in AI
GroupBy and merge are how you compute aggregate features from relational data. In ML, features like 'user's average transaction amount in the last 30 days', 'product's average rating in its category', and 'customer's total orders lifetime' are all GroupBy-based features. Merge is how you join multiple data sources into a training dataset.
Common mistakes
  • Using merge instead of join when working with indexes — pd.merge is for column-based joins; df.join(df2) is for index-based joins. Using the wrong one leads to CartesianProduct-like row multiplication.
  • Not checking for row multiplication after a merge — a left join on a non-unique key multiplies rows. Always check: assert len(merged) == len(left) after a one-to-one join.
  • GroupBy without specifying sort=False when order doesn't matter — GroupBy sorts by the group key by default (slow). Use sort=False when result order doesn't matter.

Pivot tables

Detailed explanation

A pivot table summarizes and reorganizes data by aggregating values across multiple categorical dimensions. The pd.pivot_table() function reshapes a DataFrame: rows become one categorical variable, columns become another, and the cells contain an aggregation (mean, sum, count) of a third variable. Pivot tables are essential for exploratory data analysis and feature engineering.

pd.pivot_table() syntax: pd.pivot_table(df, values='revenue', index='region', columns='product_category', aggfunc='mean', fill_value=0). values: the column to aggregate. index: column(s) that become row labels. columns: column(s) that become column labels. aggfunc: aggregation function ('mean', 'sum', 'count', or a custom function). fill_value: replace NaN with this value (use 0 for counts/sums).

Multiple aggregation functions: aggfunc=['mean', 'count'] computes both the mean and count for each cell. The result has multi-level column headers. Useful for seeing both average value and sample size — small cells with few samples should be interpreted cautiously.

Pivot tables vs groupby: groupby().agg() is more flexible (multiple groupby keys, arbitrary aggregations). pivot_table is more convenient for the specific case of two categorical variables + one numeric variable. pd.pivot_table(df, ...) is equivalent to df.groupby(['region', 'product'])['revenue'].mean().unstack('product').

Pivot table for feature engineering: cross-tabulation of user behavior. A user × product_category interaction matrix where each cell = number of purchases. pd.pivot_table(purchases, values='quantity', index='user_id', columns='category', aggfunc='sum', fill_value=0) creates a user-item interaction matrix. This is the starting point for collaborative filtering and recommendation systems.

Why this matters in AI
Pivot tables are one of the most powerful EDA tools. They immediately reveal patterns: which region has the highest average order value, which product categories have the most purchases, and where are the gaps in the data. In feature engineering, pivot tables create user-level aggregate features from transaction-level data.
Common mistakes
  • Not using fill_value=0 for count/sum aggregations — missing combinations get NaN by default, which causes downstream errors in ML models. Use fill_value=0 for numeric aggregations.
  • Pivoting too many categories — a pivot table with 1000 product IDs as columns creates a sparse matrix with many NaN values. For high-cardinality columns, group into broader categories first.
  • Confusing pivot_table with pivot — df.pivot(index, columns, values) is simpler (no aggregation) and raises an error if there are duplicate (index, columns) combinations. Use pivot_table when you need aggregation.

Datetime operations

Detailed explanation

Datetime features are extremely important in ML because many patterns are time-dependent: purchase behavior peaks on weekends, fraud occurs at unusual hours, weather changes seasonally. Raw timestamps are not useful as model features — they must be decomposed into meaningful components (hour, day_of_week, month, is_weekend) and engineered into temporal features (days_since_last_purchase, days_until_renewal).

Pandas datetime basics: pd.to_datetime(series) converts strings to datetime. df['date'] = pd.to_datetime(df['date']). Then access components: df['hour'] = df['date'].dt.hour, df['day_of_week'] = df['date'].dt.dayofweek (0=Monday, 6=Sunday), df['month'] = df['date'].dt.month, df['year'] = df['date'].dt.year, df['is_weekend'] = df['date'].dt.dayofweek.isin([5, 6]).astype(int).

Time difference features: (date2 - date1) produces a Timedelta. To get days: (date2 - date1).dt.days. Example: df['days_since_signup'] = (pd.Timestamp.now() - df['signup_date']).dt.days. df['renewal_in_days'] = (df['renewal_date'] - pd.Timestamp.now()).dt.days. These time-to-event features are among the most predictive for churn, subscription renewal, and fraud.

Seasonal and cyclical encoding: raw month (1–12) has a discontinuity — month 1 (January) and month 12 (December) are numerically far but temporally close (adjacent months). Use cyclical encoding: month_sin = sin(2π × month / 12), month_cos = cos(2π × month / 12). This encodes the cyclic nature of months. Apply to: hour of day (24), day of week (7), month (12), minute (60).

Resampling and rolling windows: for time series data, resample to a fixed frequency: df.set_index('date').resample('1D').agg({'revenue': 'sum'}) computes daily total revenue. Rolling windows: df['revenue_30d_avg'] = df.set_index('date')['revenue'].rolling('30D').mean() computes 30-day moving average. Shift for lag features: df['revenue_lag1'] = df['revenue'].shift(1) — previous period's revenue.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import pandas as pd
import numpy as np
df['date'] = pd.to_datetime(df['date'])
# Datetime components
df['hour'] = df['date'].dt.hour
df['day_of_week'] = df['date'].dt.dayofweek  # 0=Mon, 6=Sun
df['month'] = df['date'].dt.month
df['year'] = df['date'].dt.year
df['is_weekend'] = df['date'].dt.dayofweek.isin([5, 6]).astype(int)
df['quarter'] = df['date'].dt.quarter
# Cyclical encoding
df['month_sin'] = np.sin(2 * np.pi * df['month'] / 12)
df['month_cos'] = np.cos(2 * np.pi * df['month'] / 12)
df['hour_sin'] = np.sin(2 * np.pi * df['hour'] / 24)
df['hour_cos'] = np.cos(2 * np.pi * df['hour'] / 24)
# Time difference features
now = pd.Timestamp.now()
df['days_since_signup'] = (now - df['signup_date']).dt.days
df['account_age_years'] = df['days_since_signup'] / 365
# Rolling features
df_ts = df.set_index('date').sort_index()
df_ts['revenue_7d'] = df_ts['revenue'].rolling('7D').mean()
df_ts['revenue_30d'] = df_ts['revenue'].rolling('30D').sum()
Why this matters in AI
Datetime features are among the most important features in many ML problems. Fraud occurs at unusual hours. Churn is predicted by recency (days since last activity). Demand forecasting requires seasonal patterns. Extracting rich features from timestamps is a high-value feature engineering skill.
Common mistakes
  • Using raw datetime values as features — a model cannot learn from raw timestamps (nanoseconds since epoch). Always decompose into meaningful components: hour, day_of_week, month.
  • Not using cyclical encoding for periodic features — hour=23 and hour=0 are adjacent in time but 23 apart numerically. Cyclical encoding (sin/cos) fixes this.
  • Computing time differences in the test set using future data — if you compute 'days since last purchase' using the full dataset, test samples 'see' future purchases. Compute time-based features using only data available at prediction time.

apply map

Detailed explanation

Pandas apply() and map() are the two main methods for applying functions element-wise or row/column-wise to pandas objects. map() is used on a Series to apply a function to each element. apply() is more flexible: on a Series it works like map(), on a DataFrame it applies a function to each row (axis=1) or each column (axis=0). Understanding when to use each is essential for data transformation.

Series.map(): applies a function or a dictionary/mapping to each element. df['category'].map({'cat': 0, 'dog': 1}) replaces values using a dictionary — values not in the dict become NaN. df['age'].map(lambda x: 'adult' if x >= 18 else 'minor') applies a function. map() is vectorized and faster than apply() for element-wise transformations.

DataFrame.apply() with axis=1 (row-wise): applies a function to each row. def compute_bmi(row): return row['weight'] / (row['height'] ** 2). df.apply(compute_bmi, axis=1). This is slower than vectorized operations because it still iterates rows internally. Prefer vectorized operations when possible: df['weight'] / (df['height'] ** 2) is 10× faster than apply() for this simple case.

When to use apply(): for complex row-level logic that cannot be vectorized — multiple output columns from one input row (using pd.Series), conditional logic involving multiple columns, calling external functions on each row. df.apply(lambda row: parse_address(row['address']), axis=1) — parsing a string field that requires complex logic is a valid use of apply().

applymap() / map() for DataFrames: df.applymap(str) applies a function to every element in the entire DataFrame. Renamed to df.map() in newer pandas versions. Useful for type conversion or string operations across an entire DataFrame. Slower than vectorized operations — avoid on large DataFrames.

Worked example

Follow the numeric walkthrough below. It starts with a tiny function or training situation, calculates the important quantity, and then connects that number back to model learning.

import pandas as pd
df = pd.DataFrame({'age': [25, 35, None], 'income': [30000, 60000, 90000], 'gender': ['M', 'F', 'M']})
# map on Series
df['gender_enc'] = df['gender'].map({'M': 0, 'F': 1})
df['age_group'] = df['age'].map(lambda x: 'young' if x < 30 else 'senior' if x >= 50 else 'adult')
print(df)
# apply row-wise
def risk_score(row):
    score = 0
    if row['age'] > 60: score += 2
    if row['income'] < 30000: score += 1
    return score
df['risk'] = df.apply(risk_score, axis=1)

# BETTER: vectorized equivalent
df['risk_fast'] = ((df['age'] > 60) * 2 + (df['income'] < 30000) * 1)
# applymap / map for entire DataFrame
df_str = df[['age', 'income']].map(lambda x: f'{x:.0f}' if pd.notna(x) else 'N/A')
print(df_str)
Why this matters in AI
apply() and map() are the escape hatch for transformations that cannot be expressed as vectorized operations. Understanding their behavior — and when to use them vs vectorized alternatives — is essential for efficient pandas code. Over-reliance on apply() on large DataFrames causes performance problems.
Common mistakes
  • Using apply(axis=1) for simple arithmetic — df.apply(lambda row: row['a'] + row['b'], axis=1) is 50× slower than df['a'] + df['b']. Use vectorized operations for simple arithmetic.
  • Not handling None/NaN in apply() functions — if the function doesn't handle None values, it raises TypeError on NaN rows. Add: if pd.isna(x): return np.nan at the start of the function.
  • Using map() with a function that returns multiple values — map() returns a Series of scalar values. To return multiple columns from one apply(), use result_type='expand': df.apply(lambda row: {'col1': ..., 'col2': ...}, axis=1, result_type='expand').
