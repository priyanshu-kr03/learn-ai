Variables and data types

Detailed explanation

A variable is a named container that stores a value. In Python, variables are created by assignment: x = 42. Unlike C/Java, Python is dynamically typed — you do not declare the type; Python infers it from the value. The type can change: x = 42 (int), then x = 'hello' (now str). The type() function returns the current type. Variables point to objects in memory — when you write x = y, both x and y point to the same object.

Python's core data types: int (integer: 42, -7, 0), float (floating-point: 3.14, -0.5, 1e-6), str (string: 'hello', 'machine learning'), bool (boolean: True, False), NoneType (the absence of a value: None). In ML, you will constantly deal with these types: model hyperparameters are often floats (learning_rate=0.01), class labels are ints (0, 1, 2), feature names are strings, and missing values are None or NaN.

Type conversion: int('42') converts a string to int. float('3.14') converts to float. str(42) converts int to string. bool(0) is False; bool(1) is True; bool('') is False; bool('hello') is True. These conversions are critical when processing CSV data where all values are read as strings and must be converted to the correct type for computation.

Python variables are references, not values. When you assign a list: a = [1, 2, 3]; b = a, both a and b point to the same list. Modifying b modifies a too: b.append(4); print(a) → [1, 2, 3, 4]. To create an independent copy, use b = a.copy() or b = list(a). This reference behavior causes subtle bugs in ML code when you modify model parameters or dataset splits without copying.

Naming conventions: Python uses snake_case for variables and functions (learning_rate, train_model), PascalCase for classes (LogisticRegression, DataLoader), and UPPER_CASE for constants (MAX_EPOCHS = 100). Descriptive names make code self-documenting: batch_size = 32 is clear; b = 32 is not. The Python style guide PEP 8 specifies these conventions — follow it for professional code.

Why this matters in AI
Variables and data types are the building blocks of every Python program. Understanding types prevents common errors (adding a string to a number, passing the wrong type to a function) and enables efficient data processing in ML pipelines.
Common mistakes
  • Comparing floats with == — 0.1 + 0.2 == 0.3 is False due to floating-point precision. Use abs(a - b) < 1e-9 or math.isclose(a, b).
  • Mutating a shared list unintentionally — when you pass a list to a function, the function can modify the original. Use .copy() to prevent this.
  • Using Python 2-style integer division — in Python 3, 7/2 = 3.5 (float division). Use 7//2 = 3 for integer division.

Numbers and operators

Detailed explanation

Python has three numeric types: int (arbitrary precision integers: 1, -5, 10**100), float (64-bit IEEE 754: 3.14, -2.7, 1e-10), and complex (a + bj: 3+4j). For ML, floats are most common — model parameters are floats. NumPy adds float32 (single precision, less GPU memory), float64 (default), float16 (half precision, fast on modern GPUs), int32, int64.

Arithmetic operators: + (addition), - (subtraction), * (multiplication), / (true division — 7/2 = 3.5), // (floor division — 7//2 = 3), % (modulo — 7%2 = 1), ** (exponentiation — 2**10 = 1024). Python 3 always does true division — unlike Python 2 where 7/2 = 3.

Floating point precision: 0.1 + 0.2 = 0.30000000000000004 in Python — this is IEEE 754 binary floating point, not a bug. When comparing floats, never use ==: use math.isclose(a, b, rel_tol=1e-9) or np.isclose(a, b). This matters in training loops when comparing loss values.

Operator precedence: ** (highest), then unary -, then * / // %, then + - (lowest). Use parentheses to be explicit: (a + b) * c vs a + b * c gives different results. For ML formulas, parentheses prevent bugs: MSE = np.mean((y_pred - y_true)**2).

Comparison and logical operators: == (value equality), != (not equal), <, >, <=, >=. 'and', 'or', 'not'. 'is' (identity — same object) vs == (value). In ML: (y_pred == y_true).sum() counts correct predictions. Never use 'is' for value comparison.

Why this matters in AI
Python's number types and operators are the foundation of all numerical computing in ML. Float precision issues cause bugs in loss computation. Integer vs true division differences cause subtle preprocessing errors.
Common mistakes
  • Using == to compare floats — floating point arithmetic is imprecise. Use np.isclose(a, b) or math.isclose(a, b) for scalar comparison.
  • Assuming int division: in Python 3, 7/2 = 3.5. Use 7//2 = 3 for integer division. This trips up programmers from Python 2.
  • NumPy integer overflow: np.int32(2**31) wraps around to -2147483648. Python ints are arbitrary precision but NumPy/PyTorch integers do overflow.

Strings and text cleaning

Detailed explanation

Strings are sequences of characters. Python strings are immutable — you cannot modify a character in place; you must create a new string. Key string methods: lower() (convert to lowercase), upper(), strip() (remove leading/trailing whitespace), split(delimiter) (split into a list), join(iterable) (join a list into a string), replace(old, new), find(substring) (returns index or -1), startswith(prefix), endswith(suffix).

Text cleaning for NLP: raw text from the web, user input, or scraped data is messy. A standard pipeline: (1) lowercase: text.lower(), (2) remove HTML tags: re.sub('<[^>]+>', '', text), (3) remove special characters: re.sub('[^a-zA-Z0-9\s]', '', text), (4) normalize whitespace: ' '.join(text.split()), (5) remove stopwords (common words like 'the', 'a', 'is' with low discriminative value), (6) stemming/lemmatization (reduce 'running' and 'runs' to 'run').

f-strings (formatted string literals) are Python's cleanest way to embed variables in strings: name = 'Alice'; score = 0.87; message = f'User {name} achieved {score:.2%}'. The :.2% format specifier converts 0.87 to '87.00%'. f-strings are faster than .format() and % formatting and are now the standard.

String splitting for tokenization: sentence.split() splits on any whitespace. sentence.split(',') splits on commas. For more complex tokenization (handling punctuation correctly), use a dedicated tokenizer: from transformers import AutoTokenizer; tokenizer = AutoTokenizer.from_pretrained('bert-base-uncased'); tokens = tokenizer.tokenize('Hello, world!').

Regular expressions (regex) are patterns for matching and extracting text. re.findall(pattern, text) returns all matches. re.sub(pattern, replacement, text) replaces matches. Common patterns: \d+ (one or more digits), \w+ (word characters), \s+ (whitespace), [A-Za-z]+ (letters only), ^start$ (anchors). For email extraction: re.findall(r'[\w.-]+@[\w.-]+\.\w+', text).

Why this matters in AI
Text cleaning is the first step in any NLP pipeline. Raw text data from the real world is inconsistent — different capitalizations, punctuation, HTML entities, emojis. A model trained on uncleaned text learns noise patterns. The quality of text cleaning directly impacts NLP model performance.
Common mistakes
  • Applying text cleaning that removes information — removing all punctuation strips important meaning from sentiment analysis ('great!' vs 'great?') and code (operators). Clean thoughtfully, not blindly.
  • Cleaning text after vectorization — always clean text before feeding it to TF-IDF or tokenizers. The vectorizer should see the cleaned text.
  • Using str.replace() for regex patterns — str.replace() is literal; re.sub() handles patterns. 'a.b'.replace('.', '') removes all dots; re.sub(r'\.', '', 'a.b') removes the literal dot.

Lists tuples sets dictionaries

Detailed explanation

Python has four built-in collection types. List (list): ordered, mutable, allows duplicates. features = ['age', 'income', 'education']. Tuple (tuple): ordered, immutable, allows duplicates. coordinates = (40.7128, -74.0060). Set (set): unordered, mutable, no duplicates — automatically removes duplicates. unique_labels = {0, 1, 2}. Dictionary (dict): key-value pairs, ordered (Python 3.7+), mutable, keys must be unique. model_config = {'lr': 0.01, 'epochs': 100}.

When to use each: list when you need an ordered sequence you will modify (training data batches, epoch losses). Tuple when the data is fixed (RGB color (255, 128, 0), model input/output shapes). Set when you need membership testing or deduplication: if word in vocab_set is O(1); if word in vocab_list is O(n). Dictionary for structured data with named fields (model hyperparameters, per-class metrics).

List operations: append(item) adds to the end, insert(idx, item) adds at index, remove(item) removes first occurrence, pop(idx) removes and returns item at index. List comprehension: squares = [x**2 for x in range(10)]. Slicing: features[1:4] (indices 1,2,3), features[::-1] (reversed), features[::2] (every other item).

Dictionary operations: d[key] accesses a value (KeyError if not found), d.get(key, default) returns default if not found (safe), d.items() gives (key, value) pairs, d.keys() gives keys, d.values() gives values. Dictionary comprehension: {k: v*2 for k, v in metrics.items()}. Nested dictionaries are common in ML config: config = {'model': {'type': 'rf', 'n_estimators': 100}, 'training': {'epochs': 10}}.

Common ML patterns: use a list of dicts for a dataset (one dict per sample), a dict of lists for columnar data (one key per feature), sets for vocabulary building, and tuples for immutable model configurations. Pandas DataFrames are essentially optimized dicts of lists with powerful operations built on top.

Why this matters in AI
Lists, tuples, sets, and dicts are the data structures underlying all ML code. Training datasets are stored in lists or arrays. Model configurations are dicts. Vocabulary sets enable fast word lookup. Knowing when to use each prevents performance bottlenecks (using a list where a set would be O(n) vs O(1)).
Common mistakes
  • Using a list for membership testing in a large collection — 'if x in my_list' is O(n). Convert to a set first: my_set = set(my_list); 'if x in my_set' is O(1).
  • Mutating a list while iterating over it — skips elements. Iterate over a copy: for item in list(my_list).
  • Using mutable defaults in function signatures — def fn(data=[]): data.append(1) — the default list is shared across all calls. Use def fn(data=None): if data is None: data = [].

Input and output

Detailed explanation

Python's built-in I/O functions handle interaction with the user and the file system. input() reads a line from the user at the command line and always returns a string. print() outputs to the console. For ML applications, command-line I/O is used for: script arguments, interactive debugging sessions, progress bars (tqdm), and logging.

input(): always returns a string regardless of what the user types. '42' not 42. Always convert: age = int(input('Enter your age: ')). Add error handling: try: age = int(input('Age: ')) except ValueError: print('Invalid input'). For ML scripts, use argparse or click for command-line arguments rather than raw input() — they support default values, type checking, and help text.

print() formatting: print(f'Accuracy: {accuracy:.4f}') uses f-strings. print('Epoch {:3d}: loss={:.6f}'.format(epoch, loss)) uses .format(). print(f'{"Name":<20} {"Score":>10}') — < aligns left, > aligns right, width=20. print(f'{value:.2e}') — scientific notation. sep=' | ' and end='' control separators and line endings: print('a', 'b', 'c', sep=', ') prints 'a, b, c'.

sys.stdin and sys.stdout: for reading piped input (cat data.txt | python script.py) and redirecting output (python script.py > results.txt). import sys; for line in sys.stdin: processes each line. sys.stdout.write() writes without newline. sys.stderr.write() writes error messages (important for keeping stdout clean in pipelines).

Progress bars with tqdm: a must-have for ML training loops. from tqdm import tqdm; for batch in tqdm(dataloader, desc='Training'): shows a progress bar with ETA. tqdm(range(1000)) wraps any iterable. For nested loops: tqdm(outer, position=0, leave=True) and tqdm(inner, position=1, leave=False). tqdm.notebook.tqdm for Jupyter display.

Why this matters in AI
Input and output are the interfaces through which ML scripts communicate. tqdm progress bars make long training loops bearable. argparse allows scripts to be configured without editing source code — essential for running experiments with different hyperparameters. Proper print formatting makes training logs readable.

Type conversion

Detailed explanation

Type conversion (casting) changes a value from one type to another. Explicit conversion: int(), float(), str(), bool(), list(), tuple(), set(), dict(). Implicit coercion: Python automatically converts in some operations (int + float → float: 3 + 2.0 = 5.0). In ML, type conversion errors are common when loading data — a CSV column that should be float is loaded as str, causing runtime errors in model training.

Common conversions for ML: str to int/float: int('5'), float('3.14'). Fails if not a valid number — use try/except or pd.to_numeric(col, errors='coerce') to handle NaN gracefully. int to float: float(5) = 5.0. float to int: int(3.9) = 3 (truncates — use round() first if rounding is needed).

bool conversions: bool(0) = False, bool(1) = True, bool('') = False, bool('hello') = True, bool(None) = False, bool([]) = False, bool([0]) = True (non-empty list is True). In ML: binary targets should be 0/1 integers — some libraries handle booleans, others do not.

NumPy dtype conversions: array.astype(np.float32) for GPU training, array.astype(np.int64) for PyTorch LongTensor (classification labels). Conversion can lose precision: float64 → float32 loses ~8 significant digits. Watch for int8 overflow: 300 wraps to 44 (int8 range is -128 to 127).

Pandas type conversion: pd.to_numeric(series, errors='coerce') converts to numeric, replacing non-convertible values with NaN. pd.to_datetime(series) parses strings. series.astype('category') reduces memory for repeated string values. Check dtypes with df.dtypes — 'object' dtype means strings, which must be converted before training.

Why this matters in AI
Type errors are among the most common ML runtime errors. 'unsupported operand type(s) for +: int and str' breaks training pipelines. 'Expected Float but got Long' breaks PyTorch loss computation. Understanding type conversion prevents hours of debugging.
Common mistakes
  • Assuming int() rounds — int(3.9) = 3, not 4. int() always truncates toward zero. Use round() first.
  • Not handling conversion errors — int('hello') raises ValueError. Use try/except or pd.to_numeric(errors='coerce') for robust pipelines.
  • Forgetting to convert categorical strings before model training — sklearn and PyTorch require numeric inputs.

Basic debugging

Detailed explanation

Debugging is the process of finding and fixing errors in code. Python errors fall into: syntax errors (code won't run — Python tells you the line), runtime errors (code runs but crashes — Traceback tells you what happened), and logic errors (code runs without error but produces wrong results — hardest to find). ML debugging adds a fourth category: silent model failures (code runs, trains, but model performs poorly — no error, just bad metrics).

Reading tracebacks: when Python raises an exception, it prints a traceback from the outermost call to the innermost error. Read it from BOTTOM to TOP — the error is at the bottom, the call that triggered it is above. The last line tells you the error type and message. The line above it tells you which file and line. Example: 'AttributeError: NoneType object has no attribute shape' at line 45 in model.py — a variable is None when it should be an array.

print-based debugging: the simplest technique. Insert print() statements to check variable values at different points. print(f'Shape of X: {X.shape}') after data loading. print(f'y unique values: {np.unique(y)}') before training. print(f'Loss at step {i}: {loss.item():.6f}') in training loop. Remove print statements or replace with logging.DEBUG after debugging.

pdb (Python debugger): interactive debugging. import pdb; pdb.set_trace() inserts a breakpoint — code pauses there and you get an interactive Python prompt. Commands: n (next line), s (step into function), c (continue), l (list code), p variable (print variable), q (quit). In Python 3.7+, use breakpoint() instead of pdb.set_trace(). In Jupyter, use %debug after an error to enter post-mortem debugging.

Debugging ML-specific issues: (1) NaN/Inf in loss — check for zero-division (add epsilon), check for log(0), check gradient explosion (gradient clipping). (2) Constant predictions — all predictions are the same? Check if the model is learning (monitor loss decreasing), check if gradients flow (compute.grad and ensure non-zero). (3) Perfect training accuracy but poor validation — data leakage or overfitting. (4) Model not converging — learning rate too high (loss oscillates) or too low (loss barely decreases).

Why this matters in AI
Debugging is the skill that separates productive programmers from frustrated ones. ML code is particularly prone to silent failures — the code runs but the model learns nothing. Systematic debugging skills (reading tracebacks, using pdb, checking for NaN/Inf, defensive assertions) prevent hours of debugging time.
Common mistakes
  • Ignoring error messages and just re-running — tracebacks contain all the information needed to fix the error. Read the error type, the line number, and the variable values before trying anything.
  • Not using assertions for data validation — assert X.shape[1] == n_features catches shape mismatches early (at data loading) rather than late (with cryptic errors during matrix multiplication).
  • Not checking for NaN during training — a single NaN in the input propagates through the network and corrupts all gradients. Monitor loss and check for NaN whenever the loss becomes unusually large or NaN.
